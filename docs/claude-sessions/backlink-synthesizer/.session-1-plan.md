### What Does NOT Exist (the gap)

1. **No link extraction from note bodies** — the chunker (`extract_fragments`) parses headings but ignores `[[wikilinks]]`, `[markdown](links)`, and `#tags` inside body text.
2. **No note-to-note edge table** — no `note_links` or `note_edges` table. Notes are isolated nodes with no graph structure.
3. **No tag normalization** — `content_nodes.tags` stores raw frontmatter JSON; no normalized tag table for graph traversal.
4. **No graph expansion in retrieval** — `retrieve_vault_fragments()` fetches chunks from explicitly selected nodes or keyword search. It cannot walk to neighbors.
5. **No related-note suggestions in the UX** — the Ghostwriter flow shows the selected note's chunks and hooks, but never suggests "this note links to X, Y, Z — want to include them?"
6. **No graph-aware ranking** — retrieval boost is per-chunk (`retrieval_boost` column). No edge-distance weighting, no shared-tag affinity scoring.
7. **No suggestion cards or reason labels** — the frontend has no component for displaying "related note X (linked via [[wikilink]] in your note)" with accept/dismiss controls.
8. **No thread-slot insertion from related notes** — thread generation uses a single `DraftContext`; no mechanism to inject related-note fragments into specific thread slots.

### Schema Touchpoints

### 2. `epic-charter.md`

**Purpose:** Explain why graph-aware retrieval is product-critical and define the scope.

**Sections:**
- **Problem Statement** — Today, selecting a note for Ghostwriter retrieves chunks from that note only. The user's vault is a knowledge graph (via wikilinks and tags), but retrieval ignores those connections. Result: drafts miss relevant context that the user's own notes already connect.
- **Competitive Edge** — No competing tool (Typefully, Hypefury, Buffer) offers graph-aware retrieval from a personal knowledge base. This turns the vault from a flat file search into a network-aware context engine.
- **User Value** — "When I select my note about distributed systems, I want my Rust implementation note and my CAP theorem note to surface too — because I linked them." The user shouldn't have to manually select every related note.
- **Scope** — In-scope: link extraction, tag normalization, 1-hop graph expansion, ranked related-note suggestions, user accept/dismiss, thread-slot insertion. Out-of-scope: multi-hop traversal (> 1 hop), embedding-based similarity, external link resolution, collaborative graph merging.
- **Success Criteria** — (1) Given a selected note with 3+ wikilinks, at least 2 related notes surface in suggestions. (2) Accepted suggestions appear in provenance with `edge_type` reason. (3) Dismissed suggestions are excluded from future retrievals for that session. (4) All privacy invariants remain satisfied.
- **Non-Goals** — Semantic similarity / embeddings (prefer deterministic graph first), LLM-based link discovery, cross-vault graph merging.

### 4. `graph-rag-architecture.md`

**Purpose:** Specify the concrete storage and retrieval design with no TBDs.

**Sections:**

#### 4a. Link Extraction

- **When:** During `chunk_node()` in `crates/tuitbot-core/src/automation/watchtower/chunker.rs`, after fragment extraction.
- **What:** Parse `body_text` for:
  - Wikilinks: `[[Target Note]]`, `[[Target Note|Display Text]]`, `[[Target Note#Heading]]`
  - Markdown links to local files: `[text](./relative/path.md)`, `[text](relative/path.md)`
  - Inline tags: `#tag-name` (already in frontmatter but also in body)
- **Output:** `Vec<RawLink>` where:
  ```rust
  struct RawLink {
      target_raw: String,      // "Target Note" or "./path.md" or "tag-name"
      link_type: LinkType,     // Wikilink, MarkdownLink, InlineTag
      display_text: Option<String>,
      heading_anchor: Option<String>,
      source_chunk_index: usize,
  }
  ```
- **File:** New function `extract_links(body: &str) -> Vec<RawLink>` in `chunker.rs` (or a new sibling `link_extractor.rs` if chunker.rs would exceed 500 lines).

#### 4b. Edge Resolution and Storage

- **New table: `note_edges`**
  ```sql
  CREATE TABLE note_edges (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      account_id TEXT NOT NULL,
      source_node_id INTEGER NOT NULL REFERENCES content_nodes(id) ON DELETE CASCADE,
      target_node_id INTEGER NOT NULL REFERENCES content_nodes(id) ON DELETE CASCADE,
      edge_type TEXT NOT NULL,       -- 'wikilink', 'markdown_link', 'shared_tag', 'backlink'
      edge_label TEXT,               -- display text or tag name
      source_chunk_id INTEGER REFERENCES content_chunks(id),
      created_at TEXT NOT NULL DEFAULT (datetime('now')),
      UNIQUE(account_id, source_node_id, target_node_id, edge_type, edge_label)
  );
  CREATE INDEX idx_note_edges_source ON note_edges(account_id, source_node_id);
  CREATE INDEX idx_note_edges_target ON note_edges(account_id, target_node_id);
  ```
- **When:** After `chunk_node()` completes, resolve raw links to `content_nodes` by matching `target_raw` against `content_nodes.title` (case-insensitive) or `content_nodes.relative_path`. Unresolvable links are logged but not stored (fail open).
- **Backlinks:** When inserting edge `A → B`, also insert `B → A` with `edge_type = 'backlink'`. This makes bidirectional queries a single-table scan.
- **File:** New `crates/tuitbot-core/src/storage/watchtower/edges.rs`

#### 4c. Tag Normalization

- **New table: `note_tags`**
  ```sql
  CREATE TABLE note_tags (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      account_id TEXT NOT NULL,
      node_id INTEGER NOT NULL REFERENCES content_nodes(id) ON DELETE CASCADE,
      tag_text TEXT NOT NULL,          -- lowercased, trimmed
      source TEXT NOT NULL DEFAULT 'frontmatter',  -- 'frontmatter' or 'inline'
      UNIQUE(account_id, node_id, tag_text)
  );
  CREATE INDEX idx_note_tags_tag ON note_tags(account_id, tag_text);
  CREATE INDEX idx_note_tags_node ON note_tags(account_id, node_id);
  ```
- **When:** During ingestion, after frontmatter parsing and after link extraction (for inline `#tags`).
- **Shared-tag edges:** After tag normalization for a node, query other nodes sharing any of its tags. Insert `shared_tag` edges (bidirectional). Limit: max 10 shared-tag edges per node to prevent noise from popular tags.
- **File:** New `crates/tuitbot-core/src/storage/watchtower/tags.rs`

#### 4d. Graph Expansion (1-hop)

- **New function:** `expand_graph_neighbors(pool, account_id, node_id, max_neighbors) -> Vec<GraphNeighbor>`
  ```rust
  struct GraphNeighbor {
      node_id: i64,
      node_title: Option<String>,
      relative_path: String,
      edge_type: String,
      edge_label: Option<String>,
      // For ranking:
      edge_count: u32,       // how many edges connect to this neighbor
      shared_tag_count: u32, // how many tags in common
  }
  ```
- **Query:** Single SQL join on `note_edges` + `content_nodes` filtered by `source_node_id = ?` or `target_node_id = ?`, grouped by target, ordered by `edge_count DESC`.
- **Limit:** `max_neighbors = 8` (configurable). Return top 8 ranked neighbors.
- **File:** New function in `crates/tuitbot-core/src/context/retrieval.rs`

#### 4e. Ranking

Rank related notes by a composite score:

```
score = (direct_link_weight * direct_links) + (shared_tag_weight * shared_tags) + (chunk_relevance_boost)
```

Where:
- `direct_link_weight = 3.0` (wikilink or markdown link from selected note)
- `shared_tag_weight = 1.0` (shared normalized tag)
- `chunk_relevance_boost = retrieval_boost` from the best chunk in the neighbor

Deterministic — no LLM involvement in ranking.

#### 4f. Provenance

Every surfaced related-note fragment gets a `ProvenanceRef` with:
- `node_id` of the related note
- `chunk_id` of the best chunk
- `source_path` and `heading_path` snapshots
- New: `edge_type` and `edge_label` fields on `vault_provenance_links` (additive ALTER TABLE)

This preserves the "why was this note included?" reason through the entire lifecycle.

#### 4g. Fallback Behavior

- If selected note has no edges → standard retrieval (today's behavior). No graph panel.
- If edges exist but neighbor notes aren't indexed → skip unindexed neighbors, log warning. Surface only indexed neighbors.
- If all neighbors are dismissed → fall back to selected-note-only context.
- Backward compatible: all existing `/api/assist/*` endpoints continue to work without `graph_neighbors` parameter.

## Key Design Decisions and Rationale

1. **Link extraction at chunk-time** — Links are stable metadata embedded in note content. Extracting during `chunk_node()` avoids re-parsing on every retrieval query. Incremental: only changed nodes get re-extracted.

2. **Bidirectional edge storage** — Storing both `A → B` and `B → A (backlink)` means `expand_graph_neighbors` is a single `WHERE source_node_id = ?` query. Trades ~2x storage for O(1) query simplicity. SQLite is the backing store so this is negligible.

3. **1-hop only in v1** — Multi-hop traversal (friends-of-friends) adds exponential complexity and diminishes relevance. 1-hop captures the strongest signal: notes the user explicitly linked. Multi-hop can be a v2 enhancement.

4. **Deterministic ranking before LLM** — Operator constraint: "Prefer deterministic graph retrieval and ranking before LLM summarization." Composite score is transparent, debuggable, and fast. No API calls needed.

5. **Max 8 neighbors** — Balances context breadth with prompt budget. The RAG prompt is capped at 2000 chars total (existing `RAG_MAX_CHARS`). 8 neighbors provides selection headroom; the user typically accepts 2-4.

6. **Shared-tag cap of 10** — Popular tags like `#ideas` could connect hundreds of notes. Capping at 10 shared-tag edges per node prevents the graph from becoming fully connected and losing signal.

7. **Additive schema only** — New tables (`note_edges`, `note_tags`) and one `ALTER TABLE` on `vault_provenance_links`. No existing columns removed or renamed. Backward compatible.

8. **Fail-open on unresolvable links** — If a `[[wikilink]]` target doesn't match any indexed note, the link is logged and skipped. The current note-centric retrieval path continues to work. This matches the operator constraint: "If a graph edge cannot be resolved to an indexed note, fail open."
