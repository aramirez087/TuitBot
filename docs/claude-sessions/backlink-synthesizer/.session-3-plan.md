# Session 3 Implementation Plan: Graph-Aware Retrieval & Privacy-Safe APIs

**Date:** 2026-03-21
**Branch:** `epic/backlink-synthesizer`
**Mission:** Build graph-aware retrieval and expose it through privacy-safe APIs that explain why each related note was selected.

## Executive Summary

Session 2 built the graph storage layer (edges, tags, ingestion). Session 3 consumes that graph at read time: expanding neighbors around a selected note, ranking them deterministically, surfacing suggestions with human-readable reasons via new API shapes, and integrating into the existing assist/compose pipeline without breaking current behavior.

**File to modify:** `crates/tuitbot-core/src/context/retrieval.rs` (~573 lines → ~700 lines after changes)

**What:**
Add `GraphNeighbor` struct and `expand_graph_neighbors()` function that performs 1-hop graph expansion around a selected node.

**New types:**

```rust
/// Reason label explaining why a related note was suggested.
#[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum SuggestionReason {
    /// Selected note links to this note via wikilink.
    LinkedNote,
    /// This note links back to the selected note.
    Backlink,
    /// Notes share one or more tags.
    SharedTag,
    /// Both a direct link and backlink exist (bidirectional).
    MutualLink,
}

/// Intent hint for the frontend to frame the suggestion.
#[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum SuggestionIntent {
    /// Related background knowledge.
    ProTip,
    /// A contrasting or alternative viewpoint.
    Counterpoint,
    /// Supporting evidence or data.
    Evidence,
    /// General related context (default).
    Related,
}

/// A related note discovered via graph expansion.
#[derive(Debug, Clone, serde::Serialize)]
pub struct GraphNeighbor {
    pub node_id: i64,
    pub node_title: Option<String>,
    pub relative_path: String,
    pub reason: SuggestionReason,
    pub reason_label: String,           // Human-readable: "linked note", "backlink", "shared tag: #rust"
    pub intent: SuggestionIntent,
    pub matched_tags: Vec<String>,      // Tags shared with selected note (empty if not tag-based)
    pub edge_count: u32,
    pub shared_tag_count: u32,
    pub score: f64,
    pub snippet: Option<String>,        // 120-char best-chunk excerpt
    // Citation-ready metadata
    pub best_chunk_id: Option<i64>,
    pub heading_path: Option<String>,
}
```

**New functions:**

1. `expand_graph_neighbors(pool, account_id, node_id, max_neighbors) -> Result<Vec<GraphNeighbor>>`:
   - Query `get_edges_for_source()` to get all outgoing edges (wikilinks, markdown_links, backlinks, shared_tags)
   - Group edges by `target_node_id`
   - For each unique target, count direct links, backlinks, shared tags
   - Compute composite score using `compute_neighbor_score()`
   - Enrich with node title, relative_path via a single batch query to `content_nodes`
   - Load best chunk per neighbor (highest `retrieval_boost`, lowest `chunk_index`)
   - Classify reason and intent
   - Sort by score DESC, edge_count DESC, node_id ASC
   - Return top `max_neighbors` (default 8)

2. `compute_neighbor_score(direct_links, backlinks, shared_tags, best_chunk_boost) -> f64`:
   - Pure function: `3.0 * direct + 2.0 * backlinks + 1.0 * shared_tags + 0.5 * chunk_boost`

3. `classify_suggestion_reason(direct_count, backlink_count, shared_tag_count) -> SuggestionReason`:
   - Both direct + backlink → `MutualLink`
   - Direct only → `LinkedNote`
   - Backlink only → `Backlink`
   - Tags only → `SharedTag`

4. `classify_suggestion_intent(reason, edge_label) -> SuggestionIntent`:
   - Heuristic: if edge_label contains "counterpoint"/"vs"/"alternative" → `Counterpoint`
   - If edge_label contains "tip"/"how-to"/"guide" → `ProTip`
   - If edge_label contains "data"/"evidence"/"study" → `Evidence`
   - Default → `Related`

5. `build_reason_label(reason, matched_tags) -> String`:
   - `LinkedNote` → `"linked note"`
   - `Backlink` → `"backlink"`
   - `SharedTag` with tags → `"shared tag: #rust, #async"`
   - `MutualLink` → `"mutual link"`

**Design decisions:**
- **Single-query approach for edges**: Use `get_edges_for_source()` (already exists) since Session 2 stores bidirectional edges — all neighbors are reachable via `source_node_id = selected_node`. No UNION needed.
- **Batch node lookup**: After grouping edges by target, fetch all target node metadata in one query (new helper in `watchtower/nodes.rs`).
- **Snippet enrichment is a separate query**: One `get_best_chunk_for_nodes()` batch query, not per-neighbor.
- **File may approach 500 lines**: If it does, extract graph expansion into `crates/tuitbot-core/src/context/graph_expansion.rs` and re-export from `retrieval.rs`.

**Rationale:** Keep graph expansion in the `context` layer (not storage) because it involves scoring logic, reason classification, and formatting — not just CRUD.

### Step 3: Ranking Policy (pure scoring in retrieval layer)

**Location:** Inside `retrieval.rs` (or `graph_expansion.rs` if extracted)

**Ranking algorithm:**
1. Group all edges from `get_edges_for_source()` by `target_node_id`
2. For each group, count:
   - `direct_links`: edges where `edge_type` is `"wikilink"` or `"markdown_link"`
   - `backlinks`: edges where `edge_type` is `"backlink"`
   - `shared_tags`: edges where `edge_type` is `"shared_tag"`
3. Compute `score = 3.0*direct + 2.0*backlinks + 1.0*shared_tags + 0.5*best_chunk_boost`
4. Sort: `score DESC`, `edge_count DESC`, `node_id ASC`
5. Cap at `max_neighbors` (default 8)
6. **Diversity rule**: No more than 3 fragments per related note in the final prompt. This limits token budget from any single neighbor dominating context.

**Token budget enforcement:**
- `MAX_GRAPH_FRAGMENTS_PER_NOTE: u32 = 3` — when graph neighbors feed into `retrieve_vault_fragments`, each neighbor contributes at most 3 chunks
- Total fragments still capped at existing `MAX_FRAGMENTS = 5`

**File to modify:** `crates/tuitbot-server/src/routes/vault/mod.rs`

**Response shape:**

```rust
#[derive(Serialize)]
pub struct NoteNeighborsResponse {
    pub node_id: i64,
    pub neighbors: Vec<NeighborItem>,
    pub total_edges: u32,
    pub graph_state: GraphState,
}

#[derive(Serialize)]
#[serde(rename_all = "snake_case")]
pub enum GraphState {
    /// Graph has neighbors; suggestions returned.
    Available,
    /// Node exists but has no graph edges.
    NoRelatedNotes,
    /// Node has edges but targets aren't indexed.
    UnresolvedLinks,
    /// Graph query failed; fell back to standard retrieval.
    FallbackActive,
    /// Node not found in content_nodes.
    NodeNotIndexed,
}

#[derive(Serialize)]
pub struct NeighborItem {
    pub node_id: i64,
    pub node_title: Option<String>,
    pub reason: String,             // "linked_note", "backlink", "shared_tag", "mutual_link"
    pub reason_label: String,       // "linked note", "shared tag: #rust, #async"
    pub intent: String,             // "pro_tip", "counterpoint", "evidence", "related"
    pub matched_tags: Vec<String>,
    pub score: f64,
    pub snippet: Option<String>,    // 120-char excerpt
    // Citation-ready
    pub best_chunk_id: Option<i64>,
    pub heading_path: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub relative_path: Option<String>,  // Omitted in Cloud mode
}
```

**Privacy handling:**
- Cloud mode: omit `relative_path` (set to `None`)
- Snippets are 120 chars max (consistent with existing `SNIPPET_MAX_LEN`)
- No raw `body_text` exposed

**Error states:**
- Node not found → `200 OK` with `graph_state: "node_not_indexed"`, empty neighbors
- No edges → `200 OK` with `graph_state: "no_related_notes"`, empty neighbors
- Graph expansion error → `200 OK` with `graph_state: "fallback_active"`, empty neighbors (logged at warn)

**Route registration in `lib.rs`:**
```rust

```
This must go BEFORE the existing `.route("/vault/notes/{id}", ...)` line (Axum literal-before-param ordering).

Wait — actually `{id}/neighbors` is more specific than `{id}` and Axum 0.7+ handles this correctly with path matching. But to be safe given the project convention about route ordering, place it before `note_detail`.

## Key Design Decisions

### D1: Separate `graph_expansion.rs` module
**Decision:** Extract graph expansion into its own module rather than growing `retrieval.rs` past 500 lines.
**Rationale:** `retrieval.rs` is already 573 lines. Adding ~200 lines of graph logic would exceed the 500-line limit. The new module handles graph-specific concerns (expansion, ranking, classification) while `retrieval.rs` keeps fragment retrieval.

### D2: Reason and intent are enums, not free strings
**Decision:** `SuggestionReason` and `SuggestionIntent` are Rust enums serialized to snake_case.
**Rationale:** Type safety prevents typos, makes pattern matching exhaustive, and the API contract is self-documenting. New variants can be added without breaking (serde's `#[serde(other)]` on the client side).

### D3: GraphState returned as top-level field, not HTTP status codes
**Decision:** All graph states return `200 OK` with a `graph_state` field.
**Rationale:** The graph expansion is supplementary. Returning 404 or 500 would confuse clients about the primary resource. The frontend can branch on `graph_state` to show appropriate UI (loading, empty, fallback, etc.).

### D4: Graph neighbors auto-included in selection response
**Decision:** `GET /vault/selection/{session_id}` auto-expands graph neighbors when `resolved_node_id` exists.
**Rationale:** Avoids a second roundtrip from the dashboard. The selection fetch is the natural moment to discover related notes. The field is optional (`skip_serializing_if`) so existing clients aren't broken.

### D5: Provenance extension is additive (nullable columns)
**Decision:** `edge_type` and `edge_label` are nullable columns added via ALTER TABLE.
**Rationale:** Backward compatible. Existing provenance rows have NULL. New graph-sourced provenance carries the reason. No data migration needed.

### D6: Diversity cap at fragment level, not neighbor level
**Decision:** Cap fragments per neighbor at 3, not cap total neighbors.
**Rationale:** We want to show up to 8 neighbor suggestions in the UI, but when feeding into the LLM, we don't want one verbose neighbor to consume the entire 5-fragment budget. `MAX_GRAPH_FRAGMENTS_PER_NOTE = 3` ensures diversity.

### D7: Intent classification is heuristic, not LLM
**Decision:** Use simple keyword matching on `edge_label` for intent classification.
**Rationale:** Per project constraints, deterministic graph retrieval before LLM summarization. The heuristic defaults to `Related` — safe fallback. Can be refined in v2 with LLM-based classification.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `retrieval.rs` exceeds 500 lines | Medium | Extract to `graph_expansion.rs` (planned). Monitor line count. |
| Graph expansion slow on large vaults (>5K nodes) | Low | Already 1-hop only, capped at 8 neighbors. Indexes exist on `(account_id, source_node_id)`. |
| `get_edges_for_source()` returns ALL edge types including shared_tag | Low | This is correct — we need all edge types for ranking. Grouping in Rust is cheap. |
| ALTER TABLE on `vault_provenance_links` in SQLite | Low | SQLite ALTER TABLE ADD COLUMN is safe and fast (no table rewrite). |
| Route ordering: `/vault/notes/{id}/neighbors` vs `/vault/notes/{id}` | Low | Axum 0.7+ handles this correctly (longer paths match first). Add before `note_detail` to be safe. |
| Intent heuristic misclassifies | Low | Defaults to `Related`. No user-facing harm — intent is a hint, not a decision. |
| Existing assist tests break | Low | All changes are additive. `selected_node_ids` signature unchanged. Run full test suite. |
