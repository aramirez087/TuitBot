### Design Rationale

- **"Include" / "Skip"** vs intent-specific buttons: The UX journey doc specifies "Include" / "Skip" as the primary actions. Intent-specific roles ("pro_tip", "evidence") should inform the *prompt*, not the user-facing button label. Users shouldn't need to decide intent at the suggestion stage — the system infers it.
- **"Refine" over "Apply"**: "Apply" is technical. "Refine" communicates the outcome (the slot gets better).
- **"Use related notes" toggle**: More scannable than "ON/OFF" suffix. Users immediately understand what the toggle controls.

### Risks

| Risk | Mitigation |
|------|------------|
| Changing action button from "Use as pro-tip" to "Include" loses the intent mapping | Keep the `role` parameter in the `onaccept` callback — the intent is still passed to the prompt builder. Only the user-facing label changes. Add a small badge or subtitle showing the intent (e.g., "pro tip", "evidence") below the "Include" button as secondary info. |
| Tooltip text not visible on mobile | Use `title` attribute which works on long-press in mobile Safari. Keep button labels descriptive enough to work without tooltip. |

### New File to Create

### Event Catalog

| Event Name | Trigger | Properties | Component |
|------------|---------|------------|-----------|
| `backlink.suggestions_shown` | Graph suggestions panel renders with ≥1 neighbor | `{ count, session_id, graph_state }` | GraphSuggestionCards |
| `backlink.suggestion_accepted` | User clicks "Include" on a suggestion card | `{ node_id, reason, intent, score, session_id }` | VaultSelectionReview |
| `backlink.suggestion_dismissed` | User clicks "Skip" on a suggestion card | `{ node_id, reason, session_id }` | VaultSelectionReview |
| `backlink.suggestion_restored` | User clicks "Undo" on a dismissed card | `{ node_id, session_id }` | VaultSelectionReview |
| `backlink.synthesis_toggled` | User toggles "Use related notes" on/off | `{ enabled, session_id }` | VaultSelectionReview |
| `backlink.slot_targeted` | User applies a suggestion to a specific slot | `{ slot_label, source_node_id, session_id }` | SlotTargetPanel (via ComposerInspector) |
| `backlink.insert_undone` | User undoes a slot insert | `{ insert_id, slot_label, session_id }` | SlotTargetPanel (via ComposerInspector) |
| `backlink.hooks_generated` | Hook generation completes with accepted neighbors | `{ accepted_count, total_neighbors, hook_count, session_id }` | VaultSelectionReview |
| `backlink.draft_completed` | User submits a draft that used backlink suggestions | `{ insert_count, accepted_count, mode, session_id }` | ComposeWorkspace |
| `backlink.empty_graph` | Empty graph state shown (no related notes) | `{ graph_state, session_id }` | GraphSuggestionCards |
| `backlink.citation_clicked` | User clicks a citation chip | `{ source_title, is_graph_insert, is_desktop }` | CitationChips |

### Implementation Pattern

```typescript
// dashboard/src/lib/analytics/backlinkFunnel.ts
import { trackFunnel } from './funnel';

export function trackSuggestionsShown(count: number, sessionId: string, graphState: string): void {
  trackFunnel('backlink.suggestions_shown', { count, session_id: sessionId, graph_state: graphState });
}

export function trackSuggestionAccepted(nodeId: number, reason: string, intent: string, score: number, sessionId: string): void {
  trackFunnel('backlink.suggestion_accepted', { node_id: nodeId, reason, intent, score, session_id: sessionId });
}
// ... etc for each event
```

### Files to Modify (add tracking calls)

| File | Events Added |
|------|-------------|
| `GraphSuggestionCards.svelte` | `suggestions_shown` (on mount/render when neighbors > 0), `empty_graph` (on empty state render) |
| `VaultSelectionReview.svelte` | `suggestion_accepted` (in `handleAcceptNeighbor`), `suggestion_dismissed` (in `handleDismissNeighbor`), `suggestion_restored` (new undo handler), `synthesis_toggled` (in `toggleSynthesis`), `hooks_generated` (after hook generation completes) |
| `ComposerInspector.svelte` | `slot_targeted` (in `handleSlotInsert`), `insert_undone` (in `handleUndoInsert`/`handleUndoInsertById`) |
| `ComposeWorkspace.svelte` | `draft_completed` (in submit handler, conditional on having used backlink flow) |
| `CitationChips.svelte` | `citation_clicked` (in `toggleExpand` and `handleOpenInObsidian`) |

### Design Rationale

- **Console-based telemetry only**: Reuses existing `trackFunnel()` infrastructure which logs to `console.info('[tuitbot:funnel]', ...)`. This is privacy-safe (data stays local), easy to verify (check DevTools console), and can be upgraded to a backend endpoint later.
- **Session ID as correlation key**: All events include `session_id` so a complete funnel can be reconstructed for a single selection flow.
- **No PII in events**: Only node IDs (numeric), reason strings, and counts. No note titles, content, or file paths in telemetry events.
- **Separate file (`backlinkFunnel.ts`) over inline calls**: Keeps event names and shapes in one place, easy to audit and extend. Components just call named functions.

### Risks

| Risk | Mitigation |
|------|------------|
| `suggestions_shown` fires on every re-render | Guard with a flag: only fire once per session. Use `$effect` with a `hasFiredSuggestionsShown` guard. |
| No backend persistence for analytics events yet | Events log to console. Task 4 adds optional backend relay. This is intentional: local-first telemetry can be analyzed via Tauri log export without a server round-trip. |

### Frontend Integration (Optional, Not Default)

Add a `flushToBackend()` function in `backlinkFunnel.ts` that batches accumulated events and POSTs them. **Not called by default** — the console logger remains primary. This function exists for future activation.

```typescript
// dashboard/src/lib/analytics/backlinkFunnel.ts
let eventBuffer: FunnelEvent[] = [];

export function bufferEvent(event: FunnelEvent): void {
  eventBuffer.push(event);
  if (eventBuffer.length >= 10) {
    flushToBackend();
  }
}

export async function flushToBackend(): Promise<void> {
  if (eventBuffer.length === 0) return;
  const batch = eventBuffer.splice(0);
  try {
    await fetch('/api/telemetry/events', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ events: batch }),
    });
  } catch {
    // Fail silently — telemetry is best-effort
    eventBuffer.unshift(...batch);
  }
}
```

### Design Rationale

- **Server logs events via `tracing::info!`** — no new database tables. This keeps the change minimal and reversible. Structured logging lets us query via any log aggregator.
- **Max 50 events per batch** — prevents abuse. Event names prefixed with `backlink.` — prevents arbitrary data injection.
- **204 No Content** — fire-and-forget semantics. Frontend doesn't wait for or depend on response body.
- **`tuitbot-server` owns zero business logic** — this route just receives and logs. Consistent with Server Boundary constraint.

### Risks

| Risk | Mitigation |
|------|------------|
| Telemetry endpoint could be used to inject arbitrary data | Validate event name prefix (`backlink.`) and max batch size. No database writes. |
| Adding a route to `mod.rs` might affect route ordering | Place after all existing routes. No path conflicts (`/api/telemetry/events` is unique). |

## Key Decisions

1. **Console-first telemetry**: Events go to `console.info` via existing `trackFunnel()`. Backend endpoint exists for future opt-in relay but is not wired by default.
2. **"Include" / "Skip" over intent-specific buttons**: Matches UX journey doc. Intent role is preserved internally for prompt building but not exposed as user-facing copy.
3. **Dismissed card recovery via collapsible section**: Matches UX journey doc's "Show skipped (N)" pattern. Session-scoped, no permanent block list.
4. **Insert prop wiring through component tree**: Completes Session 5 residual risk. Uses prop drilling (not store) to maintain pure-function pattern.
5. **No new database tables**: Telemetry endpoint logs via `tracing::info!` only. Database persistence deferred to future session.
6. **Event names prefixed with `backlink.`**: Namespace isolation. Backend validates prefix to prevent arbitrary data injection.
