# Session 4 Implementation Plan: Ghostwriter Entry Flow & Graph Suggestion UX

## Mission

Integrate the graph-aware retrieval from Session 3 into the Ghostwriter compose flow so the user instantly understands what was selected, sees related-note suggestions with reasons, and can accept or dismiss them before generating a draft.

---

## Key Design Decisions

### D1: NeighborItem types live in `types.ts`, not a separate file
**Rationale:** The existing pattern puts all API response types in `dashboard/src/lib/api/types.ts`. Adding `NeighborItem` and `GraphState` there keeps imports consistent. The file is already 900+ lines; these add ~30 lines — within tolerance.

### D2: VaultSelectionReview owns the graph suggestion UI
**Rationale:** `VaultSelectionReview.svelte` already renders after `api.vault.getSelection()` resolves. The response now includes `graph_neighbors` and `graph_state`. Extending this component (rather than creating a new top-level panel) keeps the selection → suggestions → hooks flow linear and avoids prop-threading a new component through `FromVaultPanel` → `InspectorContent` → `ComposerInspector`.

### D3: New `GraphSuggestionCards.svelte` component for the card list
**Rationale:** The suggestion card UI (title, snippet, reason badge, intent action buttons, dismiss) is complex enough to warrant its own component (~200 lines estimated). It receives `NeighborItem[]` and emits `accept`/`dismiss` events. This also makes it independently testable.

### D4: Session-scoped toggle, not global preference
**Rationale:** The epic charter says "user in control" with "session-level toggle to disable related-note synthesis." A session-scoped toggle (stored in component state, not localStorage) means: (a) it resets each compose session, (b) no settings migration, (c) aligns with the "fail open" principle — suggestions are opt-in each time.

### D5: Accepted neighbors merge into `selectedNodeIds` at generation time
**Rationale:** Session 3 handoff specifies: "Accepted neighbor IDs should be merged into `selected_node_ids` for `resolve_composer_rag_context()`." This means the existing `handleGenerateFromVault(selectedNodeIds, ...)` call in `ComposerInspector` already supports this — we just need to pass the accepted neighbor node IDs alongside the selection's own `resolved_node_id`.

### D6: Provenance includes `edge_type` and `edge_label` for accepted neighbors
**Rationale:** Session 3 added `edge_type`/`edge_label` to `ProvenanceRef` on the backend. The frontend `ProvenanceRef` type needs these two optional fields so accepted graph neighbors carry full provenance through to the draft.

### D7: No backend changes required
**Rationale:** Session 3 already ships `graph_neighbors` and `graph_state` in the `GetSelectionResponse`. The frontend just needs to consume them. The `/vault/notes/{id}/neighbors` endpoint exists for standalone use (note detail page) but the compose flow uses the auto-expanded fields from the selection response.

## Files to Create

### 1. `dashboard/src/lib/components/composer/GraphSuggestionCards.svelte` (~250 lines)
New component for rendering neighbor suggestion cards.

**Props:**
```typescript
{
  neighbors: NeighborItem[];
  graphState: GraphState;
  loading?: boolean;
  onaccept: (neighbor: NeighborItem, role: string) => void;  // role: 'pro_tip' | 'example' | 'counterpoint'
  ondismiss: (nodeId: number) => void;
}
```

**Renders:**
- **Loading state** (`loading=true`): 3 shimmer card placeholders with "Finding related notes..." label
- **Available state** (`graphState='available'`, neighbors present): List of suggestion cards
- **Empty state** (`graphState='no_related_notes'`): Calm message: "No linked notes found — using this note only."
- **Not indexed state** (`graphState='node_not_indexed'`): "This note isn't indexed yet. Related notes will appear after your next vault sync."
- **Fallback state** (`graphState='fallback_active'`): No graph section shown (silent fallback to current behavior)
- **Each card shows:**
  - Note title (from `node_title`)
  - Snippet (120 char max, from `snippet`)
  - Reason badge (from `reason_label`, e.g., "linked note", "shared tag: #rust")
  - Intent-based action buttons: "Use as pro-tip", "Use as example", "Use as counterpoint" (derived from `intent`)
  - Dismiss button (X icon)
  - Score indicator (subtle, derived from `score`)

**Styling:** Follow existing card patterns from `CitationChips.svelte` and `VaultNoteList.svelte`. Use design tokens: `--color-surface`, `--color-accent`, `--color-border-subtle`.

### 2. `dashboard/tests/unit/GraphSuggestionCards.test.ts` (~300 lines)
Unit tests for the new component.

**Test cases:**
- Renders loading shimmer when `loading=true`
- Shows "Finding related notes..." text during loading
- Renders neighbor cards when `graphState='available'`
- Shows reason badge with correct `reason_label`
- Shows snippet text (truncated)
- Fires `onaccept` with correct neighbor and role when action button clicked
- Fires `ondismiss` with correct `nodeId` when dismiss clicked
- Shows empty state message for `no_related_notes`
- Shows not-indexed message for `node_not_indexed`
- Renders nothing for `fallback_active`
- Handles empty neighbors array with `available` state gracefully
- Accessibility: cards have appropriate ARIA roles and labels

### 5. `dashboard/src/lib/api/types.ts`
**Changes:**
- Add `NeighborItem` interface (~15 lines):
  ```typescript
  export interface NeighborItem {
    node_id: number;
    node_title: string;
    reason: string;
    reason_label: string;
    intent: string;
    matched_tags: string[];
    score: number;
    snippet: string;
    best_chunk_id: number;
    heading_path: string | null;

  ```
- Add `GraphState` type alias:
  ```typescript
  export type GraphState = 'available' | 'no_related_notes' | 'node_not_indexed' | 'fallback_active';
  ```

- Extend `VaultSelectionResponse` with two optional fields:
  ```typescript
  graph_neighbors?: NeighborItem[];
  graph_state?: GraphState;
  ```
- Extend `ProvenanceRef` with two optional fields:
  ```typescript
  edge_type?: string;
  edge_label?: string;
  ```

### 6. `dashboard/src/lib/components/composer/VaultSelectionReview.svelte` (~80 lines added, ~20 modified)
**Changes:**
- Import `GraphSuggestionCards` and `NeighborItem`
- After selection loads successfully, derive `graphNeighbors` and `graphState` from the response
- Add state: `acceptedNeighbors: Map<number, { neighbor: NeighborItem; role: string }>` — tracks which suggestions the user accepted and in what role
- Add state: `dismissedNodeIds: Set<number>` — tracks dismissed suggestions (session-scoped)
- Add state: `synthesisEnabled: boolean = true` — session-level toggle
- Derive `visibleNeighbors` = `graphNeighbors` filtered by `!dismissedNodeIds.has(n.node_id)`
- Add `handleAcceptNeighbor(neighbor, role)` — adds to `acceptedNeighbors` map
- Add `handleDismissNeighbor(nodeId)` — adds to `dismissedNodeIds` set
- Modify `handleGenerate()` — when generating hooks, include accepted neighbor node IDs in the topic context
- Modify `handleHookSelected()` — pass `[selection.resolved_node_id, ...acceptedNeighborIds]` to `ongenerate` instead of just `[selection.resolved_node_id]`
- Add `GraphSuggestionCards` between the selection review and the VaultFooter
- Add synthesis toggle button (small pill: "Related notes ON/OFF") below the selection review section header
- When `synthesisEnabled=false`, hide `GraphSuggestionCards` entirely

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `VaultSelectionReview.svelte` exceeds 400-line Svelte limit | Medium | Extracting `GraphSuggestionCards` into its own component keeps VaultSelectionReview under budget. Estimated: ~370 lines after additions. |
| `graph_neighbors` response shape doesn't match TypeScript types | Low | Session 3 defined the exact JSON shape in `graph-api-contract.md`. TypeScript types mirror it exactly. Unit tests mock the response to match. |
| Accepted neighbor IDs not reaching `resolve_composer_rag_context()` | Medium | Trace the call chain: `VaultSelectionReview.handleHookSelected()` → `FromVaultPanel.ongenerate()` → `ComposerInspector.handleGenerateFromVault(nodeIds)` → `api.assist.tweet(topic, nodeIds)`. The nodeIds array already supports multiple IDs. Just merge accepted IDs. |
| Synthesis toggle causes layout shift | Low | Toggle is a small pill in a fixed-height row. `GraphSuggestionCards` uses `display: none` when toggled off (no height contribution). |
| Frontend tests mock stale API shape | Medium | Update `sampleSelection` fixture in `VaultSelectionReview.test.ts` to include `graph_neighbors` and `graph_state` fields. Tests that don't care about graph leave them undefined. |
| Provenance `edge_type`/`edge_label` not persisted by backend | None | Session 3 already added these columns to `vault_provenance_links` and the `ProvenanceRef` → `ProvenanceLink` insertion code handles them. Just need the frontend to send them. |

## Exit Criteria Checklist

- [ ] First-time user can understand the flow without external docs
  - Selection review shows clear source summary with note title
  - "Finding related notes..." loading state is visible
  - Empty/degraded states have actionable copy (not vague spinners)
  - Suggestion cards show human-readable reason labels

- [ ] Empty and degraded states are calm and actionable
  - `no_related_notes` → "No linked notes found — using this note only."
  - `node_not_indexed` → "This note isn't indexed yet. Related notes will appear after your next vault sync."
  - `fallback_active` → Silent (no UI change, standard flow continues)

- [ ] Provenance and existing compose behavior remain intact
  - `ProvenanceRef` extended with `edge_type`/`edge_label` (additive, optional)
  - Existing compose flow without `?selection=` param is completely unchanged
  - `buildComposeRequest()` passes through provenance as before
  - All existing tests pass without modification
