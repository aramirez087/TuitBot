# Session 5 Plan: Draft Insertion & Suggestion Controls

## Mission

Make accepting a related-note suggestion feel precise and reversible so users can shape tweets and threads without losing authorship.

## Current State Analysis

After Session 4, the flow supports:
- **Selection → Graph Neighbors → Accept/Dismiss → Hook Gen → Full Generation** (whole draft replaced)
- Accepted neighbors merge into `selected_node_ids` and flow to `handleGenerateFromVault()`
- Provenance refs carry `edge_type` / `edge_label` for neighbor contributions
- Undo exists at the workspace level (10s timer, restores full snapshot)
- CitationChips display `VaultCitation[]` with expand, remove, and Obsidian deep-link

**Gap:** Once the user generates a draft, the entire content is replaced atomically. There is no way to:
1. Accept a suggestion into a *specific slot* of an existing thread
2. See *which* related note influenced *which* part of the draft
3. Undo a single suggestion acceptance without losing the entire draft
4. Replace one slot's content with a different suggestion while keeping others

## Architecture Decisions

### D1: Insertion Model — "Contextual Insert" over "Prompt Mutation"

**Decision:** Accepted suggestions produce visible, discrete insertions into the draft rather than invisibly modifying the LLM prompt and regenerating everything.

**Rationale:** The session mission explicitly says "contextual inserts rather than invisible prompt mutations." This means the user must see what changed and where. The insertion is text-level, not prompt-level.

**Implementation:** A new `DraftInsert` type tracks each suggestion-sourced insertion with its target slot, original text, inserted text, and provenance. The composer maintains a stack of these inserts for undo.

### D2: Slot Targeting — Thread Blocks as Named Slots

**Decision:** For threads, each `ThreadBlock` is a targetable slot. Slots are identified by human-readable labels derived from position: "Opening hook" (index 0), "Tweet 2", "Tweet 3", ..., "Closing takeaway" (last index). For single tweets, the entire tweet is the one slot.

**Rationale:** The session instructions specify "opening hook, tweet 2, tweet 4 pro-tip, closing takeaway." Using block index + position-derived labels keeps it simple without requiring the LLM to classify slots.

### D3: Undo Granularity — Per-Insert Undo Stack

**Decision:** Each suggestion acceptance pushes an entry onto a `draftInsertHistory` stack. "Undo" pops the most recent insert and restores the slot to its previous text. This is independent of the workspace-level undo (which covers full regeneration).

**Rationale:** The session requires "preserving the previous draft state and offering an easy revert path." A stack-based model gives precise, per-action reversibility without the complexity of a full operational transform system.

### D4: Citation Chip Extension — Source Badge per Slot

**Decision:** Extend `CitationChips` to optionally show which *slot* a citation applies to, and distinguish between "primary note" citations and "related note" citations with a visual badge (e.g., a small link icon for graph-sourced citations).

**Rationale:** Session task 4 says "citation chips and provenance reflect both the original selected note and any accepted related-note suggestions."

### D5: No Backend Changes Required

**Decision:** All insertion, undo, and slot-targeting logic lives in the frontend. The backend compose endpoints already accept `provenance: Vec<ProvenanceRef>` with `edge_type`/`edge_label`, which is sufficient.

**Rationale:** The server boundary rule says server owns zero business logic. Draft insertion is a UI concern — the backend only needs the final content + provenance at submission time.

### Phase 1: Core Types & Insertion Model

#### Step 1.1: Add DraftInsert types to `types.ts`

**File:** `dashboard/src/lib/api/types.ts`

Add near the end (after existing graph types):

```typescript
/** Represents a single suggestion insertion into a draft slot. */
export interface DraftInsert {
  /** Unique ID for this insertion action. */
  id: string;
  /** The slot index (0-based) this insert targets. For tweets, always 0. */
  slotIndex: number;
  /** Human-readable slot label ("Opening hook", "Tweet 2", etc.). */
  slotLabel: string;
  /** The text content before this insertion (for undo). */
  previousText: string;
  /** The text content after this insertion. */
  insertedText: string;
  /** The neighbor node that sourced this insertion. */
  sourceNodeId: number;
  /** Title of the source note. */
  sourceTitle: string;
  /** Provenance ref for this insert. */
  provenance: ProvenanceRef;
  /** Timestamp of insertion. */
  timestamp: number;
}

/** Maps slot index → list of active DraftInserts (for multi-suggestion per slot). */
export type SlotInsertMap = Map<number, DraftInsert[]>;
```

**Lines added:** ~25
**Rationale:** These types are consumed by multiple components. Putting them in `types.ts` keeps the single source of truth.

#### Step 1.2: Create `draftInsertStore.ts` utility

**File:** `dashboard/src/lib/stores/draftInsertStore.ts` (NEW, ~120 lines)

This is a pure utility module (not a Svelte store) that provides functions for managing the insert history. It will be instantiated as component state in `ComposeWorkspace`.

```typescript
// Functions:
// - createInsertState(): returns initial { history: DraftInsert[], slotInserts: SlotInsertMap }
// - pushInsert(state, insert): returns new state with insert added
// - popInsert(state): returns { newState, undoneInsert } or null if empty
// - undoInsertById(state, insertId): removes specific insert, returns { newState, undoneInsert }
// - getInsertsForSlot(state, slotIndex): returns DraftInsert[] for a slot
// - getSlotLabel(index, total): returns human-readable label
// - clearInserts(): returns empty state
// - hasInserts(state): boolean
```

Key implementation details:
- `getSlotLabel(0, 5)` → "Opening hook"
- `getSlotLabel(1, 5)` → "Tweet 2"
- `getSlotLabel(4, 5)` → "Closing takeaway"
- `getSlotLabel(0, 1)` → "Tweet" (single tweet mode)
- History is append-only. Undo removes from the end and restores `previousText`.

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| VaultSelectionReview exceeds 400-line Svelte limit | Medium | SlotTargetPanel extracted as separate component (Step 2.2) |
| `api.assist.improve()` for slot insertion may produce text that doesn't fit the slot's role | Medium | Include slot label in prompt context: "This is the [Opening hook] of a thread. Refine it using insights from [note title]." |
| Insert undo + workspace undo interaction confusion | Low | Clear UX: insert undo pops last suggestion, workspace undo reverts entire generation. Undo message distinguishes: "Reverted [slot]" vs "Content replaced." |
| Multiple inserts to same slot create confusing provenance | Low | Stack tracks each insert; undo removes in order. Display shows aggregate "N notes applied to this slot" |
| Thread block reordering after inserts invalidates slot indices | Medium | DraftInsert stores block ID (not just index). Slot label is cosmetic; the actual target is resolved by block ID at undo time. **Action:** Use `blockId` as the stable identifier, derive `slotIndex`/`slotLabel` at display time. |
| SlotTargetPanel on mobile may be cramped | Low | Use compact layout with single-line slot labels. Mobile inspector already has scroll. |

## Key Design Refinement: Block ID vs Slot Index

After reviewing the thread reorder risk, the `DraftInsert` type should use `blockId: string` as the stable identifier instead of `slotIndex: number`. The `slotIndex` and `slotLabel` are derived at render time from the block's current position in the sorted array. This prevents stale references when the user reorders thread blocks after accepting a suggestion.

Updated `DraftInsert`:
```typescript
export interface DraftInsert {
  id: string;
  blockId: string;           // Stable: ThreadBlock.id (or 'tweet' for single tweet)
  slotLabel: string;          // Cosmetic: derived at insert time, updated at render
  previousText: string;
  insertedText: string;
  sourceNodeId: number;
  sourceTitle: string;
  provenance: ProvenanceRef;
  timestamp: number;
}
```
