# Session 05 Implementation Plan — Approval & Autopilot Scheduling Coherence

## Problem Summary

The approval queue (`approval_queue` table) has no concept of scheduling intent. When a user composes content with a `scheduled_for` time and `approval_mode` is on, the compose endpoint (`compose.rs:439-475`) enqueues the item into `approval_queue` but **discards the `scheduled_for` value entirely**. After approval, the item posts immediately via the polling engine rather than at the user's intended time. Additionally, the approval UI shows no scheduling information and the settings copy doesn't help users understand the interaction between modes.

## Key Design Decisions

### D1: Add `scheduled_for` column to `approval_queue`

**Rationale**: The approval queue must preserve the user's scheduling intent. When approval_mode is on and the user schedules content for 2pm tomorrow, we store that timestamp alongside the queued item. Upon approval:
- If `scheduled_for` is in the future → create a `scheduled_content` row with that time (status = "scheduled"), mark approval item as "approved"
- If `scheduled_for` is in the past (user waited too long to approve) → post immediately (current behavior) but warn in UI
- If `scheduled_for` is NULL → post immediately (current behavior)

**Alternative considered**: Store scheduling intent in a separate table linking approval_queue IDs to scheduled times. Rejected — adds join complexity for minimal gain. A nullable column on the existing table is simpler and backward-compatible.

### D2: Approval → scheduled_content bridge on approve

**Rationale**: Today, `approve_item` just sets status to "approved" and the posting engine picks up approved items directly from `approval_queue`. To honor scheduling intent, the approve handler must check `scheduled_for` and, if present and future, create a `scheduled_content` entry instead. The approval item transitions to a new status "scheduled" (distinct from "approved") to prevent the immediate-posting engine from also picking it up.

### D3: Compose flow preserves `scheduled_for` through approval

**Rationale**: The compose endpoints (`persist_content`, `compose_thread_blocks_flow`) currently branch: `if approval_mode → enqueue (drops scheduled_for)`. We modify all enqueue calls to pass through `scheduled_for`. This is the smallest coherent change — no refactoring of the branching logic itself.

### D4: Autopilot + Approval off does NOT suppress manual scheduling

**Rationale**: The `read_approval_mode` function in `content/mod.rs:52` reads `config.approval_mode`. This is the raw user toggle, not the effective one (which forces approval in Composer mode). Manual compose flows should remain independent of autopilot status. No code change needed for this — the current branching already works correctly. The issue was only that approval_mode=true discarded scheduled_for.

### D5: UI shows scheduling intent on ApprovalCard

**Rationale**: Users must see "Scheduled for Mar 12, 2:00 PM EST" on pending approval items to understand what will happen after approval. This requires threading `scheduled_for` from the API response into the ApprovalCard component.

### Phase 4: Server — Approval Endpoints Honor Schedule Intent

**File**: `crates/tuitbot-server/src/routes/approval.rs` (MODIFY)

#### 4a: `approve_item` function (line ~158-216)

Current: Sets status to "approved", posting engine picks it up immediately.

New logic after `update_status_with_review_for`:
```rust
// If the item has a future scheduled_for, bridge to scheduled_content
if let Some(ref sched) = item.scheduled_for {
    // Parse and check if it's in the future
    if let Ok(dt) = chrono::NaiveDateTime::parse_from_str(sched, "%Y-%m-%dT%H:%M:%SZ") {
        let now = chrono::Utc::now().naive_utc();
        if dt > now {
            // Create scheduled_content entry
            let sc_id = scheduled_content::insert_for(
                &state.db, &ctx.account_id, &item.action_type,
                &item.generated_content, Some(sched),
            ).await?;
            // Mark approval item as "scheduled" to prevent immediate posting
            approval_queue::update_status_for(
                &state.db, &ctx.account_id, id, "scheduled"
            ).await?;
            // Return with schedule info
            return Ok(Json(json!({
                "status": "scheduled",
                "id": id,
                "scheduled_content_id": sc_id,
                "scheduled_for": sched,
            })));
        }
    }
    // Else: scheduled_for is in the past, proceed with immediate post (existing behavior)
}
```

**Important**: The `update_status_for` guard checks `status = 'pending'`. We need to first approve (pending → approved), then update to "scheduled". Or: add a new `approve_and_schedule_for` query that does `pending → scheduled` in one step. The latter is cleaner.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `enqueue_with_provenance_for` has too many parameters (14) | Maintenance burden, easy to misorder | Acceptable for this session; track for builder-pattern refactor in a later session |
| Approval items with status "scheduled" may confuse existing posting engine | Items could be double-posted or never posted | Posting engine queries `status = 'approved'` — new "scheduled" status is excluded. Verify posting engine query doesn't use broad status matches. |
| Batch approve with mixed scheduled/immediate items | Partial success scenarios | Process each item individually in the batch, log per-item outcomes |
| `scheduled_for` on approval_queue could go stale if approval takes days | User intended 2pm Monday, approved on Wednesday — 2pm Monday is in the past | On approve, check if `scheduled_for < now`. If so, warn user and offer: post now or pick new time. For MVP, just post immediately with a note in the response. |
| Frontend needs timezone for formatting scheduled_for on approval cards | Extra API call to load schedule config on approval page | Load `api.content.schedule()` in `onMount` alongside items — parallel fetch, negligible cost |
