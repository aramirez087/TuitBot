# Session 4 Implementation Plan — Draft Studio & Calendar UX

## Mission

Bring the same high-quality scheduling UX to Draft Studio and the content calendar so editing, rescheduling, and slot selection feel consistent. Ensure all scheduling surfaces use shared primitives, the canonical timezone contract, and expose clear reschedule/unschedule affordances.

---

## Current State Assessment

### What Session 3 delivered
- `SchedulePicker.svelte` — shared scheduling UI (timezone badge, preferred time pills, next-free-slot)
- `timezone.ts` — `buildScheduledFor()`, `formatInAccountTz()`, `toAccountTzParts()`, `nowInAccountTz()`
- Composer fully wired: `ComposeWorkspace`, `InspectorContent`, `HomeComposerHeader`, `ComposerCanvas`
- `DraftScheduleSection` updated to import `SchedulePicker` and accept `timezone`/`preferredTimes` props
- `DraftStudioShell` loads `ScheduleConfig` on mount and threads to children

### Gaps remaining (this session's targets)
1. **DraftScheduleSection still has `buildIso()` using browser-local `new Date()`** — needs `buildScheduledFor()` from `timezone.ts`
2. **DraftMetadataSection** shows scheduled date via `toLocaleDateString()` — needs `formatInAccountTz()`
3. **DraftHistoryPanel** shows `scheduled_for` detail via `toLocaleString()` — needs `formatInAccountTz()`
4. **Calendar store (`calendar.ts`)** uses browser-local `Date` for `getDateRange()`, `formatDateISO()`, week/month boundaries — needs account-timezone-aware boundaries
5. **CalendarWeekView** `dateKey()` and `today()` use browser-local `Date.getFullYear/getMonth/getDate` — needs account-tz-aware date keys
6. **Content page (`+page.svelte`)** `buildScheduledFor()` uses browser-local Date parts — needs `timezone.ts` version
7. **ContentItem** only allows edit for `source === 'manual'` scheduled items — should allow reschedule for all scheduled items
8. **ContentItem** has no reschedule/unschedule affordances — only edit and cancel
9. **`draftStudio.svelte.ts` `rescheduleDraft()`** does sequential unschedule+schedule instead of atomic reschedule
10. **No `api.draftStudio.reschedule()` method** — the API contract doc says `PATCH /api/drafts/:id/reschedule` exists but neither the Rust route nor the client has it yet
11. **No regression tests** for timezone-aware scheduling flows

---

## Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Add `PATCH /api/drafts/:id/reschedule` server endpoint (atomic) | The Session 02 API contract documents it. The store currently does two sequential calls (unschedule → schedule) which is racy. Single endpoint creates revision snapshot + logs activity atomically. |
| D2 | Replace all browser-local Date formatting with `timezone.ts` utilities | Ensures consistency: calendar, draft studio, metadata, and history all display in account timezone. |
| D3 | Calendar store derives "today" boundary from account timezone | A user in UTC-8 viewing at 11pm should see the correct "today" column highlighted, not the browser's next-day. |
| D4 | ContentItem gains `onreschedule` callback; relax `canEdit` to include non-manual scheduled items | All scheduled items should be reschedulable regardless of source. Edit (content change) stays manual-only. |
| D5 | Calendar prefill passes timezone-correct naive datetime via URL param | `buildScheduledFor(date, time, tz)` from `timezone.ts` produces UTC ISO. But the prefill param represents the user's intended local time, so pass `YYYY-MM-DDTHH:MM:00` (account-tz-naive) and let DraftStudioShell convert to UTC on schedule. |
| D6 | DraftMetadataSection shows timezone abbreviation alongside scheduled time | Users need to see "Mar 12, 2:00 PM EST" not just "Mar 12, 2:00 PM" to confirm timezone correctness. |

### Step 1: Server — Add atomic reschedule endpoint

**File: `crates/tuitbot-server/src/routes/content/draft_studio.rs`**

### Step 7: Calendar store — Account-timezone-aware boundaries

**File: `dashboard/src/lib/stores/calendar.ts`**

- Import `nowInAccountTz`, `buildScheduledFor` from `$lib/utils/timezone`
- Add derived `accountTimezone`:
  ```ts
  export const accountTimezone = derived(schedule, ($s) => $s?.timezone ?? 'UTC');
  ```

- **`weekStart` derived**: Use account timezone for "today" computation
  - Get current date parts via `nowInAccountTz(tz).date`
  - Compute Monday of that week in account timezone
  - Keep returning `Date` objects for calendar grid rendering (these are "representative" dates for layout, not exact timestamps)

- **`formatDateISO()`**: Replace `d.toISOString().replace('Z', '')` with manual `YYYY-MM-DDTHH:MM:SS` from account-tz-adjusted parts. Actually, for the API query range, the dates should be in UTC for the server query. The calendar API accepts `from`/`to` as date boundaries. The server needs UTC boundaries.

  **Decision:** The `from`/`to` params to the calendar API represent UTC boundaries. To show the correct week in account timezone, compute the UTC equivalent of "Monday 00:00 in account tz" and "Sunday 23:59:59 in account tz". Use `buildScheduledFor(mondayDate, "00:00", tz)` for the start boundary and `buildScheduledFor(sundayDate, "23:59", tz)` for the end.

- **`getDateRange()`**: Update both week and month modes to compute timezone-aware boundaries using `buildScheduledFor()`

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `schedule_draft_for()` storage function may filter by `status = 'draft'`, breaking reschedule | Reschedule endpoint returns error | Read implementation first; create `reschedule_draft_for()` that filters by `status = 'scheduled'` if needed |
| Calendar date grid uses browser `Date` objects as layout anchors; timezone shift could cause off-by-one | Wrong day column highlighted or items in wrong column | Only use timezone conversion for item timestamps and "today" boundary, not for grid layout dates which are midnight-anchored |
| Prefill param encoding: if we pass UTC ISO in URL, DraftStudioShell converts back to display and loses the user's intended local time | Wrong time pre-filled in SchedulePicker | Pass account-tz-naive `YYYY-MM-DDTHH:MM:00` in prefill param; SchedulePicker displays directly, converts to UTC only on submit |
| CalendarWeekView item grouping by slot may break if timezone offset crosses midnight | Items appear on wrong day in week view | Use `toAccountTzParts()` for item date key extraction, not browser-local Date methods |
| `ComposeWorkspace` already at 1310 lines | Exceeds 500-line limit | Out of scope for this session; flag in handoff for future refactor |
| DraftHistoryPanel missing "rescheduled" in action meta | Activity shows generic label | Add "rescheduled" and "reschedule" entries to both maps |
