# Session 2 — Implementation Plan

**Mission**: Implement the canonical scheduling contract so manual content can be scheduled independently of direct publish capability and with timezone-safe timestamps.

---

## Current State Summary

### Timezone Handling: Broken Everywhere

| Location | What happens | Bug |
|----------|-------------|-----|
| `composeHandlers.ts:36-40` | `new Date(targetDate).setHours(h,m)` → `.toISOString().replace('Z','')` | Constructs in browser-local time, converts to UTC, strips the `Z` — creating an ambiguous bare string that looks local but is actually UTC |
| `DraftScheduleSection.svelte:48-50` | `new Date(\`${date}T${time}\`).toISOString()` | Same: browser-local → UTC conversion, then sent as ISO with `Z` suffix |
| `calendar.ts:87-89` | `formatDateISO(d)` → `.toISOString().replace('Z','')` | Calendar range boundaries computed in UTC, not account timezone |
| `compose.rs:481` | `chrono::Utc::now().format(...)` for fallback schedule | Generates UTC timestamp as bare string — correct intent but undocumented |
| `scheduled_content/mod.rs:217` | `scheduled_for <= datetime('now')` | SQLite `datetime('now')` is UTC; works if storage is UTC, but no contract enforces this |

### What's Available But Unused
- `ScheduleConfig.timezone` is returned by the `/api/content/schedule` endpoint (e.g., `"America/New_York"`)
- The `schedule` store in `calendar.ts` already holds this value
- `ComposeWorkspace.svelte` receives `schedule: ScheduleConfig | null` as a prop
- `TimePicker` receives `schedule` and could pass `schedule.timezone` downstream

### Key Architectural Facts
- `scheduled_for` is `Option<String>` in the DB — bare ISO strings, no timezone column
- `get_due_items_for` compares `scheduled_for <= datetime('now')` (UTC)
- Two scheduling paths exist: compose flow (compose.rs `persist_content`) and Draft Studio flow (draft_studio.rs `schedule_studio_draft`)
- No reschedule endpoint — clients must unschedule then schedule (two calls, race window)
- `persist_content` couples scheduling to publish capability: if `can_post` is false and no `scheduled_for`, it auto-schedules for now

---

## Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Store all `scheduled_for` values in UTC as ISO-8601 with trailing `Z` | SQLite's `datetime('now')` returns UTC. The posting engine compares against UTC. Storing in UTC makes comparisons correct without conversion. |
| D2 | Frontend converts user-selected time from account timezone to UTC before sending | The account timezone is the canonical user-facing timezone (D1 from Session 1). All UI datetime construction must route through a utility that takes the account timezone. |
| D3 | Backend validates `scheduled_for` format on all ingest paths | Reject ambiguous formats, parse and normalize to UTC. Accept both `Z`-suffixed and bare strings (backward compat), but always store with `Z`. |
| D4 | Add `PATCH /api/drafts/:id/reschedule` as atomic reschedule | Single call that validates new time, snapshots revision, and updates `scheduled_for` — no intermediate unscheduled state. |
| D5 | Past-schedule rejection uses a configurable grace period (default: 5 minutes) | Reject schedules clearly in the past but allow slight clock skew. |
| D6 | Do NOT change the DB column type in this session | Additive only. The column stays `TEXT`. We just enforce format discipline. |

```rust
/// Validate and normalize a scheduled_for timestamp.
/// Accepts:
///   - "2026-03-10T14:00:00Z" (preferred UTC)
///   - "2026-03-10T14:00:00" (bare — treated as UTC for backward compat)
///   - "2026-03-10T14:00:00+05:00" (offset — converted to UTC)
/// Returns normalized UTC string with Z suffix.
pub fn normalize_scheduled_for(raw: &str) -> Result<String, SchedulingError>

/// Reject timestamps that are more than `grace_seconds` in the past.
pub fn validate_not_past(utc_iso: &str, grace_seconds: i64) -> Result<(), SchedulingError>

/// Combined: normalize + validate.
pub fn validate_and_normalize(raw: &str, grace_seconds: i64) -> Result<String, SchedulingError>
```

Error type:
```rust
#[derive(Debug, thiserror::Error)]
pub enum SchedulingError {
    #[error("invalid timestamp format: {0}")]
    InvalidFormat(String),
    #[error("scheduled time is in the past")]
    InThePast,
}
```

**Dependencies**: `chrono` (already in `tuitbot-core`'s deps).

**Tests** (in `scheduling.rs` or `scheduling/tests.rs`):
- Valid UTC with Z → normalized unchanged
- Bare string → appends Z
- Offset string → converts to UTC with Z
- Past timestamp → rejected
- Near-past within grace → accepted
- Future timestamp → accepted
- Garbage string → InvalidFormat error
- Empty string → InvalidFormat error

### Phase 3: Atomic Reschedule Endpoint

**Task 3.1: Add `reschedule_draft_for` to storage layer**

**File**: `crates/tuitbot-core/src/storage/scheduled_content/mod.rs`

Add a new function:

```rust
/// Atomically reschedule a scheduled item to a new time.
/// Only works on items with status = 'scheduled'.
/// Returns true if the update was applied.
pub async fn reschedule_draft_for(
    pool: &DbPool,
    account_id: &str,
    id: i64,
    new_scheduled_for: &str,
) -> Result<bool, StorageError> {
    let result = sqlx::query(
        "UPDATE scheduled_content SET scheduled_for = ?, updated_at = datetime('now') \
         WHERE id = ? AND account_id = ? AND status = 'scheduled'"
    )
    .bind(new_scheduled_for)
    .bind(id)
    .bind(account_id)
    .execute(pool)
    .await
    .map_err(|e| StorageError::Query { source: e })?;
    Ok(result.rows_affected() > 0)
}
```

**Task 3.2: Add `PATCH /api/drafts/:id/reschedule` handler**

**File**: `crates/tuitbot-server/src/routes/content/draft_studio.rs`

Add new handler:

```rust
#[derive(Deserialize)]
pub struct RescheduleBody {
    pub scheduled_for: String,
}

/// `PATCH /api/drafts/:id/reschedule` — atomically change the scheduled time.
pub async fn reschedule_studio_draft(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
    Path(id): Path<i64>,
    Json(body): Json<RescheduleBody>,
) -> Result<Json<Value>, ApiError> {
    require_mutate(&ctx)?;

    let item = scheduled_content::get_by_id_for(&state.db, &ctx.account_id, id)
        .await
        .map_err(ApiError::Storage)?
        .ok_or_else(|| ApiError::NotFound(format!("Draft {id} not found")))?;

    if item.status != "scheduled" {
        return Err(ApiError::BadRequest(format!(
            "Item is in '{}' status, not 'scheduled'", item.status
        )));
    }

    let normalized = tuitbot_core::scheduling::validate_and_normalize(&body.scheduled_for, 300)
        .map_err(|e| ApiError::BadRequest(e.to_string()))?;

    // Snapshot before reschedule
    let _ = scheduled_content::insert_revision_for(
        &state.db, &ctx.account_id, id,
        &item.content, &item.content_type, "reschedule",
    ).await;

    let updated = scheduled_content::reschedule_draft_for(
        &state.db, &ctx.account_id, id, &normalized
    ).await.map_err(ApiError::Storage)?;

    if !updated {
        return Err(ApiError::BadRequest(
            "Failed to reschedule — item may have changed status".to_string()
        ));
    }

    // Log activity
    let _ = scheduled_content::insert_activity_for(
        &state.db, &ctx.account_id, id, "rescheduled",
        Some(&json!({
            "from": item.scheduled_for,
            "to": normalized
        }).to_string()),
    ).await;

    Ok(Json(json!({
        "id": id,
        "status": "scheduled",
        "scheduled_for": normalized
    })))
}
```

### Phase 4: Frontend Timezone Utility

**Task 4.1: Create `timezone.ts`**

**File**: `dashboard/src/lib/utils/timezone.ts` (new)

```typescript
/**
 * Scheduling timezone utilities.
 *
 * Contract:
 * - All scheduled_for values sent to the server are UTC ISO-8601 with Z suffix.
 * - The account timezone (from ScheduleConfig.timezone) is the canonical
 *   user-facing timezone for display and input.
 * - Browser timezone is irrelevant — never used for scheduling logic.
 */

/**
 * Build a UTC ISO-8601 string from a user-selected date and time
 * in the account timezone.
 *
 * @param date - "YYYY-MM-DD" from <input type="date">
 * @param time - "HH:MM" from <input type="time"> or time picker
 * @param timezone - IANA timezone string (e.g. "America/New_York")
 * @returns UTC ISO-8601 string with Z suffix (e.g. "2026-03-10T18:00:00Z")
 */
export function buildScheduledFor(date: string, time: string, timezone: string): string

/**
 * Format a UTC ISO-8601 timestamp for display in the account timezone.
 *
 * @param utcIso - UTC ISO-8601 string (with or without Z)
 * @param timezone - IANA timezone string
 * @param options - Intl.DateTimeFormat options override
 * @returns Formatted string in account timezone
 */
export function formatInAccountTz(
    utcIso: string,
    timezone: string,
    options?: Intl.DateTimeFormatOptions
): string

/**
 * Extract date ("YYYY-MM-DD") and time ("HH:MM") parts from a UTC
 * timestamp, converting to the account timezone for form inputs.
 */
export function toAccountTzParts(
    utcIso: string,
    timezone: string
): { date: string; time: string }

/**
 * Get current date and time parts in the account timezone.
 */
export function nowInAccountTz(timezone: string): { date: string; time: string }
```

Implementation approach: Use `Intl.DateTimeFormat` with `timeZone` option for all conversions. For `buildScheduledFor`, construct the datetime by finding the UTC offset for the given date/time/timezone combination using `Intl.DateTimeFormat` parts, then compute the UTC equivalent.

**Tests**: Create `dashboard/src/lib/utils/timezone.test.ts` with:
- `buildScheduledFor("2026-03-10", "14:00", "America/New_York")` → `"2026-03-10T19:00:00Z"` (EDT, UTC-4 in March)
- `buildScheduledFor("2026-01-10", "14:00", "America/New_York")` → `"2026-01-10T19:00:00Z"` (EST, UTC-5 in January)
- `buildScheduledFor("2026-03-10", "14:00", "UTC")` → `"2026-03-10T14:00:00Z"`
- `toAccountTzParts("2026-03-10T19:00:00Z", "America/New_York")` → `{ date: "2026-03-10", time: "14:00" }`
- Round-trip: `buildScheduledFor(parts.date, parts.time, tz)` === original UTC
- DST edge cases: times near DST transition in target timezone

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Existing bare timestamps in DB become ambiguous | Low — they were always effectively UTC (server generates UTC, compose handler generates UTC-converted-from-browser) | Session 2 makes new values correct. Data migration deferred to Session 6. Posting engine comparison still works because both sides are UTC. |
| `chrono` offset parsing edge cases | Medium — offset strings like `+00:00` vs `Z` | Comprehensive test coverage in scheduling.rs. Use `chrono::DateTime::parse_from_rfc3339` which handles both. |
| Two scheduling paths have different validation | Low — compose and Draft Studio both go through the same `validate_and_normalize` | Ensure all 4 ingest points call the shared validator. |
| Frontend callers of `buildComposeRequest` may not pass timezone | Medium — callers need to be updated | Add `timezone` to `BuildComposeRequestOpts` as required field. TypeScript compiler catches missing args. |
| `Intl.DateTimeFormat` cross-browser differences | Low — well-supported since ES2015 | Only using standard IANA timezone names and basic formatting. Test in target browsers. |
