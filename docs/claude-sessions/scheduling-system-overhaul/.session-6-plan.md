## Task 1: Telemetry Instrumentation

### Design Decision

Extend the existing `trackFunnel()` pattern from `dashboard/src/lib/analytics/funnel.ts`. This function logs structured JSON to console as `[tuitbot:funnel]` — same pattern used by onboarding (21 events). No new infrastructure needed.

### Events to Add

| Event | Trigger Location | Properties |
|-------|-----------------|------------|
| `schedule:picker-opened` | SchedulePicker mount / ScheduleComposerSheet open | `context: 'composer' \| 'draft-studio' \| 'calendar'` |
| `schedule:time-selected` | SchedulePicker `onschedule` callback | `context`, `source: 'preferred-time' \| 'custom'`, `timezone` |
| `schedule:created` | ComposeWorkspace `handleSubmit` (scheduled path) | `mode: 'tweet' \| 'thread'`, `has_approval: boolean`, `timezone` |
| `schedule:reschedule` | DraftScheduleSection `onreschedule` / ContentItem reschedule | `source: 'draft-studio' \| 'calendar'`, `timezone` |
| `schedule:unschedule` | DraftScheduleSection `onunschedule` / ContentItem unschedule | `source: 'draft-studio' \| 'calendar'` |
| `schedule:approval-bridge` | ApprovalCard approve action (when item has scheduled_for) | `scheduled_for` present flag |
| `compose:publish-now` | ComposeWorkspace `handleSubmit` (publish-now path) | `mode: 'tweet' \| 'thread'`, `has_approval: boolean` |
| `compose:save-draft` | ComposeWorkspace `handleSubmit` (save-to-calendar path) | `mode: 'tweet' \| 'thread'` |
| `compose:submit-error` | ComposeWorkspace error catch | `error_type: string`, `mode` |

### Files to Modify

| File | Change |
|------|--------|
| `dashboard/src/lib/analytics/funnel.ts` | No changes needed — existing API sufficient |
| `dashboard/src/lib/components/composer/ComposeWorkspace.svelte` | Add `trackFunnel` import; instrument `handleSubmit` branches (schedule/publish/save) and error catch |
| `dashboard/src/lib/components/SchedulePicker.svelte` | Add `trackFunnel` import; fire `schedule:picker-opened` on mount, `schedule:time-selected` on confirm. Add `context` prop. |
| `dashboard/src/lib/components/drafts/DraftScheduleSection.svelte` | Add `trackFunnel` for reschedule/unschedule in handlers |
| `dashboard/src/lib/components/ContentItem.svelte` | Add `trackFunnel` for reschedule/unschedule button clicks |
| `dashboard/src/lib/components/ApprovalCard.svelte` | Add `trackFunnel` in approve handler when `scheduled_for` is present |

### Implementation Notes

- Add a `context` prop to `SchedulePicker` (default `'composer'`) so telemetry knows the surface. Callers in DraftScheduleSection pass `'draft-studio'`, content page passes `'calendar'`.
- Keep events synchronous — `trackFunnel` is a console.info call, no async needed.
- Properties should be minimal: only what helps measure adoption and identify confusion points.

The existing pattern is a `.sr-only` div with `role="status" aria-live="polite" aria-atomic="true"` in ComposeWorkspace (line 1045). Extend this to scheduling actions.

| Component | Announcement | Trigger |
|-----------|-------------|---------|
| `ComposeWorkspace.svelte` | "Post scheduled for {time}" | After successful schedule submit |
| `ComposeWorkspace.svelte` | "Post published" / "Post sent for approval" | After successful publish/approval submit |

| `ContentItem.svelte` | "Item rescheduled" / "Item unscheduled" | After action callbacks fire |

## Task 3: QA Matrix Document

**File**: `docs/roadmap/scheduling-system-overhaul/qa-matrix.md`

Structure modeled after the onboarding QA matrix (`docs/roadmap/x-profile-prefill-onboarding/qa-matrix.md`).

### Scenario Categories

#### A. Compose → Schedule (6 scenarios)

| # | Scenario | Conditions | Expected |
|---|----------|-----------|----------|
| A1 | Tweet + schedule + no approval | Composer mode, approval off, future time | → `scheduled_content` entry, calendar shows item, response has `scheduled_for` |
| A2 | Thread + schedule + no approval | Composer mode, approval off, future time, 3 blocks | → `scheduled_content` with thread JSON |
| A3 | Tweet + schedule + approval on | Any mode, approval on, future time | → `approval_queue` with `scheduled_for`, response `queued_for_approval` |
| A4 | Publish now + no approval | Composer mode, approval off, no schedule | → direct post via X API |
| A5 | Publish now + approval on | Any mode, approval on, no schedule | → `approval_queue` with `scheduled_for = NULL` |
| A6 | Past time rejected | Any config, time in past (>5 min) | → 422 error, user sees error message |

#### B. Draft Studio Scheduling (5 scenarios)

| # | Scenario | Expected |
|---|----------|----------|
| B1 | Schedule a draft | Status → scheduled, revision created, activity logged |
| B2 | Reschedule a scheduled draft | Atomic update, new revision with from→to detail |
| B3 | Unschedule a scheduled draft | Status → draft, time cleared |
| B4 | Schedule non-draft status fails | 409 Conflict |
| B5 | Duplicate a scheduled draft | New draft created, `scheduled_for` cleared, "(copy)" suffix |

#### C. Approval + Scheduling (5 scenarios)

| # | Scenario | Expected |
|---|----------|----------|
| C1 | Approve item with future schedule | → bridges to `scheduled_content`, status "scheduled" |
| C2 | Approve item with past schedule | → immediate post (fallback), status "approved" |
| C3 | Approve item with no schedule | → immediate post, status "approved" |
| C4 | Batch approve mixed items | Each processed individually; scheduled items bridge, immediate items post |
| C5 | Reject scheduled item | Status "rejected", no scheduled_content created |

#### D. Calendar & ContentItem (4 scenarios)

| # | Scenario | Expected |
|---|----------|----------|
| D1 | Reschedule from calendar item | Navigates to Draft Studio, SchedulePicker prefilled |
| D2 | Unschedule from calendar item | API call with fallback (draftStudio → scheduledContent), item removed from calendar |
| D3 | Cancel from calendar item | Status cancelled, removed from active view |
| D4 | Edit from calendar (manual only) | Navigates to Draft Studio with draft loaded |

#### E. Timezone Edge Cases (4 scenarios)

| # | Scenario | Expected |
|---|----------|----------|
| E1 | Account tz ≠ browser tz | All displayed times use account tz, not browser |
| E2 | DST transition (spring forward) | 2:30 AM in spring-forward zone → valid UTC conversion (no "missing hour" error) |
| E3 | DST transition (fall back) | Ambiguous 1:30 AM → deterministic UTC conversion |
| E4 | UTC account timezone | No offset applied; times display as UTC |

#### F. Autopilot Interactions (3 scenarios)

| # | Scenario | Expected |
|---|----------|----------|
| F1 | Autopilot on + manual schedule | Manual compose works normally alongside autopilot |
| F2 | Autopilot on + approval on + schedule | → approval queue with `scheduled_for` preserved |
| F3 | Autopilot off + no can_post + compose | Falls back to schedule/save-to-calendar (no publish-now button) |

#### G. Telemetry Verification (verify events fire)

List all 9 events with expected properties and code paths, similar to onboarding matrix format.
