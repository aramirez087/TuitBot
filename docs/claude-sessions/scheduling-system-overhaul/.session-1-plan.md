### Critical Gaps Found

1. **Timezone handling is browser-local, not account-timezone**
   - `composeHandlers.ts:37-40`: `new Date(targetDate)` uses browser local time, then `.toISOString().replace('Z', '')` strips the timezone marker — server receives a bare datetime that looks UTC-ish but is actually browser-local
   - `DraftScheduleSection.svelte:28-29`: `d.toISOString().slice(0, 10)` and `d.toTimeString().slice(0, 5)` — mixes UTC (toISOString) with local (toTimeString) for the same datetime
   - `DraftScheduleSection.svelte:49`: `new Date(\`${scheduleDate}T${scheduleTime}\`).toISOString()` — constructs local then converts to UTC, but server stores as-is without zone annotation
   - `calendar.ts:88`: `formatDateISO` strips 'Z' suffix — date ranges sent without timezone
   - The schedule config returns `timezone` but it's only displayed as a badge, never used for conversions

2. **Publish vs. Schedule coupling**
   - `compose.rs:347-403` and `compose.rs:452-507`: The same `ComposeRequest.scheduled_for` field controls both "schedule for later" and "immediate publish." When `scheduled_for` is `None`, the backend attempts immediate publish. If X API is unavailable, it silently falls back to scheduling at `Utc::now()` — user sees "Published" toast but content is actually scheduled.
   - `HomeComposerHeader.svelte:61-88`: When time is selected, shows "Schedule" button + separate "Publish now" icon — but `onpublishnow` just calls `onsubmit` with no way to distinguish intent at the API level
   - No explicit `publish_now: bool` flag exists in `ComposeRequest`

3. **Dual API paths for the same operation**
   - Compose flow: `POST /api/content/compose` creates a new `scheduled_content` row
   - Draft Studio flow: Draft exists first, then `POST /api/drafts/:id/schedule` promotes it
   - Both end up in the same DB table but through different code paths with different validation, revision tracking, and activity logging
   - Legacy `POST /api/content/tweets` and `POST /api/content/threads` endpoints still exist and use the approval queue differently

4. **Non-atomic reschedule**
   - `draftStudio.svelte.ts`: `rescheduleDraft()` calls unschedule then schedule as two separate API calls
   - If the second call fails, the draft reverts to 'draft' status with no scheduled_for — user loses their schedule silently
   - Two revision snapshots are created unnecessarily

5. **TimePicker is time-only, not datetime**
   - `TimePicker.svelte`: Only allows picking HH:MM from preferred slots or custom input
   - Date comes from `targetDate` (defaults to today via `$derived(prefillDate ?? new Date())`)
   - No way to pick a different date from the composer — must use Draft Studio's `DraftScheduleSection` which has date+time inputs
   - Two completely different scheduling UIs for the same operation

6. **Calendar date range queries are timezone-naive**
   - `calendar.ts:91-109`: `getDateRange()` computes week/month boundaries in browser local time, sends them without timezone info
   - Server compares these bare strings against stored datetimes — mismatch if user's browser timezone differs from account timezone

### Architecture Summary

```
Frontend Surfaces:
  Home Composer ──┐
  Modal Composer ──┼──→ POST /api/content/compose ──→ approval_queue OR scheduled_content OR direct post
  Calendar Click ──┘       (all via buildComposeRequest)

  Draft Studio ──────→ POST /api/drafts/:id/schedule ──→ scheduled_content (status transition)
                 ────→ POST /api/drafts/:id/unschedule
                 ────→ PATCH /api/content/scheduled/:id (inline edit)

Backend Storage:
  scheduled_content table (shared by all paths)
    - status: draft | scheduled | posted | cancelled
    - scheduled_for: nullable ISO string (no timezone)
    - source: manual | assist | discovery | autonomous
```

## Key Design Decisions

### D1: Account timezone as canonical
All user-facing datetime construction must use the account's configured timezone (from `ScheduleConfig.timezone`), not the browser's local timezone. The backend should validate and normalize incoming timestamps.

### D2: Separate publish intent from schedule intent
Add an explicit `action: "schedule" | "publish" | "save_draft"` field to `ComposeRequest` instead of inferring intent from presence/absence of `scheduled_for`.

### D3: Atomic reschedule endpoint
Replace the two-call unschedule+reschedule pattern with a single `PATCH /api/drafts/:id/reschedule` that atomically updates `scheduled_for` without changing status.

### D4: Shared scheduling component
Merge `TimePicker.svelte` (time-only) and `DraftScheduleSection.svelte` (date+time) into a unified `SchedulePicker` component that handles both date and time selection, usable in all surfaces.

### D5: Timezone-aware date ranges
Calendar queries should include the account timezone so the server can compute correct boundaries.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Bare datetime strings already in DB | Existing scheduled items may shift when timezone normalization is added | Additive migration: new column `scheduled_for_tz` alongside existing, backfill with assumed timezone |
| Two compose paths sharing one table | Changes to scheduled_content schema affect both paths | Session 4 explicitly unifies paths; until then, changes must be compatible with both |
| Browser timezone ≠ account timezone | Times displayed/sent could be wrong for traveling users | Enforce account timezone on all datetime construction client-side starting Session 2 |
| Reschedule atomicity gap | Users can lose their schedule if second API call fails | Session 4 adds atomic reschedule endpoint; interim, add client-side retry with rollback |
| Legacy endpoints still in use | `/api/content/tweets` and `/api/content/threads` may be called by automation | Audit all callers in Session 4 before deprecating |
