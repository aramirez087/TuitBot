# Session 3 Implementation Plan: Compose Scheduling Experience

_Planning completed 2026-03-10_

---

## Mission

Make scheduling a first-class composer action with a faster and clearer UX than the current inspector-hidden time picker. Users should discover scheduling without opening the inspector, see the active timezone, and have unambiguous Publish vs Schedule actions.

---

## Current State Analysis

### What exists today

1. **`TimePicker.svelte`** (179 lines) — inspector-only scheduling with preferred time slot buttons and a custom time input. No date picker, no timezone awareness. Uses `onselect(time: string)` to pass bare `HH:MM` strings. Clearing sends empty string.

2. **`DraftScheduleSection.svelte`** (260 lines) — Draft Studio sidebar scheduling with date + time inputs. Uses browser-local `new Date()` for all datetime construction. Three states: draft (schedule), scheduled (reschedule/unschedule), posted (duplicate). No timezone display.

3. **`InspectorContent.svelte`** (230 lines) — renders `TimePicker` in the "Schedule" section of the inspector panel. Shows hint "Posts immediately unless scheduled." No timezone label.

4. **`ComposeWorkspace.svelte`** (1299 lines) — main composer. `selectedTime` is `string | null` representing bare `HH:MM`. Passes to `ComposerCanvas` and `HomeComposerHeader` for CTA rendering. Submit calls `buildComposeRequest()`.

5. **`HomeComposerHeader.svelte`** (343 lines) — embedded mode header. Three CTA states: Schedule (when `selectedTime` set), Publish (when `canPublish` and no time), Save to Calendar (fallback). Already has `onpublishnow` and `onschedule` callback props but they're unused.

6. **`ComposerCanvas.svelte`** (197 lines) — modal mode submit button. Same three-state CTA logic. No schedule trigger.

7. **`composeHandlers.ts`** (51 lines) — `buildComposeRequest()` constructs `scheduled_for` from `selectedTime` + `targetDate` using browser-local Date. Session 02 noted it accepts optional `timezone` but the current code still uses `new Date(targetDate).setHours()`.

8. **`timezone.ts`** — Created in Session 02 but **not yet on this worktree branch** (file doesn't exist here). The utilities `buildScheduledFor`, `formatInAccountTz`, `toAccountTzParts`, `nowInAccountTz` need to be created/brought in.

9. **`calendar.ts`** (238 lines) — `getDateRange()` and `formatDateISO()` use browser-local `Date` → `toISOString()`. Not timezone-aware.

### Key gaps to address

| # | Gap | Current behavior | Target behavior |
|---|-----|-----------------|-----------------|
| G1 | Schedule hidden in inspector | User must open inspector to see TimePicker | Schedule button visible in main composer header/toolbar |
| G2 | No timezone display | Bare HH:MM times, browser-local construction | Show "America/New_York" label, use account tz for all construction |
| G3 | Ambiguous CTA | Submit button changes label based on selectedTime but no explicit schedule trigger | Split button: primary "Publish" + dropdown/secondary "Schedule" |
| G4 | Two incompatible scheduling UIs | TimePicker (inspector) vs DraftScheduleSection (drafts) | One shared SchedulePicker used in both locations |
| G5 | No quick actions | Manual time picking only | "Next free slot", preferred time buttons, clear unschedule |
| G6 | composeHandlers ignores timezone | Uses browser-local Date | Uses `buildScheduledFor()` from timezone.ts |

## Design Decisions

### D1: Schedule picker as a popover sheet, not inspector-only

**Decision**: Create `ScheduleComposerSheet.svelte` — a dropdown/popover that opens from a "Schedule" button in the composer header/toolbar. The inspector still shows the schedule section, but the primary scheduling interaction no longer requires opening it.

**Rationale**: Matches Typefully's pattern where schedule is beside the publish button. Users shouldn't need to discover the inspector panel to schedule content.

### D2: Split CTA in both embedded and modal modes

**Decision**: Both `HomeComposerHeader` (embedded) and `ComposerCanvas` (modal) will show:
- **Primary action**: "Publish" (sends immediately) or "Save to Calendar" (no X creds)
- **Secondary action**: Clock icon button that opens `ScheduleComposerSheet`
- **When scheduled**: Primary becomes "Schedule for [time]", secondary becomes "Publish now" (if capable)

**Rationale**: Eliminates ambiguity — the user always sees both options. Matches Buffer/Typefully split-action patterns from competitive research.

### D3: Account timezone from ScheduleConfig

**Decision**: Thread `schedule.timezone` from `ScheduleConfig` through the compose flow. Display it in the schedule sheet. Use it in `buildComposeRequest()` via the timezone utilities.

**Rationale**: Session 02 created the timezone utilities and the `ScheduleConfig` already has a `timezone` field. The missing piece is threading it to the UI.

### D4: Shared SchedulePicker component

**Decision**: Extract a shared `SchedulePicker.svelte` component that replaces the scheduling UI in both `InspectorContent` (composer inspector) and `DraftScheduleSection` (Draft Studio). It receives timezone, preferred_times, selected date/time, and emits schedule/unschedule events.

**Rationale**: Eliminates code duplication and ensures consistent timezone handling across both surfaces.

### D5: "Next free slot" quick action

**Decision**: Add a "Next free slot" button that finds the next available preferred_time from `ScheduleConfig.preferred_times` that hasn't been used. For MVP, this checks preferred times for today/tomorrow and picks the next one after now.

**Rationale**: Typefully's highest-rated feature. Reduces scheduling to a single click.

### D6: timezone.ts creation on this branch

**Decision**: Create `timezone.ts` on this branch with the utilities designed in Session 02 (`buildScheduledFor`, `formatInAccountTz`, `toAccountTzParts`, `nowInAccountTz`). The file doesn't exist on this worktree yet.

**Rationale**: These utilities are prerequisites for timezone-correct scheduling in both the new components and `composeHandlers.ts`.

## Exact Order of Operations

### Phase 1: Foundation (timezone utilities + shared SchedulePicker)

**Step 1.1: Create `timezone.ts`**
- File: `dashboard/src/lib/utils/timezone.ts`
- Implement: `buildScheduledFor(date, time, timezone)` → UTC ISO string, `formatInAccountTz(utcIso, timezone)` → display string, `toAccountTzParts(utcIso, timezone)` → `{ date, time }` form parts, `nowInAccountTz(timezone)` → current date/time in account tz, `nextFreeSlot(preferredTimes, timezone, existingScheduled?)` → next available slot
- Use `Intl.DateTimeFormat` with `timeZone` option for all conversions (no external deps)
- Test: manual verification that conversions round-trip correctly

**Step 1.2: Update `composeHandlers.ts`**
- File: `dashboard/src/lib/utils/composeHandlers.ts`
- Add `timezone?: string` and `scheduledDate?: string` to `BuildComposeRequestOpts`
- When `timezone` is provided, use `buildScheduledFor(scheduledDate, selectedTime, timezone)` instead of browser-local Date construction
- Fallback: when no timezone, keep existing behavior for backward compat
- Keep `selectedTime` as the bare `HH:MM` string internally

**Step 1.3: Create shared `SchedulePicker.svelte`**
- File: `dashboard/src/lib/components/SchedulePicker.svelte`
- Props: `timezone: string`, `preferredTimes: string[]`, `selectedDate: string | null`, `selectedTime: string | null`, `status: 'draft' | 'scheduled' | 'posted'`, `scheduledFor: string | null` (UTC ISO for display)
- Events: `onschedule(date, time)`, `onunschedule()`, `onchangedate(date)`, `onchangetime(time)`
- Layout:
  - Timezone badge: "Scheduling in America/New_York" (derived from timezone prop)
  - Date input (native `<input type="date">`)
  - Preferred time slots as pill buttons (from ScheduleConfig.preferred_times)
  - Custom time input (`<input type="time">`)
  - "Next free slot" quick-action button
  - "Clear schedule" button (when time is selected)
  - When status is 'scheduled': show current schedule in plain language + Reschedule/Unschedule actions
- Formatting: use `formatInAccountTz()` for display, `toAccountTzParts()` for pre-populating inputs
- Accessible: all buttons have aria-labels, keyboard navigable

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `timezone.ts` doesn't exist on this worktree branch | Blocks all timezone-aware code | Create it fresh based on Session 02 design. Keep utilities minimal and tested. |
| SchedulePicker in two contexts (inspector + sheet) may have conflicting state | Schedule set in sheet doesn't reflect in inspector or vice versa | Both are controlled components receiving the same state from ComposeWorkspace. Single source of truth in parent. |
| `Intl.DateTimeFormat` timezone support varies across browsers | Incorrect timezone conversions in old browsers | All modern browsers support IANA timezone IDs. Safari 14+ is the floor. Document minimum browser requirements. |
| ComposeWorkspace is 1299 lines (over 500 line limit) | Adding more state increases complexity | Extract schedule state into a small reactive module or keep additions minimal (3-4 new state vars). ComposeWorkspace is already at limit but is the main orchestrator — splitting it would be a separate refactor. |
| DraftStudioShell doesn't load ScheduleConfig | No timezone available in Draft Studio | DraftStudioShell needs to call `loadSchedule()` on mount and pass `schedule` to DraftDetailsPanel → DraftScheduleSection. Use the existing `schedule` store from calendar.ts. |
| `selectedTime` is bare `HH:MM` string throughout the codebase | Changing to full ISO would break many call sites | Keep `selectedTime` as `HH:MM` internally. Add `scheduledDate` as separate state. Only convert to full UTC ISO at the buildComposeRequest boundary. |
| Mobile UX for schedule sheet popover | Popovers don't work well on small screens | Render as bottom sheet on mobile (detect via matchMedia, already used in ComposeWorkspace) |
