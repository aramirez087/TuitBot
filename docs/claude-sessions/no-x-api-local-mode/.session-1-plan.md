### Current state of `x_api.scraper_allow_mutations`

- Defined in `XApiConfig` (types.rs:25-27) with `#[serde(default)]` (false)
- Read and logged in MCP init (lib.rs)
- Checked in `scraper_mutation_guard()` in x_actions/mod.rs
- Reflected in `ProviderCapabilities::scraper(allow_mutations)` (capabilities.rs)
- **Not exposed** in env vars, dashboard, onboarding, config.example.toml, or validation

### Key architectural observations

1. **No `LocalModeXClient` in tuitbot-core** — the product runtime (automation loops, server) has no scraper-backed `XApiClient` implementation. Only MCP has the `ScraperReadProvider` stub.
2. **MCP provider trait is separate from core** — `SocialReadProvider` (MCP) vs `XApiClient` (core) are distinct traits. Session 03 must create a `LocalModeXClient` implementing `XApiClient` inside tuitbot-core.
3. **Settings API is generic** — `GET/PATCH /api/settings` serializes the full `Config` struct as JSON. Adding `provider_backend` and `scraper_allow_mutations` to the TypeScript types will make them flow through automatically.
4. **No deployment_mode guard** — nothing prevents `provider_backend = "scraper"` when `deployment_mode = "cloud"`. This is a Session 02 validation task.

## Files to Create

### 1. `docs/roadmap/no-x-api-local-mode/charter.md`

**Purpose:** Define the product contract — what users get, what they don't, and deployment boundaries.

**Key design decisions:**
- **Problem:** X Developer Portal is a barrier for non-technical users (founders, indie hackers). The free tier is heavily restricted, and many users abandon onboarding when asked for API credentials.
- **User promise:** Run Tuitbot's full discovery, drafting, and content workflows on desktop or LAN without an X API key. Writes degrade safely (queue or reject with guidance).
- **Non-goals:** No cloud support for scraper mode. No guarantee of X API parity. No scraping of auth-gated data (mentions, home timeline, bookmarks, DMs).
- **Deployment boundaries:**
  - Desktop: allowed
  - LAN/self-hosted: allowed
  - Cloud: disallowed (validation rejects `provider_backend = "scraper"` when `deployment_mode = "cloud"`)
- **UI framing:** The settings surface shows "X Access" with two modes: "Official X API" (requires credentials) and "Local No-Key Mode" (no credentials needed). The UI explains tradeoffs clearly.

**Risks and mitigations:**
- Risk: Scraper transport can break without notice. Mitigation: `data_confidence = "medium"`, circuit breaker on failures, clear error messages directing users to the official API path.
- Risk: Users enable mutations in scraper mode and get their account restricted. Mitigation: `scraper_allow_mutations = false` by default, behind an "Advanced" toggle with explicit warning copy.

### 2. `docs/roadmap/no-x-api-local-mode/x-access-contract.md`

**Purpose:** Lock in the v1 config contract so Sessions 02-04 have unambiguous specifications.

**Key design decisions:**

**Decision 1: Reuse `provider_backend` directly — do NOT add duplicate config fields.**

Rationale: The `XApiConfig` struct already has both fields. The MCP crate already parses and acts on them. Adding new fields (e.g., `x_access_mode`) would create a sync problem between two representations of the same concept. Instead, the UI wraps `provider_backend` with user-friendly labels:
- `"x_api"` → displayed as "Official X API"
- `"scraper"` → displayed as "Local No-Key Mode"

This means the runtime reads `config.x_api.provider_backend` everywhere. No translation layer needed.

**Decision 2: Credential rules by backend.**

| Backend | `client_id` | `client_secret` | OAuth tokens |
|---------|------------|-----------------|--------------|
| `x_api` (default) | Required | Optional (for confidential clients) | Required (via `tuitbot auth`) |
| `scraper` | Ignored (may be empty) | Ignored (may be empty) | Not needed |

Current validation requires `client_id` unconditionally (though it doesn't explicitly check — the field just defaults to empty string and the auth flow fails later). Session 02 must make validation aware of the backend.

**Decision 3: Safe write behavior in scraper mode.**

| `scraper_allow_mutations` | Write behavior |
|--------------------------|----------------|
| `false` (default) | All write operations (post_tweet, reply_to_tweet, etc.) return `ScraperMutationBlocked` error with actionable guidance. In automation: writes route to approval queue but are marked as "blocked — awaiting API credentials or mutation override". |
| `true` | Writes attempt via scraper transport. A health check runs before each mutation. If transport confidence is insufficient, the write queues instead of posting. |

**Decision 4: Feature degradation matrix.**

| Feature | `x_api` | `scraper` |
|---------|---------|-----------|
| Discovery (search_tweets) | Full | Supported (public data) |
| Profile lookup (get_user) | Full | Supported (public data) |
| Get tweet by ID | Full | Supported (public data) |
| User tweets timeline | Full | Supported (public data) |
| Mentions | Full | **Unavailable** (auth-gated) |
| Home timeline | Full | **Unavailable** (auth-gated) |
| Bookmarks | Full | **Unavailable** (auth-gated) |
| get_me (own profile) | Full | **Unavailable** (auth-gated) |
| Post tweet | Full | Gated by `scraper_allow_mutations` |
| Reply to tweet | Full | Gated by `scraper_allow_mutations` |
| Like/retweet/follow | Full | Gated by `scraper_allow_mutations` |
| Media upload | Full | **Unavailable** |
| Analytics snapshots | Full | Degraded (no own-profile stats) |

**Decision 5: Environment variable support.**

Add `TUITBOT_X_API__PROVIDER_BACKEND` and `TUITBOT_X_API__SCRAPER_ALLOW_MUTATIONS` to `env_overrides.rs`. This is required for Docker/self-host users who configure via env vars.

### 3. `docs/roadmap/no-x-api-local-mode/session-01-handoff.md`

**Purpose:** Give Session 02 everything it needs to implement without reopening scope.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Scraper transport breaks silently | Medium | High — users see stale data | Circuit breaker, data_confidence field, clear error when scraper fails |
| Users enable mutations and get account restricted | Low | High — account damage | Default mutations off, warning copy, explicit toggle |
| `ProviderBackend` enum in MCP but not core creates duplication | Medium | Low — code smell | Document that core uses string matching (via factory), MCP uses typed enum. Consider moving enum to core in future. |
| Cloud deployment with scraper bypasses safety | Low | High | Validation rejects cloud + scraper in Session 02 |
| `client_id` validation changes break existing users | Low | Medium | Only make `client_id` conditionally required (when backend = x_api). Default backend is x_api, so existing configs are unaffected. |
