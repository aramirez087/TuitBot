# Session 2 — Settings & Validation Implementation Plan

## Mission

Expose the existing `provider_backend` / `scraper_allow_mutations` config fields as a first-class settings and onboarding UX, and make config validation accept scraper mode without X developer credentials.

## Inputs from Session 01

- `docs/roadmap/no-x-api-local-mode/charter.md` — UI framing ("Official X API" / "Local No-Key Mode"), deployment boundaries, feature matrix, risk inventory.
- `docs/roadmap/no-x-api-local-mode/x-access-contract.md` — Seven binding decisions: config field reuse (D1), credential rules (D2), validation rules (D4), env var names (D5), TypeScript types (D6).

### Phase 1: Rust Config Validation (backend, no UI impact)

## File-by-File Changes

### 1. `crates/tuitbot-core/src/config/validation.rs`

**Current state:** 297 lines. No backend-aware validation at all. No `client_id` validation. No cloud+scraper guard.

**Changes:** Add three new validation blocks in `Config::validate()`, inserted after the existing content-sources validation (line ~290) but before the `if errors.is_empty()` return. This keeps the new rules grouped together and doesn't break existing logic ordering.

#### Rule 1: Backend value validation

```rust
// Validate provider_backend value
let backend = self.x_api.provider_backend.as_str();
if !backend.is_empty() && backend != "x_api" && backend != "scraper" {
    errors.push(ConfigError::InvalidValue {
        field: "x_api.provider_backend".to_string(),
        message: format!(
            "must be 'x_api' or 'scraper', got '{}'",
            self.x_api.provider_backend
        ),
    });
}
```

**Rationale:** Must validate the value early so subsequent rules can trust it. Placement: add immediately after existing schedule/MCP validation, before cloud guard.

#### Rule 2: Cloud + scraper rejection

```rust
// Reject scraper mode in cloud deployment
if self.deployment_mode == DeploymentMode::Cloud
    && self.x_api.provider_backend == "scraper"
{
    errors.push(ConfigError::InvalidValue {
        field: "x_api.provider_backend".to_string(),
        message: "Local No-Key Mode is not available in cloud deployment. \
                  Use the Official X API (provider_backend = \"x_api\").".to_string(),
    });
}
```

**Rationale:** Charter non-goal #1. `DeploymentMode` is already imported in validation.rs scope via `super::Config` → `Config.deployment_mode`. Need to add `use crate::config::types::DeploymentMode;` at the top (already used in tests but not in validation.rs itself — check if the `Config` struct gives access via `self.deployment_mode`). Since `self.deployment_mode` is of type `DeploymentMode`, the comparison `== DeploymentMode::Cloud` works directly. The `DeploymentMode` type is already in scope because it's a field of `Config`.

#### Rule 3: Backend-aware client_id

```rust
// Require client_id when using official X API backend
let is_x_api_backend = backend.is_empty() || backend == "x_api";
if is_x_api_backend && self.x_api.client_id.trim().is_empty() {
    errors.push(ConfigError::MissingField {
        field: "x_api.client_id".to_string(),
    });
}
```

**Rationale:** Contract Decision 2. Empty string or "x_api" maps to official API → `client_id` required. "scraper" → `client_id` ignored. Using `MissingField` variant (not `InvalidValue`) since it's a required-field check, matching the existing pattern for `business.product_name`.

**Risk — backward compatibility:** Users with `provider_backend = ""` (the default) and `client_id = ""` will now get a validation error. This is correct — they couldn't use the API anyway. The error message guides them to either set credentials or switch to scraper mode.

**Risk — onboarding payload:** The onboarding page currently always sends `client_id`, even if empty. In scraper mode, this is fine because validation skips the check. In x_api mode, the existing `canAdvance()` check (step 1: `data.client_id.trim().length > 0`) ensures non-empty `client_id` before submission.

**Estimated file size after changes:** ~340 lines (well under 500 limit).

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| No new config fields | Contract D1. Reuse `provider_backend` + `scraper_allow_mutations`. |
| `MissingField` for `client_id` check | Matches existing pattern for required fields. Server returns `"this field is required"`. |
| Cloud guard in validation only | UI also hides the option, but validation is the safety net. Belt and suspenders. |
| Default `provider_backend = ""` maps to x_api | Backward compat. Existing configs don't have this field — they keep working. |

| Advanced toggle for mutations | Charter R2 mitigation. User must explicitly opt in. Warning copy from charter. |
| Rename section "X API" → "X Access" | Contract UI framing. The section covers both modes now. |

## Risks & Mitigations

### Risk 1: Existing users with empty `client_id` get validation errors

**Scenario:** User has `config.toml` with `[x_api]` section but no `client_id` (or `client_id = ""`). No `provider_backend` set.

**Impact:** Validation now rejects this config. Previously it passed validation and failed at runtime during auth.

**Mitigation:** This is correct behavior (contract D2). The error message guides them: *"Get your Client ID from https://developer.x.com, or switch to Local No-Key Mode."* Test this path explicitly (Test 3).

### Risk 2: `onboarding/+page.svelte` exceeds 400-line limit

**Current:** 474 lines (already above limit).

**Mitigation:** The changes to this file are minimal (2 small edits to `canAdvance()` and `submit()`). Not the session to refactor the onboarding page. Flag for future cleanup.

### Risk 3: `tests.rs` exceeds 500-line limit

**Current:** Already ~550+ lines.

**Mitigation:** Create a new test module. Option A: Add `mod tests_backend;` in `config/mod.rs` wrapped in `#[cfg(test)]`, with the new tests in `config/tests_backend.rs`. Option B: Keep all tests in `tests.rs` since it's a test file (some projects exempt test files from the limit). **Decision at implementation time:** Use Option A if the file is already over 500 lines.

### Risk 4: Settings PATCH round-trip with `provider_backend`

**Scenario:** Frontend sends `{ "x_api": { "provider_backend": "scraper" } }`. Server merges into TOML. Does the field persist correctly?

**Analysis:** Yes. `merge_toml()` handles nested table merging. `json_to_toml()` converts strings correctly. `XApiConfig` has `#[serde(default)]` on all fields, so deserialization fills in defaults for unspecified fields. The TOML serializer includes all fields (no `skip_serializing_if`).

**Verification:** Test 11 (TOML round-trip) covers this at the data layer. Server integration test deferred to Session 04.

### Risk 5: `deploymentMode` store not loaded in settings page context

**Scenario:** `XApiSection.svelte` reads `$deploymentMode` to hide scraper option in cloud mode. Is the store populated by the time the settings page renders?

**Analysis:** Yes. The `+layout.svelte` calls `loadCapabilities()` on mount, which fetches from `/api/runtime/status` and falls back to `/api/settings/status`. Both return `deployment_mode`. The settings page loads after layout.
