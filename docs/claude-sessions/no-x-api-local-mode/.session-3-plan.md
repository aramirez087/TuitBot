# Session 03 Implementation Plan: Runtime Scraper Backend

## Mission

Make the product runtime honor `provider_backend = "scraper"` so local users can run core discovery and publishing workflows without an X API key, while writes fail closed when the transport is unsafe.

## Codebase Assessment

### Current State (from Session 02)
- Config validation done: `provider_backend = "scraper"` passes without `client_id`
- Cloud guard blocks scraper in cloud mode (validation + UI)
- Env var overrides done: `TUITBOT_X_API__PROVIDER_BACKEND=scraper`
- Dashboard UI done: settings + onboarding write `provider_backend` to config
- **No runtime client exists yet** — the config fields are wired but ignored at runtime

### Key Initialization Paths

**CLI (`deps.rs:84-293`):**
1. Loads OAuth tokens from file → fails if no tokens exist (line 95)
2. Creates `TokenManager` with `client_id` from config (line 104-108)
3. Creates `XApiHttpClient::new(current_token)` (line 133)
4. Calls `detect_tier(&x_client)` via search endpoint probe (line 134)
5. Calls `x_client.get_me()` to get `own_user_id` (line 185-188) — **hard requirement**
6. All adapters receive `Arc<dyn XApiClient>` from a single cast (line 197)

**Server (`main.rs:49-261`):**
- Does NOT create an `XApiClient` at all — the runtime route (`routes/runtime.rs`) only creates an empty `Runtime` struct
- Server is a thin layer; no X API calls during startup
- `AppState` has no `XApiClient` field

**MCP scraper (`provider/scraper.rs`):**
- Stub `ScraperReadProvider` implementing `SocialReadProvider` (different trait than `XApiClient`)
- Auth-gated methods return `ProviderError::NotConfigured`
- Public-data methods return `ProviderError::Other { message: "not yet implemented" }`
- Useful as error semantics reference, but the trait surface is different

### Architecture Constraints
- `XApiClient` trait has 26+ methods — `LocalModeXClient` must implement all of them
- Three-layer architecture: Toolkit (L1) → Workflow (L2) → Automation (L3)
- All X API calls flow through toolkit functions, which take `&dyn XApiClient`
- The adapter pattern in `automation/adapters/x_api.rs` wraps `Arc<dyn XApiClient>` into port traits
- A factory function that returns `Arc<dyn XApiClient>` can transparently substitute the client

## Design Decisions

### D1: `LocalModeXClient` implements `XApiClient` directly
**Rationale:** The product runtime uses `XApiClient` everywhere. The MCP crate uses a different trait (`SocialReadProvider`). We mirror the MCP scraper's error semantics but implement the product trait.

### D2: New error variants on `XApiError` (not a separate enum)
**Rationale:** All toolkit functions return `Result<_, XApiError>`. Adding scraper-specific variants to `XApiError` means existing error mapping in adapters (`toolkit_to_loop_error`, etc.) continues to work with minor additions. The alternatives (new enum, wrapper) would require changes across many more files.

New variants:
- `ScraperMutationBlocked` — writes disabled via `scraper_allow_mutations = false`
- `ScraperTransportUnavailable` — transport health check failed
- `FeatureRequiresAuth` — method requires authenticated user (get_me, mentions, etc.)

### D3: Factory function `create_x_client(config) -> Result<Arc<dyn XApiClient>>`
**Rationale:** A single decision point in `x_api/mod.rs` that reads `provider_backend` and returns the right client. CLI `deps.rs` and any future server runtime code call this instead of `XApiHttpClient::new()` directly.

For the official backend path, the factory still requires tokens (the caller loads them). For scraper, no tokens needed.

### D4: CLI `deps.rs` needs a parallel initialization path for scraper mode
**Rationale:** The current `RuntimeDeps::init()` assumes OAuth tokens exist, calls `detect_tier()`, calls `get_me()`. In scraper mode:
- No tokens → skip `TokenManager` creation
- No tier detection → synthetic `ApiTier::Basic` capabilities (discovery yes, mentions no)
- No `get_me()` → `own_user_id = None` or placeholder
- `MentionsLoop`, `TargetLoop`, `AnalyticsLoop` skipped (all need `user_id`)

The cleanest approach: branch early in `RuntimeDeps::init()` based on `config.x_api.provider_backend`.

### D5: `scraper_allow_mutations` enforced at `LocalModeXClient` level
**Rationale:** The client itself holds a reference to the config flag. Write methods (`post_tweet`, `reply_to_tweet`, etc.) check the flag and return `ScraperMutationBlocked` immediately when false. When true, they still return stub errors in this session (actual scraper transport is future work), but with a different error variant (`ScraperTransportUnavailable`) to distinguish "intentionally blocked" from "not yet implemented."

### D6: `LocalModeXClient` read methods return stubs initially
**Rationale:** The actual scraper transport (browser automation, HTTP scraping) is out of scope for Session 03. Read methods (`search_tweets`, `get_user_by_username`, etc.) will return `XApiError::ApiError` with a clear "scraper backend: method not yet implemented" message. This matches the MCP scraper stub pattern. The factory + trait dispatch is the Session 03 deliverable; actual scraper I/O is Session 04+.

### D7: Server `main.rs` requires minimal changes
**Rationale:** The server doesn't create an `XApiClient` at startup. The runtime route creates an empty `Runtime`. The server only needs to:
- Not crash when config has `provider_backend = "scraper"` (already true)
- Expose backend mode in runtime status endpoint (nice-to-have)

### D8: `approval_poster` needs backend awareness
**Rationale:** `run_approval_poster` receives `Arc<dyn XApiClient>`. In scraper mode with mutations disabled, it should skip posting and leave items in queue with a log message. With mutations enabled but transport unavailable, it should log and retry. The `LocalModeXClient` will return the right error variants, so the poster's existing error handling largely works — but we should add a log message for the scraper-specific error variants.

### Step 2: Create `local_mode.rs` with `LocalModeXClient`
**File:** `crates/tuitbot-core/src/x_api/local_mode.rs` (new)

Structure:
```rust
pub struct LocalModeXClient {
    allow_mutations: bool,
}

impl LocalModeXClient {
    pub fn new(allow_mutations: bool) -> Self { ... }
}

#[async_trait::async_trait]
impl XApiClient for LocalModeXClient {
    // Read methods: return stub errors with clear messages
    async fn search_tweets(...) -> Result<SearchResponse, XApiError> {
        Err(XApiError::ScraperTransportUnavailable {
            message: "search_tweets: scraper transport not yet implemented".into()
        })
    }

    // Auth-gated methods: return FeatureRequiresAuth
    async fn get_me(...) -> Result<User, XApiError> {
        Err(XApiError::FeatureRequiresAuth {
            message: "get_me requires authenticated API access".into()
        })
    }
    async fn get_mentions(...) -> Result<MentionResponse, XApiError> {
        Err(XApiError::FeatureRequiresAuth { ... })
    }
    async fn get_home_timeline(...) -> Result<SearchResponse, XApiError> {
        Err(XApiError::FeatureRequiresAuth { ... })
    }

    // Write methods: check allow_mutations flag
    async fn post_tweet(&self, text: &str) -> Result<PostedTweet, XApiError> {
        if !self.allow_mutations {
            return Err(XApiError::ScraperMutationBlocked {
                message: "post_tweet".into()
            });
        }
        Err(XApiError::ScraperTransportUnavailable {
            message: "post_tweet: scraper write transport not yet implemented".into()
        })
    }
    // ... same pattern for reply_to_tweet, quote_tweet, like_tweet, etc.

    // Media: always unavailable
    async fn upload_media(...) -> Result<MediaId, XApiError> {
        Err(XApiError::MediaUploadError {
            message: "media upload unavailable in scraper mode".into()
        })
    }
}
```

Categorization of all 26+ `XApiClient` methods:

**Auth-gated (return `FeatureRequiresAuth`):**
- `get_me()` — requires authenticated user identity
- `get_mentions()` — requires user_id from auth
- `get_home_timeline()` — requires auth
- `get_bookmarks()` / `bookmark_tweet()` / `unbookmark_tweet()` — requires auth

**Write methods (check `allow_mutations`, then return `ScraperTransportUnavailable`):**
- `post_tweet()`, `reply_to_tweet()`
- `post_tweet_with_media()`, `reply_to_tweet_with_media()`
- `quote_tweet()`
- `like_tweet()`, `unlike_tweet()`
- `follow_user()`, `unfollow_user()`
- `retweet()`, `unretweet()`
- `delete_tweet()`

**Read methods (return `ScraperTransportUnavailable` stub):**
- `search_tweets()` — will be implemented with scraper transport later
- `get_tweet()`
- `get_user_by_username()`
- `get_user_tweets()`
- `get_user_by_id()`
- `get_followers()`, `get_following()`
- `get_users_by_ids()`
- `get_liked_tweets()`
- `get_tweet_liking_users()`
- `raw_request()`

**Media (return `MediaUploadError`):**
- `upload_media()`

**Risk:** File could be long (~300 lines) due to 26+ method stubs. If over 500 lines, split into `local_mode/mod.rs` + `local_mode/trait_impl.rs` following the `client/` pattern.

**Mitigation:** Most methods are 3-5 line stubs. Group with comments. Should stay under 300 lines.

### Step 3: Add factory function to `x_api/mod.rs`
**File:** `crates/tuitbot-core/src/x_api/mod.rs`

Add:
```rust
pub mod local_mode;
pub use local_mode::LocalModeXClient;
```

Add factory function:
```rust
use crate::config::XApiConfig;

/// Create the appropriate X API client based on config.
///
/// - `provider_backend = "scraper"` → `LocalModeXClient`
/// - `provider_backend = ""` or `"x_api"` → caller provides `XApiHttpClient`
///
/// Returns `None` for official backend (caller must construct XApiHttpClient
/// with OAuth tokens). Returns `Some(Arc<dyn XApiClient>)` for scraper.
pub fn create_local_client(config: &XApiConfig) -> Option<Arc<dyn XApiClient>> {
    if config.provider_backend == "scraper" {
        Some(Arc::new(LocalModeXClient::new(config.scraper_allow_mutations)))
    } else {
        None
    }
}
```

**Decision:** The factory returns `Option` rather than `Result<Arc<dyn XApiClient>>` because the official path requires tokens that come from outside the config. The CLI caller does `create_local_client()` first; if `None`, proceeds with the existing OAuth flow.

**Risk:** Minimal — additive module registration + one function.

### Step 4: Branch CLI initialization for scraper mode
**File:** `crates/tuitbot-cli/src/deps.rs`

This is the most complex change. The current `RuntimeDeps::init()` assumes OAuth tokens and does tier detection + `get_me()`. In scraper mode, we skip all of that.

**Approach:** Add an early branch at the top of `init()`:

```rust
pub async fn init(config: &Config, dry_run: bool) -> anyhow::Result<Self> {
    // Check if we're in scraper mode
    if config.x_api.provider_backend == "scraper" {
        return Self::init_scraper_mode(config, dry_run).await;
    }
    // ... existing OAuth flow unchanged ...
}

async fn init_scraper_mode(config: &Config, dry_run: bool) -> anyhow::Result<Self> {
    // 1. Initialize database (same as official path)
    let db_path = expand_tilde(&config.storage.db_path);
    let pool = storage::init_db(&config.storage.db_path).await?;

    // 2. Create LocalModeXClient
    let local_client = Arc::new(LocalModeXClient::new(
        config.x_api.scraper_allow_mutations
    ));
    let dyn_client: Arc<dyn XApiClient> = local_client;

    // 3. Synthetic tier: discovery=true, mentions=false
    let tier = ApiTier::Basic;
    let capabilities = TierCapabilities {
        mentions: false,  // No get_me → no user_id → no mentions
        discovery: true,  // Search will work once transport is implemented
    };

    // 4. No user_id → mentions adapter gets placeholder
    // MentionsAdapter with empty user_id — loops will be skipped anyway
    let mentions_fetcher = Arc::new(XApiMentionsAdapter::new(
        dyn_client.clone(),
        String::new(),
    ));

    // 5. LLM, scoring, safety — same as official path
    // ... (copy from existing init, everything below step 8)

    // 6. No posting queue token refresh (no TokenManager)
    // Create a dummy token manager that never refreshes
    // Actually: RuntimeDeps requires `token_manager` and `x_client` fields
    // Solution: make these Optional, or create no-op stubs
}
```

**Problem:** `RuntimeDeps` has non-optional fields `token_manager: Arc<TokenManager>` and `x_client: Arc<XApiHttpClient>`. These are only used by:
- `run_token_refresh_loop(tm, xc, cancel)` — must be skipped in scraper mode
- `run_approval_poster(pool, xc, ...)` — needs `Arc<dyn XApiClient>`, not `Arc<XApiHttpClient>`

**Solution A (preferred):** Make `token_manager` and `x_client` optional:
```rust
pub token_manager: Option<Arc<TokenManager>>,
pub x_client: Option<Arc<XApiHttpClient>>,
pub dyn_client: Arc<dyn XApiClient>,  // new: the actual client (official or local)
```

This way:
- `run.rs` checks `if let Some(tm) = &deps.token_manager` before spawning token refresh
- `run.rs` uses `deps.dyn_client` for approval poster instead of casting `x_client`
- Existing official path sets `Some(...)` for both
- Scraper path sets `None` for both

**Solution B (alternative):** Keep the existing fields and create dummy no-op stubs. This is more fragile — a dummy `TokenManager` could attempt network calls.

**Going with Solution A.** Changes cascade to `run.rs` but are straightforward.

**Risk:** This is a refactor of `RuntimeDeps` struct fields, which affects `run.rs`. Must be careful to keep the official path backward-compatible.

**Mitigation:** The official path changes from `deps.x_client.clone() as Arc<dyn XApiClient>` to `deps.dyn_client.clone()` — a transparent rename. Token manager access is already only used in two places.

### Step 6: Update adapter error mapping
**File:** `crates/tuitbot-core/src/automation/adapters/helpers.rs`

Add mapping for new error variants in `toolkit_to_loop_error`:
```rust
XApiError::ScraperMutationBlocked { .. } => LoopError::Other(xe.to_string()),
```

### Step 7: Update approval poster for scraper-specific logging
**File:** `crates/tuitbot-core/src/automation/approval_poster.rs`

The approval poster calls `toolkit::write::*` which returns `XApiError` wrapped in `String`. The new error variants will appear as error strings in the existing error logging path (line 113-128). No code change strictly needed — but we should add a log hint:

```rust
// In the Err(e) match arm, before logging:
if e.contains("scraper mutation blocked") || e.contains("scraper transport unavailable") {

        id = item.id,
```

Actually, a better approach: leave the existing error path (it already logs a warning and records the action log entry). The error message from `ScraperMutationBlocked` is clear enough. **Decision: No changes to approval_poster.rs.** The existing error handling is sufficient. The error message `"scraper mutation blocked: post_tweet. Enable scraper_allow_mutations..."` will appear in the log, which is actionable.

## Risks and Mitigations

### R1: `RuntimeDeps` refactor breaks official path
**Severity:** High
**Mitigation:** Make `token_manager` and `x_client` `Option<_>`, add `dyn_client: Arc<dyn XApiClient>`. Official path sets all three. Test by running `cargo test --workspace` before committing the scraper path changes.

### R2: `local_mode.rs` exceeds 500-line limit
**Severity:** Low
**Mitigation:** Each method stub is 3-5 lines. 26 methods × 5 = 130 lines + struct + impl header + tests reference ≈ 200 lines. Well under 500. If tests push it over, extract to `local_mode/tests.rs`.

### R3: Missing error variant match arms cause clippy warnings
**Severity:** Medium
**Mitigation:** Adding variants to `XApiError` means every `match` on it must handle them. Check all match sites:
- `automation/mod.rs:103` — matches `AuthExpired` specifically, rest falls through
- `adapters/helpers.rs` — has `other =>` catch-all
- `x_api/client/mod.rs:126-139` — matches on HTTP status codes (internal, not affected)
- `workflow/mod.rs` — uses `#[error(transparent)]` passthrough

The catch-all `other =>` arms handle new variants. Clippy should be clean since we're not adding non-exhaustive patterns.

### R4: `deps.rs` scraper path misses a dependency
**Severity:** Medium
**Mitigation:** The scraper `init_scraper_mode()` must initialize ALL fields of `RuntimeDeps`. Audit the struct definition (34 fields) and ensure each is set. LLM, scoring, safety, storage adapters are all independent of X API transport.

### R5: Token file absence crashes in scraper mode
**Severity:** High
**Mitigation:** In scraper mode, `load_tokens_from_file()` is never called. The early branch in `init()` returns before reaching the token loading code. Verify this by checking the branch order.
