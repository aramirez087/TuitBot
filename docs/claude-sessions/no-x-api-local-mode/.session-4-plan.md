**New content to add:**

```toml
# --- X Access Mode ---
# Controls how Tuitbot connects to X.
#   "" or "x_api" (default) — Official X API with OAuth credentials. Full features.
#   "scraper"               — Local No-Key Mode. No API credentials needed.
#                              Read-only by default. Discovery and drafting work;
#                              posting, mentions, and analytics are unavailable.
#                              Only available in desktop and self_host deployment modes.
# provider_backend = ""
#
# Environment variable override: TUITBOT_X_API__PROVIDER_BACKEND=x_api|scraper

# Allow write operations in Local No-Key Mode (provider_backend = "scraper").
# Default: false. When false, all posting/liking/following is blocked.
# When true, writes are attempted via the scraper transport (elevated risk
# of account restrictions — the Official X API is recommended for posting).
# scraper_allow_mutations = false
#
# Environment variable override: TUITBOT_X_API__SCRAPER_ALLOW_MUTATIONS=true|false
```

**Rationale:** Documents the new config fields with their defaults, constraints, env var overrides, and deployment restrictions. Mirrors the existing documentation style used for `mode`, `approval_mode`, and `deployment_mode`.

**Risks:** None — this is additive documentation. The commented-out defaults match the actual `#[serde(default)]` behavior.

### Step 4: Update `README.md`

**File:** `/Users/aramirez/Code/ReplyGuy/README.md`

**Changes:**

1. **Add a "Local No-Key Mode" subsection** under "Getting Started" (after section 4, before "Quick Commands"). This goes between the CLI section (line 248) and the Quick Commands heading (line 252). New section:

```markdown
### 5. Local No-Key Mode (No X API Credentials)

Don't have an X Developer Portal account? Tuitbot can run discovery and
content drafting without any API credentials:

```bash
tuitbot init  # select "Local No-Key Mode" when prompted
```

Or set it manually in `config.toml`:

```toml
[x_api]
provider_backend = "scraper"
```

**What works:** Tweet discovery (public data), conversation scoring,
LLM-powered reply/tweet/thread drafting, approval queue staging, thread
planning.

**What doesn't:** Posting, liking, following, mentions, home timeline,
bookmarks, analytics, media upload. These require the Official X API —
you can switch anytime in Settings.

**Tradeoffs vs. Official API:**
- Data comes from public endpoints, not the authenticated API. Results
  may be less complete.
- Write operations are blocked by default (`scraper_allow_mutations = false`).
  Enabling them carries elevated risk of account restrictions.
- Only available in desktop and self-hosted modes (not cloud).

> **Recommendation:** Start with Local No-Key Mode to evaluate Tuitbot,
> then upgrade to the Official X API when you're ready to post.
```

2. **Update the "Choose Your Path" table** (around line 26) to add a third row:

```markdown
| **Try without an X API key** (discovery + drafting, no posting) | [Local No-Key Mode](#5-local-no-key-mode-no-x-api-credentials) |
```

3. **Renumber "Quick Commands" heading** from being after section 4 to after section 5.

**Rationale:** Users need to discover the no-key option early in the README. The "Choose Your Path" table is the primary navigation device; adding a row there ensures the option is visible. The dedicated section explains tradeoffs honestly.

**Risks:**
- Must preserve existing anchor links. The new section uses a new anchor that doesn't conflict.
- Must not overclaim capabilities — the transport stubs mean reads will also fail until a future session implements actual scraping. The copy says "public data" which is accurate about the design intent, and the transport stubs will return clear errors if users actually try to run discovery now.

**Key design decision:** Be honest about current state. The `ScraperTransportUnavailable` errors mean discovery won't actually return results yet, but the infrastructure is fully wired. The README should describe the intended behavior (what the feature does when transport is implemented) while the release-readiness doc captures the current stub state.

**Resolution:** Add a brief note that the scraper transport is under active development and discovery will return "transport unavailable" errors until the next release. This avoids misleading users while still documenting the feature.
