# Session 09 Implementation Plan — Settings & Consent-Driven Analytics Sync

**Date:** 2026-03-22
**Session:** 09 of 11
**Goal:** Add `analytics_sync_enabled` setting, distinguish it from `loop_back_enabled`, and implement a one-time consent prompt for eligible users.

## Architecture Overview

Two distinct data flows need clear user-facing separation:

1. **Publish writeback** (`loop_back_enabled`, per-source) — Writes tweet metadata (tweet_id, url, published_at) to note frontmatter immediately after publish. Already implemented. Lives on `ContentSourceEntry`.

2. **Analytics sync** (`analytics_sync_enabled`, per-source) — Periodically enriches note frontmatter with performance data (impressions, likes, engagement_rate, performance_score). Built in Session 08 as `run_forge_sync()`. **Not yet wired to config.**

This session bridges them: adds the config field, wires the UI toggle, and adds a consent prompt so users opt in with full understanding.

## Detailed Implementation Steps

### Step 1: Add `analytics_sync_enabled` to `ContentSourceEntry` (Rust)

**File:** `crates/tuitbot-core/src/config/types.rs`

Add field after `loop_back_enabled` (line ~536):

```rust
/// Whether to sync analytics data (impressions, engagement, performance
/// score) back into source file frontmatter on a periodic schedule.
///
/// Requires `loop_back_enabled` to be true (analytics sync is a superset
/// of publish writeback). Only supported for `local_fs` sources.
/// Default: false.
#[serde(default)]
pub analytics_sync_enabled: bool,
```

**Design decisions:**
- Default `false` via `#[serde(default)]` (bool defaults to false). No custom default function needed.
- Separate from `loop_back_enabled`: publish writeback is lightweight (one write per publish), analytics sync is periodic and writes more data.
- Per-source, not global: different vaults may have different preferences.
- No validation coupling: `analytics_sync_enabled: true` with `loop_back_enabled: false` is allowed at the config level. The sync engine already checks `loop_back_enabled` via the source config_json. The UI will show a hint but not block.

**Tests to add in `types.rs`:**

```rust
#[test]

    let entry: ContentSourceEntry = serde_json::from_str("{}").unwrap();
    assert!(!entry.analytics_sync_enabled);
}

#[test]
fn content_source_entry_analytics_sync_roundtrip() {
    let mut entry: ContentSourceEntry = serde_json::from_str("{}").unwrap();
    entry.analytics_sync_enabled = true;
    let json = serde_json::to_string(&entry).unwrap();
    let back: ContentSourceEntry = serde_json::from_str(&json).unwrap();
    assert!(back.analytics_sync_enabled);
}
```

### Step 8: Implement prompt state management

**File:** `dashboard/src/lib/stores/settings.ts`

Add localStorage-backed state for prompt dismissal:

```typescript
const ANALYTICS_SYNC_PROMPT_KEY = 'tuitbot:analytics-sync-prompt-dismissed';

export const analyticsSyncPromptDismissed = writable<boolean>(
    typeof localStorage !== 'undefined'
        ? localStorage.getItem(ANALYTICS_SYNC_PROMPT_KEY) === 'true'
        : false
);

export function dismissAnalyticsSyncPrompt() {
    analyticsSyncPromptDismissed.set(true);
    if (typeof localStorage !== 'undefined') {
        localStorage.setItem(ANALYTICS_SYNC_PROMPT_KEY, 'true');
    }
}

export function resetAnalyticsSyncPrompt() {
    analyticsSyncPromptDismissed.set(false);
    if (typeof localStorage !== 'undefined') {
        localStorage.removeItem(ANALYTICS_SYNC_PROMPT_KEY);
    }
}
```

**Prompt visibility logic** (computed where the prompt is rendered):

```typescript
const showAnalyticsSyncPrompt = $derived(
    hasPendingPrompt &&
    !$analyticsSyncPromptDismissed &&
    !sourceAnalyticsSync
);
```

Where `hasPendingPrompt` is set by a WebSocket event or a flag returned from the publish response.

### Step 10: "Not now" behavior (Task 5)

- `dismissAnalyticsSyncPrompt()` sets localStorage `'true'`

- This is lightweight: just add a check in the existing `saveSettings()` function.

Actually, the simplest correct behavior per the task spec ("suppress repeats until the user revisits Settings or explicitly re-enables prompting state"):

## Key Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D61 | `analytics_sync_enabled` is per-source on `ContentSourceEntry`, not a global config field | Matches `loop_back_enabled` pattern. Users with multiple vaults may want sync on one but not another. |
| D62 | Default `false` via `#[serde(default)]` | Opt-in per epic contract. No behavioral change for existing users. |
| D63 | Prompt is a banner, not a modal | Modals block workflow. A dismissable banner respects user agency (Hook Miner principle: no hidden magic, easy fallback). |
| D64 | Prompt renders on Activity page, not compose view | Activity is where users go after publish. Compose view should stay focused on drafting. |
| D65 | "Not now" suppresses until user visits Settings page | Balances not-spamming with re-discoverability. Settings visit resets the dismissal. |
| D66 | Analytics sync toggle only visible when `loop_back_enabled` is true | Analytics sync is a superset — showing it when writeback is off would be confusing. |
| D67 | Keep `run_forge_sync()` parameter-based (don't change to read from DB) | Per Session 08 handoff requirement. Config wiring happens at the call site. |
| D68 | Google Drive unsupported notice is informational, not an error | Users shouldn't feel they're doing something wrong. Just explaining a capability boundary. |
| D69 | localStorage for prompt state, not DB | Prompt state is UI-only concern. No need to persist across devices or survive data wipes. |
| D70 | Prompt triggers on WebSocket `ActionPerformed` event for publish actions | Avoids adding a new API endpoint. Leverages existing real-time event infrastructure. |
