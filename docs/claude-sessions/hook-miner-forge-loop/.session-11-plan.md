## Task 2: Compute PerformancePercentiles

**Purpose:** The Forge sync engine requires `PerformancePercentiles` (p50, p90 impressions + sufficient data flag). Currently no DB query computes this.

### Design

Add `compute_performance_percentiles_for()` to `crates/tuitbot-core/src/storage/analytics/tweet_performance.rs`.

**Approach:** Pure Rust computation over fetched rows (not SQL percentile functions, which SQLite lacks).

```rust
pub async fn compute_performance_percentiles_for(
    pool: &DbPool,
    account_id: &str,
) -> Result<PerformancePercentiles, StorageError>
```

**Algorithm:**
1. Query: `SELECT impressions FROM tweet_performance WHERE account_id = ? ORDER BY impressions`
2. If count < 10: return `PerformancePercentiles { p50: 0, p90: 0, has_sufficient_data: false }`
3. p50 = value at index `count / 2`
4. p90 = value at index `count * 9 / 10`
5. Return with `has_sufficient_data: true`

**Import note:** The `PerformancePercentiles` type is defined in `sync.rs`. To avoid a circular dependency (storage → sync), we have two options:
- **Option A (preferred):** Move `PerformancePercentiles` to `crates/tuitbot-core/src/storage/analytics/tweet_performance.rs` since it's a data type used by both storage and sync. Re-export from sync.rs for backward compatibility.
- **Option B:** Return raw `(i64, i64, bool)` tuple from the storage function and construct `PerformancePercentiles` at the call site.

**Decision:** Option A. `PerformancePercentiles` is fundamentally about storage data shape. Move it and re-export.

|------|--------|
| `crates/tuitbot-core/src/storage/analytics/tweet_performance.rs` | Add `compute_performance_percentiles_for()` function. Move `PerformancePercentiles` struct here. |
| `crates/tuitbot-core/src/automation/watchtower/loopback/sync.rs` | Replace local `PerformancePercentiles` with re-export: `pub use crate::storage::analytics::tweet_performance::PerformancePercentiles;` |

## Task 3: Wire `run_forge_sync()` into Analytics Loop

**Purpose:** After each analytics iteration, trigger Forge sync if enabled.

### Design

The analytics loop (`analytics_loop.rs`) uses trait-based port architecture. To add Forge sync, we need to:

1. **Add a new method to `AnalyticsStorage` trait:**
   ```rust
   async fn get_forge_sync_config(&self) -> Result<Option<(bool, PerformancePercentiles)>, AnalyticsError>;
   ```
   This returns `(analytics_sync_enabled, percentiles)` for the active account, or `None` if not applicable.

2. **Alternative (simpler, preferred):** Add a separate optional `ForgeSyncRunner` trait:
   ```rust
   #[async_trait::async_trait]
   pub trait ForgeSyncRunner: Send + Sync {
       async fn run_forge_sync(&self) -> Result<ForgeSyncResult, AnalyticsError>;
   }
   ```
   Pass as `Option<Arc<dyn ForgeSyncRunner>>` to `AnalyticsLoop::new()`. This keeps the analytics loop decoupled from Forge internals.

**Decision:** Use the simpler approach — pass an optional closure/trait. But to minimize changes and keep the port pattern clean, extend `AnalyticsStorage` with one new method:

```rust
async fn run_forge_sync_if_enabled(&self) -> Result<Option<ForgeSyncSummary>, AnalyticsError>;
```

The implementation in the server crate will:
1. Look up `analytics_sync_enabled` from the active content source config
2. Call `compute_performance_percentiles_for()`
3. Call `run_forge_sync()` with the parameters
4. Return the summary

This keeps the analytics loop unaware of Forge internals — it just calls the storage method and logs the result.

### Files to modify

| File | Action |
|------|--------|
| `crates/tuitbot-core/src/automation/analytics_loop.rs` | Add `run_forge_sync_if_enabled()` to `AnalyticsStorage` trait. Call it after tweet measurement in `run_iteration()`. Add `forge_synced: bool` to `AnalyticsSummary`. |
| `crates/tuitbot-core/src/automation/analytics_loop.rs` (tests) | Update `MockAnalyticsStorage` with default impl returning `Ok(None)`. |

### Integration point in `run_iteration()`

After the tweet performance measurement block (line ~302) and before the final `log_action`, add:

```rust
// 4. Forge sync (if enabled)
match self.storage.run_forge_sync_if_enabled().await {
    Ok(Some(forge_summary)) => {
        tracing::info!(
            tweets_synced = forge_summary.tweets_synced,
            threads_synced = forge_summary.threads_synced,
            "Forge sync complete"
        );
        summary.forge_synced = true;
    }
    Ok(None) => {
        // Forge sync not enabled — no action
    }
    Err(e) => {
        // Forge sync failure is non-fatal
        tracing::warn!(error = %e, "Forge sync failed");
    }
}
```

**Key constraint:** Forge sync failure must NOT fail the analytics iteration. It's a best-effort enrichment step.

## Task 4: Wire `setPendingAnalyticsSyncPrompt()` from Approval Store

**Purpose:** When a user approves content that originated from a local_fs source, show the Forge consent prompt on the Activity page.

### Design

In `dashboard/src/lib/stores/approval.ts`, the WebSocket handler at line 210 handles `ApprovalQueued` and `ApprovalUpdated`. We need to detect publish success:

- Event type: `ApprovalUpdated` with `status === 'approved'` (or a new `ActionPerformed` event)
- Check: the item's source type must be eligible for Forge (local_fs)

**Simplest approach:** When `ApprovalUpdated` fires with `updatedStatus === 'approved'`, call `setPendingAnalyticsSyncPrompt()`. The Activity page already conditionally renders the `AnalyticsSyncPrompt` banner based on `pendingAnalyticsSyncPrompt && !analyticsSyncPromptDismissed`.

**Edge case:** Only trigger if the prompt hasn't been permanently dismissed. The `AnalyticsSyncPrompt` component already checks `analyticsSyncPromptDismissed`, so we can unconditionally call `setPendingAnalyticsSyncPrompt()` on approval and let the rendering logic handle visibility.

### 5d. Forge sync success/failure events

**Where sync is triggered:** The backend runs `run_forge_sync()` and returns the summary via the `run_forge_sync_if_enabled()` trait method. The frontend doesn't directly invoke sync — it happens server-side in the analytics loop.

**Options for tracking `forge.sync_succeeded` / `forge.sync_failed`:**
1. **Backend-side logging only:** Since sync runs in the analytics loop (server-side), these events should be server-side structured logs, not frontend telemetry calls.
2. **Frontend tracks via WebSocket notification:** Add a new WS event `ForgeSyncCompleted` that the frontend handles.

**Decision:** Option 1 (backend logging only). The existing `tracing::info!` and `tracing::warn!` calls in the analytics loop provide structured log data. The frontend event helpers `trackForgeSyncSucceeded` / `trackForgeSyncFailed` can be wired later if a WebSocket notification is added. For v1, server logs are sufficient.

**Rationale:** Adding a WebSocket event for sync results is out of scope for the integration session. The helpers exist and are tested — they can be wired when a sync-status UI is built.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `AnalyticsStorage` trait change breaks downstream implementors | Medium | Only 1 implementor exists (in server crate). Add default impl returning `Ok(None)` to make it non-breaking for tests. |
| Percentile computation off-by-one | Low | Use well-known nearest-rank method. Test with known input/output pairs. |
| AngleCards missing sessionId/sourcePathStem context | Medium | Trace prop flow from composer parent. Use `'unknown'` defaults if context unavailable — telemetry still fires, just with reduced metadata. |
| Forge sync events not wired to frontend | Low | Accepted for v1. Server logs cover this. Document as known gap. |
| Test database not available for `run_forge_sync` integration test | Medium | Use unit tests for individual functions. `run_forge_sync` itself is integration-tested via the analytics loop mock. |

## Key Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D77 | `compute_performance_percentiles_for()` uses Rust sorting, not SQL | SQLite lacks `PERCENTILE_CONT`. Fetching all impressions and sorting in Rust is simpler and portable. |
| D78 | Extend `AnalyticsStorage` trait with `run_forge_sync_if_enabled()` | Keeps analytics loop decoupled from Forge internals. The concrete impl in the server crate wires DB queries + sync engine. |
| D79 | Forge sync failure is non-fatal in analytics loop | Forge is enrichment, not core. Analytics iteration must not fail because Forge had an I/O error. |
| D80 | `setPendingAnalyticsSyncPrompt()` called on any `approved` status | Simple and correct. The rendering layer (AnalyticsSyncPrompt) already gates on `!dismissed`. No need to check source type at the approval layer. |
| D81 | `forge.sync_succeeded` / `forge.sync_failed` remain server-side only for v1 | No WebSocket notification for sync results exists. Adding one is scope creep. Server structured logs are sufficient for v1 metrics. |
| D82 | Move `PerformancePercentiles` to storage module | Type is fundamentally a storage data shape. Re-export from sync.rs maintains backward compat. |
