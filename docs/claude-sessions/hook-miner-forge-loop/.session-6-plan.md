## Current State Summary

### What exists today (`LoopBackEntry` in `loopback.rs:16-28`)

The `tuitbot:` YAML array in note frontmatter currently supports:
- `tweet_id` (String) — X post ID
- `url` (String) — full X URL
- `published_at` (String) — ISO-8601 UTC
- `type` (String) — "tweet", "thread", "reply"
- `status` (Option<String>) — "posted", "deleted", etc.
- `thread_url` (Option<String>) — never populated today

### What's missing for Forge

1. **Analytics fields** on each entry: `impressions`, `likes`, `retweets`, `replies`, `engagement_rate`, `performance_score`, `synced_at`
2. **Thread fields** on thread entries: `child_tweet_ids` (array of child tweet IDs)
3. **Note-level summary fields**: `tuitbot_social_performance`, `tuitbot_best_post_impressions`, `tuitbot_best_post_url`, `tuitbot_last_synced_at`
4. **Update semantics**: Current `write_metadata_to_file()` is append-only (deduplicates by `tweet_id` but never updates existing entries). Forge needs to update analytics fields on existing entries.
5. **Thread aggregation**: No concept of aggregating child tweet metrics into a single thread entry.

### Key constraints from prior sessions

- `source_role = 'primary_selection'` determines which note receives Forge writeback (from Session 05, D30)
- Only `local_fs` sources support writes (`loopback.rs:80`)
- Provenance propagates through: `approval_queue` → `original_tweet` via `copy_links_for()`
- Thread entity type exists in provenance but is never populated by approval poster (gap from audit)

## Design Decisions

### D37: Additive analytics fields on `LoopBackEntry`

All new fields are optional with `skip_serializing_if = "Option::is_none"`. Existing entries without analytics remain valid. Fields:

| Field | Type | Source | Semantics |
|-------|------|--------|-----------|
| `impressions` | `Option<i64>` | X API | Total views |
| `likes` | `Option<i64>` | X API | Like count |
| `retweets` | `Option<i64>` | X API | Retweet count |
| `replies` | `Option<i64>` | X API | Reply count |
| `engagement_rate` | `Option<f64>` | Computed | `(likes + retweets + replies) / impressions` × 100, or `null` if impressions = 0 |
| `performance_score` | `Option<f64>` | `tweet_performance.performance_score` | Tuitbot's composite scoring |
| `synced_at` | `Option<String>` | System clock | ISO-8601 UTC timestamp of last analytics sync |
| `child_tweet_ids` | `Option<Vec<String>>` | Thread posting | Only for `type: thread` entries |

**Rationale:** Matches the existing `tweet_performance` table schema. `engagement_rate` is derived client-side for Obsidian readability — no Obsidian plugin needed to compute it.

### D38: Note-level summary fields as top-level frontmatter keys

| Key | Type | Semantics |
|-----|------|-----------|
| `tuitbot_social_performance` | `String` | Human-readable label: `"high"`, `"medium"`, `"low"`, `"none"` |
| `tuitbot_best_post_impressions` | `i64` | Impression count of highest-performing entry |
| `tuitbot_best_post_url` | `String` | URL of highest-performing entry |
| `tuitbot_last_synced_at` | `String` | ISO-8601 UTC of most recent analytics sync across all entries |

**Rationale:** Top-level keys (not inside `tuitbot:` array) enable Obsidian Dataview queries like `WHERE tuitbot_social_performance = "high"`. The `tuitbot_` prefix avoids namespace collisions with user frontmatter.

### D39: Idempotency via tweet_id matching with in-place update

Current behavior: `write_metadata_to_file()` skips if `tweet_id` exists. New behavior:
- **Publish step** (approval poster): Append-only (unchanged). Creates entry with analytics fields = `null`.
- **Forge sync step** (new): Match by `tweet_id`, update analytics fields in-place. If no match exists, skip (don't create orphan analytics entries).

This means `write_metadata_to_file()` gains a sibling: `update_entry_analytics()` that deserializes, finds the matching entry, mutates analytics fields, re-serializes.

### D40: Thread representation as single entry with aggregated metrics

A thread is **one entry** in the `tuitbot:` array:
- `tweet_id` = root tweet ID (the first tweet posted)
- `type` = `"thread"`
- `url` = URL of root tweet
- `thread_url` = same as `url` (for thread-aware clients)
- `child_tweet_ids` = array of child tweet IDs (position-ordered, excludes root)
- Analytics fields = **aggregated** across root + all children:
  - `impressions` = sum of all tweet impressions
  - `likes` = sum of all tweet likes
  - `retweets` = sum of all tweet retweets
  - `replies` = sum of all tweet replies
  - `engagement_rate` = `(total_likes + total_retweets + total_replies) / total_impressions` × 100
  - `performance_score` = average of child performance scores (weighted by impressions)

**Rationale:** One entry per outcome, not one per tweet. Users see thread performance as a single unit in their note. Individual tweet metrics are available in the dashboard analytics, not in frontmatter.

### D41: Performance tier thresholds for `tuitbot_social_performance`

| Tier | Condition |
|------|-----------|
| `"high"` | Best post impressions ≥ 90th percentile of account's historical posts |
| `"medium"` | Best post impressions ≥ 50th percentile |
| `"low"` | Best post impressions > 0 but < 50th percentile |
| `"none"` | No synced analytics data |

**Rationale:** Relative to the account's own history, not absolute numbers. A 500-follower account and a 50K-follower account both get meaningful tiers.

### D42: Summary fields always reflect highest-impression entry

When multiple entries exist in `tuitbot:` (e.g., same note produced a tweet and later a thread):
- `tuitbot_best_post_impressions` = max impressions across all entries
- `tuitbot_best_post_url` = URL of that entry
- `tuitbot_social_performance` = tier based on `tuitbot_best_post_impressions`
- `tuitbot_last_synced_at` = most recent `synced_at` across all entries

## Key Design Rationale

### Why top-level summary fields instead of nested under `tuitbot:`

Obsidian Dataview queries work best with top-level frontmatter keys. `tuitbot_social_performance` as a top-level key enables:
```dataview
TABLE tuitbot_social_performance, tuitbot_best_post_impressions
WHERE tuitbot_social_performance = "high"
SORT tuitbot_best_post_impressions DESC
```

If nested under `tuitbot:`, Dataview would need array traversal which is significantly more complex.

### Why aggregated thread metrics instead of per-tweet entries

The `tuitbot:` array represents **outcomes**, not **infrastructure**. A thread is one outcome (one piece of content, one user action). Per-tweet entries would:
- Bloat frontmatter (5-tweet thread = 5 entries)
- Make summary field computation complex (which tweet represents the thread?)
- Confuse the Dataview story (is this note performing well because of 5 entries or 1 good thread?)

### Why `update_entry_analytics()` as a separate function from `write_metadata_to_file()`

Separation of concerns:
- `write_metadata_to_file()` — called at publish time, append-only, fast, never touches analytics
- `update_entry_analytics()` — called by Forge sync, match-and-update, recomputes summaries

This keeps the publish path simple and the sync path isolated.

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| YAML serialization order may change between writes | Low | Use `serde_yaml::Mapping` with insertion-order preservation (already used in `TuitbotFrontMatter.other`) |
| Thread `child_tweet_ids` may not be available at publish time (approval poster doesn't chain replies) | Medium | Session 07 fixes thread posting. Until then, `child_tweet_ids` is populated during Forge sync from `thread_tweets` table. |
| `tuitbot_social_performance` percentile computation requires historical data | Low | Default to `"none"` until sufficient history exists (≥10 posts). |
| Large frontmatter with many entries (prolific notes) | Low | Cap `tuitbot:` array at 20 entries. Oldest entries beyond cap are archived (moved to a separate `tuitbot_archived:` key or simply dropped). Document this in the contract. |
| `engagement_rate` division by zero when impressions = 0 | Low | Contract specifies `null` when impressions = 0. |
