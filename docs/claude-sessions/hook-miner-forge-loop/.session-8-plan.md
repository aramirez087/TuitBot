# Session 8 Implementation Plan — Forge Sync Engine

**Date:** 2026-03-22
**Session:** 08 of 11
**Goal:** Build an idempotent analytics sync engine that enriches source-note frontmatter when tweet metrics arrive.

---

## Current State Assessment

- **`loopback.rs`** is 542 lines (over the 500-line limit). Adding ~200 lines of sync logic here requires splitting first.
- **`LoopBackEntry`** currently has 7 fields and derives `PartialEq, Eq`. Adding f64 fields requires removing `Eq`.
- **`TuitbotFrontMatter`** uses `#[serde(flatten)] other: serde_yaml::Mapping` — summary fields can be written via string keys into `other`.
- **`tweet_performance`** table has: `tweet_id`, `likes_received`, `retweets_received`, `replies_received`, `impressions`, `performance_score`, `measured_at`, `account_id`.
- **`vault_provenance_links`** has `entity_type`, `entity_id`, `source_path`, `source_role`, `node_id` — can query provenance for `original_tweet` entities with `primary_selection` role to find which note to write back to.
- **`get_thread_tweet_ids_by_root_for()`** returns child tweet IDs from `thread_tweets` table — fallback for entries missing `child_tweet_ids`.
- **`analytics_sync_enabled`** setting does not exist in config yet (Session 09 adds it). This session accepts it as a boolean parameter passed into the sync step.
- `write_metadata_to_file()` is append-only (publish path). The new `update_entry_analytics()` is match-and-update (sync path). Both coexist.

## Key Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D51 | Split `loopback.rs` into `loopback/mod.rs` + `loopback/sync.rs` + `loopback/tests.rs` | File is already 542 lines; adding sync logic would push to ~750. CLAUDE.md mandates 500-line max. Tests >100 lines go to `tests.rs` submodule. |
| D52 | `LoopBackEntry` changes from `PartialEq, Eq` to `PartialEq` only | f64 fields (`engagement_rate`, `performance_score`) don't implement `Eq`. No current code depends on `Eq` for `LoopBackEntry`. |
| D53 | `update_entry_analytics()` returns `UpdateResult` enum, not `Result<bool>` | Three-variant enum (`Updated`, `EntryNotFound`, `FileNotFound`) is more expressive than a boolean and matches the contract in `forge-frontmatter-contract.md`. |
| D54 | Sync step queries provenance directly (not through watchtower node/source lookup) | The existing loopback resolves path via `node_id → content_node → source_context`. The sync path starts from tweet_id → `original_tweets.id` → `vault_provenance_links` → `source_path`. Direct provenance query is more efficient for batch processing and avoids the node_id indirection. |
| D55 | Summary recomputation uses `serde_yaml::Mapping` string keys via `TuitbotFrontMatter.other` | Avoids adding explicit fields to `TuitbotFrontMatter` struct, maintaining backward/forward compat as specified in `forge-frontmatter-contract.md`. |
| D56 | Performance tier calculation accepts a percentile provider trait/closure | The tier thresholds (p50, p90) depend on account-wide historical data. Pass percentile values into the sync function rather than querying inside the file-write function. Keeps file I/O and DB I/O separate. |
| D57 | Thread aggregation sums counts from `tweet_performance` table for all tweet IDs (root + children) | Matches the contract in `forge-thread-contract.md`. Uses existing `tweet_performance` rows — no new table needed. |
| D58 | `analytics_sync_enabled` is a boolean parameter to the sync orchestrator, not read from DB | Session 09 adds the config/UI for this setting. This session builds the engine; the wiring comes later. |

### Step 8: Implement `ForgeSyncStep` orchestrator

**Goal:** The top-level sync function that ties everything together.

```rust
/// Result summary of a Forge sync iteration.
#[derive(Debug, Default)]
pub struct ForgeSyncSummary {
    pub tweets_synced: usize,
    pub threads_synced: usize,
    pub entries_not_found: usize,
    pub files_not_found: usize,
    pub non_local_skipped: usize,
}

/// Run a single Forge sync iteration.
///
/// Finds all tweets with measured performance that have provenance
/// to local_fs source notes, and writes aggregated analytics into
/// those notes' frontmatter.
pub async fn run_forge_sync(
    pool: &DbPool,
    account_id: &str,
    analytics_sync_enabled: bool,
    percentiles: &PerformancePercentiles,
) -> Result<ForgeSyncSummary, SyncError> {
    if !analytics_sync_enabled {
        return Ok(ForgeSyncSummary::default());
    }

    // 1. Get all tweet_performance rows for this account that have been
    //    measured since last sync (use cursor or scan all)
    // 2. For each tweet_id with performance data:
    //    a. Look up original_tweets.id by tweet_id
    //    b. Look up provenance with source_role = 'primary_selection'
    //    c. Check source_type = 'local_fs' (skip google_drive, cloud)
    //    d. Resolve full file path from source_context config_json + relative_path
    //    e. If type == thread:
    //       - Get child_tweet_ids from entry or thread_tweets table
    //       - Fetch performances for all IDs
    //       - Aggregate via aggregate_thread_metrics()
    //    f. Call update_entry_analytics(path, tweet_id, analytics, percentiles)
    //    g. Record result in summary
    // 3. Return summary
}
```

**Non-local skip logic:**
- `source_type != "local_fs"` → skip with `non_local_skipped += 1`
- File doesn't exist on disk → `files_not_found += 1`, no error
- Entry not in frontmatter → `entries_not_found += 1`, no error

**Fail-open behavior:** All errors in the per-tweet loop are logged and counted, never propagated. The overall sync continues even if individual notes fail.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `loopback.rs` → `loopback/` split breaks imports | Low | Module name unchanged; `pub use` in mod.rs preserves all public API paths. Run full workspace test after split. |
| `Eq` removal on `LoopBackEntry` breaks callers | Low | Searched codebase: only `LoopBackResult` uses `Eq`. Tests use `assert_eq!` which needs `PartialEq` only. |
| YAML serialization order changes after analytics fields added | Medium | `serde_yaml` serializes fields in struct order. Analytics fields come after existing fields. Existing tests verify roundtrip. Add a test that checks field ordering in output. |
| `TuitbotFrontMatter.other` mapping key collisions with new summary fields | Low | Summary keys use `tuitbot_` prefix. The `tuitbot` key (array) is a separate struct field, not in `other`. |
| Large number of notes to sync in one iteration | Medium | Sync processes notes sequentially with per-note error isolation. No unbounded memory allocation. Future optimization (batching, cursor) deferred. |
| Performance tier calculation requires account-wide query | Low | Percentiles are computed once per sync iteration, not per note. Passed as parameter to avoid repeated DB queries. |
| `tweet_performance` may not have rows for recently posted tweets (24h delay) | Low | By design: analytics loop measures ~24h after posting. Forge sync only writes to notes that have measured metrics. Entries without metrics are simply skipped. |
| Thread entry missing `child_tweet_ids` (legacy entries) | Low | Forge sync falls back to `get_thread_tweet_ids_by_root_for()` from DB. If DB also has no children, aggregation covers root only — matches contract. |
