# Session 05 Implementation Plan — Hook Miner Provenance & Draft Semantics

**Date:** 2026-03-22
**Session:** 05 of 11
**Goal:** Make Hook Miner attribution survive draft creation, revision, publish, and later Forge sync.

---

## 1. Current State Analysis

### What exists today

**Database schema (`vault_provenance_links`):**
- Core columns: `id`, `account_id`, `entity_type`, `entity_id`, `node_id`, `chunk_id`, `seed_id`, `source_path`, `heading_path`, `snippet`, `created_at`
- Graph columns (added in `20260321000200`): `edge_type`, `edge_label`

**Rust types:**
- `ProvenanceRef` (8 optional fields): `node_id`, `chunk_id`, `seed_id`, `source_path`, `heading_path`, `snippet`, `edge_type`, `edge_label`
- `ProvenanceLink` (DB row): mirrors the table columns
- `ProvenanceInput` (approval_queue bridge): `source_node_id`, `source_seed_id`, `source_chunks_json`, `refs: Vec<ProvenanceRef>`

**Frontend types:**
- `ProvenanceRef` in `types.ts`: mirrors Rust struct (all optional fields)
- `ProvenanceLink` in `types.ts`: mirrors DB row (missing `edge_type`/`edge_label`)

**Provenance flow today:**
1. User selects angle → `handleAngleSelected()` builds `neighborProv` with `{node_id, edge_type, edge_label}`
2. Calls `ongenerate(nodeIds, format, [seed_text], angle_type, neighborProv)`
3. `ComposerInspector.handleGenerateFromVault()` maps node IDs to `ProvenanceRef[]` (captures `node_id`, `edge_type`, `edge_label`)
4. `vaultHookStyle` captures `angle.angle_type` (e.g., `"story"`)
5. Parent composer POSTs to `/api/content/compose` or `/api/content/drafts` with `{provenance, hook_style}`
6. Server-side `build_provenance_input()` extracts refs → `ProvenanceInput`
7. `insert_links_for()` writes rows to `vault_provenance_links`

### What's missing

The session tasks require four new fields on provenance: `angle_kind`, `signal_kind`, `signal_text`, and `source_role`. Currently:

1. **`angle_kind`** — The `hook_style` field carries the angle type string (`"story"`, `"listicle"`, `"hot_take"`) but it's stored only on the compose request's `source` enrichment (`"assist:hook:story"`), NOT on the provenance links themselves. A later Forge sync querying `vault_provenance_links` cannot determine which angle type produced this content.

2. **`signal_kind`** — Evidence type (`"contradiction"`, `"data_point"`, `"aha_moment"`) from the selected angle's evidence items is never persisted. It's displayed in `AngleCards` but lost after angle selection.

3. **`signal_text`** — The `citation_text` from evidence items is partially captured via `snippet` on some provenance refs (where the neighbor's chunk snippet is used), but the specific evidence citation that backed the angle is not explicitly preserved.

4. **`source_role`** — All provenance refs are flat; there's no way to distinguish the primary selection note from accepted neighbor notes. The `edge_type`/`edge_label` fields hint at it (non-null = neighbor), but this is implicit, not explicit.

## 2. Design Decisions

### D29: Additive columns on `vault_provenance_links`

Add four new nullable columns via a new migration:

```sql
ALTER TABLE vault_provenance_links ADD COLUMN angle_kind TEXT;
ALTER TABLE vault_provenance_links ADD COLUMN signal_kind TEXT;
ALTER TABLE vault_provenance_links ADD COLUMN signal_text TEXT;
ALTER TABLE vault_provenance_links ADD COLUMN source_role TEXT;
```

**Rationale:** All nullable → backward compatible. Existing rows have `NULL` for these fields. No defaults needed. The `angle_kind` column name avoids collision with `edge_type` (which describes graph relationships, not content angles).

### D30: `source_role` values

Exactly two values: `"primary_selection"` and `"accepted_neighbor"`.

- `primary_selection`: The note the user originally selected in Obsidian (the `resolved_node_id`).
- `accepted_neighbor`: A related note the user accepted from `GraphSuggestionCards`.

**Rationale:** Later Forge sync needs to know which note to write back to (the primary selection). Accepted neighbors are supporting context but shouldn't receive writeback.

### D31: `angle_kind` is set on ALL provenance refs for a given draft

When an angle-sourced draft is created, every provenance link for that entity gets the same `angle_kind` value. This is because the angle type is a property of the generation, not of individual source notes.

**Rationale:** Simpler querying — `SELECT DISTINCT angle_kind FROM vault_provenance_links WHERE entity_type = ? AND entity_id = ?` gives the angle. No need to join or filter.

### D32: `signal_kind` and `signal_text` only set on evidence-bearing refs

Only provenance links that correspond to evidence items get `signal_kind`/`signal_text`. The primary selection note and neighbors without extracted evidence keep these `NULL`.

**Rationale:** These fields describe the evidence relationship, which only exists for items the LLM extracted evidence from.

### D33: Frontend passes enriched provenance

The `ProvenanceRef` type (both Rust and TypeScript) gains the four new optional fields. The frontend constructs them during angle selection, before calling `ongenerate`. This avoids server-side re-derivation.

### D34: `copy_links_for` automatically copies new columns

The existing `copy_links_for` SQL uses `SELECT ... FROM vault_provenance_links`. We must add the new columns to the SELECT clause. Same for the INSERT column list.

### D35: Backward compatibility via `#[serde(default)]`

All new fields use `#[serde(default)]` in `ProvenanceRef`, same pattern as existing fields. Deserialization of old payloads (without the new fields) produces `None`.

### Step 9: Enrich frontend provenance during angle selection

**File:** `dashboard/src/lib/components/composer/VaultSelectionReview.svelte`

In `handleAngleSelected()`, modify the `neighborProv` construction to include `source_role`:
```typescript
// Primary selection node gets source_role: 'primary_selection'
// Accepted neighbors get source_role: 'accepted_neighbor'
```

Pass the selected angle's evidence metadata through the `ongenerate` callback. This requires extending the callback signature to include evidence items OR pre-building enriched `ProvenanceRef[]` with `angle_kind`, `signal_kind`, `signal_text`, and `source_role` set.

**Design choice:** Pre-build the enriched provenance in `VaultSelectionReview` and pass it through `ongenerate`'s existing `neighborProvenance` parameter, extended to carry the new fields.

Specifically, in `handleAngleSelected`:
```typescript
async function handleAngleSelected(angle: MinedAngle, format: 'tweet' | 'thread') {
    // ... existing nodeIds + neighborProv setup ...

    // Enrich neighborProv with source_role
    neighborProv.forEach(np => {
        np.source_role = 'accepted_neighbor';
    });

    // Add evidence-level fields: for each evidence item in the selected angle,
    // find its matching neighbor provenance and set signal_kind + signal_text
    for (const ev of angle.evidence) {
        const match = neighborProv.find(np => np.node_id === ev.source_node_id);
        if (match) {
            match.signal_kind = ev.evidence_type;
            match.signal_text = ev.citation_text;
        }
    }

    // Set angle_kind on all provenance refs
    neighborProv.forEach(np => {
        np.angle_kind = angle.angle_type;
    });

    await ongenerate(nodeIds, format, [angle.seed_text], angle.angle_type, neighborProv);
}
```

## 6. Key Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Migration order conflicts with concurrent branches | Low | Timestamp `20260322000100` chosen to follow existing `20260321000200`. Check for newer migrations before implementing. |
| `ProvenanceLink` `FromRow` derive may fail if column order matters | Low | SQLx `FromRow` maps by name, not position. New nullable columns with defaults work fine. |
| `copy_links_for` SELECT * would break if column set changes | Medium | Already uses explicit column list. New columns must be added to both INSERT and SELECT. |
| `VaultSelectionReview` might exceed 500-line limit | Medium | Session 04 handoff noted it's at ~430 lines. Adding ~15 lines of enrichment logic keeps it under 450. |
| Evidence items may reference node_ids not in `acceptedNeighbors` map | Low | The angle mining pipeline validates evidence `source_node_id` against accepted set. Frontend can defensively skip unmatched evidence. |
| Old clients sending ProvenanceRef without new fields | None | `#[serde(default)]` + nullable DB columns ensure full backward compat. |
| `get_links_for` uses `SELECT *` | Medium | Uses `query_as::<_, ProvenanceLink>` which maps by column name. Adding columns to the struct + table is sufficient. `SELECT *` will include new columns automatically. |

## 8. What This Enables for Future Sessions

### Session 06 (Forge Data Contract)
With `source_role = 'primary_selection'`, Forge knows which vault note to target for writeback. The `angle_kind` field lets Forge record which angle type produced the best-performing content.

### Session 08 (Forge Sync Engine)
Query pattern for Forge:
```sql
SELECT DISTINCT source_path, source_role
FROM vault_provenance_links
WHERE entity_type = 'original_tweet' AND entity_id = ?
  AND source_role = 'primary_selection'
```
This resolves the exact note file to update with performance data.

### Session 10 (Instrumentation)
Analytics can aggregate by `angle_kind` to show which angle types produce the best engagement, and by `signal_kind` to show which evidence types correlate with performance.
