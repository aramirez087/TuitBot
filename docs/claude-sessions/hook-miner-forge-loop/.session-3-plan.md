### 1. `crates/tuitbot-core/src/content/angles.rs` (new module)

**Purpose:** Domain types for the angle mining system.

**Contents:**
- `AngleType` enum: `Story`, `Listicle`, `HotTake` with `serde(rename_all = "snake_case")`
- `EvidenceType` enum: `Contradiction`, `DataPoint`, `AhaMoment` with `serde(rename_all = "snake_case")`
- `EvidenceItem` struct: `evidence_type`, `citation_text` (max 120 chars), `source_node_id`, `source_note_title`, `source_heading_path: Option<String>`, `confidence: f64`
- `MinedAngle` struct: `angle_type`, `seed_text` (max 280 chars), `char_count`, `evidence: Vec<EvidenceItem>`, `confidence: String` ("high"/"medium"), `rationale`
- `AngleMiningOutput` struct: `angles: Vec<MinedAngle>`, `fallback_reason: Option<String>`, `evidence_quality_score: f64`, `usage: TokenUsage`, `model: String`, `provider: String`
- Constants: `MIN_EVIDENCE_QUALITY: f64 = 0.3`, `MIN_EVIDENCE_COUNT: usize = 2`
- Helper: `assign_angle_confidence(evidence: &[EvidenceItem]) -> String` — returns "high" if 2+ items AND avg confidence >= 0.6, else "medium"
- Implement `Display` for `AngleType` and `EvidenceType` (for logging)

**Design decisions:**
- All types derive `Debug, Clone, Serialize, Deserialize` for round-trip testing and API use.
- `confidence` is a String ("high"/"medium") not an enum — matches existing `HookOption.confidence` pattern.
- `TokenUsage` is re-exported from `crate::llm::TokenUsage`.
- Constants live here in `tuitbot-core`, not in the server layer (per contract D8).

### 2. `crates/tuitbot-core/src/content/evidence.rs` (new module)

**Purpose:** Evidence extraction engine — regex pre-filter + LLM extraction + validation.

**Contents:**

#### Regex pre-filter (`pre_filter_data_points`)
- Scan neighbor content for numeric patterns as candidate `DataPoint` evidence:
  - Percentages: `\d+(\.\d+)?%`
  - Dollar amounts: `\$[\d,]+(\.\d+)?`
  - Multipliers: `\d+(\.\d+)?x`
  - ISO dates: `\d{4}-\d{2}-\d{2}`
  - Counts: `\d+\s+(users|customers|downloads|revenue|sales|companies|teams|projects)`
- Returns `Vec<CandidateDataPoint>` with `text: String`, `source_node_id: i64`, `source_note_title: String`
- Pure function, no async, no LLM — unit-testable.

#### LLM evidence extraction (`extract_evidence`)
- Signature: `async fn extract_evidence(provider: &dyn LlmProvider, business: &BusinessProfile, topic: &str, neighbors: &[NeighborContent], candidates: &[CandidateDataPoint]) -> Result<Vec<EvidenceItem>, LlmError>`
- `NeighborContent` struct: `node_id: i64`, `note_title: String`, `heading_path: Option<String>`, `snippet: String` (up to 500 chars)
- Builds a system prompt instructing the LLM to:
  1. Read the topic and neighbor snippets
  2. Identify `Contradiction`, `DataPoint`, `AhaMoment` evidence items
  3. Confirm/reject candidate data points from the regex pre-filter
  4. Return a JSON array of evidence items
- Uses `GenerationParams { max_tokens: 500, temperature: 0.3 }` (low temp for structured output)
- Parses JSON from LLM response with `serde_json::from_str` + fallback regex extraction for robustness

#### Validation (`validate_evidence`)
- Reject items referencing `node_id`s not in the accepted set
- Truncate `citation_text` to 120 chars (not reject — truncate to preserve partial value)
- Reject items with `confidence < 0.1` (noise floor)
- Deduplicate by `(evidence_type, source_node_id)` — keep highest confidence
- Returns filtered `Vec<EvidenceItem>`

#### Prompt construction
- The evidence extraction prompt will be explicit:
  ```
  You are an evidence mining engine. Given a topic and related note snippets,
  extract evidence items that could support social media content angles.

  Topic: {topic}

  Related notes:
  [1] (node_id: {id}, title: "{title}") "{snippet}"
  [2] ...

  Candidate data points found by scanning:
  - "{text}" from "{title}"
  ...

  Extract evidence as a JSON array. Each item:
  {
    "evidence_type": "contradiction" | "data_point" | "aha_moment",
    "citation_text": "exact quote or close paraphrase, max 120 chars",
    "source_node_id": <integer from the list above>,
    "confidence": <0.0-1.0>
  }

  Rules:
  - Only reference node_ids from the list above.
  - citation_text must be grounded in the source snippet.
  - For contradictions, identify opposing claims across different notes.
  - For aha_moments, identify non-obvious connections.
  - For data_points, confirm the candidate data points are relevant to the topic.
  - Return [] if no meaningful evidence found.
  ```

**Estimated lines:** ~250

### 3. `crates/tuitbot-core/src/content/generator/angles.rs` (new submodule)

**Purpose:** Angle generation from extracted evidence, parallel to existing hook generation.

**Contents:**

#### `generate_mined_angles` function
- Signature: `pub async fn generate_mined_angles(&self, topic: &str, neighbors: &[NeighborContent], selection_context: Option<&str>) -> Result<AngleMiningOutput, LlmError>`
- This is a method on `ContentGenerator` (added via the new submodule, called from mod.rs).
- Pipeline:
  1. If `neighbors` is empty → return fallback `"no_neighbors_accepted"`
  2. Run regex pre-filter on all neighbor snippets
  3. Call `extract_evidence()` LLM call
  4. Run `validate_evidence()`
  5. Check `MIN_EVIDENCE_COUNT` — if < 2, return fallback `"insufficient_evidence"`
  6. Compute `evidence_quality_score` = average confidence
  7. Check `MIN_EVIDENCE_QUALITY` — if < 0.3, return fallback `"low_evidence_quality"`
  8. Call angle generation LLM with evidence + topic + voice/persona
  9. Parse response into `Vec<MinedAngle>`
  10. Filter out angles with empty evidence
  11. If all filtered → return fallback `"all_angles_filtered"`

#### Angle generation prompt
- System prompt pattern matches existing `generate_hooks`:
  ```
  You are {product_name}'s social media voice. {product_description}.
  {audience_section}
  {voice_section}
  {persona_section}

  Task: Generate up to 3 content angles from the evidence below.
  Each angle must be one of: story, listicle, hot_take.
  Generate exactly one angle per type IF the evidence supports it.
  Do not pad with unsupported angles.

  Evidence items:
  [1] ({evidence_type}) "{citation_text}" from "{source_note_title}" [confidence: {confidence}]
  ...

  Output format:
  ANGLE_TYPE: story
  SEED_TEXT: <opening tweet, max 280 chars>
  RATIONALE: <1 sentence why this angle works>
  EVIDENCE_IDS: 1, 3
  ---
  (repeat for each angle)
  ```

#### Angle response parser (`parse_angles_response`)
- Similar pattern to `parse_hooks_response` in `parser.rs`
- Extracts `ANGLE_TYPE:`, `SEED_TEXT:`, `RATIONALE:`, `EVIDENCE_IDS:` blocks separated by `---`
- Maps evidence IDs back to the actual `EvidenceItem` objects
- Assigns confidence per the rules in the contract (2+ evidence items + avg >= 0.6 → "high")

**Estimated lines:** ~200

### 4. `crates/tuitbot-server/src/routes/assist/angles.rs` (new module)

**Purpose:** `POST /api/assist/angles` endpoint handler.

**Contents:**

#### Request/Response DTOs
- `AssistAnglesRequest`: `topic: String`, `accepted_neighbor_ids: Vec<i64>`, `session_id: Option<String>`, `selected_node_ids: Option<Vec<i64>>`
- `AssistAnglesResponse`: `angles: Vec<MinedAngleDto>`, `fallback_reason: Option<String>`, `topic: String`, `vault_citations: Vec<VaultCitation>`
- `MinedAngleDto`: `angle_type: String`, `seed_text: String`, `char_count: usize`, `evidence: Vec<EvidenceItemDto>`, `confidence: String`, `rationale: String`
- `EvidenceItemDto`: `evidence_type: String`, `citation_text: String`, `source_node_id: i64`, `source_note_title: String`, `source_heading_path: Option<String>`

#### Handler (`assist_angles`)
- Validates: `topic` non-empty (400), `accepted_neighbor_ids` non-empty (400)
- Resolves RAG context using the same `resolve_selection_rag_context` / `resolve_composer_rag_context` pattern as `hooks.rs`
- Fetches neighbor content for `accepted_neighbor_ids` via `get_chunks_for_nodes_with_context`
- Builds `NeighborContent` structs from chunk data
- Calls `ContentGenerator::generate_mined_angles()`
- Maps `AngleMiningOutput` → `AssistAnglesResponse`
- Builds `vault_citations` from evidence items' source nodes

#### Privacy rules
- Only surfaces `citation_text` (max 120 chars), `source_note_title`, and `heading_path` — never raw note bodies
- Account scoping: all DB queries go through `account_id` from `AccountContext`
- No cross-account data leakage: neighbor content is fetched with `account_id` filter

**Estimated lines:** ~150

## Key Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D11 | Place `angles.rs` and `evidence.rs` as siblings under `content/`, not nested under `generator/` | Evidence extraction is not generation — it's a distinct pipeline stage. The generator submodule `angles.rs` handles only the generation half, keeping separation of concerns. |
| D12 | Evidence extraction uses a single LLM call, not per-neighbor calls | Reduces latency (1 call vs N). The LLM sees all neighbors at once and can identify cross-note contradictions. Token budget (2000 input) easily fits 3-8 neighbor snippets at 500 chars each. |
| D13 | Angle response parser follows the `parse_hooks_response` pattern (STYLE/HOOK prefix lines, --- separators) | Proven robust parser with fallback. Same code patterns = less cognitive overhead. New prefixes: `ANGLE_TYPE:`, `SEED_TEXT:`, `RATIONALE:`, `EVIDENCE_IDS:`. |
| D14 | Endpoint named `/api/assist/angles` (matching contract), not `/api/assist/hook-miner` (from session task description) | The contract (locked in S02) specifies `/api/assist/angles`. The session task description uses "hook-miner" as a conceptual name. Contract wins. |
| D15 | `NeighborContent` struct lives in `evidence.rs`, not in `graph_expansion.rs` | It's a projection of graph neighbor data for the extraction pipeline. The graph expansion module shouldn't know about evidence mining concerns. Simple struct, no DB dependency. |
| D16 | Truncate over-long citation_text instead of rejecting | Partial citations still provide value. The 120-char limit is for UI display; a truncated citation with "..." is better than dropping evidence entirely. |
| D17 | Use `temperature: 0.3` for evidence extraction, `0.8` for angle generation | Extraction should be precise and deterministic. Angle generation benefits from creativity (matching existing hook generation at 0.9, slightly lower since angles are evidence-constrained). |

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| LLM returns malformed JSON for evidence extraction | Medium | Wrap `serde_json::from_str` with a regex fallback that extracts JSON arrays from fenced code blocks. Log raw response on parse failure for debugging. |
| `generate_mined_angles` adds a second LLM call (extraction + generation), increasing latency | Medium | Token budgets are modest (2500 input + 1300 output total). No retry loops in V1 extraction — single attempt with validation. Contract allows 15s client timeout. |
| Existing generator mod.rs exceeds 500-line limit | Low | Pre-existing. We only add ~10 lines (delegation method + mod declaration). The angles submodule keeps new logic separate. A future session can refactor. |
| Evidence items reference node_ids the frontend doesn't know about | Low | Validation step rejects evidence with node_ids not in the `accepted_neighbor_ids` set. |
| Regex pre-filter for data points is too aggressive (false positives) | Low | Pre-filter only produces candidates. LLM confirms/rejects relevance. False positives are harmless — they just give the LLM more context. |
