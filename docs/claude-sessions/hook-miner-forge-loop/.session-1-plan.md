## Key Design Decisions

### D1: Hook Miner is an evidence-first upgrade to the existing HookPicker

The current HookPicker generates 5 generic hooks via `POST /api/assist/hooks` using `TweetFormat` styles. Hook Miner replaces the "5 generic hooks" first step with "3 mined angles" backed by evidence from the vault graph. The fallback to the existing generic hook picker is preserved for sparse notes.

**Rationale:** The session-02 plan already defines the angle taxonomy (`story`, `listicle`, `hot_take`) and evidence taxonomy (`contradiction`, `data_point`, `aha_moment`). Hook Miner must extend `generate_hooks` rather than replace it, keeping the generic path as fallback.

### D2: Forge extends the existing `tuitbot:` frontmatter loopback — not a parallel system

The current `LoopBackEntry` writes `tweet_id`, `url`, `published_at`, `type`, `status`, and optional `thread_url` to note frontmatter. Forge adds analytics fields (`impressions`, `likes`, `retweets`, `replies`, `engagement_rate`, `performance_score`, `synced_at`) and note-level summary fields to the same entries. No new database, no new sync daemon.

**Rationale:** Operator rule: "Treat Forge as an extension of the current tuitbot writeback contract, not a parallel metadata system."

### D3: Thread publish normalization is a prerequisite for Forge

The current `approval_poster.rs` routes `thread_tweet` action_type to `post_tweet()` — it posts individual tweets, not a reply chain. There is no thread-aware posting path in the approval poster. The `threads` and `thread_tweets` tables exist but are only populated by the `thread_loop` generator, not by Ghostwriter compose → approve → publish flows. This means:
- Ghostwriter-composed threads posted through approval queue don't create `thread`/`thread_tweets` records
- Provenance is propagated to `original_tweets` but not to `threads`
- Analytics can't aggregate thread child tweets because they aren't linked

**This must be fixed before Forge can aggregate thread metrics.**

### D4: Analytics has no thread-level aggregation

`tweet_performance` and `reply_performance` tables track individual tweet metrics. There's no `thread_performance` table or aggregation logic. When a thread is posted, each tweet's performance is tracked independently with no linkage. Forge needs aggregated thread metrics to write meaningful frontmatter.

### D5: Audit separates Backlink Synthesizer (done) from Hook Miner + Forge (new)

The Backlink Synthesizer work is fully shipped:
- `note_edges` and `note_tags` tables (migration 20260321)
- `graph_expansion.rs`: `expand_graph_neighbors()`, `GraphNeighbor`, `SuggestionReason`, `SuggestionIntent`
- `GraphSuggestionCards.svelte`: accept/dismiss/restore UX
- `VaultSelectionReview.svelte`: synthesis toggle, slot targeting
- Provenance `edge_type` and `edge_label` columns

Hook Miner builds ON TOP of this graph by using accepted neighbors as evidence sources for angle generation. Forge builds ON TOP of the existing loopback by adding analytics enrichment.

### Step 3: Write `epic-charter.md`

**Mission:** Turn TuitBot's Ghostwriter from "generic hooks" to "evidence-mined angles with closed-loop analytics" by (a) replacing the first compose step with Hook Miner's evidence-backed angle cards, and (b) extending the existing publish loopback into Forge — an opt-in analytics enrichment layer that writes performance data back to source notes.

**Success Metrics:**
1. Angle-to-draft conversion ≥ 50% (mined angles that produce a draft)
2. Fallback-to-generic rate ≤ 30% (sparse notes that fall back to generic hooks)
3. Forge sync success rate ≥ 95% (eligible notes that update without error)
4. Thread outcome completeness: 100% of Ghostwriter threads persist root + child IDs

**Non-Goals:**
- Multi-hop graph expansion (stay at 1-hop per Backlink Synthesizer decision)
- Real-time analytics streaming (Forge syncs on a periodic schedule, not live)
- Engagement prediction or scoring in Hook Miner (evidence display only, no ML)
- New parallel metadata system (Forge extends `tuitbot:` frontmatter, not a sidecar DB)
- Redesigning the Obsidian plugin (selection ingress is complete)

**Rollout Stance:**
- Hook Miner: additive to existing HookPicker, behind angle generation flag
- Forge: opt-in via `analytics_sync_enabled` setting, off by default
- Thread normalization: transparent fix, no user-facing toggle
