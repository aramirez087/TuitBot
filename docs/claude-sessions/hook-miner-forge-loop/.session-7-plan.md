## Design Decisions

### D43: Thread content stored as single approval item, parsed at post time

The approval poster receives `action_type = "thread"` with `generated_content` containing serialized block JSON (or legacy string array). At post time, the poster must:
1. Parse the content into individual tweets
2. Post sequentially as reply chain (root standalone, children reply to previous)
3. Collect all tweet IDs

**Rationale:** This matches how `post_scheduled_thread` and `try_post_thread_now` already work. No schema change to `approval_queue`.

### D44: Thread record creation happens after all tweets are posted

The poster creates `threads` and `thread_tweets` records only after the entire chain (or partial chain) is posted. This avoids orphan thread records for failed posts.

**Rationale:** Matches existing transactional pattern in `insert_thread_tweets_for()`.

### D45: Root tweet ID is canonical key for both `original_tweets` and `threads`

After posting:
- One `original_tweets` row with `tweet_id = root_tweet_id` (for analytics/topic compatibility)
- One `threads` row with `root_tweet_id = root_tweet_id`
- N `thread_tweets` rows with each tweet's ID and position

**Rationale:** Existing topic and analytics queries use `original_tweets.tweet_id`. Root tweet ID as canonical key aligns with forge-thread-contract.md.

### D46: `LoopBackEntry` gains `child_tweet_ids: Option<Vec<String>>`

Only this one new field is added in Session 7. The 7 analytics fields (`impressions`, `likes`, etc.) are deferred to Session 8 (Forge sync implementation).

**Rationale:** Minimal change surface. Session 7 is about thread normalization, not analytics. Adding all fields now would require changing `PartialEq, Eq` derives prematurely and touching code paths that don't need it yet.

### D47: Provenance copies to both `original_tweet` AND `thread` entity types

After posting a thread, provenance links are copied from `approval_queue` to:
1. `original_tweet` (the root OT record) — preserves existing analytics/Forge writeback path
2. `thread` (the new thread record) — enables future Forge thread aggregation queries

**Rationale:** Additive, not replacement. Existing code that queries by `entity_type = 'original_tweet'` continues to work.

### D48: Partial thread failure creates records for posted tweets only

If a 4-tweet thread fails at tweet 3, the poster:
- Creates a `threads` row with `status = "partial"`, `tweet_count = 2`
- Creates 2 `thread_tweets` rows (positions 0 and 1)
- Creates 1 `original_tweets` row for the root tweet
- Writes loopback with `child_tweet_ids` containing only the successfully posted child

**Rationale:** Matches forge-thread-contract.md edge case: "Only successfully posted children appear in `child_tweet_ids`."

## Implementation Order

### Step 1: Extend `LoopBackEntry` with `child_tweet_ids`

**File:** `crates/tuitbot-core/src/automation/watchtower/loopback.rs`

1. Add field to `LoopBackEntry`:
   ```rust
   #[serde(default, skip_serializing_if = "Option::is_none")]
   pub child_tweet_ids: Option<Vec<String>>,
   ```

2. The `Eq` derive stays for now — `Vec<String>` implements `Eq`. It will change to `PartialEq` only when `f64` analytics fields are added in Session 8.

3. Update `execute_loopback()` signature to accept optional thread metadata:
   ```rust
   pub async fn execute_loopback(
       pool: &DbPool,
       node_id: i64,
       tweet_id: &str,
       url: &str,
       content_type: &str,
   ) -> LoopBackResult
   ```
   No signature change needed — the function constructs `LoopBackEntry` internally. Instead, add a new function:
   ```rust
   pub async fn execute_loopback_thread(
       pool: &DbPool,
       node_id: i64,
       root_tweet_id: &str,
       url: &str,
       child_tweet_ids: Vec<String>,
   ) -> LoopBackResult
   ```
   This builds the entry with `content_type = "thread"`, `thread_url = Some(url)`, and `child_tweet_ids = Some(child_tweet_ids)`.

4. Update `sample_entry()` in tests to include `child_tweet_ids: None`.

5. Add test for thread entry serialization with `child_tweet_ids`.

6. Add test for thread entry idempotency (same root_tweet_id detected).

### Step 2: Add thread query helper to storage

**File:** `crates/tuitbot-core/src/storage/threads.rs`

Add a query function for Forge sync fallback (documented in forge-thread-contract.md):

```rust
pub async fn get_thread_tweet_ids_by_root_for(
    pool: &DbPool,
    account_id: &str,
    root_tweet_id: &str,
) -> Result<Vec<String>, StorageError>
```

Query:
```sql
SELECT tt.tweet_id FROM thread_tweets tt
JOIN threads t ON tt.thread_id = t.id
WHERE t.account_id = ? AND t.root_tweet_id = ?
ORDER BY tt.position ASC
```

This excludes the root (position 0 has `tweet_id` = root's tweet_id, but the forge contract says `child_tweet_ids` excludes root). Filter by `position > 0` in the query.

Add tests for this query.

### Step 3: Thread-aware approval poster

**File:** `crates/tuitbot-core/src/automation/approval_poster.rs`

This is the main changeset. The file is currently 677 lines (within 500 limit after tests are excluded, but tests bring it over). The approach:

#### 3a: Add a `post_thread` function

```rust
async fn post_thread(
    x_client: &dyn XApiClient,
    content: &str,
) -> Result<Vec<String>, String>
```

- Parse content using `deserialize_blocks_from_content()` (block JSON) or `serde_json::from_str::<Vec<String>>()` (legacy)
- Post first tweet standalone via `post_tweet()`
- Post subsequent tweets as replies to previous via `post_reply()`
- Return all successfully posted tweet IDs
- On failure, return partial results in the error message (or use a result enum)

Use a result type for partial failures:
```rust
struct ThreadPostResult {
    posted_ids: Vec<String>,
    status: &'static str, // "sent" or "partial"
    error: Option<String>,
}
```

#### 3b: Update the main match block

Change the `match item.action_type.as_str()` at line 76:

```rust
"thread" => {
    post_thread_and_persist(
        &pool, &*x_client, &account_id, &item, &media_ids
    ).await
}
"reply" if !item.target_tweet_id.is_empty() => { ... }
_ => { post_tweet(...) }
```

#### 3c: Add `post_thread_and_persist` orchestrator

```rust
async fn post_thread_and_persist(
    pool: &DbPool,
    x_client: &dyn XApiClient,
    account_id: &str,
    item: &ApprovalItem,
    media_ids: &[String],
) -> Result<String, String>
```

This function:
1. Parses thread content from `item.generated_content`
2. Posts tweets as reply chain, collecting `Vec<String>` of tweet IDs
3. Determines status: `"sent"` (all posted) or `"partial"` (some failed)
4. Creates `threads` row with root_tweet_id, tweet_count, status
5. Creates `thread_tweets` rows for each posted tweet
6. Creates `original_tweets` row for the root tweet (for analytics compatibility)
7. Returns the root_tweet_id as the canonical ID

The return value is `Ok(root_tweet_id)` which plugs into the existing `mark_posted_for` and `propagate_provenance` calls.

#### 3d: Update `propagate_provenance` for thread entity

After creating the thread record, copy provenance links to both:
- `"original_tweet"` entity (existing behavior, using the OT row ID)
- `"thread"` entity (new, using the thread row ID)

#### 3e: Update `execute_loopback_for_provenance` for threads

When `item.action_type == "thread"`, call `execute_loopback_thread` instead of `execute_loopback`, passing `child_tweet_ids`.

Store the child_tweet_ids on the item's processing context (either as a field on a local struct or by querying `thread_tweets` after insertion).

#### 3f: File size management

The file is currently 677 lines. Adding ~100 lines of thread logic will push it to ~780. Since tests account for ~300+ lines, the code portion stays under 500. However, if it exceeds 500 total, extract thread posting into a submodule: `approval_poster/thread_posting.rs`. Monitor and split if needed.

### Step 4: Thread persistence in direct compose path

**File:** `crates/tuitbot-server/src/routes/content/compose/transforms.rs`

The `try_post_thread_now` function at line 474 already posts as reply chains correctly, but doesn't persist `threads`/`thread_tweets`/`original_tweets` records.

Add persistence after successful posting:
1. After collecting all `tweet_ids`, create a `threads` row
2. Create `thread_tweets` rows
3. Create `original_tweets` row for root tweet
4. If provenance refs exist on the request, insert provenance links for both `original_tweet` and `thread` entities
5. Execute loopback for thread (with `child_tweet_ids`)

**Note:** This path doesn't go through the approval poster, so it needs its own persistence logic. Extract shared thread persistence into a helper function in `tuitbot-core` to avoid duplication:

```rust
// In crates/tuitbot-core/src/storage/threads.rs or a new threads/persist.rs
pub async fn persist_thread_records(
    pool: &DbPool,
    account_id: &str,
    topic: &str,
    tweet_ids: &[String],  // root at index 0, children at 1..N
    tweet_contents: &[String],
    status: &str,
) -> Result<(i64, i64), StorageError>  // (thread_id, original_tweet_id)
```

This creates the `threads` row, `thread_tweets` rows, and `original_tweets` row atomically.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Thread content parsing diverges between approval poster and compose path | Medium | Extract shared `parse_thread_content()` function into `tuitbot-core::content` |
| File size limit exceeded in `approval_poster.rs` | Low | Monitor line count; extract thread module if > 500 code lines |
| `content_loop/publisher.rs` scheduled thread path not normalized | Medium | Document as known limitation. That path uses `ThreadPoster` trait abstraction. Normalization requires a follow-up after Session 7 or as part of Session 8. |
| `thread_tweets` position 0 = root tweet; query for child IDs must exclude position 0 | Low | Use `WHERE position > 0` in `get_thread_tweet_ids_by_root_for()` |
| Partial thread failure leaves orphan tweets on X | Low | Already the case in `try_post_thread_now`. Document: cannot undo posted tweets. |
| `mark_posted_for` expects a single tweet_id but threads have multiple | Low | Pass root_tweet_id. The `posted_tweet_id` column on `approval_queue` stores the root ID. |
| Concurrent approval poster instances could double-post | Low | Existing mitigation: `get_next_approved_for` uses `LIMIT 1` with status check. Thread posting is slower but still atomic per item. |
