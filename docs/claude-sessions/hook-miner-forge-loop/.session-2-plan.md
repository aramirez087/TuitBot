## Key Design Decisions

### D6: Angle Taxonomy — 3 exact categories

The current hook generation uses 5 style-based categories (`question`, `contrarian_take`, `tip`, `stat`, `teaser`). Hook Miner replaces these with 3 evidence-based angle types:

| Enum variant | Definition | When to use | Example seed text |
|---|---|---|---|
| `story` | A narrative angle rooted in a personal experience, anecdote, or timeline drawn from the source notes. | When evidence includes a sequence of events, a before/after, or a lived experience. | "I spent 3 months migrating from X to Y. Here's what nobody tells you about..." |
| `listicle` | A structured angle that surfaces multiple discrete takeaways from the evidence. | When evidence yields 2+ distinct data points, tips, or observations. | "3 things I learned about caching after reading [note title]:" |
| `hot_take` | A provocative angle built on a contradiction, counterintuitive finding, or strong opinion supported by evidence. | When evidence contains a contradiction, a data point that defies conventional wisdom, or a polarizing claim. | "Everyone says you should X. But [data point from note] suggests the opposite." |

**Rationale:** These three cover the main content archetypes on X (narrative, list, opinion) while being evidence-grounded rather than stylistic. Three is the right number because it forces ranking by evidence strength, not combinatorial explosion.

### D7: Evidence Taxonomy — 3 exact categories

Each angle card cites 1–3 evidence items extracted from accepted graph neighbors:

| Enum variant | Definition | Extraction rule |
|---|---|---|
| `contradiction` | Two claims in the evidence set that conflict or create tension. | LLM identifies opposing statements across ≥2 neighbor snippets, or within the selection + a neighbor. |
| `data_point` | A specific number, percentage, metric, date, or measurable fact. | Regex pre-filter for numeric patterns + LLM confirmation of relevance to the topic. |
| `aha_moment` | A non-obvious insight, connection, or reframe that the user's notes surface. | LLM identifies a surprising implication or novel synthesis across neighbor content. |

**Rationale:** These three map to the most common reasons a tweet goes viral (conflict, specificity, surprise). They are extraction-oriented — the LLM is told what to look for, not asked to generate creative takes from nothing.

### D8: Fallback Threshold

The system falls back to the existing generic hook picker when:

1. **No accepted neighbors.** The user dismissed all graph suggestions or the graph had no neighbors (sparse vault).
2. **Evidence quality score < 0.3.** The LLM-assigned confidence that meaningful evidence was extracted from the accepted neighbors is below threshold.
3. **Fewer than 2 evidence items extracted across all neighbors.** Even if quality is acceptable, a single evidence item doesn't give enough material for 3 differentiated angles.

The threshold constant is `MIN_EVIDENCE_QUALITY = 0.3` and `MIN_EVIDENCE_COUNT = 2`. These live in the extraction engine, not the API layer.

**Fallback is non-destructive:** accepted neighbors, synthesis toggle state, and selection context are preserved. The user sees explicit messaging explaining why they got generic hooks and can return to the angle flow by accepting more neighbors.

### D9: Related-Note Influence

Accepted graph neighbors influence angle generation as evidence sources but **never auto-insert content into the draft**:

1. When the user clicks "Include" on a GraphSuggestionCard, that neighbor's `node_id`, `snippet`, `heading_path`, and `best_chunk_id` are added to the angle generation context.
2. The angle generation LLM receives these as `evidence_sources` alongside the selection text.
3. The generated angle cards reference specific evidence from specific neighbors (with attribution).
4. When the user selects an angle and proceeds to draft generation, the angle text (including its evidence framing) becomes the hook/highlight. The neighbor content is available as RAG context but the draft generator decides what to weave in based on the hook.
5. No neighbor text is ever pasted verbatim into the draft. The user's voice/persona settings govern the draft's tone.

**Rationale:** Auto-insertion would break the Ghostwriter's voice consistency and surprise users with content they didn't write. The influence is directional (evidence → angle → hook → draft) not insertional.

### Step 2: Write `hook-miner-ux.md`

This is the primary deliverable. It must contain:

1. **Flow overview** — How angle mining replaces the first step of the compose flow (from VaultFooter "Generate" → angle cards instead of hook cards). The existing HookPicker remains as a separate component for the fallback path.

2. **Angle card anatomy** — Exact card layout spec:
   - Top row: Angle type pill (`STORY` / `LISTICLE` / `HOT TAKE`) — same visual treatment as current `hook-style-pill`
   - Body: Seed text (the angle's opening line, max 120 chars, generated by LLM)
   - Evidence section: 1–3 evidence badges, each showing:
     - Evidence type pill (`contradiction` / `data_point` / `aha_moment`)
     - Truncated citation text (40 chars max, from the source neighbor's snippet)
     - Source attribution: `from "Note Title"` in `--color-text-subtle`
   - Footer: Confidence badge (reuse existing `getConfidenceBadge()` pattern) + char count
   - Selection: Same click-to-select pattern as HookPicker cards

3. **Loading states** — 3 shimmer cards (matching current `hook-card-shimmer` pattern) with the label "Mining angles from your notes..."

4. **Sparse-note / weak-signal fallback** — When evidence threshold is not met:
   - Show a non-alarming state with heading "Not enough signal"
   - Body text explaining the situation
   - Two actions: "Use generic hooks" (opens HookPicker) and "← Back to related notes" (returns to GraphSuggestionCards so user can accept more neighbors)
   - Preserves all state: accepted neighbors, synthesis toggle, selection

5. **Error state** — Same pattern as current HookPicker error: red banner with message + "Retry" button

6. **Recovery states:**
   - API timeout → "Mining took too long. Try again or use generic hooks."
   - LLM returned malformed angles → "Couldn't parse mined angles. Try again or use generic hooks."
   - All angles filtered (0 evidence each) → treated as fallback state

7. **How accepted neighbors flow into angles:**
   - Sequence: Selection review → accept/dismiss neighbors → "Generate" → call `POST /api/assist/angles` → show angle cards or fallback
   - The "Generate" button on VaultFooter calls angles endpoint (not hooks) when ≥1 neighbor is accepted and synthesis is enabled
   - When 0 neighbors accepted or synthesis off, Generate calls the existing hooks endpoint directly

8. **Format toggle** — Preserved from HookPicker. Tweet/Thread toggle on angle cards works identically.

9. **Non-goals for V1:**
   - No drag-to-reorder angles
   - No inline evidence editing
   - No "explain this evidence" expansion
   - No multi-select (pick one angle, not combine angles)
   - No angle history / bookmarking
   - No evidence quality override by user

5. **Fallback logic (pseudocode):**

```
if accepted_neighbor_ids.is_empty():
    return early with fallback_reason = "no_neighbors_accepted"

evidence_items = extract_evidence(neighbors, selection_text)

if evidence_items.len() < MIN_EVIDENCE_COUNT:
    return with fallback_reason = "insufficient_evidence"

quality_score = evidence_items.avg(|e| e.confidence)
if quality_score < MIN_EVIDENCE_QUALITY:
    return with fallback_reason = "low_evidence_quality"

angles = generate_angles(topic, evidence_items, voice, persona)
filter out angles with 0 evidence
return angles (may be 0 → client treats as fallback)
```

6. **Frontend routing logic (in VaultSelectionReview):**

```
on "Generate" click:
    if synthesisEnabled AND acceptedNeighbors.size > 0:
        call POST /api/assist/angles with accepted neighbor IDs
        if response.fallback_reason is non-null:
            show fallback state (NOT HookPicker immediately)
            user can click "Use generic hooks" → call POST /api/assist/hooks (existing)
            user can click "← Back" → return to neighbor cards
        else:
            show AngleCards
    else:
        call POST /api/assist/hooks (existing path, unchanged)
```

7. **How selected angle feeds into draft generation:**

When user selects an angle and clicks "Use this angle":
- The angle's `seed_text` is passed as the `highlights` parameter in `ongenerate()`
- The angle's `angle_type` is passed as the `hookStyle` parameter (replacing generic styles like "question")
- The accepted neighbors' `node_id`s remain in the `nodeIds` parameter (for RAG context)
- The angle's evidence citations are passed as `neighborProvenance` (for provenance tracking)
- Draft generation proceeds through the existing `ongenerate → api.assist.tweet/thread` pipeline

This means the existing compose → approve → publish pipeline is untouched. Only the first step (hook selection) is upgraded.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Evidence extraction quality varies by vault density | Medium | Fallback contract ensures sparse vaults degrade gracefully. `MIN_EVIDENCE_COUNT = 2` prevents single-evidence angles. |
| 3 angles may feel limited vs. 5 hooks | Low | 3 evidence-backed options > 5 generic options. User can always fall back to generic hooks if needed. |
| `hot_take` angle type could produce inflammatory content | Medium | The business profile's `voice` and `persona` sections constrain tone. The `hot_take` prompt explicitly says "supported by evidence" — not "be controversial for engagement." |
| Fallback flow has an extra click vs. current flow | Low | Acceptable trade-off: the fallback state is informative (tells user why) and the "Use generic hooks" button is a single click. |
| AngleType/EvidenceType enums may need expansion later | Low | Enums are `#[non_exhaustive]` candidates but V1 ships with exactly 3+3. Expansion is a future session concern. |
