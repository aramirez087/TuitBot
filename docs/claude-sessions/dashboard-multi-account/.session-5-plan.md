# Session 5 — Implementation Plan: Dashboard Shell & Account Switching

## Mission

Build the dashboard shell that boots, persists, and switches active account context without stale data leaking across routes.

## Design Decisions

### DD1: WsEvent gets a top-level `account_id` field (wrapper, not per-variant)

Instead of adding `account_id` to every WsEvent variant (11 variants, invasive change), introduce a wrapper struct:

```rust
#[derive(Clone, Debug, Serialize, Deserialize)]
pub struct AccountWsEvent {
    pub account_id: String,
    #[serde(flatten)]
    pub event: WsEvent,
}
```

The broadcast channel type changes from `broadcast::Sender<WsEvent>` to `broadcast::Sender<AccountWsEvent>`. All send sites wrap their event. The frontend receives JSON like `{ "account_id": "...", "type": "ApprovalQueued", ... }`.

**Rationale:** Flat serde means zero breaking change to the JSON shape — just one new field. Per-variant `account_id` fields are redundant and error-prone. The wrapper keeps WsEvent clean for testing.

### DD2: Client-side WebSocket filtering (per charter D4)

The `websocket.ts` store filters incoming events by `account_id` against `currentAccountId`. Events without `account_id` (e.g., Error) pass through. This is simpler than server-side channel-per-account and adequate for low event volume.

### DD3: Account switch triggers store reset + refetch

`switchAccount()` becomes the orchestration point. When called:
1. Update `currentAccountId` store + localStorage + HTTP client header
2. Clear the WebSocket `events` buffer (stale events from prior account)
3. Emit a `accountSwitched` custom event on `window`
4. Pages listen to this event via a reactive subscription and refetch their data

This avoids coupling the accounts store to every page store. Each page is responsible for its own refetch.

### DD4: Bootstrap validation against `/api/accounts`

On boot, `initAccounts()` calls `fetchAccounts()`, then validates the persisted `currentAccountId` against the fetched list. If the persisted ID is not in the list (deleted account, stale localStorage), fall back to the default account or the first active account.

### DD5: AccountSwitcher always visible (not just >1 account)

Per charter D5, the AccountSwitcher shows for all account counts:
- **0 accounts** (shouldn't happen — default always exists): Show "Add Account"
- **1 account**: Show current account identity + "Add Account" button
- **>1 accounts**: Full switcher dropdown + "Add Account"

### DD6: Scope of this session — NO credential linking UI

This session focuses on the switching shell: boot validation, switch-driven invalidation, sidebar/shell UX, and WebSocket scoping. The credential linking UI (Add Account flow → OAuth/scraper session import) is deferred to a follow-up since it requires more UI components and can be layered on top of the switching foundation.

### Phase 4: Frontend — Switch-driven store invalidation

**Step 4.2: Update data-loading pages to refetch on switch**

Each page that fetches account-scoped data on mount must also refetch when the account switches. The pattern:

```typescript
onMount(() => {
    loadPageData();
    const handler = () => loadPageData();
    window.addEventListener('tuitbot:account-switched', handler);
    return () => window.removeEventListener('tuitbot:account-switched', handler);
});
```

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Circular dependency: `websocket.ts` imports `accounts.ts` | Medium | Use `get(currentAccountId)` at call time, not module-level subscription. Or pass account_id as parameter to filtering function. |
| `broadcast::Sender` type change breaks compilation | Low | All send/subscribe sites are in the server crate. Systematic find-and-replace. Compiler will catch all mismatches. |
| Flash of stale data between clear and refetch | Medium | Each page clears stores to initial values immediately in the switch handler, before the async fetch resolves. Show loading skeletons if stores are null. |
| Page-level event listeners leak if not cleaned up | Low | Use `onDestroy` cleanup pattern consistently. Svelte's `onMount` return value handles cleanup. |
| AccountSwitcher dropdown z-index conflicts | Low | Existing z-index (100) should be sufficient. Test in both collapsed and expanded sidebar states. |
| `initAccounts` failure strands the UI | Medium | Catch errors in `initAccounts`, fall back to default account, set `bootstrapState = 'ready'` anyway so the app renders. |

### Frontend

```bash
npm --prefix dashboard run check
npm --prefix dashboard run build
```

Manual verification scenarios:
1. **Boot with valid persisted ID**: App loads normally, account shown in sidebar
2. **Boot with stale/deleted persisted ID**: Falls back to default, localStorage updated
3. **Boot with empty localStorage**: Uses default account
4. **Switch accounts**: All page data refreshes, no stale content visible
5. **WebSocket events filtered**: Events for non-active account don't trigger store updates
6. **Single account**: AccountSwitcher shows identity + "Add Account"
7. **Multiple accounts**: Full dropdown works, switching updates sidebar identity
8. **Collapsed sidebar**: Account avatar visible, switcher accessible
