# Session 4 Implementation Plan: Runtime & Service Isolation

## Objective

Make runtime start/stop/status, AI assist, discovery generators, and watchtower consume account-specific effective config and isolated credentials instead of default-account singletons. A second account must be able to run with different config/credentials without mutating the default account's behavior.

---

## Current State Analysis

### What Session 03 Delivered
- `read_effective_config(state, account_id)` in `routes/content/mod.rs` ŌĆö merges base config with DB overrides
- `can_post_for(state, account_id)` / `require_post_capable(state, account_id)` ŌĆö per-account credential checks
- Account-scoped token paths via `account_token_path()` / `account_scraper_session_path()`
- `content_generators: Mutex<HashMap<String, Arc<ContentGenerator>>>` in AppState (keyed by account_id)
- `runtimes: Mutex<HashMap<String, Runtime>>` in AppState (keyed by account_id)
- X auth linking endpoints per account

### Remaining Gaps (This Session)
1. **Content generator lazy-init**: Only default account's generator is created at startup (`main.rs:160-163`). Non-default accounts get "LLM not configured" from `get_generator()` in both `assist.rs` and `discovery.rs`.
2. **`resolve_composer_rag_context`** (`assist.rs:44-79`): Loads config via `Config::load()` instead of `read_effective_config`, ignoring account overrides.
3. **`get_mode`** (`assist.rs:294-309`): Same problem ŌĆö loads global config, ignores account.
4. **Runtime `start`**: Creates empty `Runtime::new()` with no loops ŌĆö comment says "full loop setup requires X API client and LLM provider." No account-specific config is loaded.
5. **Runtime `status`**: Returns global `state.provider_backend` instead of the account's effective provider backend.
6. **WatchtowerLoop**: Single global instance using base `content_sources` config. Not account-scoped. Charter rates this S5 (MEDIUM) ŌĆö content sources attribution.
7. **Default-account-only `provider_backend` field** on AppState (`state.rs:70`): Used by `status` endpoint but is a global singleton.

## Design Decisions

### D1: Lazy Content Generator Per Account
Add a helper method `AppState::get_or_create_content_generator(account_id)` that:
1. Checks `content_generators` HashMap for existing entry ŌåÆ return clone
2. Loads effective config for the account
3. Creates LLM provider from config
4. Constructs `ContentGenerator` with account's `business` profile
5. Inserts into HashMap and returns

**Rationale**: Mirrors the existing `get_x_access_token` lazy-init pattern for TokenManagers. Avoids eagerly creating generators for all accounts at startup.

### D2: Runtime Start Loads Account Config
The `start` endpoint should load effective config for the account and pass the resolved `Config` to any spawned loops. Currently it creates an empty Runtime. This session will:
- Load effective config on start
- Return the effective `provider_backend` and `mode` in the start response
- Store the account's resolved provider_backend alongside the Runtime (or just compute it on-the-fly for status)

**Not in scope**: Actually spawning automation loops (discovery, content, mentions, etc.) ŌĆö that requires wiring up XApiClient, LLM, and all the port trait implementations in the server crate. The server currently cannot do this (see the comment at `runtime.rs:45`). This session focuses on making the config and credential resolution account-aware so that when loops are eventually spawned, they use the right config.

### D3: Watchtower Stays Global (This Session)
The charter rates watchtower scoping as S5 (MEDIUM) and schedules it for Session 6. Content sources in `config.toml` are instance-level infrastructure. This session will **not** scope watchtower per-account, but will document the future path in the plan.

### D4: Remove `provider_backend` Global From AppState
The `provider_backend: String` field on AppState is a default-account singleton. Replace its usage in `runtime::status` with per-account effective config lookup (already done for `can_post`). The field can be removed from AppState entirely since no other code path uses it after Session 03.

## Detailed Code Sketches

### `AppState::get_or_create_content_generator`

```rust
impl AppState {
    pub async fn get_or_create_content_generator(
        &self,
        account_id: &str,
    ) -> Result<Arc<ContentGenerator>, String> {
        // Fast path: already cached
        {
            let generators = self.content_generators.lock().await;
            if let Some(gen) = generators.get(account_id) {
                return Ok(gen.clone());
            }
        }

        // Load effective config for the account
        let config = self.load_effective_config(account_id).await
            .map_err(|e| format!("failed to load config: {e}"))?;

        let provider = create_provider(&config.llm)
            .map_err(|e| format!("LLM not configured: {e}"))?;

        let gen = Arc::new(ContentGenerator::new(provider, config.business));

        self.content_generators
            .lock()
            .await
            .insert(account_id.to_string(), gen.clone());

        Ok(gen)
    }

    /// Load effective config for an account (internal helper).
    async fn load_effective_config(&self, account_id: &str) -> Result<Config, String> {
        let contents = std::fs::read_to_string(&self.config_path).unwrap_or_default();
        let base: Config = toml::from_str(&contents).unwrap_or_default();

        if account_id == DEFAULT_ACCOUNT_ID {
            return Ok(base);
        }

        let account = accounts::get_account(&self.db, account_id)
            .await
            .map_err(|e| e.to_string())?
            .ok_or_else(|| format!("account not found: {account_id}"))?;

        effective_config(&base, &account.config_overrides)
            .map(|r| r.config)
            .map_err(|e| e.to_string())
    }
}
```

**Note:** This duplicates the logic from `routes::content::read_effective_config`. Consider extracting to a shared helper on AppState to avoid duplication. The `routes::content` version returns `ApiError`; the AppState version returns `String` for flexibility. Alternatively, make `read_effective_config` public and have the AppState method delegate to it (but it needs `&AppState` not `Arc<AppState>`).

**Decision:** Add `load_effective_config` as a method on AppState (it already has `config_path` and `db`). Then `routes::content::read_effective_config` can delegate to it, reducing duplication. Both callers map the error to their own type.

### `runtime::status` updated

```rust
pub async fn status(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
) -> Result<Json<Value>, ApiError> {
    let runtimes = state.runtimes.lock().await;
    let runtime = runtimes.get(&ctx.account_id);
    let running = runtime.is_some();
    let task_count = runtime.map_or(0, |r| r.task_count());
    let capabilities = state.deployment_mode.capabilities();

    let can_post = crate::routes::content::can_post_for(&state, &ctx.account_id).await;

    // Load provider_backend from effective config (not global singleton)
    let provider_backend = match state.load_effective_config(&ctx.account_id).await {
        Ok(config) => config.x_api.provider_backend,
        Err(_) => String::new(),
    };

    Ok(Json(json!({
        "running": running,
        "task_count": task_count,
        "deployment_mode": state.deployment_mode,
        "capabilities": capabilities,
        "provider_backend": provider_backend,
        "can_post": can_post,
    })))
}
```

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Removing `provider_backend` from AppState breaks other code | Low | Grep confirms only `main.rs` and `runtime.rs` use it. Cargo check catches any missed references. |
| Lazy content generator creates stale generators after config change | Medium | Same issue as TokenManager cache. Accept for now; config reload (Session 03 open issue #4) will address both. |
| `load_effective_config` on AppState duplicates `read_effective_config` in routes | Low | Refactor routes::content to delegate to AppState method. Single source of truth. |
| Test flakiness from concurrent HashMap access | Low | All access is behind `Mutex`. Tests are single-threaded per test. |
| Non-default account with no LLM config gets confusing error from assist | Low | Error message already says "LLM not configured." Lazy init just moves the check from startup to first use. |
