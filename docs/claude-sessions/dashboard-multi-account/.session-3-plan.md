## Task 1: Account-Aware `can_post` and Config Resolution Helpers

### Problem

Multiple code paths determine `can_post` by checking `state.provider_backend` (a global string) and `state.data_dir.join("scraper_session.json")` (global file). These need to resolve per-account.

### Locations to Fix

| File | Line(s) | Current Singleton | Fix |
|------|---------|-------------------|-----|
| `routes/runtime.rs` | 32-36 | `state.provider_backend`, `state.data_dir.join("scraper_session.json")` | Load effective config for `ctx.account_id`, check account-specific scraper session path |
| `routes/content/mod.rs` | 49-63 (`require_post_capable`) | `read_config(state)` reads global `config.toml`, checks global scraper path | Accept `&AccountContext`, load effective config for account, check account-specific paths |
| `routes/content/mod.rs` | 40-42 (`read_approval_mode`) | Same global config read | Accept `&AccountContext`, use effective config |
| `routes/content/mod.rs` | 67-71 (`read_config`) | Reads global `config.toml` only | Add `read_effective_config(state, account_id)` that merges overrides for non-default accounts |
| `routes/content/compose.rs` | 413-418, 454-459 | Global `can_post` + global `data_dir` for `LocalModeXClient::with_session` | Use account-scoped paths |
| `routes/discovery.rs` | 175-183 (`queue_reply`) | Global `can_post` check | Use `require_post_capable` with account context |

### Design

Add a shared helper in `routes/content/mod.rs`:

```rust
/// Load the effective config for the given account.
///
/// Default account: reads config.toml directly (backward compat).
/// Non-default: merges config.toml base with account's config_overrides from DB.
fn read_effective_config_for(state: &AppState, account_id: &str) -> Result<Config, ApiError>
```

This helper:
1. Reads `config.toml` as the base.
2. If `account_id == DEFAULT_ACCOUNT_ID`, returns it directly (existing behavior).
3. Otherwise, fetches the account's `config_overrides` from DB (blocking query via `tokio::task::block_in_place` or cached), calls `effective_config()`.

**Decision: Sync vs Async.** The existing `read_config()` is sync (called from sync helper functions). Converting all callers to async would be a large refactor. Instead, use `state.db` with `block_in_place` + `tokio::runtime::Handle::current().block_on()` only for non-default accounts. For the default account, the fast path remains sync (file read only, no DB).

**Alternative (preferred):** Make the helpers async and update compose/discovery callers. The callers are all already async handler functions — the `read_config` and `read_approval_mode` helpers are sync only by convention, not necessity. Converting them to async is straightforward since they're internal helpers.

**Decision: Go async.** Rename `read_config` → async `read_effective_config`, update all callers. This is cleaner than `block_in_place`.

### Updated Signatures

```rust
// routes/content/mod.rs
async fn read_effective_config(state: &AppState, account_id: &str) -> Result<Config, ApiError>;
async fn read_approval_mode(state: &AppState, account_id: &str) -> Result<bool, ApiError>;
async fn require_post_capable(state: &AppState, account_id: &str) -> Result<(), ApiError>;
```

`require_post_capable` changes:
```rust
async fn require_post_capable(state: &AppState, account_id: &str) -> Result<(), ApiError> {
    let config = read_effective_config(state, account_id).await?;
    let can_post = match config.x_api.provider_backend.as_str() {
        "x_api" => {
            let token_path = account_token_path(&state.data_dir, account_id);
            token_path.exists()
        }
        "scraper" => {
            let session_path = account_scraper_session_path(&state.data_dir, account_id);
            session_path.exists()
        }
        _ => false,
    };
    if !can_post { return Err(ApiError::BadRequest(...)); }
    Ok(())
}
```

## Task 2: Account-Aware Posting Paths

### Problem

`try_post_now` in compose.rs (line 446) uses the global `state.data_dir` for `LocalModeXClient::with_session` and resolves `token_path` per account only for `x_api` backend. The scraper backend needs the same account-scoped path.

### Fix

In `try_post_now`:
```rust
"scraper" => {
    // Use account-scoped data dir for cookie-auth.
    let account_data = account_data_dir(&state.data_dir, &ctx.account_id);
    let client = LocalModeXClient::with_session(
        config.x_api.scraper_allow_mutations,
        &account_data,  // was &state.data_dir
    );
    // ...
}
```

Wait — `LocalModeXClient::with_session` looks for `scraper_session.json` in the given directory. For default account, `account_data_dir` returns `state.data_dir` (backward compat). For non-default, it returns `data_dir/accounts/{id}/`. This is exactly right.

## Task 3: X Credential-Linking Endpoints (OAuth PKCE)

### Problem

There are no server-side endpoints for initiating X API OAuth PKCE flow for a specific account. The CLI `auth` command exists but writes to the global `tokens.json`. The server needs endpoints so the dashboard can link X credentials per account.

### Design

Add new routes under `/api/accounts/{id}/x-auth/`:

```
POST /api/accounts/{id}/x-auth/start    → Start OAuth PKCE flow, return auth URL
POST /api/accounts/{id}/x-auth/callback  → Exchange code for tokens, save to account path
GET  /api/accounts/{id}/x-auth/status    → Check if account has valid X credentials
```

### New File: `crates/tuitbot-server/src/routes/x_auth.rs`

```rust
//! X API credential linking endpoints for per-account OAuth PKCE flows.

/// POST /api/accounts/{id}/x-auth/start
/// - Validates account exists
/// - Generates PKCE challenge using `tuitbot_core::startup::generate_pkce`
/// - Stores PendingOAuth in `state.pending_oauth` keyed by state param
///   (augmented with account_id so callback knows which account)
/// - Builds auth URL using `tuitbot_core::startup::build_auth_url`
/// - Returns { authorization_url, state }

/// POST /api/accounts/{id}/x-auth/callback
/// - Accepts { code, state } body
/// - Looks up PendingOAuth by state, validates account_id matches
/// - Calls `tuitbot_core::startup::exchange_auth_code` to get tokens
/// - Resolves account token path via `account_token_path()`
/// - Creates parent dir, saves tokens via `tuitbot_core::x_api::auth::save_tokens`
///   (converts StoredTokens → auth::Tokens format)
/// - Evicts any existing TokenManager for this account (force reload)
/// - Auto-triggers sync_profile logic
/// - Returns { status: "linked", username }

/// GET /api/accounts/{id}/x-auth/status
/// - Checks if token file exists at account path
/// - If exists, loads and checks expiry
/// - Returns { linked: bool, expired: bool, username, expires_at }
```

### Augment PendingOAuth

Add `account_id: String` to `PendingOAuth` so the callback can verify which account initiated the flow:

```rust
pub struct PendingOAuth {
    pub code_verifier: String,
    pub created_at: Instant,
    pub account_id: String,  // NEW
}
```

This requires updating the connector flow in `connectors.rs` where `PendingOAuth` is created — set `account_id: String::new()` or `"connector".to_string()` there since connector OAuth is not account-scoped.

### Token Format Bridge

`startup.rs` uses `StoredTokens` (with `Option<String>` refresh_token, `Option<DateTime>` expires_at).
`x_api::auth.rs` uses `Tokens` (with `String` refresh_token, `DateTime<Utc>` expires_at).

The callback handler should use `exchange_auth_code` from `startup.rs` (which returns `StoredTokens`), then convert to `auth::Tokens` for saving via `auth::save_tokens` (which handles 0600 permissions). Or alternatively, save `StoredTokens` directly — `AppState::get_x_access_token` uses `auth::load_tokens` which expects `auth::Tokens`.

**Decision:** Use `auth::Tokens` format for the saved file since that's what `get_x_access_token` → `auth::load_tokens` expects. The callback should exchange the code using `startup::exchange_auth_code` (reusing the existing PKCE helpers), convert `StoredTokens` → `auth::Tokens`, and save with `auth::save_tokens`.

### Route Registration

In `lib.rs`, add:
```rust
.route("/accounts/{id}/x-auth/start", post(routes::x_auth::start_link))
.route("/accounts/{id}/x-auth/callback", post(routes::x_auth::complete_link))
.route("/accounts/{id}/x-auth/status", get(routes::x_auth::link_status))
```

Place these BEFORE the `/accounts/{id}` catch-all route to avoid routing conflicts (Axum route ordering).
