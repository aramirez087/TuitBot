# Session 8 Plan: Dashboard Credential-Linking Flows

## Goal

Each account can manage its own X credentials (OAuth tokens + scraper sessions) entirely from the dashboard, with immediate UI feedback and no cross-account leakage.

## Current State Analysis

### Backend (ready)
- `POST /api/accounts/{id}/x-auth/start` -- generates PKCE challenge, returns `authorization_url` + `state` (x_auth.rs:32-84)
- `POST /api/accounts/{id}/x-auth/callback` -- exchanges code for tokens, saves to account-specific path (x_auth.rs:100-169)
- `GET /api/accounts/{id}/x-auth/status` -- returns `AccountAuthStatus` with `oauth_linked`, `oauth_expired`, `scraper_linked`, etc. (x_auth.rs:175-211)
- Scraper session endpoints are already account-scoped via `AccountContext` (scraper_session.rs)
- `GET /api/runtime/status` already returns per-account `can_post` and `provider_backend` (runtime.rs:30-36)
- All API calls include `X-Account-Id` header from http.ts

### Frontend (gaps)
- `AccountsSection.svelte` shows credential status badges but has **no connect/disconnect/reconnect actions**
- `BrowserSessionSection.svelte` uses `api.settings.scraperSession` which routes through the account-scoped endpoint but is only rendered inside `XApiSection` (scraper mode)
- `api.accounts.authStatus(id)` exists in client.ts but there's no `startAuth` or `completeAuth` method
- `RuntimeStatus.can_post` is fetched but not refreshed after credential changes
- No OAuth disconnect (token deletion) endpoint exists

## Implementation Plan

### Phase 1: Backend -- Add Missing Endpoints

#### 1a. Add OAuth disconnect endpoint

**File:** `crates/tuitbot-server/src/routes/x_auth.rs`

Add `DELETE /api/accounts/{id}/x-auth/tokens` handler:
- Validates account exists
- Deletes the token file at `account_token_path(data_dir, id)`
- Evicts `TokenManager` from `state.token_managers`
- Returns `{ "deleted": true }`

**File:** `crates/tuitbot-server/src/lib.rs`

Register the new route:
```rust
.route(
    "/accounts/{id}/x-auth/tokens",
    delete(routes::x_auth::unlink),
)
```

Place it **before** the catch-all `/accounts/{id}` route (alongside existing x-auth routes at line ~258).

**Rationale:** Without a disconnect endpoint, users can't unlink OAuth credentials from the dashboard. The scraper session already has a delete endpoint.

#### 1b. No other backend changes needed

The existing endpoints cover:
- OAuth start/callback/status
- Scraper session get/import/delete
- Runtime status with `can_post`

### Phase 2: Frontend -- API Client Types & Methods

#### 2a. Add API methods to `client.ts`

**File:** `dashboard/src/lib/api/client.ts`

Add to the `accounts` namespace:
```typescript
startAuth: (id: string) =>
    request<{ authorization_url: string; state: string }>(
        `/api/accounts/${id}/x-auth/start`, { method: 'POST' }
    ),
completeAuth: (id: string, code: string, state: string) =>
    request<{ status: string; token_path: string }>(
        `/api/accounts/${id}/x-auth/callback`, {
            method: 'POST',
            body: JSON.stringify({ code, state })
        }
    ),
unlinkOAuth: (id: string) =>
    request<{ deleted: boolean }>(
        `/api/accounts/${id}/x-auth/tokens`, { method: 'DELETE' }
    ),
```

These methods call the account-specific endpoints. The `X-Account-Id` header is automatically set by `http.ts`.

#### 2b. No new types needed

`AccountAuthStatus` already exists in `types.ts` (line 683-689). The start/callback/unlink responses are simple enough to inline.

### Phase 3: Frontend -- Credential Management UI in AccountsSection

This is the main UI work. The `AccountsSection.svelte` credential detail section (lines 379-399) will be replaced with an interactive credential management panel.

**File:** `dashboard/src/routes/(app)/settings/AccountsSection.svelte`

#### Design decisions:

1. **Expand the credential detail section** into a per-account credential management card with actions:
   - OAuth: "Link X Account" / "Relink" / "Unlink" buttons based on status
   - Scraper: "Import Session" / "Remove Session" buttons based on status

2. **OAuth flow:** "Link X Account" calls `api.accounts.startAuth(id)`, opens the authorization URL in a new window/tab via `window.open()`, then shows a "Paste callback code" input for the user to complete the flow. This matches the existing PKCE pattern where the user is redirected to X, authorizes, and gets a code to paste back.

   **Alternative considered:** Using `auth.mode: local_callback` with a redirect URI back to the dashboard. Rejected because this requires coordinating server-side callback handling with per-account context, and the current `local_callback` flow in `startup.rs` is CLI-oriented. The paste-code approach is simpler and already proven.

3. **Scraper session management:** Embed a simplified version of the `BrowserSessionSection` import form directly in the credential card, but call the scraper session endpoints scoped to the specific account (not the "active" account). This requires passing the account ID explicitly rather than relying on `AccountContext` from the global `X-Account-Id` header.

   **Important:** The scraper session endpoints use `AccountContext` which reads from the `X-Account-Id` header. Since the credential section shows ALL accounts, we need to temporarily override the account ID when making scraper calls for a non-active account. We'll use direct `request()` calls with the target account's ID, or briefly set the account ID around the call.

   **Decision:** Add helper functions that wrap the scraper session API calls with an explicit account ID parameter. These will use `request()` directly with a custom `X-Account-Id` header override.

4. **Immediate feedback after credential changes:**
   - After successful OAuth link/unlink or scraper import/delete, refresh the auth status for that specific account
   - Call `syncAccountProfile(id)` after OAuth linking to pull avatar/display name
   - Refresh runtime status to update `can_post`

#### Component structure:

The credential detail section will be restructured into expandable per-account cards:

```
Credential Status
  @username1 (Default)
    OAuth: linked (expires Mar 10) [Relink] [Unlink]
    Scraper: none               [Import Session]

  @username2
    OAuth: none                  [Link X Account]
    Scraper: linked (@user)     [Remove]
```

Each account row in the credential detail will have:
- Status indicators (existing badge pattern)
- Action buttons that trigger inline flows (OAuth code input, scraper import form)
- Loading states during async operations
- Error/success messages per-account

### Phase 4: Account-Scoped Scraper Session API Helpers

**File:** `dashboard/src/lib/api/client.ts`

Add account-scoped scraper session methods alongside the existing `settings.scraperSession` namespace. These explicitly take an account ID:

```typescript
// Inside api.accounts namespace:
scraperSession: {
    get: (id: string) =>
        request<{ exists: boolean; username?: string; created_at?: string }>(
            '/api/settings/scraper-session',
            { headers: { 'X-Account-Id': id } }
        ),
    import: (id: string, data: { auth_token: string; ct0: string; username?: string }) =>
        request<{ status: string; username?: string; created_at?: string }>(
            '/api/settings/scraper-session',
            { method: 'POST', body: JSON.stringify(data), headers: { 'X-Account-Id': id } }
        ),
    delete: (id: string) =>
        request<{ deleted: boolean }>(
            '/api/settings/scraper-session',
            { method: 'DELETE', headers: { 'X-Account-Id': id } }
        ),
},
```

**Important:** The `request()` function in `http.ts` already merges `options.headers` with the default headers (line 63-66). The explicit `X-Account-Id` in the options will override the default from the module-level `accountId` variable. This is the cleanest approach without modifying the HTTP layer.

### Phase 5: Runtime Status Refresh After Credential Changes

**File:** `dashboard/src/lib/stores/runtime.ts`

The `loadCapabilities()` function has a `fetched` guard that prevents re-fetching. We need a way to force a refresh after credential changes.

Add `reloadCapabilities()`:
```typescript
export async function reloadCapabilities(): Promise<void> {
    fetched = false;
    fetching = false;
    await loadCapabilities();
}
```

**File:** `dashboard/src/routes/(app)/settings/AccountsSection.svelte`

After successful OAuth link/unlink or scraper import/delete for the **currently active** account, call `reloadCapabilities()` to update `can_post` in the runtime store. Non-active account changes don't need this since `can_post` is per-active-account.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| OAuth PKCE flow requires user to paste code manually | This is the same pattern used by `tuitbot auth` CLI. Future sessions can add `local_callback` mode with auto-redirect. |
| `X-Account-Id` header override in `request()` may not work if `headers` merge is order-dependent | Verify by reading `http.ts` -- spread operator puts options.headers last, so it overrides defaults. Confirmed at line 63-66. |
| Scraper session endpoints don't return updated auth status | After scraper import/delete, explicitly call `api.accounts.authStatus(id)` to refresh. |
| AccountsSection.svelte may exceed 400-line limit | Credential management adds significant UI. Extract inline forms into separate `CredentialActions.svelte` component if needed. |
| Runtime `can_post` stale after credential change on non-active account | Only refresh runtime for the active account. Non-active accounts' `can_post` is checked when they become active. |
| Token file deletion on unlink needs parent directory cleanup | Don't clean up parent dirs -- they may hold scraper sessions. Only delete the specific file. |
