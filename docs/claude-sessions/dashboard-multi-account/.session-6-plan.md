# Session 6 Plan — Account Management Flows

## Goal

Add a full account-management surface to the dashboard so users can create, rename, archive, and inspect accounts from the UI — replacing the current placeholder "Add Account" navigation to `/settings`.

## Current State Summary

- **Backend**: Full account CRUD at `/api/accounts` (list, create, get, update, delete). X credential linking at `/api/accounts/{id}/x-auth/{start,callback,status}`. Profile sync at `/api/accounts/{id}/sync-profile`. All routes are registered and tested.
- **Frontend**: `accounts.ts` store has `fetchAccounts()`, `switchAccount()`, `initAccounts()`, `syncCurrentProfile()`. API client has `api.accounts.{list,get,create,update,delete,syncProfile}`. `AccountSwitcher.svelte` shows accounts and has "Add Account" button that navigates to `/settings` (placeholder).
- **Gap**: No UI for account management actions (create, rename, archive, credential status). "Add Account" is a dead-end navigation to the general settings page.

## Design Decisions

### D1: Account roster as a new SettingsSection vs. dedicated page

**Decision**: Add an `AccountsSection.svelte` component within the existing settings page, positioned first in the section list (before Workspace). This follows the established pattern of all other settings sections using `SettingsSection` wrapper with `id`, `title`, `description`, `icon` props.

**Rationale**: The settings page already has the section-nav pattern with IntersectionObserver highlighting. Adding a new section is consistent, avoids a new route, and keeps account management co-located with other configuration. The "Add Account" button in `AccountSwitcher` will scroll to this section.

### D2: Create-account flow

**Decision**: Inline form within AccountsSection. User types a label, clicks "Create". Backend creates the account, frontend receives the new Account object, adds it to the accounts store, switches to it, and scrolls to show credential-linking status.

**Rationale**: Simple, no modals. The label can always be renamed later. Immediate switch makes the new account feel real.

### D3: Credential status display

**Decision**: Each account row shows credential status badges by calling `GET /api/accounts/{id}/x-auth/status`. Status is fetched on section mount for all accounts and cached locally.

**Rationale**: The x-auth status endpoint already exists and returns `{oauth_linked, oauth_expired, scraper_linked, has_credentials}`. No new backend work needed.

### D4: Archive guards

**Decision**: Cannot archive the default account (backend enforces, frontend hides archive button). Cannot archive the currently active account (frontend guard — must switch away first). Archive requires confirmation (inline "type to confirm" pattern from DangerZoneSection).

**Rationale**: Prevents users from stranding themselves. Backend already rejects deleting DEFAULT_ACCOUNT_ID, but the frontend should not even offer the option.

### D5: "Add Account" button in AccountSwitcher

**Decision**: Change from `goto('/settings')` to `goto('/settings#accounts')` so the section scrolls into view. The section's `id="accounts"` attribute enables this.

### D6: No backend changes needed

The existing API endpoints cover all Session 6 requirements:
- `POST /api/accounts` — create
- `PATCH /api/accounts/{id}` — rename (label update)
- `DELETE /api/accounts/{id}` — archive (soft-delete)
- `POST /api/accounts/{id}/sync-profile` — refresh profile
- `GET /api/accounts/{id}/x-auth/status` — credential status

No new Rust code required. This session is purely frontend + documentation.

### Step 3: Add account management helpers to store

**File: `dashboard/src/lib/stores/accounts.ts`**

Add three exported functions:

```typescript
export async function createAccount(label: string): Promise<Account | null> {
    try {
        const account = await api.accounts.create(label);
        await fetchAccounts();
        switchAccount(account.id);
        return account;
    } catch {
        return null;
    }
}

export async function renameAccount(id: string, label: string): Promise<Account | null> {
    try {
        const updated = await api.accounts.update(id, { label });
        accounts.update((list) =>
            list.map((a) => (a.id === updated.id ? updated : a))
        );
        return updated;
    } catch {
        return null;
    }
}

export async function archiveAccount(id: string): Promise<boolean> {
    try {
        await api.accounts.delete(id);
        await fetchAccounts();
        // If we archived the current account, switch to default
        const currentId = getPersistedAccountId();
        if (currentId === id) {
            switchAccount(DEFAULT_ACCOUNT_ID);
        }
        return true;
    } catch {
        return false;
    }
}
```

Note: `createAccount` auto-switches to the new account (per session instructions: "selects the new account immediately"). `archiveAccount` handles the edge case where the archived account was active.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| AccountsSection exceeds 400-line limit | Medium | Extract sub-components: `AccountRow.svelte` for individual account display, `CreateAccountForm.svelte` for the creation form. But only if needed — prefer single file if under 400 lines. |
| Auth status endpoint returns 401 for accounts without credentials | Low | The `link_status` endpoint doesn't require account credentials — it just checks file existence. Safe to call for any account. |
| Race condition: archive account while it's being switched to | Low | `archiveAccount()` re-fetches accounts list after delete, and `switchAccount()` validates against list. If the archived account ID is stale, `initAccounts()` fallback handles it. |
| Hash navigation (`#accounts`) doesn't scroll on same-page | Low | If already on `/settings`, `goto('/settings#accounts')` may not scroll. Add `setTimeout(() => document.getElementById('accounts')?.scrollIntoView({ behavior: 'smooth' }), 100)` as fallback. |
| Store helpers don't surface error messages to UI | Medium | Return error messages from store functions (not just null/false), or let the component call API directly and handle errors. Prefer: store functions throw, component catches and sets error state. |

### States for each account row

- **Default view**: shows identity, badges, credential status, action buttons
- **Rename mode**: label becomes editable input, save/cancel buttons replace rename
- **Archive confirming**: shows "Type ARCHIVE to confirm" input + button
- **Syncing**: sync button shows spinner
- **Error**: red error text below the row

### Credential badge logic

```typescript
function credentialBadge(status: AccountAuthStatus) {
    if (!status.has_credentials) return { label: 'No credentials', color: 'warning' };
    if (status.oauth_expired) return { label: 'Token expired', color: 'warning' };
    return { label: 'Linked', color: 'success' };
}
```
