### Singleton Seams Blocking Full Multi-Account

#### S1: Config Loading (CRITICAL)
- **File:** `crates/tuitbot-core/src/config/mod.rs:161-185`
- **Problem:** Single global `config.toml` loaded at startup. No per-account config resolution.
- **Existing hook:** `accounts.config_overrides` column (JSON text) exists but is unused.
- **Decision needed:** Which config fields are account-scoped vs instance-scoped?

#### S2: Token & Scraper Session Files (CRITICAL)
- **File:** `crates/tuitbot-server/src/routes/scraper_session.rs:30,64`
- **Problem:** Hardcoded `data_dir.join("scraper_session.json")` and `data_dir.join("tokens.json")` — no per-account paths.
- **Existing hook:** `accounts.token_path` column exists; `sync-profile` uses it as fallback.
- **Decision needed:** Per-account file naming convention (e.g., `tokens_{account_id}.json`).

#### S3: Automation Loops Don't Accept account_id (CRITICAL)
- **File:** `crates/tuitbot-core/src/automation/approval_poster.rs:42,95,104`
- **Problem:** `run_approval_poster()` calls `get_next_approved(&pool)` (default account), not `get_next_approved_for(&pool, account_id)`. Same for `mark_posted()`, `log_action()`.
- **Also:** `discovery_loop.rs`, `content_loop.rs`, `mentions_loop.rs`, `thread_loop.rs`, `target_loop.rs`, `analytics_loop.rs` — none accept `account_id`.
- **Scope:** Every automation loop signature needs an `account_id: String` parameter, threaded through all storage calls.

#### S4: WebSocket Events Unscoped (HIGH)
- **File:** `crates/tuitbot-server/src/ws.rs:22-84`
- **Problem:** `WsEvent` enum has no `account_id` field. All events broadcast to all clients.
- **Also:** `handle_ws()` subscribes to global broadcast — no client-side account filtering.
- **Decision needed:** Add `account_id` to each variant, or wrap in an outer envelope?

#### S5: Watchtower is Global (MEDIUM)
- **File:** `crates/tuitbot-server/src/main.rs:209-217`
- **Problem:** Single `WatchtowerLoop` spawned globally, no account-scoping.
- **Content sources in config are instance-level, not per-account.

#### S6: Settings UX is Instance-Level (MEDIUM)
- **File:** `dashboard/src/routes/(app)/settings/+page.svelte`
- **Problem:** Settings page reads/writes the global `config.toml`. No per-account overrides UI.
- **Decision needed:** Which settings sections should show per-account controls?

#### S7: Content Generator Init (LOW)
- **File:** `crates/tuitbot-server/src/main.rs:160-163`
- **Problem:** Only default account's content generator is initialized at startup.
- **Fix:** Lazy-init pattern (already exists for token managers, just extend to content generators).

#### S8: CLI Has No Account Awareness (LOW)
- **File:** `crates/tuitbot-cli/` — no `account_id` usage found.
- **Impact:** CLI `run` command boots a single-account runtime. Fine as fallback.

## Design Decisions to Document in Charter

### D1: Account-Scoped vs Instance-Scoped Config

**Instance-scoped (shared across accounts):**
- `server` (host, port)
- `storage` (db path, retention)
- `logging`
- `deployment_mode`
- `connectors` (OAuth app credentials)
- `llm` (provider, API key, model) — shared LLM billing
- `circuit_breaker`

**Account-scoped (per-account overrides via `config_overrides` JSON):**
- `business` (profile, keywords, persona)
- `scoring` (weights, thresholds)
- `limits` (rate limits)
- `intervals` (loop timings)
- `mode` (autopilot/composer)
- `approval_mode`
- `schedule` (active hours)
- `x_api` (client_id, provider_backend)
- `content_sources` (which sources belong to which account)
- `targets` (monitored accounts)

**Rationale:** This keeps the product single-user (one LLM key, one server) while allowing multiple X accounts with distinct personas and behaviors.

### D2: Effective Config Resolution

```
fn effective_config(base: &Config, account: &Account) -> Config {
    let overrides: Value = serde_json::from_str(&account.config_overrides)?;
    let mut base_toml = toml::to_string(base)?;
    // JSON-merge overrides into base
    merge_patch(&mut base_toml, &overrides);
    toml::from_str(&merged)
}
```

- New function in `tuitbot-core::config` (not server).
- Called by runtime bootstrap and settings endpoints.
- `config_overrides` is validated against Config schema before save.

### D3: Per-Account File Layout

```
~/.tuitbot/
  config.toml                          # Instance config
  tuitbot.db                           # Shared DB (account_id columns)
  tokens.json                          # Default account tokens (backward compat)
  scraper_session.json                 # Default account session (backward compat)
  accounts/
    {account_id}/
      tokens.json                      # Per-account OAuth tokens
      scraper_session.json             # Per-account browser session
```

- Default account (`00000000-...`) uses root-level files (backward compat).
- New accounts get `accounts/{id}/` subdirectory.
- `account.token_path` column stores the resolved path.

### D4: WebSocket Account Scoping

Add `account_id: String` to every `WsEvent` variant. Client-side filtering in the WebSocket store:

```typescript
// dashboard/src/lib/stores/websocket.ts
socket.onmessage = (e) => {
    const event = JSON.parse(e.data);
    if (event.account_id && event.account_id !== get(currentAccountId)) return;
    // ...dispatch
};
```

Server broadcast remains global (single broadcast channel). Filtering is client-side to avoid per-account channels complexity.

### D5: Credential Linking Flow (In-Dashboard)

1. User clicks "Add Account" in AccountSwitcher.
2. Dashboard creates account via `POST /api/accounts`.
3. Dashboard shows credential entry UI (OAuth flow or scraper session import).
4. On success, `POST /api/accounts/{id}/sync-profile` fetches X profile.
5. Account appears in switcher with avatar and @username.

### D6: Default Account Backward Compatibility

- Default account ID (`00000000-...`) always exists, cannot be deleted.
- Missing `X-Account-Id` header defaults to it (already implemented).
- Existing `config.toml`, `tokens.json`, `scraper_session.json` map to default account.
- No migration of existing data required — all rows already have default account_id.
