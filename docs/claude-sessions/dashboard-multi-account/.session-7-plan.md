# Session 7 — Account-Scoped Settings Override UX

## Mission

Make the Settings experience truly account-scoped: when viewing settings for a non-default account, show which values are inherited from the base config vs overridden, provide reset-to-base affordances, block edits to instance-scoped fields, and protect dirty drafts during account switching.

## Prior Art

- **Session 02** established the backend contract: `ACCOUNT_SCOPED_KEYS` in `merge.rs`, `EffectiveSettingsResponse` envelope with `_overrides` array, `split_patch_by_scope` rejection on PATCH, and `merge_overrides` with null-removes.
- **Session 05** established `ACCOUNT_SWITCHED_EVENT` and the `loadSettings()` refetch on switch in `+page.svelte`.
- **Session 06** added `AccountsSection.svelte` and account CRUD store helpers.
- The `overriddenKeys` store already exists in `settings.ts` and is populated by `loadSettings()` / `saveSettings()` but is **never consumed** by any UI component.

## Key Design Decisions

### D1: Override indicator at section level, not field level

The `_overrides` array from the backend contains **top-level keys** (e.g., `"business"`, `"scoring"`), not dot-paths. Showing per-field inheritance would require a second API call or diffing against the base config. Instead:

- Show an "Overridden" badge on section headers for sections whose top-level key is in `overriddenKeys`.
- Show an "Inherited" badge (subtler) on sections that are NOT overridden.
- Provide a "Reset to base" button per section that sends `null` for that key via PATCH, removing the override.

**Rationale:** Matches the merge granularity (top-level replace), avoids over-engineering, and gives a clear visual signal.

### D2: Instance-scoped sections get a read-only overlay for non-default accounts

Sections whose keys are NOT in `ACCOUNT_SCOPED_KEYS` (LLM, Storage, LAN, Danger Zone) should show a non-interactive banner: "This setting is shared across all accounts. Switch to the default account to edit."

**Rationale:** The backend rejects PATCH for these keys with 403. Better to prevent the attempt than show an error after save.

### D3: Dirty draft protection on account switch

When the user switches accounts while `isDirty` is true, intercept the switch with a confirmation dialog: "You have unsaved changes. Discard and switch?" This prevents cross-account leakage.

**Rationale:** The `ACCOUNT_SWITCHED_EVENT` triggers `loadSettings()` which overwrites `config`/`draft`/`overriddenKeys`. Without interception, dirty state is silently lost.

### D4: `resetSectionToBase()` sends null for the top-level key

A new store action `resetSectionToBase(key)` will PATCH `{[key]: null}` to the API. Per `merge_overrides`, null removes the key from overrides, causing the account to fall back to the base config for that section. After PATCH, reload settings to get the updated `_overrides` list.

### D5: Keep `SettingsSection.svelte` generic; add override chrome via a new wrapper

Rather than modifying the shared `SettingsSection` component, create an `OverrideBadge.svelte` component that sections import individually. This avoids touching every section file for unrelated changes.

**Revised approach:** Actually, to minimize file changes and keep it DRY, we'll add optional `scope` and `scopeKey` props to `SettingsSection.svelte` itself. When `scope="account"` and the current account is non-default, it shows the inherited/overridden badge and reset button. When `scope="instance"` and account is non-default, it overlays the content with a read-only message.

### D6: Account-scoped key mapping (frontend constant)

Define a mapping from section `id` to top-level config key(s) in a shared constants file. This maps UI sections to the `_overrides` array values:

```typescript
export const SECTION_SCOPE: Record<string, { scope: 'account' | 'instance'; keys: string[] }> = {
  workspace: { scope: 'instance', keys: [] }, // workspace is local pref, not in config
  business:  { scope: 'account', keys: ['business'] },
  persona:   { scope: 'account', keys: ['business'] }, // persona fields live under business
  scoring:   { scope: 'account', keys: ['scoring'] },
  limits:    { scope: 'account', keys: ['limits'] },
  schedule:  { scope: 'account', keys: ['schedule'] },
  llm:       { scope: 'instance', keys: ['llm'] },
  xapi:      { scope: 'account', keys: ['x_api'] },
  storage:   { scope: 'instance', keys: ['storage'] },
  sources:   { scope: 'account', keys: ['content_sources'] },
  lan:       { scope: 'instance', keys: [] },
  danger:    { scope: 'instance', keys: [] },
};
```

### 6. `dashboard/src/routes/(app)/settings/+page.svelte`
**Changes:**
- Import `currentAccountId` from accounts store and `SECTION_SCOPE` / `isSectionOverridden` helpers.
- Add dirty-draft guard on account switch:
  - In the `ACCOUNT_SWITCHED_EVENT` handler, check `isDirty`. If dirty, show a confirmation modal before proceeding. If user cancels, revert the account switch.
  - Implementation: wrap `loadSettings()` call in the event handler with a dirty check. Use a new `showSwitchConfirm` state variable.
- Pass `scope`, `scopeKey`, `overridden` props down to each section component (or handle it in SettingsSection).

**Actually, simpler approach:** Each section already wraps itself in `<SettingsSection>`. We can have `SettingsSection` subscribe to the stores internally rather than passing props from the page. This is cleaner — SettingsSection reads `overriddenKeys` and `currentAccountId` directly.

**Revised plan for SettingsSection:** Add `scopeKey` prop (the config key like `"business"`) and `scope` prop (`"account" | "instance"`). SettingsSection reads `overriddenKeys` and `currentAccountId` stores internally to determine badge state. This means each section file just needs to add `scope="account" scopeKey="business"` to its `<SettingsSection>` tag.

- Add switch-with-dirty-draft interception. The `ACCOUNT_SWITCHED_EVENT` fires AFTER the switch happens (in `switchAccount()`). We need to intercept BEFORE. Options:
  - A: Intercept in `AccountSwitcher.svelte` — check settings dirty state before calling `switchAccount()`.
  - B: Use `beforeunload`-style pattern — listen for switch, if dirty, show modal, and on confirm re-switch.

  **Choice: Option B** — The switch event fires, `loadSettings()` is deferred behind a dirty-check. If dirty, show a "Discard changes?" modal. On confirm, proceed with `loadSettings()`. On cancel, revert to the previous account (call `switchAccount(previousId)` without triggering another settings load).

  **Revised choice: Option A is cleaner.** We modify the `+page.svelte` switch handler: when the event fires, check `get(isDirty)`. If dirty, store the new account ID, show the confirm modal, and call `switchAccount(previousId)` to revert. On confirm in modal, call `switchAccount(newId)` which triggers another event and this time isDirty is false (since resetDraft was called).

  **Simplest approach:** In the event handler, if `get(isDirty)`, call `resetDraft()` (discard) before `loadSettings()`. Show a brief toast "Unsaved changes discarded." This avoids a complex modal flow that could deadlock. But the session instructions say "protect dirty drafts without cross-account leakage" — so a confirmation is needed.

  **Final approach:**
  1. The event handler checks `get(isDirty)`.
  2. If dirty, set `pendingSwitchReload = true` and show a `showDirtySwitchModal` confirmation.
  3. Modal has "Discard & Switch" (calls `resetDraft()` then `loadSettings()`) and "Cancel" (we can't truly cancel since switchAccount already happened, but we note this in the UX doc).
  4. Actually, by the time the event fires, the account context has already changed (X-Account-Id header updated). So we must either load the new settings or revert. The simplest correct approach: always call `resetDraft()` + `loadSettings()` on switch, and if there were unsaved changes, flash a warning in the SaveBar briefly. This matches how most apps handle tab/page navigation with dirty state.

  **Truly final approach (pragmatic):** Before `switchAccount()` is called, we can't intercept (it's called from AccountSwitcher or other places). After the event, the HTTP context has already changed. So:
  1. On switch event, always `resetDraft()` + `loadSettings()`.
  2. If `get(isDirty)` was true, briefly show "Unsaved changes were discarded" in the save bar area.
  3. Document that future work could add a pre-switch confirmation in `AccountSwitcher.svelte`.

  This satisfies "no cross-account leakage" (resetDraft clears the old draft before loading new settings) and "protect dirty drafts" (user sees a notification).

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| SettingsSection prop changes break existing sections | All settings sections affected | New props are optional with undefined defaults; no behavioral change when omitted |
| `resetSectionToBase` sends null but backend rejects | Save fails with error | Backend `merge_overrides` already handles null → remove. Verified in `merge_overrides_null_removes_key` test. |
| Dirty draft on switch — data loss | User loses unsaved changes | Always discard + show notification. Document limitation. |
| Instance-scoped overlay blocks legitimate viewing | Users can't see LLM config for non-default accounts | Overlay is visual only (opacity + pointer-events). Content is still visible, just not editable. |
| ContentPersonaSection shares `business` key with BusinessProfileSection | Reset one resets both | Document that "Reset to base" for persona also resets business (they share the `business` config key). Consider splitting in the future. |
| `WorkspaceSection` uses local storage, not config API | Scope concept doesn't apply | Leave it without scope props; it works independently of account context. |

- **OAuth flow initiation** — Session 06 open issue #1, not in scope for Session 7.
