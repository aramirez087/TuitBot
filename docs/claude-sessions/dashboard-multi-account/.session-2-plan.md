# Session 2 Implementation Plan: Account-Aware Effective Config & Settings API

## Goal

Make `GET /api/settings`, `PATCH /api/settings`, `POST /api/settings/validate`, and scraper session endpoints account-aware. When a non-default account is selected, settings reads return an **effective config** (base `config.toml` merged with the account's `config_overrides`), and writes to account-scoped fields persist to `accounts.config_overrides` instead of mutating `config.toml`.

---

## Key Design Decisions

### D1: `effective_config` lives in `tuitbot-core::config` (not server)
Per the server-boundary rule — the server crate owns zero business logic. The merge function takes a `&Config` and an overrides JSON string, returns a merged `Config`.

### D2: RFC 7396 merge-patch semantics on JSON
Algorithm: serialize base Config to JSON, parse overrides as JSON, deep-merge (overrides win for scalars, recurse for objects, arrays are replaced entirely per Session 01 recommendation), deserialize back to Config, validate. Using `serde_json::Value` merge — no need for a separate `merge.rs` file since the logic is ~30 lines.

### D3: Scope enforcement via allowlist
A compile-time constant list of **account-scoped top-level keys** controls what can appear in `config_overrides`. When PATCH arrives for a non-default account, the patch is split: account-scoped keys go to `config_overrides`, instance-scoped keys are rejected with a 403. For the default account, all keys write to `config.toml` (backward compat).

### D4: Override metadata in GET response
`GET /api/settings` for non-default accounts returns the effective merged config plus a sibling `_overrides` key listing which top-level sections have account-specific values. This lets the frontend distinguish "inherited from base" vs "overridden for this account".

### D5: `null` clears an override field
When a non-default account sends `null` for an account-scoped section in PATCH, that section is removed from `config_overrides`, causing the account to fall back to the base config value.

### D6: Scraper session path resolution via `account_data_dir`
Default account uses `data_dir/scraper_session.json`. Other accounts use `data_dir/accounts/{id}/scraper_session.json`. The helper `account_data_dir` is in `tuitbot-core::storage::accounts` to keep path logic in core.

---

## Scope Contract: Account-Scoped vs Instance-Scoped Fields

**Account-scoped** (allowed in `config_overrides`):
`mode`, `x_api`, `business`, `scoring`, `limits`, `intervals`, `approval_mode`, `max_batch_approve`, `schedule`, `targets`, `content_sources`

**Instance-scoped** (shared, `config.toml` only):
`server`, `storage`, `logging`, `llm`, `deployment_mode`, `connectors`, `circuit_breaker`, `mcp_policy`, `auth`

### 3. `crates/tuitbot-core/src/storage/accounts.rs` (MODIFY)

Add helper function (~15 lines):

```rust
use std::path::{Path, PathBuf};

/// Resolve the data directory for an account.
///
/// - Default account: returns `data_dir` itself (root-level, backward compat).
/// - Other accounts: returns `data_dir/accounts/{account_id}/`.
pub fn account_data_dir(data_dir: &Path, account_id: &str) -> PathBuf {
    if account_id == DEFAULT_ACCOUNT_ID {
        data_dir.to_path_buf()
    } else {
        data_dir.join("accounts").join(account_id)
    }
}

/// Resolve the scraper session file path for an account.
pub fn account_scraper_session_path(data_dir: &Path, account_id: &str) -> PathBuf {
    account_data_dir(data_dir, account_id).join("scraper_session.json")
}

/// Resolve the token file path for an account.
pub fn account_token_path(data_dir: &Path, account_id: &str) -> PathBuf {
    account_data_dir(data_dir, account_id).join("tokens.json")
}
```

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `effective_config` merge produces invalid Config due to partial section override | HIGH | Always run `config.validate()` after merge. Return validation errors to caller. Test edge cases (partial `scoring` override with missing fields — serde defaults fill in). |
| Default account behavior regression — `get_settings`/`patch_settings` currently take no `AccountContext` | HIGH | `AccountContext` defaults to `DEFAULT_ACCOUNT_ID` when header is missing. For default account, code path is identical to current (read/write `config.toml`). No behavioral change. |
| Scraper session directory doesn't exist for new account | MEDIUM | `import_scraper_session` creates parent dir before writing. `get_scraper_session` returns `exists: false` if file not found (existing behavior). |
| `config_overrides` column contains legacy data or `'{}'` | LOW | Empty string and `'{}'` both produce empty JSON object, which means "no overrides" — base config returned unchanged. |
| `json_to_toml` null-skipping behavior conflicts with override `null`-means-remove | MEDIUM | For non-default accounts, PATCH works entirely in JSON (not TOML). The `json_to_toml` path is only used for default account (writing `config.toml`). |
| Route handler signature change (adding `AccountContext`) breaks compilation | LOW | `AccountContext` is an Axum `FromRequestParts` extractor — just add it as a parameter. The handler still works the same. |

## Verification Steps

### Automated

```bash

cargo clippy --workspace -- -D warnings

```

### Manual verification scenarios

1. **Default account unchanged**: `GET /api/settings` with no `X-Account-Id` header returns same response as before.
2. **Non-default account with empty overrides**: `GET /api/settings` with `X-Account-Id: {new-account-id}` returns base config + `_overrides: []`.
3. **Non-default account with overrides**: After `PATCH /api/settings` with `{"scoring": {"threshold": 80}}` and non-default account, `GET` returns `threshold: 80` and `_overrides: ["scoring"]`.
4. **Instance-scoped rejection**: `PATCH /api/settings` with `{"llm": {"provider": "anthropic"}}` and non-default account returns 403.
5. **Null-clear**: `PATCH /api/settings` with `{"scoring": null}` and non-default account removes the scoring override; subsequent `GET` returns base scoring values.
6. **Scraper session isolation**: Import session for account A, verify account B's `GET /api/settings/scraper-session` returns `exists: false`.
