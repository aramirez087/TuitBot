# Session 6 Plan: Hook-First Compose Workflow

## Mission

Integrate hook generation into the Ghostwriter compose flow so users can generate 5 hook options, compare them, choose one, and produce a grounded tweet or thread — all without losing source context.

## Current State Summary

- **Backend**: `POST /api/assist/hooks` exists and returns 3–5 styled hook options with `style`, `text`, `char_count`, `confidence` fields. Accepts `topic`, `session_id`, and `selected_node_ids`.
- **Frontend types**: `HookOption` and `AssistHooksResponse` already defined in `types.ts`. `api.assist.hooks()` client method exists in `client.ts`.
- **Compose flow**: `FromVaultPanel` → chunk selection → `VaultHighlights` → generate. Selection path goes through `VaultSelectionReview` when `selectionSessionId` is set. Both paths call `ComposerInspector.handleGenerateFromVault()` which writes to `tweetText` or `threadBlocks`.
- **No hook step exists yet**: The flow goes directly from highlights/selection to final content generation.

## Architecture Decision: Where the Hook Step Lives

**Decision**: Insert a new `HookPicker` step between highlights extraction and final generation, reachable from both the chunk-selection path and the Obsidian selection path.

**Rationale**: The hook step is a UI-layer concern. The backend already generates hooks; we need a picker component and a state machine transition in `FromVaultPanel` and `VaultSelectionReview`. The hook picker is a new component (`HookPicker.svelte`) because it has distinct UX (card grid, selection, regenerate) that doesn't belong in `VaultHighlights` or `VaultFooter`.

**Alternative rejected**: Embedding hook selection inside `VaultHighlights`. This would bloat that component and conflate two distinct user decisions (which highlights matter vs. which hook style to use).

## Detailed Implementation Plan

### Step 1: Create `hookStyles.ts` utility

**File**: `dashboard/src/lib/utils/hookStyles.ts`

```typescript
// Style display name mapping
const STYLE_LABELS: Record<string, string> = {
  question: 'Question',
  contrarian_take: 'Hot Take',
  tip: 'Quick Tip',
  list: 'List',
  most_people_think_x: 'Myth Buster',
  storytelling: 'Story',
  before_after: 'Before/After',
  general: 'General',
};

export function getStyleLabel(style: string): string {
  return STYLE_LABELS[style] ?? style.replace(/_/g, ' ').replace(/\b\w/g, c => c.toUpperCase());
}

export function getConfidenceBadge(confidence: string): { label: string; class: string } {
  if (confidence === 'high') return { label: 'Strong', class: 'confidence-high' };
  return { label: 'Good', class: 'confidence-medium' };
}
```

**Test**: `dashboard/tests/unit/hookStyles.test.ts` — covers all known styles, unknown style fallback, confidence mapping.

### Step 2: Create `HookPicker.svelte` component

**File**: `dashboard/src/lib/components/composer/HookPicker.svelte`

**Props**:
```typescript
{
  hooks: HookOption[];
  outputFormat: 'tweet' | 'thread';
  loading?: boolean;
  error?: string | null;
  onselect: (hook: HookOption, format: 'tweet' | 'thread') => void;
  onregenerate: () => void;
  onback: () => void;
  onformatchange: (format: 'tweet' | 'thread') => void;
}
```

**UI Structure**:
- Header row: back arrow + "Choose a Hook" label (matches `VaultHighlights` pattern)
- Scrollable card list (max-height: 280px, overscroll-behavior: contain)
- Each card shows:
  - Style label pill (from `getStyleLabel`)
  - Hook text (13px, line-height 1.45)
  - Footer row: char count + confidence badge
  - Selected state: accent border + subtle background
- Footer:
  - Format toggle (tweet/thread) — reuses the same pattern as `VaultHighlights`
  - "Regenerate" button (secondary style, calls `onregenerate`)
  - "Use this hook" button (primary, disabled until a hook is selected)
- Loading state: 5 shimmer cards (same shimmer pattern as `VaultSelectionReview`)
- Error state: inline error message with retry button

**Estimated lines**: ~200 (script) + ~150 (styles) = ~350 total. Under 400-line Svelte limit.

**Design decisions**:
- Single-select, not multi-select. User picks one hook to seed downstream generation.
- Selected hook index tracked as `$state<number | null>(null)`. Selection resets on regenerate.
- No individual hook regeneration in v1 — regenerate-all only. Per-hook regen is a future enhancement (Session 5 handoff item 4 noted this as "consider"). Reason: individual regen requires a new backend parameter (`preferred_style`) which doesn't exist yet.

### Step 3: Wire hook step into `FromVaultPanel.svelte`

**Current flow**: search → select chunks → extract highlights → generate content
**New flow**: search → select chunks → extract highlights → **generate hooks → pick hook** → generate content

**State additions**:
```typescript
let hookStep = $state<HookOption[] | null>(null);
let hookLoading = $state(false);
let hookError = $state<string | null>(null);
let selectedHook = $state<HookOption | null>(null);
```

**Changes to `handleGenerateFromHighlights`**:
- Rename to `handleGenerateHooksFromHighlights`
- Instead of calling `ongenerate(cachedNodeIds, outputFormat, enabledHighlights)`, call `api.assist.hooks()` with:
  - `topic`: Join enabled highlights with newline (same text that would have gone to generation)
  - `opts.selectedNodeIds`: `cachedNodeIds`
- On success: set `hookStep = response.hooks`
- On error: set `hookError`

**New function `handleHookSelected`**:
- Receives the chosen `HookOption` and current `outputFormat`
- Calls `ongenerate(cachedNodeIds, outputFormat, [hook.text])` — the hook text is passed as the "highlights" parameter, which `ComposerInspector.handleGenerateFromVault` already handles as highlight context

**New function `handleRegenerateHooks`**:
- Resets `selectedHook` to null
- Re-calls `api.assist.hooks()` with same parameters
- Preserves `cachedNodeIds` and `highlightsStep` (source context retained for retries)

**New function `handleBackToHighlights`**:
- Sets `hookStep = null`, `hookError = null`, `selectedHook = null`
- Returns to highlights step (highlights data preserved in `highlightsStep`)

**Template change**: Add a new conditional branch:
```svelte
{:else if hookStep}
  <HookPicker
    hooks={hookStep}
    {outputFormat}
    loading={hookLoading}
    error={hookError}
    onselect={handleHookSelected}
    onregenerate={handleRegenerateHooks}
    onback={handleBackToHighlights}
    onformatchange={(f) => { outputFormat = f; }}
  />
```

**Context preservation**: `cachedNodeIds`, `highlightsStep`, and `selectedChunks` are all preserved during hook generation and retries. The user can go back to highlights, toggle highlights on/off, and re-generate hooks without losing their chunk selections.

### Step 4: Wire hook step into `VaultSelectionReview.svelte`

**Current flow**: load selection → "Generate from selection" → generate content
**New flow**: load selection → "Generate hooks" → **pick hook** → generate content

**State additions**:
```typescript
let hookStep = $state<HookOption[] | null>(null);
let hookLoading = $state(false);
let hookError = $state<string | null>(null);
```

**Changes to `handleGenerate`**:
- First click: generate hooks via `api.assist.hooks()` with:
  - `topic`: `selection.selected_text` or `selection.note_title` or `selection.heading_context` (first available as topic)
  - `opts.sessionId`: `sessionId` (passes through to backend for RAG resolution)
- Sets `hookStep = response.hooks` on success
- Shows `HookPicker` instead of generating content directly

**New function `handleHookSelected`**:
- Receives chosen `HookOption`
- Calls `ongenerate(nodeIds, outputFormat, [hook.text])` — same pattern as `FromVaultPanel`
- `nodeIds` derived from `selection.resolved_node_id` (same as current `handleGenerate`)

**New function `handleRegenerateHooks`**:
- Re-calls `api.assist.hooks()` with same selection context
- Selection data (`selection` state) is preserved — no re-fetch needed

**Button label change**: `VaultFooter` button says "Generate hooks" instead of "Generate from selection" when `selectionMode=true`.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `FromVaultPanel.svelte` exceeds 400-line limit after adding hook state | Medium | Must extract subcomponent | Current file is 374 lines. Hook step adds ~30 lines of script + ~10 lines of template. Should stay under 400. If exceeded, extract hook state management into a `hookFlow.ts` utility. |
| Hook generation is slow (LLM call) and user sees no feedback | Medium | UX feels broken | `HookPicker` has explicit loading state with shimmer cards. `hookLoading` flag shows immediately. |
| User selects "thread" format but hook is a single tweet | Low | Confusing output | Hook text is used as the seed/opener for thread generation. The downstream `api.assist.thread()` call handles expanding a single hook into a multi-tweet thread. This is the correct behavior — the hook IS the thread opener. |
| Highlights step feels redundant now that hooks exist | Low | UX friction | Highlights serve a different purpose: letting users curate which ideas matter. Hooks are about finding the right opening angle. Both are valuable. If users skip highlights, the selection path (`VaultSelectionReview`) goes directly to hooks. |
| Test mocking `api.assist.hooks` requires new mock setup | Low | Test brittleness | Follow existing pattern in `FromVaultPanel.test.ts` which already mocks `api.vault.*` and `api.assist.highlights`. Add `api.assist.hooks` to the same mock factory. |
