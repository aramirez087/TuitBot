# Session 7 Plan: Native Workflow Polish — Provenance Propagation

## Goal

Ensure Ghostwriter provenance and chosen-hook context survive the full content lifecycle: hook selection → compose → draft creation → scheduling → publish → approval queue → posted tweet. Surface citation affordances in Draft Studio without overloading the UI. Add regression coverage. Document the native workflow for Session 08's privacy gating.

---

## Current State (Gaps Identified)

### Backend Gaps

1. **Approval → Scheduled Content bridge drops provenance.** `handlers.rs:158` calls `scheduled_content::insert_for()` which has no provenance parameter. When a scheduled approval item is bridged to `scheduled_content`, its `vault_provenance_links` rows are not copied.

2. **Draft duplication drops provenance.** `duplicate_draft_for()` in `scheduled_content/drafts.rs` creates a new draft row but never calls `provenance::copy_links_for()`.

3. **Thread compose endpoint has no `provenance` field.** `ComposeThreadRequest` (compose/mod.rs:122) lacks `provenance: Option<Vec<ProvenanceRef>>`. Thread content composed via the legacy endpoint loses provenance.

4. **`compose/transforms.rs` scheduled-content path drops provenance.** When approval_mode is off, `persist_content()` calls `scheduled_content::insert_for()` without provenance.

5. **Hook style not persisted.** When a user picks a hook, only `hook.text` flows downstream. The `hook.style` (e.g., `contrarian_take`) is discarded by the frontend and never reaches the backend.

6. **Draft detail endpoint returns no provenance.** `GET /api/drafts/:id` returns `ScheduledContentItem` which has no provenance field. The frontend has no way to display citations for an existing draft.

### Frontend Gaps

7. **`buildComposeRequest()` never sets `provenance`.** The compose handler in `composeHandlers.ts` builds `ComposeRequest` but omits the `provenance` field entirely.

8. **`handleGenerateFromVault()` doesn't capture ProvenanceRef.** After vault-assisted generation, `ComposerInspector.svelte:142` sets `tweetText`/`threadBlocks` but discards the vault node IDs and chunk IDs that should become provenance refs.

9. **`HydrationPayload` has no provenance.** `draftStudioParse.ts` defines `HydrationPayload` without citation data. When a draft is loaded into Draft Studio, provenance is invisible.

10. **`DraftMetadataSection` shows source label but no citations.** It renders "AI Assist" / "Manual" but never shows which vault notes sourced the content.

11. **`CitationChips` is display-only, never wired into compose flow.** It can render citations but is never used in Draft Studio or the compose submit path.

## Design Decisions

### D1: Add `hook_style` as optional metadata, not a schema column

Store the chosen hook style in `source` field enrichment: `"assist:hook:contrarian_take"` instead of bare `"assist"`. This avoids a migration, reuses the existing `source TEXT` column on `scheduled_content`, and gives downstream surfaces (DraftMetadataSection, analytics) a way to distinguish hook-sourced content.

**Why:** Adding a new column for a single string that's useful only for Ghostwriter content is over-engineered. The `source` field already distinguishes `manual` / `assist` / `discovery` — extending it with a sub-tag is additive and backward-compatible.

### D2: Add a `GET /api/drafts/:id/provenance` endpoint

Rather than bloating the `ScheduledContentItem` struct or the existing GET endpoint, add a dedicated provenance sub-resource. The frontend calls it when Draft Studio opens a draft. This is a read-only, privacy-safe endpoint that returns `ProvenanceLink[]`.

**Why:** Provenance data is optional context, not core draft data. Separate endpoint avoids N+1 on list views and keeps the draft list response lean.

### D3: Copy provenance links when bridging approval → scheduled_content

In `handlers.rs`, after `insert_for()`, call `provenance::copy_links_for()` from `approval_queue` entity to `scheduled_content` entity. Same for `duplicate_draft_for()`.

**Why:** Without this, scheduled content loses its source attribution. This is the most critical gap — it breaks the "provenance survives lifecycle" contract.

### D4: Pass vault context through compose as `ProvenanceRef[]`

Wire `handleGenerateFromVault` to capture `selectedNodeIds` as `ProvenanceRef[]` and stash them in component state. `buildComposeRequest` reads this state and includes `provenance` in the `ComposeRequest`. For the hook path, also include `hook.style` as metadata.

**Why:** The backend already accepts `provenance` on `ComposeRequest` and `CreateDraftRequest`. The frontend just isn't populating it.

### D5: Display citation chips in DraftStudioDetailsPane, not DraftMetadataSection

DraftMetadataSection is a compact summary strip. Citations deserve their own collapsible section in the details pane, using the existing `CitationChips` component.

**Why:** Keeps the metadata section clean. CitationChips already handles expand/collapse, Obsidian deep links, and chip rendering.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| `source` field string parsing is fragile | Medium — bad parse shows wrong label | Use a strict regex `^assist:hook:(\w+)$` with fallback to plain "AI Assist" |
| Approval bridge provenance copy fails silently | High — scheduled content has no attribution | Use `?` propagation, log warning on copy failure, do not block the bridge operation |
| `HydrationPayload` changes break existing draft loading | Medium — draft studio won't open | Provenance is fetched separately (D2), HydrationPayload unchanged |
| Thread compose provenance is new API surface | Low — additive field with serde(default) | Backward compatible: old clients send no provenance, new field defaults to None |
| `duplicate_draft_for` is in tuitbot-core, provenance copy is in server | Low — layering concern | Copy in the server route handler, not in core. Matches existing pattern in `publish_draft`. |
