# Session 4 Plan: Dashboard Ghostwriter UX â€” Block Selection, Ingress Hydration, Citations

## Overview

Turn the current vault panel into a Ghostwriter workspace where users can:
1. See and select exact source blocks (not just whole notes)
2. Receive Obsidian plugin selections as first-class compose state via `?selection=<session_id>`
3. Get clear CTAs for "use source", citation actions, and provenance rendering
4. Have full test coverage for selection state, ingress hydration, and citation affordances

## Architecture Context

Session 3 delivered the backend: `POST /api/vault/send-selection` stores selections in `vault_selections` (30-min TTL), resolves block identity, emits `SelectionReceived { session_id }` via WebSocket, and returns `composer_url: "/compose?selection=<uuid>"`. Session 4 wires the frontend to consume this.

## Design Decisions

### Decision 1: Selection ingress via URL param + WebSocket listener

**Approach**: Two ingress paths to the composer:
- **URL param**: `/?selection=<session_id>` â€” user clicks the composer URL from Obsidian plugin response
- **WebSocket**: `SelectionReceived` event auto-opens vault panel with selection pre-loaded when dashboard is already open

**Rationale**: URL param covers the primary flow (Obsidian â†’ browser). WebSocket covers the "dashboard already open" case without requiring a page reload.

### Decision 2: New `api.vault.getSelection()` client method

**Approach**: Add `getSelection(sessionId: string)` to `api.vault` namespace in `client.ts`. Returns a `VaultSelection` type (new in `types.ts`) containing `session_id`, `file_path`, `selected_text`, `heading_context`, `note_title`, `resolved_node_id`, `resolved_chunk_id`.

**Rationale**: Mirrors the existing `GET /api/vault/selection/{session_id}` endpoint from Session 3. The `selected_text` field will be `null` in Cloud mode (privacy gate enforced server-side).

### Decision 3: Selection hydration as pre-populated highlight state

**Approach**: When a selection is received (via URL or WebSocket), the vault panel enters a "selection review" state â€” showing the selected text as a single pre-checked highlight with source metadata. User can then generate directly or refine.

**Why not skip to generation?** Power users want to see what's being used before generation. This also handles Cloud mode gracefully â€” when `selected_text` is null, we show metadata-only with a "Generate from source" CTA.

### Decision 4: ProvenanceRef construction at generation time

**Approach**: When generating from a selection, construct `ProvenanceRef` from the selection's `resolved_node_id`/`resolved_chunk_id` and pass through the existing compose/draft provenance pipeline.

**Rationale**: ProvenanceRef already has all fields. No new pipeline needed â€” just populate from selection metadata.

### Decision 5: CitationChips enhanced with heading-anchor deep links

**Approach**: Extend `buildObsidianUri` to accept optional `heading` param for `obsidian://open?vault=X&file=Y#heading` deep links. CitationChips already has the Obsidian open button â€” just needs the heading fragment.

### Decision 6: WebSocket store update for SelectionReceived

**Approach**: Add `'SelectionReceived'` to the `WsEvent.type` union in `websocket.ts`. Add a subscriber in `DraftStudioShell.svelte` that reacts to this event by fetching the selection and opening the vault panel.

## Risks & Mitigations

### Risk 1: Cloud mode selection UX degradation
**Problem**: In Cloud mode, `selected_text` is null in GET response. User sees metadata-only.
**Mitigation**: Show file path + heading context as the "source preview". CTA says "Generate from source" (not "from selection"). Clear messaging: "Text not shown in cloud mode for privacy."

### Risk 2: Selection TTL expiry during compose
**Problem**: Selection expires after 30 minutes. User might take longer.
**Mitigation**: Fetch selection immediately on param/WS receipt. Store the full response in component state (not just session_id). If fetch fails (404 expired), show "Selection expired â€” select blocks manually" and fall back to normal vault panel.

### Risk 3: WebSocket event race with URL navigation
**Problem**: `SelectionReceived` WS event fires, but user already navigated to `/?selection=X`.
**Mitigation**: Deduplicate by tracking `lastConsumedSelectionId`. If WS event session_id matches current param, skip. If different, prompt user: "New selection received â€” switch?"

### Risk 4: File size limits on modified Svelte files
**Problem**: `FromVaultPanel.svelte` is 353 lines. Adding selection hydration logic might push it past 400.
**Mitigation**: Extract selection hydration logic into a helper function in a separate `$lib/utils/selectionHydration.ts` utility if needed. Keep the Svelte component focused on rendering.

### Risk 5: ProvenanceRef attachment to compose pipeline
**Problem**: Need to ensure ProvenanceRef from selection flows through to `api.content.compose()` and `api.drafts.create()`.
**Mitigation**: Both APIs already accept `provenance?: ProvenanceRef[]`. Just construct and pass. The `ComposeRequest` type already has the field.
