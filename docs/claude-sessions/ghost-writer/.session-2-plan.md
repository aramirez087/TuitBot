# Session 2 Implementation Plan: Obsidian-Side Ghostwriter Entry Path

## Mission

Create the Obsidian plugin (`plugins/obsidian-tuitbot/`) that captures editor selection and block context, sends it to the local TuitBot runtime via HTTP POST, and documents the transport contract for Session 3 to implement the receiving side.

## Key Design Decisions

### Decision 1: Transport — HTTP POST to localhost API

**Chosen**: `POST http://127.0.0.1:3001/api/vault/send-selection`

**Rejected alternatives**:
- **Clipboard bridge**: Fragile, platform-specific, no structured payload, user loses clipboard content. No acknowledgment of receipt.
- **File-system watcher**: Operator rules explicitly prohibit background daemons and sync engines. A watched drop-file would be a sync engine in disguise.
- **Custom IPC / Unix socket**: Adds a new transport layer that doesn't exist in the codebase. TuitBot already has a full HTTP API on `127.0.0.1:3001`. Using it is zero new infrastructure.
- **obsidian:// URI callback**: Obsidian's URI scheme is one-way (open files). It cannot receive structured data back.
- **WebSocket push**: Adds connection lifecycle management to the plugin. HTTP is request-response and simpler for a command-driven action.

**Rationale**: The TuitBot server already runs on `127.0.0.1:3001` in Desktop mode with Bearer token auth. The HTTP API is the existing integration surface (used by the Tauri frontend, MCP server, and CLI). A single POST with JSON body is the simplest possible transport — one request, one response, no persistent connection. Session 01's Decision 4 already selected this approach.

### Decision 2: Auth — Read `~/.tuitbot/api_token` file

The server authenticates via `Authorization: Bearer <token>` where the token is read from `~/.tuitbot/api_token` (see `crates/tuitbot-server/src/auth/token.rs`). The Obsidian plugin will:
1. Read `~/.tuitbot/api_token` from disk on plugin load.
2. Include `Authorization: Bearer <token>` header on every request.
3. Show a notice if the token file is missing (TuitBot not installed or never started).

This matches how the Tauri desktop app and CLI authenticate — no new auth mechanism needed.

### Decision 3: Plugin packaging — Obsidian community plugin conventions

Obsidian plugins require:
- `manifest.json` (Obsidian-specific: `id`, `name`, `version`, `minAppVersion`, `author`, `description`)
- `main.js` (compiled entry point)
- `package.json` (dev tooling)

We follow Obsidian's conventions (not the OpenClaw plugin conventions) because this plugin runs inside Obsidian's runtime, not Node.js. Key differences from the openclaw-tuitbot plugin:
- Entry point is a class extending `Plugin` with `onload()`/`onunload()`, not an ES module export
- Build uses `esbuild` (Obsidian community standard) instead of bare `tsc`, because Obsidian needs a single bundled `main.js`
- `manifest.json` follows Obsidian's schema, not OpenClaw's `configSchema` pattern
- TypeScript types come from `obsidian` npm package

### Decision 4: Minimal command surface

Two commands, no settings tab, no ribbon icon:
1. **"Send selection to TuitBot"** — sends the current editor selection with full context
2. **"Send current block to TuitBot"** — sends the block (paragraph) at cursor when nothing is selected

This follows the operator constraint: "command-driven selection handoff, no sync engine, no background daemon." The plugin does nothing unless the user explicitly triggers a command.

### Decision 5: Configurable server URL via plugin settings

Default: `http://127.0.0.1:3001`. Configurable for self-host users who run TuitBot on a different host/port. Settings:
- `serverUrl` (string, default `http://127.0.0.1:3001`)
- `apiTokenPath` (string, default `~/.tuitbot/api_token`)

No settings tab UI in Session 2 — these are JSON-editable in `data.json`. A settings tab can be added later if needed.

**Key implementation details**:

#### Payload construction (`buildPayload`)

```typescript
interface GhostwriterPayload {
  vault_name: string;
  file_path: string;
  selected_text: string;
  heading_context: string | null;
  selection_start_line: number;
  selection_end_line: number;
  // Additional context for richer provenance
  note_title: string | null;
  frontmatter_tags: string[] | null;
}
```

- `vault_name`: From `this.app.vault.getName()`
- `file_path`: From `file.path` (relative to vault root, includes `.md`)
- `selected_text`: From `editor.getSelection()` or extracted block
- `heading_context`: Walk backward from cursor to find nearest heading (e.g., `## Setup`)
- `selection_start_line` / `selection_end_line`: From `editor.getCursor('from').line` / `editor.getCursor('to').line`
- `note_title`: From frontmatter `title` field or filename
- `frontmatter_tags`: From `this.app.metadataCache.getFileCache(file)?.frontmatter?.tags`

#### Heading resolution (`resolveHeadingContext`)

Uses Obsidian's metadata cache to find the heading hierarchy at the cursor position:
```typescript
const cache = this.app.metadataCache.getFileCache(file);
const headings = cache?.headings ?? [];
// Find the deepest heading whose line is <= cursor line
// Build heading path like "# Title > ## Section > ### Subsection"
```

This is more reliable than regex-parsing the editor content because Obsidian's metadata cache handles edge cases (YAML frontmatter, code blocks containing `#`).

#### Block extraction (`extractBlockAtCursor`)

When no text is selected, extract the "block" at cursor — defined as the contiguous non-blank lines around the cursor:
```typescript
const cursorLine = editor.getCursor().line;
let start = cursorLine;
let end = cursorLine;
// Walk up while line is non-empty
while (start > 0 && editor.getLine(start - 1).trim() !== '') start--;
// Walk down while line is non-empty
while (end < editor.lineCount() - 1 && editor.getLine(end + 1).trim() !== '') end++;
// Collect lines start..end
```

This captures the semantic "block" (paragraph, list, code block) without requiring Obsidian's internal block IDs.

#### HTTP transport (`sendToTuitBot`)

```typescript
async sendToTuitBot(payload: GhostwriterPayload): Promise<SendSelectionResponse> {
  const token = await this.readApiToken();
  const response = await requestUrl({
    url: `${this.settings.serverUrl}/api/vault/send-selection`,
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
    },
    body: JSON.stringify(payload),
  });
  return response.json;
}
```

Uses Obsidian's `requestUrl` API (not `fetch`) because Obsidian restricts direct network access but provides `requestUrl` for plugin HTTP calls.

#### Error handling

- **TuitBot not running**: `requestUrl` throws on connection refused → show Notice: "TuitBot server not reachable at {url}. Is TuitBot running?"
- **Auth failure (401)**: Show Notice: "TuitBot authentication failed. Check your API token at {tokenPath}."
- **Token file missing**: Show Notice on load: "TuitBot API token not found at {path}. Start TuitBot once to generate it."
- **No active editor**: Show Notice: "No active editor. Open a note first."
- **Empty selection + empty block**: Show Notice: "Nothing to send. Select text or place cursor in a paragraph."

## Risks and Mitigations

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **Obsidian `requestUrl` may not reach localhost in some configurations** | Medium | `requestUrl` bypasses CORS and browser restrictions. Tested on macOS/Windows/Linux. If blocked, fall back to Node.js `http` module (available in desktop Obsidian). Document both paths. |
| 2 | **`~/.tuitbot/api_token` path varies on Windows** | Medium | Use `process.env.HOME || process.env.USERPROFILE` for cross-platform home dir. Or use Obsidian's `app.vault.adapter` for filesystem access. Document the Windows path (`%USERPROFILE%\.tuitbot\api_token`). |
| 3 | **Plugin can't be tested in CI without Obsidian runtime** | Low | Unit tests cover pure logic. Build quality gate (`esbuild` compiles successfully) verifies type correctness. Manual testing in Obsidian for integration. |
| 4 | **`metadataCache` may be stale for recently edited notes** | Low | Obsidian updates the metadata cache incrementally. For heading resolution, the cache is accurate enough — worst case, the user sees a slightly stale heading path. Not a correctness issue since the selected text is the primary payload. |
| 5 | **Server endpoint doesn't exist yet — plugin always errors** | Expected | This is by design. The plugin is the sender half. Document this clearly in README: "Requires TuitBot v0.X.X+ with Ghostwriter support." The contract document bridges to the receiver implementation. |
| 6 | **Obsidian types package version mismatch** | Low | Pin `obsidian` dev dependency to `^1.7.2` (current stable). Test build with `minAppVersion: 1.4.0` as minimum. |

## Transport Contract Preview

This is the contract that `obsidian-plugin-contract.md` will formalize:

### Request: `POST /api/vault/send-selection`

**Headers**:
- `Content-Type: application/json`
- `Authorization: Bearer <api_token>`

**Body**:
```json
{
  "vault_name": "my-vault",
  "file_path": "notes/async-rust.md",
  "selected_text": "The key insight is that async in Rust gives you zero-cost abstractions for concurrent code. Unlike Go's goroutines, Rust's async model compiles down to state machines with no runtime overhead.",
  "heading_context": "## The Async Story > ### Zero-Cost Abstractions",
  "selection_start_line": 45,
  "selection_end_line": 52,
  "note_title": "Async Patterns in Rust",
  "frontmatter_tags": ["rust", "async", "programming"]
}
```

**Field constraints**:
- `vault_name`: 1-255 chars, required
- `file_path`: relative to vault root, required, must end in `.md`
- `selected_text`: 1-10000 chars, required (trimmed)
- `heading_context`: nullable, markdown heading path with ` > ` separator
- `selection_start_line`: >= 0, required
- `selection_end_line`: >= `selection_start_line`, required
- `note_title`: nullable, extracted from frontmatter or filename
- `frontmatter_tags`: nullable, array of strings

### Response (Success): `200 OK`
```json
{
  "status": "received",
  "session_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "composer_url": "/compose?selection=a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### Response (Error): `401 Unauthorized`
```json
{
  "error": "invalid_token",
  "message": "Missing or invalid Authorization header"
}
```

### Response (Error): `422 Unprocessable Entity`
```json
{
  "error": "validation_error",
  "message": "selected_text must not be empty"
}
```
