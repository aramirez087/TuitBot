## Key Design Decisions

### D1: Fix only real regressions, not aspirational gaps

This session validates what exists. "No E2E Playwright tests" is a known limitation, not a regression. Only fix things that would break a user at runtime (mismatched field names, broken serialization, incorrect privacy labels). Document everything else as follow-up.

**Rationale**: Session 9 is a release gate, not a feature session. Adding new tests or features here would change the risk profile of the release.

### D2: Code-tracing over runtime E2E tests

The four journeys are verified by reading code paths, not by running the full application with a real vault. This is sufficient because:
- All unit and integration tests already pass (5,377 Rust + 686 frontend)
- The system is composed of well-tested modules with clear contracts
- Runtime E2E testing requires a configured Obsidian vault and is out of scope for automated sessions

**Rationale**: The charter's non-goals explicitly exclude building a full E2E test harness. The code-tracing approach validates contracts and data flow without new infrastructure.

### D3: Separate follow-up work from release decision

The release-readiness report explicitly lists follow-up work (plugin settings tab, publish audit trail, analytics by hook style) in a separate section from the go/no-go decision. This prevents "scope creep by review" where deferred features are mistaken for release blockers.

### D4: QA matrix organized by privacy invariant, not by session

The matrix groups verification by cross-cutting concern (privacy, provenance, hooks, deep-links) rather than by session number. A maintainer making a release decision needs to see "are all privacy invariants satisfied?" not "what did session 5 do?"
