# Session 8 Plan: Deployment-Aware Privacy & Local-First Ghostwriter

## Summary

Implement deployment-aware privacy gating so Desktop + local vault = defensible local-first claim, while Self-host and Cloud modes are explicit about their privacy envelope. This touches config types, server state, vault/selection routes, Tauri setup, frontend settings/composer, and the Obsidian plugin.

---

## Key Design Decisions

### D1: Add `is_local_first()` method to `DeploymentMode`
**What:** A computed method `DeploymentMode::is_local_first() -> bool` that returns `true` only for `Desktop`. Self-host has local filesystem access but data crosses a network boundary (browser → server), so it cannot claim local-first. Cloud is obviously not local-first.

**Rationale:** The privacy matrix doc says "never market a path as local-first unless the runtime can guarantee it." Desktop with embedded server on `127.0.0.1:3001` is the only mode where data never leaves the machine. Self-host may be on LAN/VPN — still user-controlled but not same-machine.

### D2: Add `privacy_envelope` field to `DeploymentCapabilities`
**What:** A new string field `privacy_envelope: String` with values `"local_first"`, `"user_controlled"`, `"provider_controlled"`. Returned in the capabilities API so the frontend can display accurate privacy descriptions.

**Rationale:** The frontend already consumes `DeploymentCapabilities` from the runtime status endpoint. Adding this field lets settings UI and composer show honest privacy copy without hardcoding deployment mode checks in multiple places.

### D3: Add `ghostwriter_local_only` field to `DeploymentCapabilities`
**What:** Boolean flag indicating whether the Ghostwriter vault-to-compose pipeline runs entirely on the local machine (true only for Desktop + local_fs). The frontend uses this to gate the "Local-First" badge and privacy copy.

**Rationale:** The Ghostwriter flow (vault read → chunk retrieval → hook generation → compose) is local-first only when: (a) the server is embedded (Desktop), (b) the vault source is `local_fs`, and (c) the LLM could be local (but we don't gate on this — the user chose their LLM provider). This flag covers (a); the frontend must also check the source type for (b).

### D4: Server-side `selected_text` gating already exists — extend with privacy label
**What:** The `get_selection` endpoint already omits `selected_text` in Cloud mode (line 260 of `selections.rs`). We add a `privacy_notice` field to the response for Self-host mode to communicate that text is on the user's server but crossed a network.

**Rationale:** Self-host users see selected text (they control the server), but the UI should note it's not same-machine local-first. Cloud users never see selected text — already implemented.

### D5: Frontend privacy banner in ContentSourcesSection and FromVaultPanel
**What:** Conditional banners in settings and composer that show the privacy envelope:
- Desktop + local_fs: "Local-first — your notes never leave this machine"
- Self-host + local_fs: "Self-hosted — notes stay on your server"
- Self-host + google_drive: "Self-hosted — notes fetched from Google Drive to your server"
- Cloud: "Cloud — notes processed server-side, only generated content returned"

**Rationale:** Honest copy per the operator constraint: "never market a path as local-first unless the runtime can guarantee it."

### D6: Obsidian plugin transport validation
**What:** The plugin already sends to `http://127.0.0.1:3001` by default. Add a warning notice when the configured `serverUrl` is not localhost/127.0.0.1, informing the user that selections will cross a network boundary.

**Rationale:** Users may point the Obsidian plugin at a remote self-hosted server. The plugin should be transparent about where data goes.

---

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Adding `privacy_envelope` to `DeploymentCapabilities` is a breaking API change | Low — field is additive, clients ignore unknown fields | Serde `default` ensures backward compat for deserialization |
| Frontend privacy banners feel preachy or cluttering | Medium — users may find them annoying | Use subtle, dismissible notice-info style, only show once per session or per settings page load |
| Self-host users may feel their setup is "second class" | Medium — marketing concern | Copy should be positive: "Self-hosted — you control the server" rather than "Not local-first" |
| Obsidian plugin network warning fires for `localhost:3001` custom port setups | Low | Check for `localhost` or `127.0.0.1` hostnames specifically, not port |

## Implementation Order

### Phase 1: Core — DeploymentMode Privacy Extensions
**Files:** `crates/tuitbot-core/src/config/types.rs`, `crates/tuitbot-core/src/config/tests/deployment.rs`

1. Add `is_local_first(&self) -> bool` method to `DeploymentMode` impl block:
   ```rust
   pub fn is_local_first(&self) -> bool {
       matches!(self, DeploymentMode::Desktop)
   }
   ```

2. Add `privacy_envelope(&self) -> &'static str` method to `DeploymentMode`:
   ```rust
   pub fn privacy_envelope(&self) -> &'static str {
       match self {
           DeploymentMode::Desktop => "local_first",
           DeploymentMode::SelfHost => "user_controlled",
           DeploymentMode::Cloud => "provider_controlled",
       }
   }
   ```

3. Add two new fields to `DeploymentCapabilities`:
   ```rust
   pub privacy_envelope: String,       // "local_first" | "user_controlled" | "provider_controlled"
   pub ghostwriter_local_only: bool,   // true only for Desktop
   ```

4. Update `DeploymentMode::capabilities()` match arms to populate new fields.

5. Add tests in `tests/deployment.rs`:
   - `is_local_first` returns true only for Desktop
   - `privacy_envelope` returns correct values for all three modes
   - `ghostwriter_local_only` true only for Desktop
   - Serde roundtrip for new `DeploymentCapabilities` fields
   - Backward compat: deserializing old JSON without new fields succeeds (add `#[serde(default)]`)

### Phase 2: Server — Privacy-Aware Route Behavior
**Files:** `crates/tuitbot-server/src/routes/vault/mod.rs`, `crates/tuitbot-server/src/routes/vault/selections.rs`, `crates/tuitbot-server/src/routes/sources.rs`

1. **`vault/mod.rs` — `vault_sources` handler:**
   - Add `deployment_mode` and `privacy_envelope` fields to `VaultSourcesResponse` so the frontend knows the privacy context when rendering vault source status.
   - Omit `path` field in Cloud mode (already conditional on `local_fs`, but add explicit Cloud guard for defense in depth).

2. **`vault/selections.rs` — `get_selection` handler:**
   - Already gates `selected_text` on Cloud mode. Add a `privacy_envelope` string field to `GetSelectionResponse` so the frontend can render the right copy.
   - For Self-host mode, include `selected_text` (user controls server) but set `privacy_envelope: "user_controlled"`.
   - For Desktop mode, include `selected_text` and set `privacy_envelope: "local_first"`.

3. **`sources.rs` — `source_status` handler:**
   - Add `deployment_mode` to `SourceStatusResponse` (informational, helps frontend without extra round-trip).

4. **Tests:** Add integration tests to `vault/mod.rs::tests` for:
   - `vault_sources` with Desktop mode returns `privacy_envelope: "local_first"`
   - `vault_sources` with Cloud mode returns `privacy_envelope: "provider_controlled"` and no local path
   - Test with modified `test_state()` helper that accepts a `DeploymentMode` parameter

### Phase 3: Server State — Validation Tightening
**Files:** `crates/tuitbot-server/src/state.rs`

1. Add a `is_local_first(&self) -> bool` convenience method to `AppState`:
   ```rust
   pub fn is_local_first(&self) -> bool {
       self.deployment_mode.is_local_first()
   }
   ```

2. In `restart_watchtower()`, add a log line when starting with local_fs in non-Desktop mode to surface the privacy envelope mismatch for operators who care:
   ```rust
   if !self.deployment_mode.is_local_first() {
       tracing::info!("Note: local_fs source in {} mode — data is user-controlled but not local-first", self.deployment_mode);
   }
   ```

### Phase 4: Tauri — Desktop Deployment Mode Hardening
**Files:** `dashboard/src-tauri/src/lib.rs`

1. The Tauri setup already sets `deployment_mode: DeploymentMode::Desktop` (line 209). This is correct and needs no change.

2. Add a `get_privacy_envelope` Tauri command that returns `"local_first"` — gives the frontend a way to confirm Desktop mode without an API round-trip:
   ```rust
   #[tauri::command]
   fn get_privacy_envelope() -> String {
       "local_first".to_string()
   }
   ```

3. Register in `invoke_handler`: add `get_privacy_envelope` to the generate_handler macro.

4. **Decision:** Keep this minimal. The frontend already gets `deployment_mode` from the runtime status API. The Tauri command is a fast sync fallback before the API is ready.

### Phase 5: Frontend API Types & Client
**Files:** `dashboard/src/lib/api/types.ts`, `dashboard/src/lib/api/client.ts`

1. **`types.ts`:**
   - Add `privacy_envelope` and `ghostwriter_local_only` to `DeploymentCapabilities` interface
   - Add `privacy_envelope` to `VaultSelectionResponse` type
   - Add `deployment_mode` and `privacy_envelope` to `VaultSourcesResponse` type (if exists, or create)

2. **`client.ts`:**
   - No new API methods needed — existing `api.runtime.status()` and `api.vault.sources()` already return the data; we just need the types to match

### Phase 6: Frontend Runtime Store
**Files:** `dashboard/src/lib/stores/runtime.ts`

1. Add derived stores:
   ```ts
   export const privacyEnvelope = derived(capabilitiesData, ($d) => $d?.capabilities?.privacy_envelope ?? 'local_first');
   export const isLocalFirst = derived(capabilitiesData, ($d) => $d?.capabilities?.ghostwriter_local_only ?? true);
   ```

2. Update `DESKTOP_DEFAULTS` to include new fields:
   ```ts

   ```

### Phase 7: Frontend Settings — Privacy Copy
**Files:** `dashboard/src/routes/(app)/settings/ContentSourcesSection.svelte`

1. Import `privacyEnvelope` and `isLocalFirst` from runtime store.

2. Add a privacy banner below the health summary, before the source type selector:
   ```svelte
   {#if isDesktop && sourceType === 'local_fs'}
     <div class="notice notice-success">
       Your notes are processed locally — content never leaves this machine.
     </div>
   {:else if isSelfHost}
     <div class="notice notice-info">
       Self-hosted — notes stay on your server. Content is processed within your infrastructure.
     </div>
   {:else if isCloud}
     <div class="notice notice-info">
       Cloud mode — notes are processed server-side. Only generated content is returned to your browser.
     </div>
   {/if}
   ```

3. Add `notice-success` CSS class for the local-first positive badge (green-tinted).

### Phase 8: Frontend Composer — Privacy Indicator
**Files:** `dashboard/src/lib/components/composer/FromVaultPanel.svelte`

1. Import `deploymentMode` from runtime store.

2. Add a subtle privacy indicator below the "From Vault" label in the header:
   ```svelte
   <span class="vault-privacy-badge">
     {#if $deploymentMode === 'desktop'}
       Local-first
     {:else if $deploymentMode === 'self_host'}
       Self-hosted
     {:else}
       Cloud
     {/if}
   </span>
   ```

3. Style as a tiny muted badge next to "From Vault" label, using existing design tokens.

### Phase 9: Obsidian Plugin — Transport Awareness
**Files:** `plugins/obsidian-tuitbot/src/main.ts`

1. Add a `isLocalTransport()` method to `TuitBotPlugin`:
   ```ts
   private isLocalTransport(): boolean {
     try {
       const url = new URL(this.settings.serverUrl);
       return url.hostname === 'localhost' || url.hostname === '127.0.0.1';
     } catch {
       return false;
     }
   }
   ```

2. In the `send()` method, before the actual send, show an informational notice if `!isLocalTransport()`:
   ```ts
   if (!this.isLocalTransport()) {
     new Notice('TuitBot: Sending selection to remote server — text will cross the network.', 3000);
   }
   ```

3. This is non-blocking (doesn't prevent the send), just transparent about the transport.
