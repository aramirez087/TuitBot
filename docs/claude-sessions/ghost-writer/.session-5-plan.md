# Session 5 Plan: Hook Generation Backend

## Goal

Add a `POST /api/assist/hooks` endpoint that accepts Ghostwriter provenance (selection context or selected node IDs) and returns 5 differentiated hook options, each tagged with style metadata for downstream UI ranking.

## Design Decisions

### D1: Reuse `TweetFormat` as hook style taxonomy

The existing `TweetFormat` enum (`list`, `contrarian_take`, `most_people_think_x`, `storytelling`, `before_after`, `question`, `tip`) already defines 7 differentiated tweet structures with prompt fragments. Rather than inventing a new "hook style" taxonomy, we pick 5 of these formats per request to produce differentiated hooks. This keeps the system composable — a hook is just a single-tweet generation with an assigned format.

**Tradeoff**: Reusing `TweetFormat` means hooks share the same prompt engineering as standalone tweets. This is intentional: a hook IS a tweet, and users will use the best one directly or as a thread opener.

### D2: Single LLM call with structured multi-hook output (not 5 parallel calls)

Generate all 5 hooks in a single LLM call with a structured prompt that asks for 5 distinct hooks labeled by style. This is cheaper (1 call vs 5), faster, and naturally produces differentiated output since the LLM sees all hooks in context and avoids repetition.

**Format**: Expect output as `STYLE: <name>\nHOOK: <text>\n---` blocks (similar to existing `parse_seed_response` in `seed_worker.rs`).

**Fallback**: If the LLM returns fewer than 5 hooks, pad with a retry or return what we have (minimum 3). If it returns more than 5, truncate to 5.

### D3: `generate_hooks` lives in `ContentGenerator` (tuitbot-core)

The new method goes on `ContentGenerator` alongside `generate_tweet_inner`, `extract_highlights`, etc. This keeps business logic in `tuitbot-core` per the server boundary constraint. The server route is a thin adapter.

### D4: Response includes lightweight metadata, not analytics

Each hook in the response carries: `style` (the TweetFormat name), `text` (the hook content), `char_count`, and `confidence` (a simple heuristic: "high" if under 240 chars and matches a known format, "medium" otherwise). No engagement prediction or analytics — that's a future session concern per the session instructions.

### D5: Selection context flows through existing RAG pipeline

When the request includes a `session_id` (from a Ghostwriter selection), the handler:
1. Fetches the selection via `vault_selections` (reusing Session 3 infra)
2. If `resolved_node_id` exists, uses it as `selected_node_ids` for RAG context
3. If only `selected_text` is available (unresolved selection), injects it directly as context
4. Falls back to standard keyword-based RAG when no selection context

This reuses `resolve_composer_rag_context` from `rag_helpers.rs` for the node-based path, and adds a small extension for raw text injection.

### D6: Hook style selection strategy

Select 5 styles from the 7 `TweetFormat` variants, biased toward:
1. `Question` (always include — highest engagement format for hooks)
2. `ContrarianTake` (always include — strongest hook energy)
3. 3 randomly selected from remaining 5 formats

This ensures every generation includes the two most "hook-like" formats while varying the rest.

## Risks and Mitigations

### R1: LLM output parsing fragility
**Risk**: The LLM may not follow the `STYLE: ...\nHOOK: ...` format consistently.
**Mitigation**: Parser is lenient — strips prefixes, trims, falls back to "general" style if STYLE line is missing. Minimum 3 hooks enforced with one retry. Same approach as `parse_seed_response` which is proven in production.

### R2: File size limits
**Risk**: `assist.rs` is currently 508 lines (right at the 500-line limit). Adding ~80 lines for the hooks handler pushes it over.
**Mitigation**: Consider extracting hooks into a submodule if assist.rs exceeds 500 lines. If it's close, we can split `assist.rs` into `assist/mod.rs` + `assist/hooks.rs` following the vault module pattern from Session 3.

### R3: Selection session_id resolution adds DB call to hot path
**Risk**: When `session_id` is provided, we do an extra DB lookup (fetch selection) before the LLM call.
**Mitigation**: Single indexed query on `session_id` — negligible latency vs. the LLM call that follows. Same pattern already used by `GET /api/vault/selection/{session_id}`.

### R4: `ContentGenerator` doesn't have direct DB access for selection fetch
**Risk**: `ContentGenerator` only takes business profile + LLM provider, not a DB pool. Selection resolution must happen at the server layer.
**Mitigation**: This is correct by design — `ContentGenerator` stays DB-free. The server handler resolves context first (selection → node IDs → RAG), then passes the assembled text context to `generate_hooks`. Same pattern as all other assist handlers.

### R5: Coverage thresholds
**Risk**: New code without tests drops coverage below 75% threshold.
**Mitigation**: Comprehensive test plan (8+ tests for core, 3+ for server). Using the same MockProvider/PromptCapturingProvider patterns proven in existing tests.
