## Current State Summary (from codebase audit)

### What exists today

**Vault ingestion pipeline:**
- `content_nodes` table stores ingested Obsidian/markdown notes (title, body_text, relative_path, tags, front_matter_json)
- `content_chunks` table stores heading-level fragments (heading_path, chunk_text, chunk_hash, chunk_index, retrieval_boost)
- Chunks are addressed by heading hierarchy (`"# Title/## Section"`) — no line-range or byte-offset addressing
- `draft_seeds` table stores LLM-extracted tweetable hooks from content nodes
- `seed_worker.rs` runs background extraction (1-3 hooks per note via LLM)

**Retrieval & RAG:**
- `retrieval.rs`: keyword-based vault fragment retrieval with selected-note bias via `selected_node_ids`
- `winning_dna/mod.rs`: three-tier draft context (winning ancestors + vault fragments + cold-start seeds)
- `rag_helpers.rs`: server-side RAG resolver that loads config keywords, builds DraftContext
- Fragment prompt injection capped at 1000 chars

**Composer flow (frontend):**
- `FromVaultPanel.svelte`: search notes → expand → select chunks (max 3) → extract highlights → generate
- `VaultNoteList.svelte`: note expansion with chunk checkboxes
- `VaultHighlights.svelte`: toggle extracted highlights before generation
- `ComposerInspector.svelte`: orchestrates AI assist, vault generation, voice cue
- `CitationChips.svelte`: displays vault citations with Obsidian deep-links (desktop only)

**Deep-link & desktop:**
- `obsidianUri.ts`: builds `obsidian://open?vault=...&file=...` URIs from vault path + relative path
- `lib.rs` (Tauri): `open_external_url` command restricted to `obsidian://` and `file://` schemes
- Desktop mode sets `deployment_mode: DeploymentMode::Desktop`

**Provenance:**
- `vault_provenance_links` table: polymorphic (entity_type + entity_id) with snapshot fields
- `ProvenanceRef`: carried through API layer, stored as immutable snapshots
- Provenance survives source note deletion

**Compose & scheduling:**
- `compose/mod.rs`: unified compose with provenance support
- Draft Studio: full CRUD, autosave, revisions, activity log, tags, scheduling
- Approval queue integration

**Privacy model:**
- Vault routes are account-scoped via `AccountContext`
- No raw note bodies in API responses — only titles, paths, tags, heading_paths, truncated snippets (120 chars)
- `content_sources` config supports `local_fs` and Google Drive

### Key gaps for Ghostwriter edge

1. **No block-level send**: Users select whole chunks (heading sections), not arbitrary text blocks or paragraphs
2. **No hook-first thread drafting**: Draft seeds exist but aren't exposed in the composer — only used in autopilot context
3. **No heading-anchor deep-links**: Obsidian URI builder doesn't include heading anchors (`#heading`)
4. **Chunk identity is heading-only**: No support for sub-heading blocks, callouts, or paragraph-level selection
5. **Highlight extraction is LLM-round-trip**: Adds latency; no cached/pre-computed highlights in the DB
6. **No Obsidian → TuitBot selection handoff**: No clipboard bridge or URI-based selection push from Obsidian

## Key Design Decisions

### Decision 1: Extend existing chunk_id + heading_path, do not introduce byte-offset block addressing

**Rationale:** The current `heading_path` model already captures the primary semantic unit users think in (markdown sections). Adding byte offsets or line ranges would:
- Break the existing retrieval pipeline
- Require a new chunking strategy
- Add complexity without proportional user value

Instead, enrich the existing model with:
- **Sub-heading paragraphs**: Split large chunks into paragraph-level sub-chunks (new `paragraph_index` field)
- **Heading-anchor deep-links**: Append `#heading` to Obsidian URIs for section-level navigation
- **Pre-computed highlights**: Cache seed worker outputs as selectable items in the vault panel

### Decision 2: Hook-first thread drafting as a new composer entry point

**Rationale:** Draft seeds (`draft_seeds` table) already contain 1-3 tweetable hooks per note but are only consumed by autopilot. Expose them as a "Hook → Thread" flow:
1. User browses vault → sees pre-extracted hooks per note
2. Selects a hook → system generates a full thread expanding that hook
3. Thread is editable in the existing thread composer

This reuses `seed_worker.rs` infrastructure and the existing thread generation API.

### Decision 3: Privacy boundary — never expose raw note bodies over HTTP in Cloud mode

**Current state:** Vault API already returns only titles, paths, heading_paths, and 120-char snippets. This is correct.

**Extension:** For the Ghostwriter flow, block-level "send to composer" must:
- **Desktop**: Read chunk_text from local DB (same process, no network boundary)
- **Self-host**: Read chunk_text from local DB (user controls the server)
- **Cloud**: Return only pre-processed summaries/highlights, never raw chunk_text. The LLM processes raw text server-side and returns generated content only.

### Decision 4: Obsidian selection handoff via command palette, not sync daemon

**Rationale:** Per constraints — "minimal: command-driven selection handoff, no sync engine, no background daemon."

The handoff flow:
1. User selects text in Obsidian
2. Obsidian command palette → "Send to TuitBot" (via community plugin or URI scheme)
3. Selection is sent as a `POST /api/vault/send-selection` with heading context
4. TuitBot composer opens with the selection pre-loaded

This requires a thin Obsidian plugin (out of scope for initial sessions) but the API contract should be designed now.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Hook quality varies by note quality | Low conversion from hooks to good threads | Allow manual hook editing before thread generation; fall back to full-note context |
| Paragraph sub-chunking breaks existing chunk_id references | Existing provenance links become stale | Make paragraph_index additive (default 0 = whole chunk); never re-ID existing chunks |
| Cloud privacy boundary hard to enforce consistently | Privacy violation in cloud mode | Explicit deployment_mode checks in every new API endpoint; test matrix per mode |
| Obsidian plugin review/approval timeline unknown | Delays selection handoff feature | Design API-first; plugin is optional — manual copy-paste works without it |
