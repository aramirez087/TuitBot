# Session 3 Plan: Backend Contracts for Ghostwriter Selections

## Mission

Implement the server-side contracts that receive exact Ghostwriter selections from the Obsidian plugin, resolve them against indexed vault data, preserve provenance through compose and draft workflows, and add the `vault_selections` transient storage layer.

## Current State Assessment

**What exists:**
- `ProvenanceRef` struct with `node_id`, `chunk_id`, `seed_id`, `source_path`, `heading_path`, `snippet` (all optional) — fully implemented in `provenance.rs`
- `vault_provenance_links` table with polymorphic entity linking — fully migrated
- `insert_links_for()`, `get_links_for()`, `copy_links_for()`, `delete_links_for()` — all account-scoped
- Compose and draft endpoints already accept `Option<Vec<ProvenanceRef>>` and thread it through `enqueue_with_provenance_for()`
- Assist endpoints accept `selected_node_ids: Option<Vec<i64>>` and return `vault_citations`
- Nodes keyed by `(source_id, relative_path)` with account scoping
- Chunks indexed by `(node_id, heading_path)` with account scoping
- No `vault_selections` table, no `POST /api/vault/send-selection` endpoint, no `GET /api/vault/selection/{session_id}` endpoint

**What the Obsidian plugin sends** (from `obsidian-plugin-contract.md`):
```json
{
  "vault_name": "string",
  "file_path": "notes/async-rust.md",
  "selected_text": "...",
  "heading_context": "# Title > ## Section",
  "selection_start_line": 45,
  "selection_end_line": 52,
  "note_title": "string | null",
  "frontmatter_tags": ["tag1"] | null
}
```

## Key Design Decisions

### Decision 1: Selection storage as transient table with TTL

**Choose**: New `vault_selections` table with 30-minute TTL, hourly cleanup.
**Rationale**: Matches the contract spec exactly. Selections are ephemeral — the user sends from Obsidian, the dashboard consumes within minutes. No long-lived state needed. Account-scoped via `account_id` column.

### Decision 2: Best-effort block identity resolution

**Choose**: When receiving a selection, attempt to resolve `file_path` → `content_node` → best-matching `content_chunk` by heading_path overlap. Store resolved `node_id` and `chunk_id` on the selection row. If no match (note not indexed yet), store NULL — the selection text itself is the payload.
**Rejected**: Requiring prior indexing (too fragile — user may send from an un-indexed vault or newly created note).
**Rationale**: Resolution is best-effort enrichment, not a gate. The `selected_text` is always the authoritative payload. Resolved IDs enable richer provenance when available.

### Decision 3: Provenance flows through existing `ProvenanceRef` pipeline

**Choose**: When selection is used for compose/draft, construct `ProvenanceRef` from the stored selection (resolved `node_id`, `chunk_id`, `source_path`, `heading_path`, `snippet`). No new provenance types needed.
**Rejected**: New provenance struct for selections.
**Rationale**: `ProvenanceRef` already has all required fields. The existing compose/draft provenance pipeline handles it unchanged.

### Decision 4: Cloud mode privacy gate on GET endpoint

**Choose**: `GET /api/vault/selection/{session_id}` omits `selected_text` when `deployment_mode` is Cloud. Returns metadata only (file_path, heading_context, resolved IDs).
**Rationale**: Privacy invariant from architecture doc. Raw note content must not persist or be retrievable via API in Cloud deployments.

### Decision 5: No new core crate module for selections

**Choose**: Add selection storage functions to a new file `crates/tuitbot-core/src/storage/vault_selections.rs` and re-export from `storage/mod.rs`.
**Rationale**: Follows existing pattern (e.g., `provenance.rs`, `scheduled_content.rs` are storage peers). Keeps tuitbot-server as thin routing layer.

## Implementation Order

### Step 1: Migration — `vault_selections` table

**File**: `crates/tuitbot-core/migrations/20260321000100_vault_selections.sql`

```sql
CREATE TABLE vault_selections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL,
    session_id TEXT NOT NULL,
    vault_name TEXT NOT NULL,
    file_path TEXT NOT NULL,
    selected_text TEXT NOT NULL,
    heading_context TEXT,
    selection_start_line INTEGER NOT NULL DEFAULT 0,
    selection_end_line INTEGER NOT NULL DEFAULT 0,
    note_title TEXT,
    frontmatter_tags TEXT,          -- JSON array of strings, nullable
    resolved_node_id INTEGER REFERENCES content_nodes(id),
    resolved_chunk_id INTEGER REFERENCES content_chunks(id),
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    expires_at TEXT NOT NULL
);

CREATE UNIQUE INDEX idx_vault_selections_session ON vault_selections(session_id);
CREATE INDEX idx_vault_selections_account ON vault_selections(account_id, created_at);
```

**Key details:**
- `session_id` is UUID v4, unique indexed for fast lookup
- `resolved_node_id` and `resolved_chunk_id` are nullable (best-effort resolution)
- `expires_at` = `created_at` + 30 minutes
- `frontmatter_tags` stored as JSON string (matches SQLite pattern used elsewhere)
- Account-scoped index for cleanup queries

**File**: `crates/tuitbot-core/src/storage/watchtower/chunks.rs`

Add function:
```rust
/// Find the best-matching chunk for a heading context within a node.
/// Uses longest prefix match on heading_path.
/// Returns None if no chunks exist or heading_context is None.
pub async fn find_best_chunk_by_heading_for(
    pool: &DbPool,
    account_id: &str,
    node_id: i64,
    heading_context: Option<&str>,
) -> Result<Option<ContentChunk>, StorageError>
```

Strategy:
1. Get all chunks for the node (already have `get_chunks_for_node`)
2. If `heading_context` is None, return the first chunk (document root)
3. Otherwise, find the chunk whose `heading_path` is the longest prefix match against the provided `heading_context`
4. This is a small in-memory operation (typically <20 chunks per note)

**File**: `crates/tuitbot-core/src/context/retrieval.rs`

Add orchestration function:
```rust
/// Resolve a Ghostwriter selection payload to the best available indexed block identity.
/// Returns (Option<node_id>, Option<chunk_id>) — both None if the note isn't indexed.
pub async fn resolve_selection_identity(
    pool: &DbPool,
    account_id: &str,
    file_path: &str,
    heading_context: Option<&str>,
) -> Result<(Option<i64>, Option<i64>), StorageError>
```

Steps:
1. Call `find_node_by_path_for(pool, account_id, file_path)`
2. If node found, call `find_best_chunk_by_heading_for(pool, account_id, node.id, heading_context)`
3. Return `(Some(node.id), chunk.map(|c| c.id))`
4. If no node found, return `(None, None)`

### Step 5: Vault route handlers

**File**: `crates/tuitbot-server/src/routes/vault.rs`

Add two handlers:

#### `POST /api/vault/send-selection`

```rust
#[derive(Deserialize)]
pub struct SendSelectionRequest {
    pub vault_name: String,
    pub file_path: String,
    pub selected_text: String,
    #[serde(default)]
    pub heading_context: Option<String>,
    pub selection_start_line: i64,
    pub selection_end_line: i64,
    #[serde(default)]
    pub note_title: Option<String>,
    #[serde(default)]
    pub frontmatter_tags: Option<Vec<String>>,
}

#[derive(Serialize)]
pub struct SendSelectionResponse {
    pub status: String,          // "received"
    pub session_id: String,      // UUID v4
    pub composer_url: String,    // "/compose?selection={session_id}"
}
```

Handler logic:
1. Validate payload:
   - `vault_name`: 1-255 chars
   - `file_path`: non-empty, ends in `.md`
   - `selected_text`: 1-10000 chars (413 if over 10000)
   - `selection_end_line >= selection_start_line`
2. Rate limit: `count_recent_for(pool, account_id, 60) >= 10` → 429
3. Generate `session_id` = `uuid::Uuid::new_v4().to_string()`
4. Resolve block identity: `resolve_selection_identity(pool, account_id, file_path, heading_context)`
5. Compute `expires_at` = now + 30 minutes (UTC ISO 8601)
6. Insert into `vault_selections`
7. Emit `WsEvent::SelectionReceived { session_id }` via `event_tx`
8. Return 200 with `{ status: "received", session_id, composer_url }`

## Risks and Mitigations

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **Node resolution fails for un-indexed vaults** | Expected | By design — resolution is best-effort. `selected_text` is the authoritative payload. NULL resolved IDs are valid. |
| 2 | **Heading context format mismatch** | Medium | Plugin uses `"# Title > ## Section"` format. Chunks store `heading_path` in same format. Longest-prefix match handles partial matches. Test with exact match, partial match, and no-match cases. |
| 3 | **Race condition: selection created before node indexed** | Low | Node may be indexed after selection is stored. Resolution happens at insert time only. If dashboard needs re-resolution later, it can call `resolve_refs` with the node_id. Acceptable for MVP. |
| 4 | **File path normalization across platforms** | Medium | Plugin sends forward-slash paths (Obsidian normalizes). Server stores forward-slash paths in `content_nodes.relative_path`. No normalization needed. |
| 5 | **Migration ordering collision** | Low | Using timestamp `20260321000100` — check no existing migration at this timestamp. Latest existing is `20260312000101`. No conflict. |
| 6 | **vault.rs exceeds 500-line limit after additions** | Medium | Current file is 568 lines. Adding ~120 lines for two handlers + DTOs would push to ~690. Extract selection handlers to `crates/tuitbot-server/src/routes/vault/selections.rs` as a submodule if needed. Or use the module directory pattern: `vault.rs` → `vault/mod.rs` + `vault/selections.rs`. **Decision**: Convert to module directory only if post-implementation file exceeds 500 lines. |
