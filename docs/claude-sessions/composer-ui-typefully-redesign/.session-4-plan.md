## Problems Identified

### P1: `Cmd+J` wipes drafts without undo (CRITICAL)
`handleInlineAssist()` at ComposeWorkspace:339 calls `api.assist.improve()` and replaces content. In tweet mode, if nothing is selected (`start === end`), it replaces the entire `tweetText`. In thread mode, `ThreadFlowLane.handleInlineAssist()` replaces the entire focused block text when nothing is selected. No undo snapshot is taken. No undo banner is shown.

Charter requirement (G4): "No keyboard shortcut may silently destroy or replace content without an undo path."

### P2: `Cmd+Enter` meaning is context-dependent but tips tray doesn't reflect this
The tips tray shows "Split into thread → ⌘Enter" regardless of whether the user is in tweet or thread mode. In tweet mode, `Cmd+Enter` submits the tweet. The tip is misleading.

### P3: `SHORTCUT_CATALOG` has stale entries
- `cmd+n` label says "New compose" but that's handled at layout level, not in the workspace. It's in the catalog but no workspace handler uses it.
- `cmd+enter` label says "Publish (tweet) / Split below (thread)" — accurate but lengthy.
- `cmd+j` label says "Improve with AI" — should note selection-aware behavior.

### P4: `handleInlineAssist` in ThreadFlowLane has no undo mechanism
The parent's undo system (`undoSnapshot`/`showUndo`) currently only covers "from notes" generation. There's no mechanism for block-level undo after inline assist in thread mode.

## Design Decisions

### D1: Add undo snapshot to `Cmd+J` in tweet mode
Before calling `api.assist.improve()`, snapshot `{ mode, text: tweetText, blocks: [...threadBlocks] }` into `undoSnapshot`. After successful replacement, set `showUndo = true` and start a 10-second timer. Reuse the existing undo banner and `handleUndo()` function.

**Rationale**: The undo mechanism already exists for "from notes" generation. Extending it to cover inline assist requires zero new UI components.

### D2: Add undo snapshot to `Cmd+J` in thread mode
`ThreadFlowLane.handleInlineAssist()` needs to snapshot the focused block's text before calling the API. Introduce an `undoBlockSnapshot: { blockId: string; text: string } | null` state in ThreadFlowLane. Expose an `onundoavailable` callback so ComposeWorkspace can show the undo banner, and an `undoInlineAssist()` method.

**Alternative considered**: Have ComposeWorkspace snapshot all thread blocks before inline assist. This is simpler (reuse existing `undoSnapshot`) but loses the user's cursor position in other blocks. However, since the current "from notes" undo already snapshots all blocks, this level of fidelity is acceptable for consistency.

**Decision**: Use the ComposeWorkspace-level `undoSnapshot` for both tweet and thread mode inline assist. This keeps the undo mechanism in one place.

### D3: `Cmd+J` with no content does nothing
Currently, `handleInlineAssist` has an early return if `!selectedText.trim()` returns empty. This is safe — no content is destroyed. But if `tweetText` is non-empty and nothing is selected, the entire text gets replaced. The fix: always snapshot before replacement when content exists.

### D4: Tips tray becomes mode-aware
The `ComposerTipsTray` receives a `mode` prop. Tips are filtered:
- Tweet mode: show "Publish → ⌘Enter", "Command palette → ⌘K", "AI improve → ⌘J"
- Thread mode: show "Split / add post → ⌘Enter", "Command palette → ⌘K", "AI improve → ⌘J"

This eliminates the misleading "Split into thread" tip when the user is in tweet mode.

### D5: AI button tooltip clarifies selection behavior
HomeComposerHeader AI button tooltip changes from `"Improve with AI (⌘J)"` to `"AI improve selection or post (⌘J)"`. Command palette label for `ai-improve` changes from `"Improve with AI"` to `"AI improve (selection or full post)"`.

### D6: `SHORTCUT_CATALOG` updated for accuracy
- Remove `cmd+n` entry (handled at layout level, not composer-scoped).
- Update `cmd+j` label to `"AI improve (selection or full post)"`.
- `cmd+enter` split into two entries: `"Publish"` (when: `tweet`) and `"Split / add post below"` (when: `thread`).
- Add `cmd+shift+s` as secondary split shortcut in catalog (already implemented in ThreadFlowLane but not in catalog — wait, it IS in the catalog at line 118).

### D7: Command palette action labels updated
- `ai-improve` label: `"Improve with AI"` → `"AI improve (selection or post)"`
- `submit` shortcut display: already mode-aware (`mode === 'thread' ? 'cmd+shift+enter' : 'cmd+enter'`) — correct.
- `split` label: `"Split below"` → `"Split / add post"` to match the behavior (split at cursor, or add new block if cursor is at end).

### Step 2: Add undo snapshot to `handleInlineAssist` in ComposeWorkspace

#### 2a: Modify `handleInlineAssist()` (lines 339–362)

Replace the function body to snapshot before API call:

```typescript
async function handleInlineAssist() {
    if (mode === 'tweet') {
        const textarea = document.querySelector('.compose-input') as HTMLTextAreaElement | null;
        if (!textarea) return;
        const start = textarea.selectionStart;
        const end = textarea.selectionEnd;
        const selectedText = start !== end ? tweetText.slice(start, end) : tweetText;
        if (!selectedText.trim()) return;

        // Snapshot before replacement for undo
        undoSnapshot = { mode, text: tweetText, blocks: [...threadBlocks] };

        assisting = true; submitError = null;
        try {
            const result = await api.assist.improve(selectedText, voiceCue || undefined);
            if (start !== end) {
                tweetText = tweetText.slice(0, start) + result.content + tweetText.slice(end);
            } else {
                tweetText = result.content;
            }
            voicePanelRef?.saveCueToHistory();
            showUndo = true;
            if (undoTimer) clearTimeout(undoTimer);
            undoTimer = setTimeout(() => { showUndo = false; }, 10000);
        } catch (e) {
            submitError = e instanceof Error ? e.message : 'AI assist failed';
            undoSnapshot = null; // Clear snapshot on failure — nothing was changed
        } finally { assisting = false; }
    } else {
        // Thread mode: snapshot all blocks before delegating
        undoSnapshot = { mode, text: tweetText, blocks: [...threadBlocks] };
        try {
            await threadFlowRef?.handleInlineAssist(voiceCue || undefined);
            showUndo = true;
            if (undoTimer) clearTimeout(undoTimer);
            undoTimer = setTimeout(() => { showUndo = false; }, 10000);
        } catch {
            undoSnapshot = null;
        }
    }
}
```

Key changes:
- **Tweet mode**: `undoSnapshot` is set before the API call. On success, `showUndo = true` + 10s timer. On failure, snapshot is cleared.
- **Thread mode**: Snapshot all blocks before delegating to `threadFlowRef.handleInlineAssist()`. Same undo banner behavior.

## Risks and Mitigations

### R1: Undo banner timing conflict
**Risk**: If the user triggers "from notes" and then immediately `Cmd+J`, two undo operations might conflict since they share `undoSnapshot`.

**Mitigation**: The second action overwrites the snapshot. The banner always reflects the most recent destructive action. The 10-second timer resets. This is acceptable — the user expects undo to reverse the last action, not a sequence.

### R2: Thread mode undo restores all blocks, not just the affected one
**Risk**: `undoSnapshot` stores all thread blocks. After a single-block AI improve, undo replaces all blocks, potentially losing edits to other blocks made during the 10-second window.

**Mitigation**: This is the same behavior as the existing "from notes" undo. The 10-second window is short. The snapshot captures state immediately before the AI call. Any edits during the API call (while the block is in assisting state with `pointer-events: none`) are not possible on the affected block. Edits to other blocks during the 10-second banner window are a theoretical concern but acceptable for this scope.

### R3: `Cmd+D` browser bookmark conflict
**Risk**: On some browsers, `Cmd+D` opens the bookmark dialog.

**Mitigation**: The handler calls `e.preventDefault()` in `ThreadFlowLane.handleCardKeydown()` (line 292). This only fires when a thread card textarea is focused. If focus is elsewhere (e.g., URL bar), the browser default fires. This is correct — the shortcut is scoped to the thread editor.

### R4: Tips tray `$derived` instead of static array
**Risk**: Svelte 5 reactivity — using `$derived` for the tips array instead of `const` means the array rebuilds when `mode` changes.

**Mitigation**: Minimal cost. The array has 3 elements. Mode changes are infrequent. No performance concern.

### R5: `ComposerTipsTray` import of `Send` icon
**Risk**: Adding a new Lucide icon import could increase bundle size.

**Mitigation**: `Send` is already imported in multiple other components (`ComposerCanvas`, `HomeComposerHeader`). Tree-shaking ensures only one copy. No bundle impact.
