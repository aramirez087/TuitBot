## Key Design Decisions

### D1: Overlay renders inside ComposeWorkspace, not as a portal

The `ComposerPreviewSurface` uses `position: fixed; inset: 0` to cover the full viewport. It renders as a direct child of the ComposeWorkspace template, after the `{/if}` for embedded/modal branching. This means:
- In **embedded mode**: The overlay is a child of `.embedded-workspace` but covers the entire viewport via `position: fixed`.
- In **modal mode**: The overlay is a child of `ComposerShell` but again covers the entire viewport. It renders _above_ the shell's `z-index: 1000` backdrop by using `z-index: 2000`.

This approach avoids needing a Svelte portal, keeps the component tree simple, and lets the overlay access the same reactive scope as ComposeWorkspace.

### D2: `previewCollapsed` renamed to `previewMode` with inverted semantics

Current: `previewCollapsed = false` (default) means "preview is not collapsed" (but nothing renders since Session 2 removed the inline rail).

New: `previewMode = false` (default) means "preview overlay is not open". `true` means overlay is visible.

The `togglePreview()` function becomes:
```js
function togglePreview() {
    previewMode = !previewMode;
}
```

Every reference to `previewCollapsed` changes:
- `previewCollapsed` → `previewMode` (6 occurrences in ComposeWorkspace)
- The prop `previewVisible={hasPreviewContent && !previewCollapsed}` becomes `previewVisible={previewMode}` — the overlay opens regardless of content state. If there's no content, the overlay shows an empty-state message. This is simpler and matches Typefully's behavior.

Actually — on reflection, gating the preview button's _active_ styling on `previewMode` is correct, but gating whether the button _works_ on content is not desirable. The button should always toggle, and the overlay handles the empty state internally. The `hasPreviewContent` derived remains for potential future use (e.g., disabling the button) but is not used as a gate for `previewMode`.

### D3: Preview button shows Eye when preview is closed, EyeOff never used

Current behavior: Eye icon when preview is "visible", EyeOff when hidden. This is confusing — EyeOff suggests "click to hide" but the button actually opens/closes.

New behavior: The preview button always shows `Eye` icon. Its `.active` class is applied when `previewMode === true`. The aria-label reads "Open preview" / "Close preview". This is clearer.

Wait — examining the current code more carefully: `HomeComposerHeader` shows `Eye` when `previewVisible` is true and `EyeOff` when false. That means "Eye = preview is on, EyeOff = preview is off". This is actually the standard toggle pattern. Keep it as-is: `Eye` when `previewMode` is true, `EyeOff` when false. The icon reflects state, the user clicks to toggle.

### D4: Escape key priority

The `handleKeydown` function in ComposeWorkspace currently checks Escape after all other shortcuts. The preview overlay must be the **first** thing checked when Escape is pressed, since it's the most immediate "modal" layer.

New Escape handling order:
1. `previewMode` is true → close preview overlay (set `previewMode = false`)
2. `showFromNotes` is true → close notes panel
3. `isMobile && inspectorOpen` → close inspector
4. `!embedded && focusMode` → exit focus mode
5. `!embedded` → close modal

### D5: Focus management on open/close

**On open**: When `previewMode` becomes true, focus moves to the overlay's close button. This is standard dialog behavior and works with the `focusTrap` action.

**On close**: Focus returns to the element that triggered the preview. Since the trigger can be:
- The preview button in HomeComposerHeader (embedded)
- The preview button in ComposerHeaderBar (modal)
- The `Cmd+Shift+P` shortcut (focus may be in textarea)

The simplest correct approach: capture `document.activeElement` when opening, restore it when closing. The ComposerPreviewSurface component handles this internally via `onMount`/close.

### D6: No z-index conflict with CommandPalette

The CommandPalette uses `position: absolute` with `z-index: 10` within the compose workspace. The preview overlay uses `position: fixed` with `z-index: 2000`. Since `position: fixed` creates a new stacking context at the viewport level, the overlay always appears above the palette. However, the CommandPalette should not open while preview is active — the `handleKeydown` already returns early if `paletteOpen` is true, but we also need to prevent `Cmd+K` from opening the palette when preview is open.

New guard in `handleKeydown`:
```js
if (previewMode) {
    if (e.key === 'Escape') { e.preventDefault(); previewMode = false; return; }
    if (matchEvent(e, 'cmd+shift+p')) { e.preventDefault(); togglePreview(); return; }
    return; // Block all other shortcuts while preview is open
}
```

This means the preview overlay "swallows" all keyboard shortcuts except Escape and the toggle itself. This is correct — the overlay is a modal dialog.

### D7: Overlay visual design

The overlay uses X's dark theme as reference:
- **Backdrop**: `position: fixed; inset: 0; background: rgba(0, 0, 0, 0.85)` — darker than the compose modal's `0.6` to create visual separation
- **Container**: Centered, `max-width: 600px` (matching X's content column), `width: 100%`, with `16px` horizontal padding
- **Header**: Minimal — "Preview" label left-aligned, close button (X icon) right-aligned, 48px height, border-bottom
- **Scroll area**: `overflow-y: auto`, `max-height: calc(100vh - 80px)`, smooth scrolling
- **Empty state**: Centered message "Nothing to preview — start writing" in muted text
- **Reduced motion**: All transitions respect `prefers-reduced-motion: reduce`

### D8: Thread connectors in preview

The existing `TweetPreview` component already handles thread connectors via the `showConnector` derived (`index < total - 1`). The `ComposerPreviewSurface` passes the sorted, filtered blocks array with correct `index` and `total` values. No new connector logic needed.

### D9: Media preview URLs in overlay

For tweet mode, `attachedMedia` contains `AttachedMedia[]` with `previewUrl` (object URLs) and `path` (server paths). The `TweetPreview` component accepts `mediaPaths` and `localPreviews`. The overlay maps:
- `mediaPaths` = `attachedMedia.map(m => m.path)`
- `localPreviews` = `tweetMediaPreviewMap` (the existing derived `Map<string, string>` from `attachedMedia`)

For thread mode, each block's `media_paths` are passed directly to `TweetPreview`. Thread blocks don't have local previews (media is uploaded inline per-card in ThreadFlowCard) — `localPreviews` is omitted.

### D10: Single vs thread rendering decision

The overlay checks `mode`:
- **Tweet mode**: Renders a single `TweetPreview` if `tweetText.trim()` or `attachedMedia.length > 0`, otherwise shows empty state
- **Thread mode**: Uses `sortedPreviewBlocks` (already derived in ComposeWorkspace — sorted by order, filtered to non-empty). Renders one `TweetPreview` per block with thread connectors. If no blocks have content, shows empty state.

## Risks and Mitigations

### R1: z-index conflict with ComposerShell backdrop

**Risk**: The ComposerShell uses `z-index: 1000` on its `.backdrop`. If the preview overlay's `z-index: 2000` doesn't stack above it, the overlay could be hidden behind the modal.

**Mitigation**: `position: fixed` with `z-index: 2000` always wins over `position: fixed` with `z-index: 1000` in the same stacking context (both are children of the viewport). Verified by reading ComposerShell — it uses `position: fixed; inset: 0; z-index: 1000`. The preview overlay at `z-index: 2000` will correctly layer above it.

**Additional safeguard**: The overlay renders as a sibling to the embedded/modal branch in ComposeWorkspace, not inside the ComposerShell. Since both use `position: fixed`, DOM order doesn't matter for stacking — the higher z-index wins.

### R2: Focus trap interaction with ComposerShell's focus trap

**Risk**: ComposerShell already uses `use:focusTrap`. When the preview overlay opens inside the modal, there could be two active focus traps competing.

**Mitigation**: The preview overlay uses `position: fixed` and renders conditionally. When it's open, its focus trap captures Tab events. The ComposerShell's focus trap remains active but since focus is inside the overlay (which is a descendant of the shell), Tab events are caught by the overlay's trap first (due to event bubbling — the inner trap's keydown handler fires first). This is correct behavior.

Additionally, we're rendering the `ComposerPreviewSurface` outside the `ComposerShell` block (after the `{/if}` on line 583), so it won't be inside the shell's focus trap at all. This cleanly avoids the conflict.

### R3: Object URL revocation when preview is open

**Risk**: If the user has attached media with blob URLs and then somehow the URLs get revoked while the preview is displaying them, images would break.

**Mitigation**: Object URLs are only revoked in `onDestroy` (ComposeWorkspace line 190) and during `handleSubmit` reset (line 257). Both cases unmount the preview overlay first (since ComposeWorkspace unmounts or `previewMode` resets to false). No risk.

### R4: Server-side rendering

**Risk**: The overlay uses `document.activeElement` in `onMount`. SSR would fail.

**Mitigation**: The dashboard is SPA-only (no SSR per CLAUDE.md). The component only renders when `previewMode === true`, which requires user interaction. No SSR concern.

### R5: Mobile viewport coverage

**Risk**: On mobile, the overlay might not fully cover the viewport, especially with mobile browser chrome (address bar, bottom bar).

**Mitigation**: Use `position: fixed; inset: 0` which covers the visual viewport. The container uses `max-height: 100vh` on mobile. Add `overscroll-behavior: contain` on the scroll area to prevent background scrolling.

## State Transition Rules

### Opening the preview

| Trigger | Condition | Effect |
|---------|-----------|--------|
| `Cmd+Shift+P` | Not in palette, not already in preview | `previewMode = true` |
| Preview button (header) | Click | `previewMode = true` |
| Command palette → "Toggle preview" | Palette action | `previewMode = true` |

### Closing the preview

| Trigger | Condition | Effect |
|---------|-----------|--------|
| `Escape` | Preview is open | `previewMode = false` |
| `Cmd+Shift+P` | Preview is open | `previewMode = false` |
| Close button click | — | `previewMode = false` |
| Backdrop click | Click on backdrop, not container | `previewMode = false` |
| Submit | Auto-reset in handleSubmit | `previewMode = false` |
| ComposeWorkspace unmount | — | Component destroyed |

### Draft state during preview

| Aspect | Behavior |
|--------|----------|
| `tweetText` | Read by overlay via props, not modified |
| `threadBlocks` | Read via `sortedPreviewBlocks` derived, not modified |
| `attachedMedia` | Read for `tweetMediaPaths` and `tweetLocalPreviews`, not modified |
| Autosave | Continues to fire (watches `mode`, `tweetText`, `threadBlocks`) — preview doesn't change these |
| Editor textarea | Still mounted behind the overlay, retains cursor position and selection |

### Mode-specific behavior

| Mode | Preview Content | Empty State |
|------|----------------|-------------|
| Tweet, has text | Single TweetPreview with text + media | — |
| Tweet, no text, has media | Single TweetPreview with media only | — |
| Tweet, no text, no media | — | "Nothing to preview — start writing" |
| Thread, has non-empty blocks | N TweetPreviews with connectors | — |
| Thread, all blocks empty | — | "Nothing to preview — start writing" |

### Scheduled drafts

When a schedule time is selected via the inspector, the preview shows the same content. The schedule time is NOT displayed in the preview — the preview only shows what the post will look like on X, not scheduling metadata. This matches Typefully's behavior.
