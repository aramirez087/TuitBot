## Key Design Decisions

### KD1: No focus indicator on borderless textarea
The TweetEditor textarea gets `outline: none` on focus and no replacement focus ring/border. This is acceptable because:
- The textarea is the primary interaction target; focus is implied by cursor presence
- The over-limit state uses `box-shadow` for visual feedback when needed
- Adding a focus ring would reintroduce the "form input" feel we're removing
- The thread card's `.card-writing-area.focused` visual (spine dot color change) provides adequate focus indication in thread mode

### KD2: Spine positioning arithmetic
With `padding-left: 32px` and `lane-spine left: 15px`:
- Spine dot (8px wide) needs `left: -(32 - 15) - 4 = -21px` from card content edge to center on spine
- Between-zone plus needs `margin-left: -(32 - 15) + (14/2 - 1) = -25px` to roughly center on spine (slight offset is acceptable since exact centering would be -24px but -25px matches visual weight better with the border)

### KD3: previewCollapsed state preserved but inactive
The `previewCollapsed` state and `togglePreview()` remain wired to shortcuts and buttons but have no visible effect after the inline preview section is removed. This is a deliberate transitional state:
- Session 3 replaces `previewCollapsed` with `previewMode` and drops in the `PreviewOverlay`
- Users who press `Cmd+Shift+P` in the interim see the preview button toggle its active state (visual feedback that something happened) but no panel appears. This is a minor temporary regression acknowledged in the handoff.

### KD4: Schedule pill removed, not hidden
The schedule pill is deleted from `HomeComposerHeader`, not hidden behind a media query or conditional. The schedule workflow is accessible through:
- Inspector panel (`Cmd+I` → schedule section)
- Command palette (`Cmd+K` → "Schedule")
This reduces the header from 10 interactive elements to 8 (handle, meta, dot, publish pill, preview, AI, inspector, palette).

### KD5: ComposerTipsTray and ComposerPromptCard retained
Both components serve useful purposes (onboarding tips, writing inspiration) and don't compete with the canvas because:
- TipsTray renders between header and canvas, auto-dismisses permanently
- PromptCard renders below the canvas, only visible when composer is empty
- Neither adds chrome to the writing surface itself

## Risks and Mitigations

### R1: Borderless textarea may lose visual affordance
**Risk:** Users might not immediately recognize the textarea as editable without a border.
**Mitigation:** The placeholder text ("What's on your mind?") and cursor change on hover provide sufficient affordance. X's own composer uses the same borderless pattern. Thread cards already use borderless textareas, so this is consistent.

### R2: Spine dot repositioning may be off by a pixel
**Risk:** The spine dot, lane spine, and between-zone plus may not align perfectly after the padding/positioning changes.
**Mitigation:** All three are positioned relative to the same coordinate system (lane padding-left and spine left). The arithmetic is documented in KD2. Visual verification during implementation — if 1px off, adjust `left` values.

### R3: Removing schedule pill may surprise users who rely on it
**Risk:** Users who frequently schedule posts may miss the always-visible schedule pill.
**Mitigation:** The publish pill already shows "Schedule" as its label when a time is selected (`publishLabel` derived). The inspector panel's schedule section is the canonical place for time selection. The command palette provides quick keyboard access.

### R4: Character counter disappearing may cause users to exceed limits without warning
**Risk:** Users writing near the 240–280 range may not notice the counter appearing.
**Mitigation:** The counter appears at 240, giving 40 characters of buffer. The over-limit state (`box-shadow` for tweet mode, `spine-dot.over-limit` for thread mode) provides strong visual feedback. This matches X's own behavior where the ring indicator only appears as you approach the limit.

### R5: svelte-check may flag removed props
**Risk:** Removing `onschedule` from `HomeComposerHeader` and passing it from `ComposeWorkspace` will cause a type error if not done atomically.
**Mitigation:** Step 4 removes both the prop definition and the prop passing in the same step. Run `npm --prefix dashboard run check` after each step to catch issues immediately.
