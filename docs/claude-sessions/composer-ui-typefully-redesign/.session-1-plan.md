## Key Design Decisions and Rationale

### 1. The Compose Surface IS the Writing Canvas

**Decision:** The editor textarea itself must visually resemble the rendered post. The user sees their content styled with the same font, line-height, and visual rhythm as a published tweet while typing.

**Rationale:** Typefully achieves this by making the editor area look like the rendered post. The current TweetEditor uses a generic `<textarea>` with `border: 1px solid var(--color-border)`, `font-size: 14px`, and `padding: 10px 12px` — it looks like a form input, not a tweet. ThreadFlowCard is closer (borderless, transparent background) but still wrapped in heavy chrome (spine dots, separators, char counters visible at all times).

**Current gaps identified from code audit:**
- `TweetEditor.svelte:109-117` — Standard bordered textarea with generic placeholder
- `ThreadFlowCard.svelte:117-128` — Better (transparent bg, no border) but wrapped in card-separator chrome (lines 132-168)
- `ComposeWorkspace.svelte:474-487` — Inline preview section with "Preview"/"Hide" toggle adds visual noise
- `HomeComposerHeader.svelte:63-149` — Dense toolbar with 6+ icon buttons, schedule pill, publish pill
- `ComposerCanvas.svelte:78-84` — `padding: 0 20px 20px` plus `padding-top: 4px` via `:global(:first-child)`

### 2. X-Accurate Preview as Dedicated Full-Screen Mode

**Decision:** Remove the inline `ThreadPreviewRail` from the compose canvas. Replace it with a dedicated full-screen preview overlay that reuses the same `$state` from `ComposeWorkspace` — no draft duplication, no route change.

**Rationale:** The operator rules explicitly state: "the X-accurate preview must live in a dedicated full-screen preview mode that reuses the same draft state instead of an inline side-by-side panel."

**Architecture:**
- New component: `PreviewOverlay.svelte` — full-screen dark overlay showing `TweetPreview` components
- Triggered by `Cmd+Shift+P` or eye icon button (replaces current preview toggle)
- Receives the same `tweetText`, `threadBlocks`, `attachedMedia` as read-only props
- Escape or close button returns to editing — no state cloning, no route change
- The existing `TweetPreview.svelte` and `MediaCropPreview.svelte` are reused inside the overlay

### 3. Shortcut Policy (Cmd+J Analysis)

**Decision:** Document exactly why `Cmd+J` can replace content unexpectedly and propose a fix.

**Trace through code:**
1. `ComposeWorkspace.svelte:298` — `matchEvent(e, 'cmd+j')` → calls `handleInlineAssist()`
2. `ComposeWorkspace.svelte:331-354` — `handleInlineAssist()` in tweet mode:
   - Gets `textarea.selectionStart` and `textarea.selectionEnd`
   - If `start !== end` (text selected): replaces just the selection with AI result
   - **If `start === end` (no selection)**: `selectedText = tweetText` (entire content), then `tweetText = result.content` — **replaces ALL content with AI rewrite**
3. Thread mode: delegates to `ThreadFlowLane.handleInlineAssist()` which does the same per-block (lines 321-346) — if no selection in the focused block, the entire block text is replaced.

**The problem:** When the user presses `Cmd+J` without a text selection, the entire tweet or focused block is silently replaced with an AI-improved version. There is no confirmation, no undo (the `undoSnapshot` mechanism only covers "from notes" generation, not inline assist), and no visual indicator of what's about to happen.

**Proposed fix (for Session 4):**
- If no selection exists, show a brief confirmation tooltip or require a double-press
- Add undo support for inline assist (snapshot before replacement)
- The shortcut policy document will specify: "Cmd+J with selection = replace selection; Cmd+J without selection = improve whole content with undo snapshot"

## Exact Order of Operations

### Step 1: Write `benchmark-notes.md`

This document captures the gap analysis between the current composer and the Typefully target.

**Content structure:**
1. **Current State Inventory** — enumerate all composer surfaces (home embedded, modal, focus mode) with their component trees
2. **Chrome Density Audit** — list every visible UI chrome element (borders, separators, labels, counters) and classify as essential/removable
3. **Visual Rhythm Analysis** — compare current editor typography/spacing to X's actual post rendering
4. **Preview Behavior Audit** — document how `ThreadPreviewRail` works inline and why it doesn't match the target
5. **Typefully Reference Points** — the specific qualities to emulate: clean writing surface, minimal distractions, WYSIWYG feel

**Key findings to document:**

- **TweetEditor chrome:** Bordered textarea (`border: 1px solid var(--color-border)`) with `padding: 10px 12px`. Character counter always visible below. Media attach section always visible. Feels like a form, not a post.
- **ThreadFlowCard chrome:** Spine dots (`.spine-dot`), card separators (`.card-separator`) with char counter + drag handle + merge/remove buttons, between-zone "+" buttons. Total of 24px separator height per card + 20px between-zone = 44px of non-writing chrome between each post.
- **HomeComposerHeader:** 10px vertical padding, 7+ interactive elements (handle, meta, schedule pill, publish pill, preview toggle, AI button, inspector toggle, palette button). The header has more UI than the editor.
- **Inline preview:** `ThreadPreviewRail` renders below the editor with a "Preview" label and "Hide" button. It's a secondary rendering of the same content, not an enhancement of the writing experience.
- **Modal shell:** `ComposerShell` adds `border-radius: 12px`, `box-shadow: 0 16px 48px`, and a header bar with close/focus/inspector/preview buttons. The modal itself is 640px wide (900px with inspector).

### Step 2: Write `charter.md`

**Content structure:**
1. **Problem Statement** — The current composer feels like a form filling exercise, not a writing experience. The editor and preview are separate concerns instead of a unified canvas.
2. **Goals**
   - The compose surface should visually approximate the rendered post while writing
   - X-accurate preview lives in a dedicated full-screen mode, not inline
   - Chrome (toolbars, counters, action buttons) retreats to periphery or appears on demand
   - All shortcuts are safe — no shortcut silently destroys content
   - Thread mode feels as clean as tweet mode
3. **Non-Goals**
   - No backend/Rust changes
   - No new API contracts (preserve `ThreadBlock[]`, `ComposeRequest`, autosave format)
   - No route changes (preview is an overlay, not a page)
   - No Tauri-specific changes
4. **Acceptance Criteria** (per session, referencing the roadmap)
5. **Interaction Model**
   - Writing: minimal chrome, editor styled to approximate post appearance
   - Preview: `Cmd+Shift+P` → full-screen overlay with X-accurate rendering
   - Thread: flowing cards without heavy separators
   - AI: `Cmd+J` always has undo, clear visual feedback
6. **Contracts Preserved** — list each contract and confirmation it won't change

### Step 4: Write `session-01-handoff.md`

**Content structure:**
1. **What Changed** — four documentation files created, no source code modified
2. **Open Questions**
   - Should the tweet editor use a `contenteditable` div instead of a `<textarea>` for richer WYSIWYG? (Recommendation: No — `textarea` is safer for plain text, and the styling improvements alone close most of the gap. A `contenteditable` rewrite is out of scope and high-risk.)
   - Should thread cards show character count only when approaching the limit (e.g., >240 chars)? (Recommendation: Yes — this is a simple derived check and reduces always-visible chrome.)
   - Should the preview overlay include an edit button to jump back with cursor placement? (Recommendation: Yes — Escape closes, but a visible button is good for discoverability.)
3. **Shortcut Policy** (to implement in S4)
   - `Cmd+J` (inline assist): Always snapshots current content for undo before AI call. If no selection, replaces full content. Undo available via `Cmd+Z`-like mechanism (10-second undo banner, matching existing notes undo pattern).
   - `Cmd+Enter`: Tweet mode = submit. Thread mode = split at cursor (current behavior, no change).
   - `Cmd+Shift+Enter`: Always submit (both modes).
   - `Cmd+Shift+P`: Opens full-screen preview overlay (replaces inline toggle).
   - `Cmd+K`: Command palette (unchanged).
   - `Cmd+I`: Inspector toggle (unchanged).
   - `Cmd+Shift+F`: Focus mode (modal only, unchanged).
   - No shortcut should silently destroy content without undo.
4. **Next Session Inputs**
   - Session 2 starts from: `charter.md`, `ui-architecture.md`, `benchmark-notes.md`
   - First task: Restyle `TweetEditor.svelte` to remove borders, increase font size, and match post visual rhythm
   - Second task: Simplify `ThreadFlowCard.svelte` separator chrome
   - Third task: Simplify header bars
   - Validation: `npm --prefix dashboard run check` must pass
