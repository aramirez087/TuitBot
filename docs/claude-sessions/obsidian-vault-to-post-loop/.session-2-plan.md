# Session 02 — Vault Domain Foundation: Implementation Plan

## Mission

Implement the durable account-scoped vault domain model that supports fragments,
provenance, and loop-back without breaking existing content flows.

---

## Key Design Decisions

### D1: Combined Foundation Migration
A single migration (`20260308010000_vault_domain_foundation.sql`) creates the
`content_chunks` table, adds `chunk_id` to `draft_seeds`, and adds provenance
columns to `approval_queue`. Rationale: these three changes form a single
coherent domain shift. Splitting into multiple migrations adds ordering risk
with no benefit since they're shipping in the same session.

### D2: Account-Scoped _for Pattern
Follow the existing `fn_for(pool, account_id, ...) / fn(pool, ...)` pattern
used in `threads.rs` and `approval_queue/queries.rs`. Every new watchtower
function takes `account_id` as a parameter. Existing functions get `_for`
variants; the old signatures become thin wrappers that pass `DEFAULT_ACCOUNT_ID`.
This preserves backward compatibility while making account isolation the default
path for new code.

### D3: No Intermediate `chunked` Status Yet
The `content_nodes.status` lifecycle adds `chunked` as a valid value in the
schema documentation, but the seed worker query change (`pending` → `chunked`)
happens in a later session when the chunker is implemented. This session only
lays the schema and storage foundation — the chunker automation is Session 02
of the original plan (now a future session).

### D4: Provenance Columns Are Nullable
All provenance columns (`source_node_id`, `source_seed_id`, `source_chunks_json`,
`chunk_id`) are nullable with sensible defaults. Existing rows are unaffected.
No backfill needed.

### D5: content_chunks.retrieval_boost Default 1.0
New chunks start with neutral boost (1.0). The [0.1, 5.0] range is enforced in
application code, not in the schema, to keep the migration simple and avoid
CHECK constraints that differ across SQLite versions.

## Files to Create

### 1. `migrations/20260308010000_vault_domain_foundation.sql`
### 2. `crates/tuitbot-core/migrations/20260308010000_vault_domain_foundation.sql`

Identical content, mirrored in both directories per project convention.

```sql
-- Vault Domain Foundation: content_chunks, provenance links, account-scoped chunk_id

-- Fragment table: heading-delimited sections of content_nodes
CREATE TABLE IF NOT EXISTS content_chunks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL DEFAULT '00000000-0000-0000-0000-000000000000',
    node_id INTEGER NOT NULL REFERENCES content_nodes(id) ON DELETE CASCADE,
    heading_path TEXT NOT NULL DEFAULT '',
    chunk_text TEXT NOT NULL,
    chunk_hash TEXT NOT NULL,
    chunk_index INTEGER NOT NULL DEFAULT 0,
    retrieval_boost REAL NOT NULL DEFAULT 1.0,
    status TEXT NOT NULL DEFAULT 'active',
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_content_chunks_node
    ON content_chunks(account_id, node_id);
CREATE INDEX IF NOT EXISTS idx_content_chunks_status
    ON content_chunks(account_id, status);
CREATE INDEX IF NOT EXISTS idx_content_chunks_hash
    ON content_chunks(node_id, chunk_hash);

-- Provenance: link draft_seeds to specific chunks
ALTER TABLE draft_seeds ADD COLUMN chunk_id INTEGER REFERENCES content_chunks(id);

-- Provenance: link approval_queue items back to source material
ALTER TABLE approval_queue ADD COLUMN source_node_id INTEGER REFERENCES content_nodes(id);
ALTER TABLE approval_queue ADD COLUMN source_seed_id INTEGER REFERENCES draft_seeds(id);
ALTER TABLE approval_queue ADD COLUMN source_chunks_json TEXT DEFAULT '[]';
```

**Rationale:**
- `ON DELETE CASCADE` on `node_id`: when a content node is deleted, its chunks
  are cleaned up automatically.
- `idx_content_chunks_hash` on `(node_id, chunk_hash)`: enables efficient dedup
  during re-chunking (check if a chunk already exists for this node).
- `source_chunks_json` defaults to `'[]'` (empty JSON array) — matches the
  `serialize_json_string` pattern already used for `media_paths` and
  `detected_risks`.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Module split breaks imports elsewhere | Medium | Medium | `pub use` re-exports in `mod.rs` maintain identical public API. Grep for all `storage::watchtower::` imports before splitting. |
| Migration conflicts with upstream changes | Low | High | Use unique timestamp `20260308010000`. Check both migration dirs for ordering conflicts. |
| `ApprovalRow` FromRow derivation breaks with new nullable columns | Medium | Medium | SQLite returns NULL for new columns on old rows; `Option<i64>` handles this. `source_chunks_json` uses `COALESCE` in SELECT. |
| File size limits exceeded after adding chunks.rs | Low | Low | Module split ensures each file stays under 500 lines. |
| `ON DELETE CASCADE` on content_chunks causes unexpected data loss | Low | Medium | Only triggers when a `content_node` row is deleted — which only happens via manual DB intervention. Document this behavior. |
| sqlx compile-time query checking fails with new migration | Low | Medium | Project uses runtime queries (string-based), not compile-time checked. No risk. |
