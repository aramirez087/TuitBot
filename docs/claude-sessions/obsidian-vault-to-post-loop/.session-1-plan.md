### What's Missing (The Gaps)

| # | Gap | Impact |
|---|-----|--------|
| G1 | **No chunk-level retrieval** — ingestion stores whole note body; retrieval is seed-based (LLM-extracted hooks), not chunk-based | Seed extraction is lossy; retrieval can't cite specific paragraphs |
| G2 | **No explicit provenance chain** — approval_queue has no FK to content_nodes or draft_seeds | Cannot trace post back to source note |
| G3 | **No draft_seed usage tracking** — `draft_seeds.used_at` never set when a seed feeds a draft | Loop-back metrics can't attribute performance to specific seeds |
| G4 | **No "From Vault" composer surface** — composer sends topic string, gets back LLM output; no way to browse/select vault notes | User can't intentionally ground a post in a specific note |
| G5 | **No vault health UI** — no dashboard view of source_contexts status, content_nodes count, sync state, errors | User can't tell if vault sync is working |
| G6 | **No citation metadata in responses** — assist endpoints return plain text, no source references | User doesn't know which notes influenced the output |
| G7 | **Reply assist has no RAG context** — `compose_reply` in discovery.rs doesn't call `resolve_composer_rag_context` | Discovery replies don't benefit from vault knowledge |
| G8 | **No loop-back state visibility** — loopback writes metadata but dashboard doesn't show it | User can't see which notes produced successful posts |
| G9 | **No Obsidian-specific affordances** — no vault structure awareness (folders, links, tags), no Obsidian URI support | Obsidian users get generic file-sync, not an Obsidian-native experience |
| G10 | **No per-account vault isolation enforcement** — storage queries use DEFAULT_ACCOUNT_ID constant | Multi-account vault data could leak |

### Step 3: Write `product-model.md`

**Path:** `docs/roadmap/obsidian-vault-to-post-loop/product-model.md`

**Content structure — Entity definitions:**

1. **Source** — A registered content source (row in `source_contexts`)
   - Types: `local_fs` (Obsidian vault), `google_drive`, `manual`
   - Per-account isolated via `account_id`
   - Tracks sync state: `status`, `sync_cursor`, `error_message`

2. **Note** — An ingested file (row in `content_nodes`)
   - Belongs to exactly one Source
   - Stores: `relative_path`, `content_hash`, `title`, `body_text`, `front_matter_json`, `tags`
   - Status: `pending` → `chunked` → `processed`
   - Note-level metadata extracted from YAML front-matter

3. **Fragment** (NEW) — A heading-delimited section of a Note (row in `content_chunks`)
   - Schema: `id`, `account_id`, `node_id` (FK), `heading_path` (e.g. "## Key Insight/### Detail"), `chunk_text`, `chunk_hash`, `chunk_index` (ordering), `retrieval_boost` (default 1.0), `status`, `created_at`, `updated_at`
   - Retrieval unit for RAG context
   - `retrieval_boost` updated by loop-back learning

4. **Seed** — An LLM-extracted tweetable hook from a Fragment (row in `draft_seeds`, extended)
   - Existing: `node_id`, `seed_text`, `archetype_suggestion`, `engagement_weight`
   - Add: `chunk_id` (FK to `content_chunks`) for fragment-level provenance

5. **Selected Reference** — A fragment chosen by the RAG retriever for a specific generation call
   - Not persisted as a separate table; captured as a JSON array in the draft/approval record
   - Format: `[{chunk_id, note_title, heading_path, relevance_score}]`

6. **Citation** — User-facing attribution in assist responses
   - Format: `{note_title, heading, chunk_preview, obsidian_uri?}`
   - Returned in API response alongside generated content

7. **Draft Provenance** — Link from a draft to its source material
   - Approval queue gains: `source_node_id`, `source_seed_id`, `source_chunks_json` (selected references)
   - Tracks which vault material influenced the draft

8. **Post Provenance** — Link from a posted tweet to its source material
   - `original_tweets.source_node_id` already exists
   - Add: `posted_tweet_id` in approval_queue already exists
   - Chain: Note → Fragment → Seed → Draft (approval_queue) → Post (original_tweets/posted_tweet_id)

9. **Loop-Back State** — Performance data flowing back to vault
   - When `analytics_loop` measures a post's performance:
     - Look up `approval_queue.source_chunks_json` for that post
     - Update `content_chunks.retrieval_boost` for each referenced chunk
     - Update `draft_seeds.engagement_weight` if a specific seed was used
   - Loopback writer (`loopback.rs`) already writes YAML metadata to source files
   - Extend: include performance score in loopback entry

**Lifecycle diagram:**

```
Note File (Obsidian)
  → [Watchtower ingest] → content_nodes (Note)
  → [Chunker] → content_chunks (Fragments)
  → [Seed Worker] → draft_seeds (Seeds, linked to chunks)
  → [RAG Retriever] → selected references (chunks ranked by retrieval_boost × relevance)
  → [Generator] → draft text + citations
  → [Approval Queue] → approval_queue (with provenance FKs)
  → [Poster] → posted tweet
  → [Analytics Loop] → performance score
  → [Loop-Back] → update chunk retrieval_boost + write YAML metadata
  → (cycle repeats)
```

## Key Design Decisions & Rationale

### D1: Heading-Based Chunking Over Sentence/Paragraph Splitting
**Choice:** Split notes on H1-H3 headings (`^#{1,3} ` regex).
**Why:** Obsidian users organize content under headings. Heading-delimited chunks preserve semantic coherence and map naturally to citation paths ("Note Title > Heading"). Paragraph-based splitting would be too granular (noisy retrieval); whole-note would be too coarse (already the status quo).
**Risk:** Notes without headings become a single chunk. Mitigated by fallback: if no headings, split on double-newline paragraphs with 500-char max.

### D2: Keyword-Based Retrieval Without Embeddings
**Choice:** LIKE-based keyword search on chunk_text, ranked by `keyword_match_count × retrieval_boost`.
**Why:** Avoids adding an embedding model dependency, vector store, or external service. The existing `product_keywords` + `competitor_keywords` + `industry_topics` from BusinessProfile provide good retrieval signals. SQLite FTS5 can be added later as an optimization without changing the API.
**Risk:** Lower precision than semantic search. Mitigated by: small chunk count (max 3), retrieval_boost learning loop, and seed extraction as a parallel retrieval path.

### D3: Provenance as Optional FKs, Not a Separate Table
**Choice:** Add nullable `source_node_id`, `source_seed_id`, `source_chunks_json` columns to `approval_queue`.
**Why:** Simplest change; no new joins needed for basic queries; backward compatible (existing rows have NULL). A separate `draft_provenance` table would add complexity for minimal benefit at current scale.
**Risk:** JSON array for chunks is not fully normalized. Acceptable because it's audit data, not a hot query path.

### D4: Retrieval Boost Instead of Absolute Weight
**Choice:** `retrieval_boost` is a multiplicative factor (default 1.0, max 5.0) applied to keyword-match relevance score.
**Why:** Keeps retrieval_boost independent of the relevance scoring algorithm. Can be tuned without re-indexing. Bounded range prevents runaway amplification.

### D5: Citations as Optional Response Fields
**Choice:** Add `citations?: Citation[]` to assist response types.
**Why:** Backward compatible — existing clients ignore the new field. Structured JSON allows rich UI rendering. Obsidian URI is conditionally populated based on deployment mode.

### D6: No Embedding Infrastructure in This Epic
**Choice:** Defer embedding-based retrieval to a future epic.
**Why:** Keyword matching + retrieval_boost learning provides a meaningful improvement over the current seed-only approach. Embedding infrastructure (model, vector store, indexing pipeline) is a significant complexity increase that should be its own initiative.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Heading-based chunking produces bad results for flat notes | Medium | Low | Fallback to paragraph splitting for notes without headings |
| Keyword retrieval returns irrelevant chunks | Medium | Medium | Bounded to max 3 chunks; retrieval_boost learning self-corrects over time |
| Migration breaks existing installs | Low | High | All new columns are nullable; additive-only migrations |
| Loop-back boost creates filter bubble | Low | Medium | Bounded range (0.1-5.0); recency decay still applied; cold-start seeds remain as fallback |
| File size limits exceeded by chunker module | Low | Low | Chunker is a small module (~150 lines); watchtower/mod.rs is already 857 lines but can be split if needed |
| Account isolation audit reveals deep changes needed | Medium | Medium | Session 06 is dedicated to this; DEFAULT_ACCOUNT_ID replacement is mechanical |
| Obsidian URI scheme varies across platforms | Low | Low | Only generated on Desktop mode; follows documented Obsidian URI spec |
