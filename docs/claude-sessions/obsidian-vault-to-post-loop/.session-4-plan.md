# Session 4 тАФ Fragment Extraction & Indexing: Implementation Plan

## Objective

Extract durable note fragments during ingest so retrieval can cite specific headings or blocks instead of vague note-level seeds. Ingested vault content should have stable, queryable fragments with predictable update semantics.

---

## Key Design Decisions

### 1. No new crate dependency for markdown parsing

The codebase has no `pulldown-cmark` or similar markdown parser. Rather than adding one, we use a lightweight regex-based heading splitter. Rationale:
- Obsidian markdown is standard CommonMark headings (`# `, `## `, etc.)
- We only need to split on headings, not render HTML
- A ~80-line parser avoids a new dependency and keeps the surface area small
- If we later need link/embed resolution, we can add `pulldown-cmark` then

### 2. Fragment identity: `(node_id, heading_path, chunk_hash)`

- `heading_path` is slash-delimited heading hierarchy: `"## Section/### Subsection"`
- `chunk_hash` is SHA-256 of `chunk_text` тАФ used for dedup on re-chunk
- `chunk_index` is 0-based sequential position in the note for ordering
- A fragment is "the same" across re-ingest if it has the same `chunk_hash` for the same `node_id` тАФ the existing `upsert_chunks_for_node` already handles this

### 3. Plain-text fallback for `.txt` files

- If the body has no markdown headings, the entire body becomes a single chunk with `heading_path = ""` (root)
- This handles `.txt` files and flat markdown notes identically
- Preserves backward compat: every ingested node always gets at least one chunk

### 4. Chunking runs inline after ingest, not as a separate worker

- When `ingest_content` returns `Inserted` or `Updated`, immediately chunk the node
- On `Updated`: mark existing chunks stale first, then create new chunks тАФ preserving unchanged chunks via hash dedup
- On `Skipped`: no chunking needed
- This avoids a separate polling loop and ensures chunks are always consistent with content

### 5. Front-matter context attached to every chunk's heading_path

- The root chunk (index 0) carries the note's front-matter context (title, tags) in its `heading_path` as just `""` тАФ the title/tags are already on the parent `ContentNode`
- We do NOT duplicate front-matter into chunk_text; retrieval queries join with `content_nodes` for context

## Files to Create

### `crates/tuitbot-core/src/automation/watchtower/chunker.rs` (NEW, ~150 lines)

Core fragment extraction module.

**Public API:**

```rust
/// A parsed fragment from a note.
pub struct Fragment {
    pub heading_path: String,  // e.g. "## Intro/### Background"
    pub text: String,          // body text under this heading
    pub index: usize,          // 0-based position
}

/// Extract fragments from markdown/plain-text body.
/// Does NOT include front-matter (caller strips it before calling).
pub fn extract_fragments(body: &str) -> Vec<Fragment>

/// Chunk a content node: extract fragments, mark stale, upsert new chunks.
/// Called after ingest on Inserted/Updated results.
pub async fn chunk_node(
    pool: &DbPool,
    account_id: &str,
    node: &ContentNode,
) -> Result<Vec<i64>, ChunkerError>
```

**Extraction algorithm:**
1. Scan body line by line
2. Detect heading lines via regex: `^(#{1,6})\s+(.+)$`
3. Maintain a heading stack for hierarchy tracking
4. On each new heading, flush accumulated text as a fragment
5. After all lines, flush the final fragment
6. If no headings found, emit one root fragment with `heading_path = ""`
7. Skip fragments that are empty or whitespace-only

**Heading path construction:**
- Maintain a stack of `(level, text)` tuples
- When encountering `## Foo`, pop all entries with level >= 2, push `(2, "## Foo")`
- `heading_path` = stack entries joined with `/`
- Examples:
  - `# Title` тЖТ `"# Title"`
  - `## Section` under `# Title` тЖТ `"# Title/## Section"`
  - `### Sub` under `## Section` тЖТ `"# Title/## Section/### Sub"`
  - Back to `## Other` тЖТ pops `### Sub`, becomes `"# Title/## Other"`

**`chunk_node` flow:**
1. Call `extract_fragments(&node.body_text)`
2. Compute SHA-256 hash for each fragment's text
3. Call `store::mark_chunks_stale(pool, account_id, node.id)` (idempotent on first chunk)
4. Build `Vec<NewChunk>` from fragments
5. Call `store::upsert_chunks_for_node(pool, account_id, node.id, &chunks)`
6. Call `store::mark_node_chunked(pool, account_id, node.id)`
7. Return chunk IDs

**Error type:**
```rust
#[derive(Debug, thiserror::Error)]
pub enum ChunkerError {
    #[error("storage error: {0}")]
    Storage(#[from] crate::error::StorageError),
}
```

## Files to Modify

### `crates/tuitbot-core/src/automation/watchtower/mod.rs`

**Changes:**
1. Add `pub mod chunker;` declaration after `pub mod loopback;`
2. Modify `ingest_content()` to accept an optional `account_id` parameter (or use default) and call `chunker::chunk_node()` after successful insert/update
3. Alternative (simpler): Add a new `pub async fn ingest_and_chunk()` wrapper that calls `ingest_content` + `chunk_node`, and update callers

**Preferred approach тАФ modify `ingest_content` return type:**

Rather than changing `ingest_content`'s signature (which would break many callers), add a new function:

```rust
/// Ingest content and extract fragments in one step.
/// Returns (UpsertResult, Option<Vec<i64>>) where the Vec is chunk IDs on Inserted/Updated.
pub async fn ingest_content_and_chunk(
    pool: &DbPool,
    source_id: i64,
    provider_id: &str,
    content: &str,
    force: bool,
) -> Result<(store::UpsertResult, Option<Vec<i64>>), WatchtowerError>
```

This function:
1. Calls existing `ingest_content()`
2. On `Inserted` or `Updated`, looks up the node by `(source_id, provider_id)` to get `node_id` and `account_id`
3. Calls `chunker::chunk_node(pool, &account_id, &node)`
4. Returns both the upsert result and chunk IDs

**Integration points in WatchtowerLoop:**
- `scan_directory` тЖТ replace `ingest_file` calls with `ingest_file_and_chunk` (new wrapper)
- `handle_event` тЖТ same replacement
- `poll_remote_sources` тЖТ replace `ingest_content` with `ingest_content_and_chunk`

Actually, the cleanest approach is:

**After `ingest_content` returns, add a post-ingest chunk step.** We need to look up the node to get its `id` and `body_text`. But `ingest_content` doesn't return the node ID.

**Better approach тАФ return node_id from upsert:**

Looking at `upsert_content_node_for` in `nodes.rs`, it can easily be modified to return the node ID on Inserted/Updated. But this changes a public API.

**Simplest approach тАФ chunk in a separate pass:**

Add a function `chunk_pending_nodes(pool, account_id, limit)` that:
1. Gets pending nodes via `get_pending_content_nodes_for`
2. Chunks each one
3. Marks as `chunked`

Then call this after initial scan and after each event batch in the watchtower loop. This is simple, idempotent, and doesn't require changing `ingest_content`.

**Decision: Use the separate-pass approach.** It's additive, doesn't change existing ingest signatures, and naturally batches work. The watchtower already has natural "after scan" hooks.

### Updated plan for `mod.rs` changes:

1. Add `pub mod chunker;`
2. Add a new private method on `WatchtowerLoop`:
   ```rust
   async fn chunk_pending_nodes(&self) {
       // Get pending nodes, chunk each, mark as chunked
   }
   ```
3. Call `self.chunk_pending_nodes()` after:
   - Initial scan of local directories (after the scan loop, line ~461)
   - Initial poll of remote sources (after `poll_remote_sources`, line ~466)
   - Each fallback scan (after the scan loop in the select, line ~567)
   - Each remote poll (after `poll_remote_sources` in the select, line ~573)
   - Each event batch (after processing all events, line ~593)

This keeps ingest and chunking decoupled but ensures chunks are created promptly.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Regex heading parser misses edge cases (e.g., headings inside code blocks) | Medium | Low | Document known limitations; code-block-aware parsing can be added later without schema changes |
| Large notes produce many small chunks | Low | Low | No hard limit needed now; future session can add min-size thresholds |
| `chunk_pending_nodes` re-processes nodes if called twice before status update | Low | Low | `upsert_chunks_for_node` is idempotent by hash; double-processing is harmless |
| watchtower mod.rs exceeds 500-line limit | Low | Medium | Current ~934 lines already violates this; `chunk_pending_nodes` adds ~15 lines. Accept for now; extract in a future refactor session |
| Chunking slows down ingest for large vaults | Low | Medium | Chunking is O(n) per note body; SHA-256 is fast. No concern until 10k+ notes |
