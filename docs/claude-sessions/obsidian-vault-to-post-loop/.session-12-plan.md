# Session 12 Plan — Desktop Obsidian Affordances

## Goal

Add desktop-specific affordances that let local-vault users move seamlessly between Tuitbot and their Obsidian notes. This includes "Open in Obsidian" from citations, clearer vault labels, and platform guards so web/cloud builds remain safe.

## Key Design Decisions

### 1. Obsidian URI Deep Linking — Tauri command approach

**Decision:** Add a new Tauri command `open_external_url` in `lib.rs` that opens arbitrary URIs via `std::process::Command::new("open")` (macOS) / `opener::open` pattern.

**Rationale:** No `tauri-plugin-opener` or `tauri-plugin-shell` is installed. Adding a plugin dependency for one function is overkill. A dedicated Tauri command gives us control over URI validation (only allow `obsidian://` and `file://` schemes). The frontend dynamically imports `@tauri-apps/api/core` + `invoke` — this pattern is already established in `CredentialCard.svelte` for `open_oauth_window`.

**URI format:** `obsidian://open?vault={vault_name}&file={relative_path}` where:
- `vault_name` = last path component of the configured source path (e.g., `/Users/alice/notes/marketing` → `marketing`)
- `relative_path` = note's `relative_path` from the API, URL-encoded

**Fallback:** When not in Tauri (web/self-host), the "Open" button is hidden. When in desktop mode but vault name is unknown, show the expanded snippet instead of a broken link.

### 2. Vault path exposure — Server-side `path` field on source status

**Decision:** Add an optional `path` field to `VaultSourceStatusItem` in the vault sources API response, populated from `config_json` for `local_fs` sources. This lets the frontend derive `vault_name` without needing to read the settings draft.

**Rationale:** The settings draft has the path, but the citation chips appear in the composer which doesn't load settings. The vault sources API already returns per-source metadata and is loaded wherever citations appear. Adding `path` is a single field addition, not a new endpoint.

### 3. CitationChips enhancement — "Open in Obsidian" button

**Decision:** Add an external-link icon button to each citation chip that fires the Obsidian URI when:
- `deploymentMode === 'desktop'`
- Source type is `local_fs`
- Vault path is known

The chip already has expand/collapse. The open-in-Obsidian action is a separate icon button (not overriding expand), placed at the trailing edge.

### 4. Platform guards

**Decision:** All desktop-only behavior uses dynamic `import('@tauri-apps/api/core')` with try/catch fallback (established pattern). The `deploymentMode` store from `runtime.ts` gates UI rendering. No compile-time guards needed — the same bundle runs everywhere, with runtime feature detection.

## Implementation Order

### Step 1: Rust — Add `open_external_url` Tauri command

**File:** `dashboard/src-tauri/src/lib.rs`

Add a new command:
```rust
#[tauri::command]
fn open_external_url(url: String) -> Result<(), String> {
    // Validate scheme: only allow obsidian:// and file://
    let parsed = url::Url::parse(&url).map_err(|e| format!("Invalid URL: {e}"))?;
    match parsed.scheme() {
        "obsidian" | "file" => {}
        scheme => return Err(format!("Scheme '{scheme}' is not allowed")),
    }
    #[cfg(target_os = "macos")]
    {
        std::process::Command::new("open")
            .arg(&url)
            .spawn()
            .map_err(|e| format!("Failed to open URL: {e}"))?;
    }
    #[cfg(target_os = "windows")]
    {
        std::process::Command::new("cmd")
            .args(["/C", "start", "", &url])
            .spawn()
            .map_err(|e| format!("Failed to open URL: {e}"))?;
    }
    #[cfg(target_os = "linux")]
    {
        std::process::Command::new("xdg-open")
            .arg(&url)
            .spawn()
            .map_err(|e| format!("Failed to open URL: {e}"))?;
    }
    Ok(())
}
```

Register in `invoke_handler`:
```rust
.invoke_handler(tauri::generate_handler![get_api_token, open_oauth_window, open_external_url])
```

### Step 4: TypeScript — Create `obsidianUri.ts` utility

```typescript
/**
 * Build an obsidian:// URI to open a note in the Obsidian app.
 * Returns null if vault name cannot be derived.
 */
export function buildObsidianUri(vaultPath: string, relativePath: string): string | null {
    const vaultName = vaultPath.split('/').filter(Boolean).pop();
    if (!vaultName) return null;
    const encodedVault = encodeURIComponent(vaultName);
    const encodedFile = encodeURIComponent(relativePath.replace(/\.md$/, ''));
    return `obsidian://open?vault=${encodedVault}&file=${encodedFile}`;
}

/**
 * Open a URI via Tauri's open_external_url command.
 * No-ops silently outside Tauri (web/self-host).
 */
export async function openExternalUrl(url: string): Promise<boolean> {
    try {
        const { invoke } = await import('@tauri-apps/api/core');
        await invoke('open_external_url', { url });
        return true;
    } catch {
        return false;
    }
}
```

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Obsidian not installed — URI opens nothing | Medium | Low | OS will show "no handler" dialog. Not our concern — document the requirement. |
| Vault name mismatch (directory name ≠ Obsidian vault name) | Low | Medium | Document that directory name must match. Most Obsidian users have this naturally. Future: could add vault name override in settings. |
| `config_json` parse failure for path extraction | Very Low | Low | `serde_json::from_str` returns `None` on failure → path is `None` → open button hidden. |
| `open` command injection via URL | Low | High | Validate URL scheme strictly — only `obsidian://` and `file://` allowed. URL is parsed by `url::Url` before being passed to OS. |
| Extra API call in ComposeWorkspace for vault path | Low | Low | Only fires in desktop mode. Single lightweight GET, cached in component state. |
| ComposeWorkspace already 947 lines (limit 400) | Known | Low | This session adds ~10 lines. The file is already over limit — splitting is tracked as future tech debt per Session 11 handoff. |
