# Session 7 Plan — Provenance-Driven Loop-Back

## Mission

Wire the persisted provenance chain into real, idempotent loop-back writes: when a tweet or thread is posted, trace provenance links back to the source note and write publishing metadata into its YAML front-matter. Keep Google Drive and manual sources explicitly read-only. Prevent self-induced re-ingest storms. Add integration tests that exercise the full provenance-to-loopback flow.

---

## Current State Analysis

### What exists
- **`loopback.rs`** — Pure file-level helpers: `write_metadata_to_file()`, `split_front_matter()`, `parse_tuitbot_metadata()`. Writes `LoopBackEntry` (tweet_id, url, published_at, content_type) into YAML front-matter idempotently.
- **`approval_poster.rs`** — `propagate_provenance()` copies provenance links from `approval_queue` to `original_tweet` entity in DB. Does NOT trigger any file writes.
- **Watchtower `CooldownSet`** — Already prevents re-ingest of recently-written files (5s TTL). But it's scoped to the watchtower event loop — the approval poster has no access to it.
- **Provenance chain** — `vault_provenance_links` stores `node_id`, `source_path`, etc. for each entity.
- **`ContentNode`** — Has `source_id` + `relative_path`. Source's `config_json` stores `{"path": "~/notes"}` for local_fs.

### The gap
After posting, `propagate_provenance()` persists links in DB but nothing calls `write_metadata_to_file()`. The loopback helpers exist but are disconnected from the posting pipeline. There's no source-type awareness (local_fs vs google_drive vs manual) at the loopback layer.

---

## Design Decisions

### D1: Loopback writes happen inside `propagate_provenance()` in `approval_poster.rs`
**Rationale:** This is the single point where we know a post succeeded and have the tweet_id. Adding loopback here keeps it colocated with provenance propagation. The alternative (a separate background worker) adds complexity for no benefit since we already have the context.

### D2: Resolve file path via DB lookup: node_id → ContentNode → SourceContext → base_path + relative_path
**Rationale:** The provenance links carry `node_id`. We look up the `ContentNode` to get `source_id` + `relative_path`, then the `SourceContext` to get `source_type` and `config_json.path` (the vault base directory). This is the only reliable way to reconstruct the file path.

### D3: Source-type gate at write time — local_fs writable, everything else no-op
**Rationale:** Google Drive files can't be written to via local filesystem. Manual sources have no backing file. The gate checks `source_type == "local_fs"` before attempting any write. Log a debug message for skipped sources to aid troubleshooting.

### D4: Use a DB-level "loopback_written_at" timestamp rather than in-process CooldownSet
**Rationale:** The watchtower's `CooldownSet` is an in-process `HashMap` inside the watchtower event loop. The approval poster runs in a different task. Sharing state via `Arc<Mutex<CooldownSet>>` would couple the two loops. Instead, we use the existing content_nodes table: after writing loopback, update `updated_at` on the node. The watchtower's normal hash-based dedup (`content_hash` comparison) already handles this — when the file is re-ingested after a loopback write, the hash will differ, triggering an `Updated` result that resets status to `pending`. This is harmless and desired (the node's front-matter did change). The existing 5s `CooldownSet` in the watchtower already suppresses the filesystem event from the write. No new mechanism needed.

### D5: Expand `LoopBackEntry` with optional `status` and `thread_url` fields
**Rationale:** The session asks for thread status and URLs. Adding `status: Option<String>` (e.g., "posted", "deleted") and making the struct extensible with `#[serde(default)]` on new fields ensures backward compatibility. Future analytics updates can add fields without destroying existing front-matter.

### D6: New public function `execute_loopback()` in `loopback.rs`
**Rationale:** Encapsulates the full DB-lookup-to-file-write flow. Takes `pool`, `account_id`, `node_id`, `tweet_id`, `url`, `content_type`. Returns `Result<LoopBackResult>` indicating whether the write happened, was skipped (idempotent), or was a no-op (non-local source).

## Files to Create/Modify

### 1. `crates/tuitbot-core/src/automation/watchtower/loopback.rs` (MODIFY)

**Changes:**
- Add `status: Option<String>` and `thread_url: Option<String>` to `LoopBackEntry` with `#[serde(default, skip_serializing_if = "Option::is_none")]`.
- Add `LoopBackResult` enum: `Written`, `AlreadyPresent`, `SourceNotWritable(String)`, `NodeNotFound`, `FileNotFound`.
- Add `pub async fn execute_loopback(pool, account_id, node_id, tweet_id, url, content_type) -> Result<LoopBackResult>`:
  1. `get_content_node(pool, node_id)` → get `source_id`, `relative_path`
  2. `get_source_context(pool, source_id)` → get `source_type`, `config_json`
  3. If `source_type != "local_fs"` → return `SourceNotWritable`
  4. Parse `config_json` to extract `path` field → expand tilde → join with `relative_path`
  5. Build `LoopBackEntry` and call `write_metadata_to_file()`
  6. Return `Written` or `AlreadyPresent`
- Add tests for `execute_loopback` with local_fs source (happy path), google_drive source (no-op), missing node.

**Estimated lines:** ~80 new lines (function + result enum + tests).

### 2. `crates/tuitbot-core/src/automation/approval_poster.rs` (MODIFY)

**Changes:**
- After `propagate_provenance()` succeeds and creates the original_tweet, call loopback for each distinct `node_id` in the provenance links.
- Add `async fn execute_loopback_for_provenance(pool, item, tweet_id)`:
  1. Get provenance links for the approval_queue entity
  2. Collect unique `node_id` values (deduplicate — multiple chunks from same note)
  3. For each node_id, call `loopback::execute_loopback()` with the tweet URL constructed as `https://x.com/i/status/{tweet_id}`
  4. Log results (Written/AlreadyPresent/SourceNotWritable) at appropriate levels
- Call this new function right after `propagate_provenance()`.

**Estimated lines:** ~40 new lines.

  2. Create provenance links referencing both node_ids
  3. Execute loopback for each
  4. Verify both files get metadata written

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| File write fails (permissions, disk full) | Low | Low | `execute_loopback` returns `Result` with IO error; approval poster logs warning but doesn't fail the post |
| `config_json` format varies or is empty | Low | Medium | Parse as `serde_json::Value`, handle missing `path` key gracefully (return `SourceNotWritable`) |
| Race between loopback write and watchtower re-ingest | Low | Low | Watchtower's existing 5s `CooldownSet` suppresses the filesystem event. Even if it misses, re-ingest just causes an `Updated` upsert which is harmless |
| `expand_tilde` doesn't handle all path formats | Very Low | Low | Already used elsewhere in the codebase (media upload); well-tested |
| Multiple provenance links point to same node | Medium | None | Deduplicate node_ids before calling loopback; `write_metadata_to_file` is idempotent anyway |
| Thread posts have multiple tweet_ids | Medium | Low | Each tweet in a thread gets its own `LoopBackEntry`. The current approval_poster posts individual items, not full threads atomically — handle current behavior first |
| `DEFAULT_ACCOUNT_ID` used for provenance queries | Known | Medium | Same limitation as Session 6. Multi-account will be addressed later |
