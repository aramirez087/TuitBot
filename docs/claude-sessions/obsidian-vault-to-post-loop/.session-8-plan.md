# Session 8 Plan — Vault API Surface for Dashboard

## Mission

Add server APIs for vault source status, note/fragment search, note detail/preview, and selected-reference retrieval. Extend generation-entry APIs to accept optional selected vault refs. Return citation and provenance metadata in generation responses. Keep error behavior privacy-safe.

---

## Design Decisions

### 1. New `vault` route module instead of extending `sources.rs` or `assist.rs`

The existing `sources.rs` is a status/reindex pair without account scoping. The vault APIs need account-scoped note search, fragment preview, and retrieval — different enough to warrant a dedicated `routes/vault.rs` module. This keeps `assist.rs` focused on generation and `sources.rs` focused on watchtower admin.

### 2. Account scoping via `AccountContext` extractor

All new vault endpoints use the existing `AccountContext` extractor for account isolation. The storage layer already provides `get_source_contexts_for(pool, account_id)`, `search_chunks_with_context(pool, account_id, ...)`, and `get_chunks_for_node(pool, account_id, ...)` — all account-scoped.

### 3. Privacy-safe responses

- Note search returns only `node_id`, `title`, `relative_path`, `tags`, `status`, `chunk_count` — **no `body_text`**.
- Fragment search returns `chunk_id`, `node_id`, `heading_path`, `snippet` (truncated to 120 chars), `retrieval_boost` — **no full `chunk_text`**.
- Note detail/preview returns title, path, tags, headings list, and chunk summaries — **not the raw body**.
- The selected-reference retrieval endpoint returns full `VaultCitation` records (same as assist responses already do).

### 4. Selected vault refs in assist requests — backward compatible

Add `selected_node_ids: Option<Vec<i64>>` to `AssistTweetRequest`, `AssistThreadRequest`, and `AssistImproveRequest` with `#[serde(default)]`. The existing `resolve_composer_rag_context` gains a `selected_node_ids` parameter that passes through to `retrieve_vault_fragments`. Existing callers that omit the field get identical behavior.

### 5. TypeScript types and client methods

Add `VaultCitation`, `ProvenanceRef`, `VaultSourceStatus`, `VaultNoteSearchResult`, `VaultFragmentResult`, `VaultNoteDetail` types to `types.ts`. Add `api.vault.*` methods to `client.ts`. Update assist response types to include `vault_citations`.

### Step 2: Server vault route module

**File:** `crates/tuitbot-server/src/routes/vault.rs` (new, ~250 lines)

Endpoints:

#### `GET /api/vault/sources` — Vault source status
- Account-scoped via `AccountContext`
- Returns sources with node counts per source
- Response: `{ sources: VaultSourceStatusItem[] }`

```rust
#[derive(Serialize)]
pub struct VaultSourceStatusItem {
    pub id: i64,
    pub source_type: String,
    pub status: String,
    pub error_message: Option<String>,
    pub node_count: i64,
    pub updated_at: String,
}
```

#### `GET /api/vault/notes?q=&source_id=&limit=` — Note search
- Account-scoped
- `q` parameter: LIKE search on title + relative_path
- `source_id` optional filter
- Returns note summaries without body text
- Response: `{ notes: VaultNoteItem[] }`

```rust
#[derive(Serialize)]
pub struct VaultNoteItem {
    pub node_id: i64,
    pub source_id: i64,
    pub title: Option<String>,
    pub relative_path: String,
    pub tags: Option<String>,
    pub status: String,
    pub chunk_count: i64,
    pub updated_at: String,
}
```

#### `GET /api/vault/notes/{id}` — Note detail/preview
- Account-scoped, returns note metadata + chunk headings
- No raw body text — only heading_path list and snippet per chunk
- Response: `VaultNoteDetail`

```rust
#[derive(Serialize)]
pub struct VaultNoteDetail {
    pub node_id: i64,
    pub source_id: i64,
    pub title: Option<String>,
    pub relative_path: String,
    pub tags: Option<String>,
    pub status: String,
    pub ingested_at: String,
    pub updated_at: String,
    pub chunks: Vec<VaultChunkSummary>,
}

#[derive(Serialize)]
pub struct VaultChunkSummary {
    pub chunk_id: i64,
    pub heading_path: String,
    pub snippet: String,          // truncated to 120 chars
    pub retrieval_boost: f64,
}
```

#### `GET /api/vault/search?q=&limit=` — Fragment search
- Account-scoped keyword search across chunks
- Returns `VaultCitation[]` — same shape as assist responses
- Uses existing `search_chunks_with_context`
- Response: `{ fragments: VaultCitation[] }`

#### `POST /api/vault/resolve-refs` — Selected-reference retrieval
- Account-scoped
- Body: `{ node_ids: i64[] }`
- Returns `VaultCitation[]` for the selected notes' chunks
- Uses existing `get_chunks_for_nodes_with_context`
- Response: `{ citations: VaultCitation[] }`

### Step 6: Dashboard API client

**File:** `dashboard/src/lib/api/client.ts`

1. Add vault methods:
```typescript
vault: {
    sources: () => request<{ sources: VaultSourceStatus[] }>('/api/vault/sources'),
    searchNotes: (params: { q?: string; source_id?: number; limit?: number } = {}) => {
        const query = new URLSearchParams();
        if (params.q) query.set('q', params.q);
        if (params.source_id) query.set('source_id', params.source_id.toString());
        if (params.limit) query.set('limit', params.limit.toString());
        const qs = query.toString();
        return request<{ notes: VaultNoteItem[] }>(`/api/vault/notes${qs ? `?${qs}` : ''}`);
    },
    noteDetail: (id: number) => request<VaultNoteDetail>(`/api/vault/notes/${id}`),
    searchFragments: (params: { q: string; limit?: number }) => {
        const query = new URLSearchParams({ q: params.q });
        if (params.limit) query.set('limit', params.limit.toString());
        return request<{ fragments: VaultCitation[] }>(`/api/vault/search?${query.toString()}`);
    },
    resolveRefs: (nodeIds: number[]) =>
        request<{ citations: VaultCitation[] }>('/api/vault/resolve-refs', {
            method: 'POST',
            body: JSON.stringify({ node_ids: nodeIds })
        })
},
```

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `build_draft_context` signature change breaks callers | Medium | Low | Only two call sites: `resolve_composer_rag_context` in assist.rs and tests in winning_dna.rs. Both updated in the same commit. |
| Note search returns too many results for large vaults | Low | Low | Default limit of 20, max limit of 100. Uses LIKE search (not full-text) so results are bounded. |
| Full chunk text leaked through note detail | Low | High | Only return snippets (120 chars) in VaultChunkSummary. Never serialize body_text to API responses. |
| Fragment search slow for large chunk tables | Low | Low | Existing `search_chunks_with_context` uses LIMIT. Same performance profile as assist endpoints. |
| Assist request shape change breaks existing clients | Low | Medium | `selected_node_ids` uses `#[serde(default)]` — existing clients that omit the field are unaffected. |
| `vault.rs` exceeds 500 line limit | Medium | Low | Response structs are compact. Handlers are thin (delegate to storage). Expected ~250 lines. If needed, split into `vault/mod.rs` + submodules. |
