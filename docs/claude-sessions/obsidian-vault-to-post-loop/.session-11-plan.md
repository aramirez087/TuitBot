# Session 11 Implementation Plan — Onboarding, Settings & Vault Health

## Mission

Make the onboarding and settings surfaces truthful about the vault runtime contract, add vault health visibility, and remove misleading optional-field UX. The vault should feel like a trustworthy product area.

## Current State Analysis

### Problems to Fix

1. **Onboarding `SourcesStep.svelte` (line 62)**: Header says "Content Source (Optional)" — but the vault is not truly optional for the core note-to-post loop. Copy says "index your content and use it for smarter replies and tweets" which is generic and doesn't describe the actual vault behavior (fragment indexing, RAG retrieval, provenance tracking, loop-back).

2. **Onboarding copy mismatch**: "Watch for changes" / "Re-index automatically when files change" doesn't mention the `change_detection` field semantics or that Google Drive always polls. "Loop back" says "Write performance metadata into file frontmatter" — too vague about what actually gets written.

3. **Settings `ContentSourcesSection.svelte` (line 94)**: Section says "Connect a content source for the Watchtower to index" — still generic "content source" language. No health status visibility at all.

4. **Google Drive Folder ID labelled "(optional)"** in both onboarding (line 116) and settings (line 156): Per `source-lifecycle.md` validation rules, enabled `google_drive` sources must have a non-empty `folder_id`. The UX says optional but runtime requires it.

5. **No vault health** anywhere in settings — no sync status, no note/fragment counts, no error messages, no reindex button.

6. **`runtime.ts`**: No `manual_local_path` in `DESKTOP_DEFAULTS` capabilities type (it does exist in the actual `DeploymentCapabilities` interface in types.ts but is missing from the defaults object — actually it IS there at line 13). Looks fine.

7. **`onboarding.ts`**: `vault_loop_back` defaults to `true` — but the handoff says loop-back is not yet fully wired (thread-level loop-back is future). Need to be truthful about current state.

- `POST /api/sources/{id}/reindex` exists server-side but has no frontend client method
- `VaultSourceStatus` type already defined in `types.ts`
- All vault API endpoints are account-scoped and working
- `source-lifecycle.md` documents the state machine clearly

## Key Design Decisions

### 1. "Knowledge Vault" not "Content Sources"

The old "Content Sources" framing treats the vault as a generic file sync. The vault is a specific product area: notes → fragments → RAG context → generation → provenance → (future) loop-back. Renaming to "Knowledge Vault" reflects this.

**Rationale:** Users should understand that connecting a vault enables a specific workflow, not just file indexing. The copy should describe what they gain (context-grounded tweets, cited sources, performance tracking) not just what happens technically (indexing).

### 2. Google Drive Folder ID: Required, Not Optional

`source-lifecycle.md` line 79 states: "Enabled `google_drive` sources must have a non-empty `folder_id`." But both onboarding and settings label it "(optional)" and the hint says "Leave blank to index your entire Drive."

**Decision:** Remove "(optional)" from the label. Change hint to: "Required. Find the folder ID in your Google Drive URL after /folders/." This matches the runtime validation contract.

**Risk:** Some users may have working configs with no folder_id (if the validation was added after their setup). Mitigation: The settings UI will still accept empty values (no frontend validation gate that blocks save) but will show a warning notice: "A folder ID is required for Google Drive syncing to work."

### 3. Loop-back Default: Off

Session 10 handoff lists "Thread-level loop-back" as future work. The provenance is tracked end-to-end, but actual file write-back is not yet complete. Defaulting to `true` in onboarding is misleading.

**Decision:** Default `vault_loop_back` to `false`. Toggle hint explains: "Writes performance data back into your notes. Currently tracks which notes were used — file write-back coming soon."

### 4. Inline Health vs Separate Page

The UX blueprint envisions both inline health in settings AND a full `/vault` page. This session focuses on inline health in settings. The full vault page is a larger effort (note browser, fragment detail, seed list) that deserves its own session.

**Decision:** Add inline health summary to `ContentSourcesSection.svelte`. Do NOT create the `/vault` page or add nav items in this session. That's Session 12+ scope.

### 5. Re-scan Button Scope

The server's `POST /api/sources/{id}/reindex` only supports `local_fs` sources. The button should only appear for local sources.

**Decision:** Show "Re-scan" button only when `source_type === 'local_fs'` and `status !== 'syncing'`. For Google Drive, rely on the poll cycle.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Google Drive folder_id "required" breaks existing users | Low | Medium | No frontend validation gate — just warning copy. Server validation may already enforce. |
| `vault_loop_back` default change affects new users only | None | None | Only affects `onboarding.ts` initial state, not existing configs. |
| Health summary fetch adds API call on settings load | Low | Low | Single lightweight `GET /api/vault/sources` call. Data is small. |
| Re-scan button without feedback on completion | Medium | Low | Button shows spinner. Poll `vault.sources()` every 3s while syncing to detect completion. Or rely on WebSocket events if available. |
| "Knowledge Vault" naming may not match all use cases (e.g., just a notes folder, not Obsidian) | Low | Low | Copy says "Knowledge Vault" as product name but accepts any markdown folder. Subtitle clarifies "your Obsidian vault, notes folder, or Google Drive." |
