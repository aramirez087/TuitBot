# Session 10 Plan — Vault-Aware Reply Surfaces & Automation Parity

## Goal

Make vault context a first-class part of every reply surface — manual assist, discovery compose, workflow drafts, and automation loops — using one coherent contract. After this session, vault context is no longer a composer-only feature.

## Design Decisions

### D1: Extend `ReplyGenerator` trait with optional RAG context

The `ReplyGenerator` trait in `loop_helpers.rs` currently takes `(tweet_text, author, mention_product)`. Rather than creating a parallel trait, add a default method:

```rust
async fn generate_reply_with_rag(
    &self,
    tweet_text: &str,
    author: &str,
    mention_product: bool,
    rag_context: Option<&str>,
) -> Result<ReplyOutput, LoopError> {
    // Default: ignore RAG, delegate to existing method
    let text = self.generate_reply(tweet_text, author, mention_product).await?;
    Ok(ReplyOutput { text, vault_citations: vec![] })
}
```

**Rationale:** Backward compatible. Existing mocks and implementations keep working. Only `LlmReplyAdapter` overrides the new method. The loops that want vault context call the new method; the old method remains for callers that don't need it.

**Alternative considered:** Separate `VaultAwareReplyGenerator` trait. Rejected because it would require duplicating the loop wiring code and mock implementations.

### D2: New `ReplyOutput` struct for richer return

Currently `ReplyGenerator::generate_reply` returns `Result<String, LoopError>`. The new `generate_reply_with_rag` returns `ReplyOutput { text: String, vault_citations: Vec<VaultCitation> }`. This lets automation loops carry citations forward to the approval queue.

The existing `generate_reply` signature stays unchanged to avoid breaking all mock implementations in tests.

### D3: Automation loops use vault context when keywords are configured

All three automation loops already have access to keywords (discovery_loop explicitly, target/mentions implicitly through config). The `LlmReplyAdapter` will be enhanced to accept an optional pre-built RAG prompt block that it injects on every call. This is built once at loop startup, not per-tweet, to avoid per-tweet DB queries in hot paths.

**Product rationale:** Automation replies should be grounded in the same vault knowledge as manual replies. A discovery reply about "Rust async" should draw from the user's vault notes about Rust, just like a manual compose would.

### D4: Target loop stays vault-aware but never mentions product

Target loop already sets `mention_product: false` for genuine engagement. Vault context still applies — it grounds the reply in domain knowledge without product-pushing. This is documented as an intentional behavior.

### D5: `DraftResult::Success` gains optional `vault_citations`

Adding `vault_citations: Vec<VaultCitation>` to the Success variant. The batch draft step already builds a `DraftContext` with citations — it just discards them. This change passes them through.

### D6: Discovery queue-reply accepts optional provenance

`QueueReplyRequest` gains an optional `provenance` field (matching the composer pattern). When present, `enqueue_with_provenance_for` is used instead of `enqueue_for`.

### D7: `LlmReplyAdapter` gets a vault-aware variant

Rather than injecting DB + config into the adapter (which would break the dependency rule — automation shouldn't reach into context), the adapter accepts a pre-built `Option<String>` RAG prompt at construction time. The runtime (CLI deps / server watchtower) builds the RAG context once and passes it in.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Adding `vault_citations` to `DraftResult::Success` breaks serialization | Low | Medium | `#[serde(default, skip_serializing_if = "Vec::is_empty")]` ensures backward compat |
| `resolve_composer_rag_context` is private to `assist.rs` | Certain | Low | Extract to shared helper in `routes/mod.rs` or make `pub(crate)` |
| Automation loops get slower due to per-tweet RAG queries | Medium | Medium | Build RAG once at adapter construction, not per-tweet. The `VaultAwareLlmReplyAdapter` caches the prompt block. |
| Dependency rule violation if automation layer imports context layer | Medium | High | The adapter lives in `automation/adapters/` which is at the automation layer. It receives a pre-built `String` prompt — no dependency on `context/` module. The RAG building happens in the CLI/server wiring layer. |
| Mock implementations need update for new trait method | Low | Low | Default method means existing mocks work without changes. Tests that want to verify vault behavior can override. |
| Discovery compose lacks account-scoped RAG helper | Certain | Low | Move `resolve_composer_rag_context` from `assist.rs` to a shared `routes/rag_helpers.rs` or use `pub(crate)` on existing function and import across route modules. |
| `DraftResult::Success` already has many fields | Low | Low | One additional field is acceptable. The variant is already rich with metadata. |

## Shared RAG Helper Strategy

The `resolve_composer_rag_context` function in `assist.rs` (lines 42-82) needs to be callable from `discovery.rs` too. Options:

**Option A (preferred): Move to `routes/mod.rs` or `routes/helpers.rs`**
- Create a `pub(crate) async fn resolve_composer_rag_context(...)` in a shared location
- Both `assist.rs` and `discovery.rs` import from there

**Option B: Make the function `pub` in assist.rs and import**
- `use crate::routes::assist::resolve_composer_rag_context;`
- Less clean but simpler

Go with **Option A** — create `routes/rag_helpers.rs` with the function and its imports. Keep `assist.rs` and `discovery.rs` focused on route handling.

## Intentional Differences (Documented in reply-integration.md)

| Surface | Vault Context | Product Mention | Provenance Stored |
|---------|--------------|-----------------|-------------------|
| Manual assist reply | ✅ when keywords configured | User-controlled | ❌ (stateless endpoint) |
| Discovery compose-reply | ✅ when keywords configured | User-controlled | ❌ (stateless endpoint) |
| Discovery queue-reply | N/A (user provides text) | N/A | ✅ when `provenance` field present |
| Workflow batch draft | ✅ automatic from keywords | Config-controlled | ✅ via DraftResult → approval queue |
| Discovery loop | ✅ via adapter | Always `true` | Via approval queue (future) |
| Target loop | ✅ via adapter | Always `false` (genuine) | Via approval queue (future) |
| Mentions loop | ✅ via adapter | Always `true` (direct ask) | Via approval queue (future) |

**Note:** Automation loops currently post directly via `PostSender::send_reply()`, bypassing the approval queue. Provenance storage in the queue for automation is deferred to when approval mode is wired into loops. This session ensures the *generation* path carries vault context, which is the prerequisite.
