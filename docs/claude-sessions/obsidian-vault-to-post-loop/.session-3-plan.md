# Session 3 Implementation Plan — Content Source Runtime & Lifecycle

## Mission

Make content-source behavior truthful, live-reloadable, and consistent across
local folders, Google Drive, and manual ingest. Fix the misleading `watch` toggle
semantics and add hot-reload capability for config changes.

---

## Current State Analysis

### The `watch` Problem

Two places filter sources on `s.watch`:

1. **`main.rs:200`** — Server startup decides whether to start Watchtower at all:
   ```rust
   s.watch && (s.path.is_some() || s.folder_id.is_some())
   ```
2. **`watchtower/mod.rs:337`** — WatchtowerLoop.run() filters local sources:
   ```rust
   .filter(|s| s.source_type == "local_fs" && s.watch && s.path.is_some())
   ```

**Problem:** Setting `watch = false` on a local source means it never gets
ingested _at all_ — not even an initial scan. Meanwhile, Google Drive sources
ignore `watch` entirely and always poll. The `watch` field conflates "enabled"
with "real-time change detection mode."

### No Hot Reload

`WatchtowerLoop` takes `ContentSourcesConfig` by value at construction time.
`AppState.watchtower_cancel` is `Option<CancellationToken>` — it can stop the
watcher but can't restart it with new config. `AppState.content_sources` is a
plain field, not behind `RwLock`/`watch`, so `PATCH /api/settings` writes config
to disk but the running Watchtower is oblivious.

### No Source Status API

`source_contexts` table has `status` and `error_message` columns
(`sources.rs:109-128`), and `update_source_status()` exists, but there's no
HTTP endpoint to read source runtime status. The dashboard has no way to show
sync state.

## Design Decisions

### D1: Replace `watch` with `enabled` + `change_detection`

The `watch: bool` field is semantically overloaded. Replace with:

- **`enabled: bool`** (default `true`) — whether this source participates in
  ingestion at all. Disabled sources are completely skipped.
- **`change_detection: String`** (default `"auto"`) — how changes are detected:
  - `"auto"` — local_fs: notify watcher + fallback poll; google_drive: poll
  - `"poll"` — poll only (useful when notify is unreliable, e.g. network mounts)
  - `"none"` — initial scan only, no ongoing monitoring

Keep `watch: bool` as a deprecated serde alias that maps to
`enabled = watch, change_detection = if watch { "auto" } else { "none" }`.
This preserves backward compatibility for existing configs.

### D2: Watchtower Restart via Cancel-and-Respawn

Rather than adding internal config-reload complexity to WatchtowerLoop, use a
simple cancel-and-respawn pattern:

1. `AppState` gains a method `restart_watchtower(&self)` that:
   - Cancels the existing token (if any)
   - Reloads `ContentSourcesConfig` from disk
   - Creates a new `WatchtowerLoop` + `CancellationToken`
   - Spawns it
   - Updates `self.watchtower_cancel` and `self.content_sources`

2. `PATCH /api/settings` calls `restart_watchtower()` after writing config
   when the patch touches `content_sources`.

This is simple, safe (WatchtowerLoop.run already handles cancellation cleanly),
and avoids shared-state complexity inside the polling loop.

### D3: Source Status as DB-Backed Runtime State

Source runtime status already lives in `source_contexts.status` and
`source_contexts.error_message`. Expose it via:

- `GET /api/sources/status` — all sources with status, last sync, error
- `POST /api/sources/{id}/reindex` — trigger a full rescan of one source

The Watchtower already calls `update_source_status()` on errors and
`update_sync_cursor()` on success. We add explicit `"syncing"` status
transitions at scan/poll start and `"active"` on completion.

### D4: `enabled` Validation

Add validation in `validation.rs`:
- `enabled = true` local_fs source must have a non-empty `path`
- `enabled = true` google_drive source must have `folder_id`
- `change_detection` must be one of `"auto"`, `"poll"`, `"none"`
- `poll_interval_seconds` must be >= 30 when set

## Files to Modify

### 1. `crates/tuitbot-core/src/config/types.rs`

**Changes:**
- Add `enabled: bool` field to `ContentSourceEntry` (default `true`).
- Add `change_detection: String` field (default `"auto"`).
- Deprecate `watch` via serde: keep `watch` with `#[serde(default, alias = "watch")]` mapping or implement a custom deserializer that reads `watch` and maps it to `enabled`/`change_detection`.
- Add `fn default_enabled() -> bool { true }` and `fn default_change_detection() -> String { "auto".into() }`.
- Add a method `ContentSourceEntry::effective_enabled(&self) -> bool` that returns `enabled` (or maps from legacy `watch` if using the alias approach).
- Add a method `ContentSourceEntry::is_poll_only(&self) -> bool`.

**Approach for backward compat:** Use `#[serde(deserialize_with)]` on a wrapper
or simply keep both fields and apply a post-deserialization normalization step.
The simplest approach: keep `watch` as deprecated, add `enabled` and
`change_detection`, and in a post-deser method on `ContentSourcesConfig`,
normalize entries where `enabled` is unset but `watch` was explicitly set.

Actually, the simplest backward-compatible approach:
- Keep `watch: bool` with `#[serde(default = "default_watch")]`.
- Add `enabled: Option<bool>` with `#[serde(default)]`. When `None`, falls back
  to `watch`. When `Some(v)`, uses `v`.
- Add `change_detection: String` with `#[serde(default = "default_change_detection")]`.
- Add `ContentSourceEntry::is_enabled(&self) -> bool` = `self.enabled.unwrap_or(self.watch)`.
- Add `ContentSourceEntry::effective_change_detection(&self) -> &str` = returns
  `&self.change_detection` (or `"none"` if not enabled).

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `RwLock` on `content_sources` causes deadlock with existing readers | Medium | High | Audit all readers; use short-lived read guards; never hold across await points |
| Watchtower restart drops in-flight ingests | Low | Medium | CancellationToken is cooperative; WatchtowerLoop completes current scan iteration before exiting |
| Backward-compat `watch` → `enabled` mapping confuses users | Low | Low | Both fields work; doc clearly states `enabled` supersedes `watch`; `watch` not removed |
| `reindex_source` for large vaults blocks for a long time | Medium | Low | Runs in spawned task; returns immediately; status set to "syncing" |
| Callers of `state.content_sources` not updated after RwLock change | Medium | High | Grep for all usages; compiler will catch type mismatches |
| AppState struct construction in tests (e.g. `assist.rs`) breaks | High | Low | Update test helpers; changes are mechanical |
