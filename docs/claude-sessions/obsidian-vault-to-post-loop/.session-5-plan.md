# Session 5 — Retrieval Engine & Context Builder v2: Implementation Plan

## Summary

Build the account-filtered fragment retrieval engine that replaces the current
global cold-start seed fallback with direct chunk retrieval, returns structured
citations, and unifies vault fragments with winning ancestors into a single
context model.

## Current State Analysis

### Key Gaps Identified

1. **No account isolation in ancestors.rs** — `get_scored_ancestors()` has zero
   `account_id` filtering. Tweets/replies are queried globally. The function
   signature doesn't accept `account_id` at all.

2. **No account isolation in winning_dna.rs** — `build_draft_context()`,
   `retrieve_ancestors()`, and `retrieve_cold_start_seeds()` take no `account_id`
   parameter. The server handler (`resolve_composer_rag_context`) receives
   `account_id` but doesn't pass it through.

3. **Seeds-only fallback, no chunk retrieval** — The cold-start path uses
   `draft_seeds` (LLM-extracted hooks) instead of raw vault fragments
   (`content_chunks`). Chunks have keyword search (`search_chunks_by_keywords`)
   and boost scoring ready but are never queried in the context builder.

4. **No citation records** — `DraftContext` carries `prompt_block` as an opaque
   string. There's no structured data connecting the prompt back to specific
   chunk IDs, node paths, or heading paths.

5. **Binary ancestor vs seed** — `build_draft_context` is either-or: if ancestors
   exist, seeds are skipped entirely. There's no mixed mode where both winning
   patterns and vault fragments enrich the same prompt.

### Existing Infrastructure Ready to Use

- `search_chunks_by_keywords(pool, account_id, keywords, limit)` — LIKE-based
  chunk retrieval, ordered by `retrieval_boost DESC`
- `get_chunks_for_node(pool, account_id, node_id)` — per-node chunk listing
- `ContentChunk` struct with `heading_path`, `chunk_text`, `retrieval_boost`
- Account-scoped `_for` variants exist in seeds.rs but not in ancestors.rs

## Design Decisions

### D1: Add `account_id` to all retrieval functions

**Rationale:** Account isolation is a hard constraint. Every query touching
user-owned data must filter by `account_id`. The current ancestor queries are
a pre-existing gap that this session must fix.

**Approach:** Add `account_id` parameter to `get_scored_ancestors()`,
`retrieve_ancestors()`, `retrieve_cold_start_seeds()`, and
`build_draft_context()`. Update all callers (server assist handler, workflow
draft step).

### D2: Three-tier context model (ancestors + fragments + seeds)

**Rationale:** The current binary (ancestors XOR seeds) model wastes the chunk
infrastructure. The new model:
1. **Winning ancestors** — always included when available (behavioral signal)
2. **Vault fragments** — keyword-matched chunks from the user's notes (knowledge signal)
3. **Content seeds** — LLM-extracted hooks as last-resort fallback

Ancestors and fragments serve different purposes and should combine. Seeds
remain as fallback when no chunks exist (e.g., chunking hasn't run yet).

**Prompt budget:** Split `RAG_MAX_CHARS` (2000) into:
- Ancestors: up to 800 chars (behavioral patterns)
- Fragments: up to 1000 chars (knowledge/expertise)
- Seeds fallback: up to 1000 chars (only when no fragments)
- Buffer: 200 chars for headers/instructions

### D3: Structured citations via `VaultCitation`

**Rationale:** The session mandate requires "structured citation records alongside
prompt text so later API and UI work can expose the exact source material used."

**Struct:**
```rust
pub struct VaultCitation {
    pub chunk_id: i64,
    pub node_id: i64,
    pub heading_path: String,
    pub source_path: String,    // relative_path from content_nodes
    pub source_title: Option<String>,
    pub snippet: String,        // first ~120 chars of chunk_text
    pub retrieval_boost: f64,
}
```

`DraftContext` gets a new field: `vault_citations: Vec<VaultCitation>`.

### D4: Keep `prompt_block` as a single formatted string

**Rationale:** The generator's `format_rag_section()` takes `Option<&str>`.
Changing this to structured input would require modifying `ContentGenerator`
across 4 generation methods + all their callers. That's scope creep. Instead,
`prompt_block` remains the formatted string, and `vault_citations` provides
structured data alongside it for API/UI consumption.

### D5: Fragment retrieval uses keyword + selected-note bias

**Rationale:** The session asks for "optionally selected-note bias." This means
the retrieval function accepts an optional `node_ids` filter that, when provided,
boosts chunks from those specific notes. Implementation: query chunks from
selected nodes with a boost multiplier, then fill remaining slots with keyword
search results.

## Files to Create/Modify

### Modified: `crates/tuitbot-core/src/context/winning_dna.rs`

Changes (estimated +120 lines, total ~850):

1. **New structs:**
   - `VaultCitation` — structured citation record (chunk_id, node_id,
     heading_path, source_path, source_title, snippet, retrieval_boost)
   - `FragmentContext` — intermediate: chunk_text + citation metadata

2. **New constants:**
   - `MAX_FRAGMENT_CHARS: usize = 1000` — budget for vault fragment section
   - `MAX_ANCESTOR_CHARS: usize = 800` — budget for ancestor section
   - `MAX_FRAGMENTS: u32 = 5` — max fragments in context

3. **New function: `retrieve_vault_fragments()`**
   ```rust
   pub async fn retrieve_vault_fragments(
       pool: &DbPool,
       account_id: &str,
       keywords: &[String],
       selected_node_ids: Option<&[i64]>,
       max_results: u32,
   ) -> Result<Vec<FragmentContext>, StorageError>
   ```
   - Calls `search_chunks_by_keywords()` with account scoping
   - If `selected_node_ids` is provided, queries those nodes first via
     `get_chunks_for_node()`, then fills remaining slots with keyword results
   - Deduplicates by chunk_id
   - Returns up to `max_results` fragments with citation metadata

4. **New function: `format_fragments_prompt()`**
   - Formats fragment text with heading paths as inline citations:
     ```
     Relevant knowledge from your notes:
     1. [Heading > Sub] (from: note-title.md): "chunk text excerpt..."
     2. ...
     Reference these insights to ground your response in your own expertise.
     ```

5. **Modified: `retrieve_ancestors()`** — add `account_id: &str` parameter,
   pass to `get_scored_ancestors()`

6. **Modified: `retrieve_cold_start_seeds()`** — add `account_id: &str`
   parameter, call `get_seeds_for_context_for()` instead of
   `get_seeds_for_context()`

7. **Modified: `build_draft_context()`** — new signature:
   ```rust
   pub async fn build_draft_context(
       pool: &DbPool,
       account_id: &str,
       topic_keywords: &[String],
       max_ancestors: u32,
       half_life_days: f64,
   ) -> Result<DraftContext, StorageError>
   ```
   New logic:
   - Retrieve ancestors (always attempted)
   - Retrieve vault fragments via keyword search (always attempted)
   - If both found → combine into prompt_block with split budgets
   - If only ancestors → ancestor-only prompt
   - If only fragments → fragment-only prompt
   - If neither → fall back to seeds
   - Populate `vault_citations` from fragment results

8. **Modified: `DraftContext`** — add `vault_citations: Vec<VaultCitation>` field

9. **New function: `build_citations()`** — extracts `VaultCitation` records from
   `FragmentContext` results (pure function)

10. **Update existing tests** — add `account_id` parameter to all test calls.
    Add new tests for fragment retrieval and combined context.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `replies_sent` lacks `account_id` column | Low | High | Check schema first; if missing, join through a different path or skip reply account filtering with a TODO |
| `winning_dna.rs` exceeds 500-line limit after changes | High | Medium | If it hits ~850 lines, extract `retrieval.rs` and `citation.rs` submodules. Tests already separated via `#[cfg(test)] mod tests` |
| Thread-through of `account_id` touches many callers | Medium | Low | The callers are few: `assist.rs` (already has it), `draft.rs` (needs it), test helpers. Manageable. |
| LIKE-based keyword search returns low-quality matches | Medium | Low | Acceptable for MVP. `retrieval_boost` ordering helps. Future session can add TF-IDF or embedding search |
| Prompt budget split (800+1000) may be too tight for rich contexts | Low | Low | Constants are tunable. Start conservative, adjust based on real usage |
| `build_draft_context` becomes too complex | Low | Medium | Keep the function as orchestrator; delegate formatting and retrieval to subfunctions |
