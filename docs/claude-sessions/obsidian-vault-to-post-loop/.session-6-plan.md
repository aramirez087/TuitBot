# Session 06 Plan — End-to-End Provenance Wiring

## Summary

Wire `VaultCitation` data through every persistence boundary so that any generated draft, approval queue item, or posted tweet can be traced back to the vault notes and chunks that influenced it. All provenance fields are optional—manual and legacy flows continue working unchanged.

## Current State

### What exists
- **Schema columns already in place** (migration `20260308010000`):
  - `approval_queue.source_node_id`, `approval_queue.source_seed_id`, `approval_queue.source_chunks_json`
  - `original_tweets.source_node_id` (migration `20260228000019`)
  - `draft_seeds.chunk_id`
- **`VaultCitation` type** in `context/retrieval.rs` with `chunk_id`, `node_id`, `heading_path`, `source_path`, `source_title`, `snippet`, `retrieval_boost`
- **`DraftContext.vault_citations`** populated by `build_draft_context()` but never forwarded beyond the prompt block

### What's missing
1. `enqueue_for()` / `enqueue_with_context_for()` don't accept provenance parameters — columns always NULL
2. `scheduled_content` table has no provenance columns (only loose `source` string)
3. `threads` table has no provenance link
4. Compose and draft API endpoints never pass provenance to storage
5. Assist endpoints return `DraftContext.vault_citations` but don't surface them in API responses or carry them through to drafts
6. `approval_poster.rs` doesn't propagate `source_node_id` to `original_tweets` when posting

---

## Design Decisions

### D1: Separate provenance link table vs inline columns
**Decision: Use a new link table `vault_provenance_links` rather than adding more columns to each table.**

Rationale:
- Multiple content tables need provenance (approval_queue, scheduled_content, original_tweets, threads)
- A link table avoids duplicating 3+ columns on every table
- Supports multiple source refs per item (a post can draw from several notes/chunks)
- The existing inline columns on `approval_queue` (`source_node_id`, `source_seed_id`, `source_chunks_json`) are kept for backward compatibility but the link table becomes the canonical provenance store

### D2: Provenance payload shape
**Decision: Use a `ProvenanceRef` Rust struct serializable to/from the link table.**

```rust
pub struct ProvenanceRef {
    pub node_id: Option<i64>,
    pub chunk_id: Option<i64>,
    pub seed_id: Option<i64>,
    pub source_path: Option<String>,
    pub heading_path: Option<String>,
    pub snippet: Option<String>,
}
```

This is derived from `VaultCitation` at creation time. It stores the snapshot values so provenance survives even if the source note is later deleted.

### D3: How provenance flows through API layer
**Decision: Accept optional `provenance` array in compose/draft/assist endpoints; never require it.**

- Assist endpoints return `vault_citations` in their response so the frontend can pass them back when saving
- Compose/draft endpoints accept `provenance: Option<Vec<ProvenanceRef>>` and persist them
- Legacy callers that omit `provenance` work exactly as before

### D4: Approval poster → original_tweets provenance forwarding
**Decision: When the approval poster successfully posts, copy `source_node_id` from the approval_queue item to `original_tweets.source_node_id`.**

This is the final link that makes the chain traceable: vault note → approval_queue → original_tweets → analytics.

### 1. `migrations/20260308020000_vault_provenance_links.sql`
### 2. `crates/tuitbot-core/migrations/20260308020000_vault_provenance_links.sql`

Both identical:

```sql
-- Provenance link table: maps any content record to vault source material.
-- entity_type + entity_id form a polymorphic FK (e.g., 'approval_queue' + 42).
CREATE TABLE IF NOT EXISTS vault_provenance_links (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL DEFAULT '00000000-0000-0000-0000-000000000000',
    entity_type TEXT NOT NULL,          -- 'approval_queue', 'scheduled_content', 'original_tweet', 'thread'
    entity_id INTEGER NOT NULL,
    node_id INTEGER REFERENCES content_nodes(id),
    chunk_id INTEGER REFERENCES content_chunks(id),
    seed_id INTEGER REFERENCES draft_seeds(id),
    source_path TEXT,                   -- snapshot of relative_path at creation
    heading_path TEXT,                  -- snapshot of heading hierarchy
    snippet TEXT,                       -- snapshot of chunk excerpt
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_provenance_entity
    ON vault_provenance_links(account_id, entity_type, entity_id);
CREATE INDEX IF NOT EXISTS idx_provenance_node
    ON vault_provenance_links(account_id, node_id);
```

## Files to Modify

### 5. `crates/tuitbot-core/src/storage/approval_queue/queries.rs`

**Changes:**
- Add `enqueue_with_provenance_for()` function that accepts `source_node_id: Option<i64>`, `source_seed_id: Option<i64>`, `source_chunks_json: &str` and includes them in the INSERT
- Update `enqueue_with_context_for()` to delegate to `enqueue_with_provenance_for()` with `None` provenance values (preserves all existing signatures)
- Add `insert_provenance_links()` helper that batch-inserts `vault_provenance_links` rows for an entity

**Approach:** The existing `enqueue_for` → `enqueue_with_context_for` delegation chain stays intact. A new top-level function `enqueue_with_provenance_for` is added at the bottom of the chain. This avoids breaking any existing callers.

### 6. `crates/tuitbot-core/src/storage/approval_queue/mod.rs`

**Changes:**
- Re-export `enqueue_with_provenance_for` from queries module
- No struct changes needed (provenance fields already on `ApprovalItem`)

### 7. `crates/tuitbot-core/src/storage/scheduled_content.rs`

**Changes:**
- Add `insert_draft_with_provenance_for()` that calls `insert_draft_for()` then inserts provenance link rows
- Keep `ScheduledContent` struct unchanged—provenance is queried via the link table, not inline

### 8. `crates/tuitbot-core/src/storage/threads.rs`

**Changes:**

- Add `set_original_tweet_source_node()` helper for the approval_poster to set `original_tweets.source_node_id` after posting

### 9. `crates/tuitbot-core/src/context/retrieval.rs`

**Changes:**
- Add `pub fn citations_to_provenance_refs(citations: &[VaultCitation]) -> Vec<ProvenanceRef>` converter
- Add `pub fn citations_to_chunks_json(citations: &[VaultCitation]) -> String` for the legacy `source_chunks_json` field
- Define `ProvenanceRef` struct here (it's a retrieval/citation concept)

### 10. `crates/tuitbot-server/src/routes/assist.rs`

**Changes:**
- Add `vault_citations: Vec<VaultCitation>` to `AssistTweetResponse` and `AssistThreadResponse`
- Modify `resolve_composer_rag_context()` to return `DraftContext` (not just `String`) so citations are available
- Update `assist_tweet`, `assist_thread`, `assist_improve` to include citations in response

### 11. `crates/tuitbot-server/src/routes/content/drafts.rs`

**Changes:**
- Add optional `provenance: Option<Vec<ProvenanceRef>>` to `CreateDraftRequest`
- In `create_draft()`, if provenance is provided, call `insert_draft_with_provenance_for()` instead of `insert_draft_for()`
- In `publish_draft()`, load provenance from link table for the draft's `scheduled_content` ID, then pass to `enqueue_with_provenance_for()` so provenance survives the draft→approval_queue transition

### 12. `crates/tuitbot-server/src/routes/content/compose.rs`

**Changes:**
- Add optional `provenance: Option<Vec<ProvenanceRef>>` to `ComposeRequest`, `ComposeTweetRequest`
- In approval-mode paths, use `enqueue_with_provenance_for()` when provenance is present
- In non-approval paths that insert `scheduled_content`, call `insert_draft_with_provenance_for()` equivalent
- Backwards compatible: all provenance fields are `Option` with `#[serde(default)]`

### 13. `crates/tuitbot-core/src/automation/approval_poster.rs`

**Changes:**
- After successfully posting and calling `mark_posted()`, check if `item.source_node_id` is set
- If so, insert an `original_tweets` record with `source_node_id` populated (or use `set_original_tweet_source_node()` if the tweet already exists)
- Also copy provenance links from `entity_type='approval_queue'` to `entity_type='original_tweet'` for the new record

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Polymorphic FK (`entity_type + entity_id`) can't use real FK constraints | Known | Low | Add index; document convention. Acceptable tradeoff for SQLite. |
| `enqueue_with_provenance_for` adds another argument to an already-long function | Medium | Low | Use a `ProvenanceInput` struct to bundle the three provenance params |
| Assist responses now include `vault_citations` — larger payload | Low | Low | Array is empty when no vault context exists; max ~5 items |
| `approval_poster` doesn't have account_id context | Medium | Medium | The poster currently uses `DEFAULT_ACCOUNT_ID`; provenance copy uses the same. Future multi-account poster work will address this |
| Provenance links could accumulate — no cleanup on entity deletion | Low | Low | `DELETE CASCADE` not possible with polymorphic FK. Add a cleanup query later if needed; rows are small |
