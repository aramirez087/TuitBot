# Session 1 Implementation Plan: Charter and UX Blueprint

## Objective

Produce four documentation deliverables that define the north-star UX and technical plan for Draft Studio. No source code changes.

## Current State Audit

### Fragmented Compose Surfaces

The codebase has **four independent compose/draft surfaces**, each with its own state management:

1. **Home Composer** (`dashboard/src/routes/(app)/+page.svelte`): Full `ComposeWorkspace` embedded on `/`. Uses `homeSurface` store to toggle between composer and analytics. State is component-local `$state` with localStorage autosave. Submits via `api.content.compose()` — creates a `scheduled_content` row directly, bypassing drafts entirely.

2. **Drafts CRUD Page** (`dashboard/src/routes/(app)/drafts/+page.svelte`): A flat card-list page with inline textarea editing. No `ComposeWorkspace` integration. Creates/edits drafts via `api.drafts.*` methods. Has its own AI generate/improve flow. No autosave, no undo, no inspector, no preview.

3. **Calendar Compose Modal** (`dashboard/src/routes/(app)/content/+page.svelte`): Opens `ComposeModal` → `ComposeWorkspace` as a dialog. Submits via `api.content.compose()` with a prefilled date. Same local-only state as home. No draft record created.

4. **Backend Draft Storage** (`crates/tuitbot-core/src/storage/scheduled_content.rs`): Drafts are rows in `scheduled_content` with `status = 'draft'`. Schema has: `id, content_type, content, scheduled_for, status, posted_tweet_id, source, created_at, updated_at, qa_*`. No title, notes, tags, archive flag, or revision history columns.

### Key Problems

- **Two independent writing experiences**: `ComposeWorkspace` (rich, 893 lines, 16 shortcuts, autosave, undo, preview, inspector) vs drafts page (basic textarea, no shortcuts, no autosave).
- **Home compose creates orphan state**: Content in the home composer has no server-side draft record. If you close the tab, only localStorage recovery can save you.
- **Calendar compose is equally ephemeral**: Same problem — no draft record until final submit.
- **No draft selection UX**: The drafts page shows a flat list of cards. No way to "open" a draft into the full composer. Editing is inline-only.
- **Schema is thin**: `scheduled_content` lacks metadata columns needed for a studio experience (title, notes, tags, archive status, revision snapshots).

## Deliverables and Order of Operations

### Step 1: Write `docs/roadmap/draft-studio-beyond-typefully/charter.md`

**Key design decisions to lock:**

| Decision | Default | Rationale |
|----------|---------|-----------|
| One canonical draft workspace | `/drafts` route | Eliminates fragmentation. Home becomes a shortcut into Draft Studio, not a parallel composer. |
| Server-backed draft records as source of truth | Every compose action creates or selects a server draft | No more ephemeral component state. Autosave writes to server, localStorage is crash recovery only. |
| No independent home-composer state | Home links to `/drafts` or creates a new draft there | Prevents the current split where home compose and drafts are disconnected systems. |
| Extend `scheduled_content` table | Add columns, don't create parallel table | Preserves existing draft/schedule/post lifecycle. Status field already covers `draft → scheduled → posted → cancelled`. |
| Additive, reversible migrations | New columns nullable or with defaults | No data loss risk. Old queries still work against new schema. |
| Local autosave as crash recovery only | Debounced server saves (1-2s), localStorage as fallback | Server is truth. Local recovery banner only appears when server state is stale. |

**Goals (G1-G5):**
- G1: `/drafts` is the canonical writing workspace — a three-zone layout (rail + composer + inspector).
- G2: Server-backed drafts are the single source of truth for all content in progress.
- G3: Draft-to-scheduled-to-posted transitions are explicit, visible, and reversible.
- G4: Every AI action has undo; every destructive action has confirmation or restore.
- G5: Beat Typefully on sync visibility, revision safety, and keyboard-first navigation.

**Non-goals:**
- NG1: Don't remove `ComposeWorkspace` — it becomes the composer surface inside Draft Studio.
- NG2: Don't break calendar compose — it creates a new draft and opens Draft Studio.
- NG3: Don't change existing API contracts for `ComposeRequest`, `ThreadBlock[]`.
- NG4: Don't introduce contenteditable.
- NG5: Don't change Tauri bridge or desktop chrome.

### Step 2: Write `docs/roadmap/draft-studio-beyond-typefully/ux-blueprint.md`

**Screens to define:**

1. **Empty state** — First visit to `/drafts` with no drafts. Prominent "New Draft" CTA. Suggest AI generate or start from blank.

2. **Draft rail (left panel)** — Scrollable list of drafts grouped by status tab (Active / Scheduled / Archive). Each entry shows: title or content preview (first ~60 chars), content type badge, relative timestamp, sync indicator. Selected draft highlighted. Keyboard: `↑/↓` to navigate, `Enter` to select, `N` to create new.

3. **Composer surface (center)** — Full `ComposeWorkspace` bound to the selected draft's server record. Changes debounce-save to server. Sync badge shows "Saved" / "Saving..." / "Offline — local backup". All existing composer features (inspector, preview, AI assist, shortcuts) preserved.

4. **New draft** — Click "New Draft" or press `N` from rail. Creates a blank server draft record immediately. Composer hydrates from the new record. User starts typing — debounced server saves begin.

5. **Draft selection** — Click a draft in the rail. Composer swaps to that draft. Unsaved changes to the previous draft are flushed first. URL updates to include draft ID (e.g., `/drafts?id=42`).

6. **Thread editing** — Same `ThreadFlowLane` experience. Blocks stored as `ThreadBlocksPayload` in the draft's `content` column. Server-side thread validation on save.

7. **Schedule** — Inspector schedule picker sets `scheduled_for` on the draft record. Status transitions to `scheduled`. Draft moves from Active tab to Scheduled tab in the rail. Can be unscheduled (returns to Active).

8. **Archive** — Soft delete. Draft status changes to `archived`. Disappears from Active tab, appears in Archive tab. Can be restored.

9. **Restore** — From Archive tab, restore returns draft to `draft` status in Active tab.

10. **Keyboard navigation** — Full spec:
    | Context | Key | Action |
    |---------|-----|--------|
    | Rail | `↑` / `↓` | Navigate drafts |
    | Rail | `Enter` | Select draft |
    | Rail | `N` | New draft |
    | Rail | `Delete` / `Backspace` | Archive selected |
    | Composer | All existing shortcuts | Unchanged |
    | Global | `⌘K` | Command palette (extended with draft actions) |
    | Global | `Escape` | Focus cascade: palette → inspector → rail |

11. **Mobile behavior** — Rail collapses to a top drawer or tab. Draft list shows as a compact picker above the composer. Swipe gestures for archive.

**Typefully differentiators to exceed:**
- **Explicit sync state**: Typefully doesn't show save status. Draft Studio shows "Saved 2s ago" / "Saving..." / "Offline — local backup" in the composer header.
- **Safer AI undo**: Typefully has no undo after AI rewrite. Draft Studio snapshots before every AI action; 10-second undo banner; plus revision history for longer-term recovery.
- **Revision restore**: Typefully has no revision history. Draft Studio stores snapshots on meaningful events (AI rewrite, schedule, unschedule) in a `content_revisions` table. Viewable in the inspector or a side panel.
- **Frictionless transitions**: Typefully separates drafts and scheduled posts. Draft Studio keeps them in one table with status transitions: `draft → scheduled → posted`. Schedule/unschedule is a single click, no data duplication.

**Data model additions (Session 02):**

```sql
-- Additive columns on scheduled_content
ALTER TABLE scheduled_content ADD COLUMN title TEXT DEFAULT NULL;
ALTER TABLE scheduled_content ADD COLUMN notes TEXT DEFAULT NULL;
ALTER TABLE scheduled_content ADD COLUMN archived_at TEXT DEFAULT NULL;

-- Revision history
CREATE TABLE content_revisions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id),
    account_id TEXT NOT NULL,
    content TEXT NOT NULL,
    content_type TEXT NOT NULL,
    trigger TEXT NOT NULL,  -- 'autosave', 'ai_rewrite', 'schedule', 'manual'
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Tags
CREATE TABLE content_tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL,
    name TEXT NOT NULL,
    color TEXT DEFAULT NULL,
    UNIQUE(account_id, name)
);

CREATE TABLE content_tag_assignments (
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id),
    tag_id INTEGER NOT NULL REFERENCES content_tags(id),
    PRIMARY KEY (content_id, tag_id)
);

-- Activity log
CREATE TABLE content_activity (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id),
    account_id TEXT NOT NULL,
    action TEXT NOT NULL,  -- 'created', 'edited', 'ai_rewrite', 'scheduled', 'unscheduled', 'archived', 'restored', 'posted'
    detail TEXT DEFAULT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Step 4: Write `docs/roadmap/draft-studio-beyond-typefully/session-01-handoff.md`

**What changed**: Four planning documents created. No code changes.

**What remains**: Sessions 02-11 as defined above.

**Risks:**
1. `scheduled_content` table is used by multiple systems (content poster, approval queue, calendar queries). Schema changes must not break any existing queries. Mitigation: all new columns nullable with defaults; test all existing storage functions against migrated schema.
2. `ComposeWorkspace` is 893 lines and tightly coupled to component-local state. Refactoring to accept server-backed draft will require careful state management. Mitigation: Session 05 is dedicated to this; use a prop-based hydration pattern rather than store injection.
3. Autosave race conditions: debounced server saves + localStorage fallback + draft switching could produce stale writes. Mitigation: use monotonic version counter on PATCH; reject stale writes server-side.
4. The existing composer redesign roadmap (`docs/roadmap/composer-ui-typefully-redesign/`) has its own session plan (S1-S5 focused on visual polish). Draft Studio is a larger scope that subsumes those visual changes. Mitigation: Draft Studio sessions 02-11 can incorporate the visual polish goals from the redesign charter as a parallel concern, but the architectural work takes priority.
