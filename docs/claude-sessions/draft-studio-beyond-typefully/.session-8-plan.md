## Design Decisions

### D1: Schedule action lives in the Details Panel, not the Inspector
**Rationale**: In embedded studio mode, the ComposeWorkspace Inspector's TimePicker is designed for the calendar compose flow (select time â†’ submit â†’ create new scheduled item). For studio drafts, the natural flow is: write â†’ pick time in details panel â†’ click "Schedule" button â†’ status transitions. This avoids repurposing the existing `handleSubmit()` path which resets composer state after submit.

**Alternative considered**: Reuse Inspector TimePicker for studio scheduling. Rejected because `handleSubmit()` clears the composer (lines 402-419), which is wrong for a draft that should persist after scheduling.

**Implementation**: Add a "Schedule" section to `DraftDetailsPanel.svelte` with a date-time picker and schedule/unschedule buttons. When a draft is `status: 'draft'`, show the schedule controls. When `status: 'scheduled'`, show the scheduled time with unschedule/reschedule buttons.

### D2: `handleDraftSubmit()` becomes the "publish now" flow
**Rationale**: When the user presses `Cmd+Shift+Enter` in the studio composer, this should trigger "publish now" (immediate post) if the draft has content. This maps to the existing `onsubmit` callback, which can call the legacy compose endpoint to post immediately. For Session 8, this will be stubbed as a "not yet implemented" toast; the schedule flow uses the details panel instead.

**Revised approach**: Actually, `handleDraftSubmit` should be wired to schedule the draft using the Inspector's `selectedTime`. If the user picks a time in the Inspector and hits Cmd+Shift+Enter, it schedules. If no time is selected, it's a no-op (or shows a toast). This keeps the Inspector relevant for power users who want keyboard-driven scheduling.

### D3: Calendar â†’ Studio navigation uses `goto('/drafts?id={id}')`
**Rationale**: The simplest integration. When a user clicks "Edit" on a scheduled item in the calendar, we navigate to Draft Studio with that item pre-selected. No need for a modal or embedded editor in the calendar â€” the studio IS the editor.

### D4: Calendar quick-compose creates a draft first, then schedules
**Rationale**: This ensures all content flows through the canonical draft system. Instead of `api.content.compose()` (which creates an anonymous scheduled item), the flow becomes: create draft â†’ set content â†’ schedule it. However, this would change the calendar compose UX significantly and risks breaking the existing flow.

**Pragmatic decision**: Keep calendar compose as-is for Session 8. Calendar compose creates items via the legacy path. Those items still appear in the calendar AND in the Draft Studio's Scheduled tab (they share the same `scheduled_content` table). Session 10 can optionally redirect calendar compose to Draft Studio. The key requirement is that *editing* a scheduled item from the calendar opens it in the studio.

### D5: Posted tab shows "Duplicate as draft" action
**Rationale**: Posted items can't be edited (already published). The useful action is to duplicate a posted item back as a new draft for iteration. This gives the Posted tab purpose beyond being a read-only log.

### Step 1: Add store actions (`draftStudio.svelte.ts`)

Add after the `duplicateDraft` function (line 274):

```typescript
export async function scheduleDraft(id: number, scheduledFor: string): Promise<boolean> {
    try {
        const result = await api.draftStudio.schedule(id, scheduledFor);
        // Update the draft in collection
        collection = collection.map((d) =>
            d.id === id
                ? { ...d, status: result.status, scheduled_for: result.scheduled_for }
                : d
        );
        tab = 'scheduled';
        return true;
    } catch (e) {
        error = e instanceof Error ? e.message : 'Failed to schedule draft';
        return false;
    }
}

export async function unscheduleDraft(id: number): Promise<boolean> {
    try {
        await api.draftStudio.unschedule(id);
        collection = collection.map((d) =>
            d.id === id
                ? { ...d, status: 'draft', scheduled_for: null }
                : d
        );
        tab = 'active';
        return true;
    } catch (e) {
        error = e instanceof Error ? e.message : 'Failed to unschedule draft';
        return false;
    }
}

export async function rescheduleDraft(id: number, scheduledFor: string): Promise<boolean> {
    try {
        // Server requires status='draft' to schedule, so unschedule first
        await api.draftStudio.unschedule(id);
        const result = await api.draftStudio.schedule(id, scheduledFor);
        collection = collection.map((d) =>
            d.id === id
                ? { ...d, status: result.status, scheduled_for: result.scheduled_for }
                : d
        );
        return true;
    } catch (e) {
        error = e instanceof Error ? e.message : 'Failed to reschedule draft';
        return false;
    }
}
```

### Step 3: Wire shell handlers (`DraftStudioShell.svelte`)

Add handler functions in the `<script>` section:

```typescript
async function handleSchedule(scheduledFor: string) {
    const id = studio.getSelectedId();
    if (id === null) return;
    // Flush any pending autosave before scheduling
    // (handled by the store action via API call)
    const success = await studio.scheduleDraft(id, scheduledFor);
    if (success) {
        // Refresh the full draft to get updated status
        await fetchDraft(id);
    }
}

async function handleUnschedule() {
    const id = studio.getSelectedId();
    if (id === null) return;
    const success = await studio.unscheduleDraft(id);
    if (success) {
        await fetchDraft(id);
    }
}

function handleDuplicateFromDetails() {
    const id = studio.getSelectedId();
    if (id !== null) handleDuplicate(id);
}
```

### Step 6: Verify `handleDraftSubmit` data flow

The `ComposeWorkspace` calls `onsubmit(data)` where `data` is a `ComposeRequest` that includes `scheduled_for` if the user selected a time in the Inspector. The shell's `handleDraftSubmit` currently takes no arguments. Need to update signature to accept `ComposeRequest`:

In `DraftStudioShell.svelte` line 300:
```svelte
onsubmit={handleDraftSubmit}
```

The `onsubmit` prop type is `(data: ComposeRequest) => void | Promise<void>`. So `handleDraftSubmit` must accept `ComposeRequest`. Update:

```typescript
async function handleDraftSubmit(data: ComposeRequest) {
    const id = studio.getSelectedId();
    if (id === null) return;
    if (data.scheduled_for) {
        await studio.scheduleDraft(id, data.scheduled_for);
        await fetchDraft(id);
    }
}
```

This means Cmd+Shift+Enter with a selected time in the Inspector will schedule the draft. Without a time, it's a no-op (acceptable for now; publish-now is a later session).
