# Session 9 — Activity History & Restore: Implementation Plan

## Current State

The backend infrastructure for revisions and activity is **fully implemented**:
- `content_revisions` and `content_activity` tables exist (migration `20260308000100`)
- Storage functions: `insert_revision_for`, `list_revisions_for`, `insert_activity_for`, `list_activity_for` in `tuitbot-core`
- API endpoints: `GET /api/drafts/:id/revisions`, `POST /api/drafts/:id/revisions`, `GET /api/drafts/:id/activity` in `draft_studio.rs`
- Frontend API client methods: `api.draftStudio.revisions(id)`, `api.draftStudio.createRevision(id)`, `api.draftStudio.activity(id)` in `client.ts`
- TypeScript types: `ContentRevision`, `ContentActivity` in `types.ts`

**What's missing:** Frontend UI for the history panel, store actions for revisions/restore-from-revision, a server endpoint to restore content from a revision, and the safety-first restore flow with confirmation.

## Key Design Decisions

### 1. Separate `DraftHistoryPanel.svelte` instead of extending `DraftDetailsPanel`
**Rationale:** DraftDetailsPanel is already 935 lines (over the 400-line Svelte limit). Adding revision/activity UI would push it to ~1200+ lines. Instead, create a standalone `DraftHistoryPanel.svelte` that toggles in the details zone. The Shell will manage which panel is visible.

### 2. Restore-from-revision creates a new revision before overwriting
**Rationale:** The session mission says "Make restore safer than any existing undo path by preserving the current state before any destructive revert." On restore: (a) snapshot current content as a `pre_restore` revision, (b) overwrite content with the selected revision's content, (c) log a `revision_restored` activity. This makes restore non-lossy — the user can always go back.

### 3. Server-side restore endpoint (not client-side content patching)
**Rationale:** Restoring from a revision involves: read current content → create pre-restore snapshot → update content → log activity. Doing this atomically on the server prevents race conditions with autosave. A new `POST /api/drafts/:id/revisions/:rev_id/restore` endpoint handles this.

### 4. Confirmation dialog before restore
**Rationale:** Restore replaces current content. Show a simple confirm dialog with the revision's trigger_kind and timestamp, plus a note that the current state will be preserved as a snapshot.

### 5. AI changes get explicit `ai_rewrite` trigger_kind labels
**Rationale:** The session requires AI-generated changes to be surfaced clearly. Revisions with `trigger_kind = 'ai_rewrite'` get a sparkle icon and "AI Rewrite" label in the history panel, making them visually distinct.

## Detailed Implementation

### `restore_from_revision` Handler (draft_studio.rs)

```rust
/// `POST /api/drafts/:id/revisions/:rev_id/restore` — restore content from a revision.
pub async fn restore_from_revision(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
    Path((id, rev_id)): Path<(i64, i64)>,
) -> Result<Json<scheduled_content::ScheduledContent>, ApiError> {
    require_mutate(&ctx)?;

    // 1. Fetch current draft
    let current = scheduled_content::get_by_id_for(&state.db, &ctx.account_id, id)
        .await
        .map_err(ApiError::Storage)?
        .ok_or_else(|| ApiError::NotFound(format!("Draft {id} not found")))?;

    // 2. Fetch target revision (verify ownership)
    let revisions = scheduled_content::list_revisions_for(&state.db, &ctx.account_id, id)
        .await
        .map_err(ApiError::Storage)?;
    let target_rev = revisions.iter().find(|r| r.id == rev_id)
        .ok_or_else(|| ApiError::NotFound(format!("Revision {rev_id} not found")))?;

    // 3. Snapshot current state as pre_restore
    let _ = scheduled_content::insert_revision_for(
        &state.db, &ctx.account_id, id,
        &current.content, &current.content_type, "pre_restore",
    ).await;

    // 4. Update content to revision's content
    let _ = scheduled_content::autosave_draft_for(
        &state.db, &ctx.account_id, id,
        &target_rev.content, &target_rev.content_type, &current.updated_at,
    ).await.map_err(ApiError::Storage)?;

    // 5. Log activity
    let _ = scheduled_content::insert_activity_for(
        &state.db, &ctx.account_id, id, "revision_restored",
        Some(&json!({"from_revision_id": rev_id}).to_string()),
    ).await;

    // 6. Return updated draft
    let updated = scheduled_content::get_by_id_for(&state.db, &ctx.account_id, id)
        .await
        .map_err(ApiError::Storage)?
        .ok_or_else(|| ApiError::NotFound(format!("Draft {id} not found")))?;
    Ok(Json(updated))
}
```

### Store additions (draftStudio.svelte.ts)

```typescript
// State
let revisions = $state<ContentRevision[]>([]);
let activity = $state<ContentActivity[]>([]);

// Getters
export function getRevisions(): ContentRevision[] { return revisions; }
export function getActivity(): ContentActivity[] { return activity; }

// Actions
export async function loadRevisions(): Promise<void> {
    if (selectedId === null) { revisions = []; return; }
    try { revisions = await api.draftStudio.revisions(selectedId); }
    catch { revisions = []; }
}

export async function loadActivity(): Promise<void> {
    if (selectedId === null) { activity = []; return; }
    try { activity = await api.draftStudio.activity(selectedId); }
    catch { activity = []; }
}

export async function restoreFromRevision(revisionId: number): Promise<boolean> {
    if (selectedId === null) return false;
    try {
        const updated = await api.draftStudio.restoreRevision(selectedId, revisionId);
        fullDraft = updated;
        // Update collection preview
        updateDraftInCollection(selectedId, { updated_at: updated.updated_at });
        await loadRevisions();
        await loadActivity();
        return true;
    } catch (e) {
        error = e instanceof Error ? e.message : 'Failed to restore revision';
        return false;
    }
}
```

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| DraftDetailsPanel at 935 lines already over 400 limit | Already true | Low for this session | Not adding to it — creating separate DraftHistoryPanel instead. Follow-up: extract DraftDetailsPanel sections into sub-components in Session 10/11. |
| DraftStudioShell at 488 lines nearing 400 limit | Medium | Low | Panel switching logic is ~30 lines. Still under 520 lines. |
| autosave race with restore | Low | Medium | Restore uses current `updated_at` for the autosave call, so it succeeds atomically. If autosave fires concurrently, one will get a 409 — the conflict resolution UI handles this. |
| Revision list grows unbounded | Low | Low | Document retention strategy in `history-and-restore.md`. For now, SQLite handles thousands of rows fine. Future: add pagination or age-based cleanup. |
