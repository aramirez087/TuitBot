## Design Decisions

### 1. Store Pattern: Svelte 5 Runes (not writable/derived)

The existing stores (approval, calendar, etc.) use Svelte 4 `writable`/`derived` from `svelte/store`. The CLAUDE.md mandates Svelte 5 runes for new code. The `draftStudio` store will use module-level `$state` and exported functions, following the rune pattern.

**Rationale:** Forward-looking; avoids `get()` wrapper calls; reactive by default in Svelte 5 components.

**Trade-off:** Different pattern from existing stores. Acceptable since this is a new feature module with no store interop needed.

### 2. URL-based Selection via `$state` + `replaceState`

The selected draft ID lives in the URL search params (`?id=42`). The store reads the initial value from the URL on mount, and writes back via `history.replaceState` (not `pushState`) to avoid polluting browser history.

**Why not SvelteKit `$page.url`?** We need imperative control over when the URL updates (e.g., after creating a draft). Using `$page.url.searchParams` for reading the initial value on mount, then managing via `replaceState` is the cleanest approach.

**Why `replaceState` not `goto`?** `goto` with `replaceState: true` triggers a SvelteKit navigation which re-renders the `{#key $page.url.pathname}` block in the layout, causing a fade transition. We want silent URL updates. Direct `history.replaceState` avoids this.

### 3. Tab as In-Memory State (not URL param)

The tab (`active` | `scheduled` | `archive`) is component state, not a URL param. Rationale: bookmarking a tab is not useful — users always want to land on "Active". The tab resets to "Active" on navigation.

### 4. Layout: CSS Grid Three-Zone

The shell uses CSS grid with three zones. The right zone is a placeholder for future metadata/details panel (sessions 07-09). For now it renders nothing — the composer surface fills the remaining space.

```
grid-template-columns: 260px 1fr;
```

The rail is 260px fixed. The composer surface gets `1fr`. No right zone column yet — will be added in session 07 when metadata panel is built.

### 5. Composer Surface: Placeholder in Session 04

The center zone shows a read-only preview of the selected draft's content (title + content preview from `DraftSummary`). The full `ComposeWorkspace` binding with server hydration and autosave is Session 05. This avoids a massive session scope.

**What the placeholder shows:**
- If a draft is selected: the draft's content preview text, content type badge, and timestamps. A clear "Editing coming in next update" message is NOT shown — instead, the content just renders as a styled preview area that looks intentional.
- If no draft selected: `DraftEmptyState` with "Select a draft or create a new one".

### 6. Collection Loading Strategy

On mount, the store loads all non-archived drafts (`GET /api/drafts`). Switching to the Archive tab fetches with `?archived=true`. The collection is filtered client-side by tab:
- Active tab: `status === 'draft'`
- Scheduled tab: `status === 'scheduled'`
- Archive tab: all items from the `?archived=true` fetch

### 7. Account Switching

Listen to `ACCOUNT_SWITCHED_EVENT` and reload the collection, clearing the selection.

## Implementation Order

### Step 1: Create `draftStudio.ts` store

**Path:** `dashboard/src/lib/stores/draftStudio.ts`

```typescript
// Module-level $state (Svelte 5 runes)
// Exports:
//   State (read via getter functions or direct import):
//     collection: DraftSummary[]
//     selectedId: number | null
//     tab: 'active' | 'scheduled' | 'archive'
//     loading: boolean
//     error: string | null
//     syncStatus: 'saved' | 'saving' | 'unsaved' | 'offline' | 'conflict'
//
//   Derived:
//     activeDrafts: DraftSummary[]  — collection filtered to status === 'draft'
//     scheduledDrafts: DraftSummary[] — collection filtered to status === 'scheduled'
//     archivedDrafts: DraftSummary[] — the archived collection
//     currentTabDrafts: DraftSummary[] — whichever tab is active
//     selectedDraft: DraftSummary | null — matching selectedId
//     tabCounts: { active: number, scheduled: number, archive: number }
//
//   Actions:
//     loadDrafts(): Promise<void>
//     selectDraft(id: number | null): void  — updates selectedId + URL
//     setTab(tab): void — switches tab, loads archive if needed
//     createDraft(): Promise<number | null> — POST, add to collection, select, return id
//     archiveDraft(id: number): Promise<void>
//     restoreDraft(id: number): Promise<void>
//     duplicateDraft(id: number): Promise<void>
//     initFromUrl(url: URL): void — reads ?id= param on mount
//     reset(): void — clear state (for account switch)
```

Key implementation details:

- Use `$state` for `collection`, `selectedId`, `tab`, `loading`, `error`, `syncStatus`, `archivedCollection`.
- Use `$derived` for filtered lists and `selectedDraft`.
- `selectDraft()` calls `history.replaceState(null, '', newUrl)` to update URL without SvelteKit navigation.
- `createDraft()` calls `api.draftStudio.create()`, inserts a new `DraftSummary` optimistically at the top of the collection, then selects it.
- `archiveDraft()` calls `api.draftStudio.archive(id)`, removes from `collection`, adds to `archivedCollection`. If the archived draft was selected, select the next draft or null.
- Two separate collections: `collection` (active + scheduled) and `archivedCollection`. This avoids re-fetching when switching between Active/Scheduled tabs.
- `loadDrafts()` fetches non-archived drafts into `collection`. The Archive tab triggers a separate `loadArchived()` that populates `archivedCollection`.

## Detailed Store Design

### State Shape

```typescript
let collection = $state<DraftSummary[]>([]);        // active + scheduled drafts
let archivedCollection = $state<DraftSummary[]>([]); // archived drafts (lazy-loaded)
let selectedId = $state<number | null>(null);
let tab = $state<'active' | 'scheduled' | 'archive'>('active');
let loading = $state(true);
let archiveLoaded = $state(false);                   // whether archive was fetched
let error = $state<string | null>(null);
let syncStatus = $state<'saved' | 'saving' | 'unsaved' | 'offline' | 'conflict'>('saved');
```

### Derived Values

```typescript
const activeDrafts = $derived(
  collection.filter(d => d.status === 'draft').sort(byUpdatedDesc)
);
const scheduledDrafts = $derived(
  collection.filter(d => d.status === 'scheduled').sort(byUpdatedDesc)
);
const currentTabDrafts = $derived(
  tab === 'active' ? activeDrafts
  : tab === 'scheduled' ? scheduledDrafts
  : archivedCollection.sort(byUpdatedDesc)
);
const selectedDraft = $derived(
  [...collection, ...archivedCollection].find(d => d.id === selectedId) ?? null
);
const tabCounts = $derived({
  active: activeDrafts.length,
  scheduled: scheduledDrafts.length,
  archive: archivedCollection.length
});
```

### Key Action: `selectDraft(id)`

```typescript
export function selectDraft(id: number | null) {
  selectedId = id;
  const url = new URL(window.location.href);
  if (id !== null) {
    url.searchParams.set('id', String(id));
  } else {
    url.searchParams.delete('id');
  }
  history.replaceState(null, '', url.toString());
}
```

### Key Action: `createDraft()`

```typescript
export async function createDraft(): Promise<number | null> {
  try {
    const result = await api.draftStudio.create({ content_type: 'tweet' });
    const newDraft: DraftSummary = {
      id: result.id,
      title: null,
      content_type: 'tweet',
      content_preview: '',
      status: 'draft',
      scheduled_for: null,
      archived_at: null,
      updated_at: result.updated_at,
      created_at: result.updated_at,
    };
    collection = [newDraft, ...collection];
    tab = 'active';
    selectDraft(result.id);
    return result.id;
  } catch (e) {
    error = e instanceof Error ? e.message : 'Failed to create draft';
    return null;
  }
}
```

### Key Action: `archiveDraft(id)`

```typescript
export async function archiveDraft(id: number) {
  try {
    const result = await api.draftStudio.archive(id);
    const draft = collection.find(d => d.id === id);
    if (draft) {
      collection = collection.filter(d => d.id !== id);
      archivedCollection = [{ ...draft, archived_at: result.archived_at }, ...archivedCollection];
    }
    // If archived draft was selected, select next available or null
    if (selectedId === id) {
      const remaining = tab === 'active' ? activeDrafts : scheduledDrafts;
      selectDraft(remaining[0]?.id ?? null);
    }
  } catch (e) {
    error = e instanceof Error ? e.message : 'Failed to archive draft';
  }
}
```

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `history.replaceState` doesn't update `$page.url` | Low | We only read `$page.url` on initial mount via `initFromUrl()`. Subsequent reads use our own `selectedId` state. |
| Layout padding negation breaks on other pages | None | The negative margins are scoped to `.studio-shell` which only renders on `/drafts`. |
| Archive tab shows stale data after restoring | Medium | `restoreDraft()` moves the item from `archivedCollection` back to `collection`. Both arrays stay in sync. |
| Svelte 5 rune store module — reactivity across components | Low | Module-level `$state` is reactive when imported directly. Use getter functions if needed. Export the `$state` variables directly since Svelte 5 supports this pattern with `$state` at module scope. |
| ComposeWorkspace not bound to drafts yet | Expected | Session 04 scope is the shell and selection only. The composer surface shows a preview placeholder. Session 05 handles binding. |

### Reactivity Clarification for Rune Store

Svelte 5 module-level `$state` requires using `.svelte.ts` extension for the store file to enable rune compilation. The file must be named `draftStudio.svelte.ts`.

**Updated file path:** `dashboard/src/lib/stores/draftStudio.svelte.ts`

This is critical — `.ts` files don't get rune compilation. The `$state` and `$derived` keywords only work in `.svelte.ts` files.
