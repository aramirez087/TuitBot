#### 1b. Handlers

| Handler | Method+Path | Logic |
|---------|-------------|-------|
| `list_studio_drafts` | `GET /api/drafts` | Query `DraftListQuery`, return `Vec<DraftSummary>`. If `archived=true`, call `list_archived_drafts_for`; otherwise `list_drafts_for`. Apply `status`/`search` filtering in-code (or extend SQL later). Truncate `content` to 60 chars for `content_preview`. |
| `get_studio_draft` | `GET /api/drafts/:id` | Call `get_by_id_for`, return full `ScheduledContent`. Return 404 if not found. |
| `create_studio_draft` | `POST /api/drafts` | Accept optional content (default empty tweet `" "`), call `insert_draft_for`, optionally set title via `update_draft_meta_for`, log `created` activity. Return `{ id, updated_at }`. |
| `autosave_draft` | `PATCH /api/drafts/:id` | **Key endpoint.** Compare `body.updated_at` against DB `updated_at`. If DB is newer → 409 `stale_write`. Otherwise `update_draft_for` + return `{ id, updated_at }`. No revision, no activity — this is side-effect free. |
| `patch_draft_meta` | `PATCH /api/drafts/:id/meta` | Call `update_draft_meta_for`. Return updated item. |
| `schedule_studio_draft` | `POST /api/drafts/:id/schedule` | Validate item is `draft` status. Insert revision (trigger: `schedule`). Call `schedule_draft_for`. Log `scheduled` activity. Return `{ id, status, scheduled_for }`. |
| `unschedule_studio_draft` | `POST /api/drafts/:id/unschedule` | Validate item is `scheduled`. Insert revision (trigger: `unschedule`). Transition status back to `draft` + clear `scheduled_for`. Log `unscheduled` activity. Return `{ id, status }`. |
| `archive_studio_draft` | `POST /api/drafts/:id/archive` | Call `archive_draft_for`. Log `archived` activity. Return `{ id, archived_at }`. |
| `restore_studio_draft` | `POST /api/drafts/:id/restore` | Call `restore_draft_for`. Log `restored` activity. Return `{ id }`. |
| `duplicate_studio_draft` | `POST /api/drafts/:id/duplicate` | Call `duplicate_draft_for`. Log `created` activity on new draft. Return `{ id }` (new draft id). |
| `list_draft_revisions` | `GET /api/drafts/:id/revisions` | Call `list_revisions_for`. Return `Vec<ContentRevision>`. |
| `create_draft_revision` | `POST /api/drafts/:id/revisions` | Get current content, call `insert_revision_for` with trigger_kind from body (default `manual`). Return `{ id }`. |
| `list_draft_activity` | `GET /api/drafts/:id/activity` | Call `list_activity_for`. Return `Vec<ContentActivity>`. |

#### 1c. Design decisions for autosave

**Conflict detection via `updated_at` string comparison:**
- Client sends `updated_at` from its last known state in the PATCH body.
- Server fetches the row, compares `updated_at` strings.
- If they don't match → 409 with `{ error: "stale_write", server_updated_at }`.
- This is simpler than `If-Unmodified-Since` headers and works with SQLite's text timestamps.

**Autosave is side-effect free:**
- No revision created (revisions are for meaningful lifecycle events only).
- No activity logged (would flood the activity log).
- Only updates `content`, `content_type`, and `updated_at`.

**Unschedule needs a new storage function:**
- `unschedule_draft_for(pool, account_id, id)` — sets `status = 'draft'`, `scheduled_for = NULL`, updates `updated_at`.
- This doesn't exist yet in the storage layer. Add it to `crates/tuitbot-core/src/storage/scheduled_content/mod.rs` alongside existing functions.

### Step 2: Add `unschedule_draft_for` to storage layer

**File:** `crates/tuitbot-core/src/storage/scheduled_content/mod.rs`

Add one function:

```rust
pub async fn unschedule_draft_for(
    pool: &SqlitePool,
    account_id: &str,
    id: i64,
) -> Result<bool, StorageError> {
    let result = sqlx::query(
        "UPDATE scheduled_content SET status = 'draft', scheduled_for = NULL, \
         updated_at = datetime('now') \
         WHERE id = ? AND account_id = ? AND status = 'scheduled'"
    )
    .bind(id)
    .bind(account_id)
    .execute(pool)
    .await?;
    Ok(result.rows_affected() > 0)
}
```

Also need a query to get `updated_at` for conflict detection in autosave. The existing `get_by_id_for` returns the full `ScheduledContent` which includes `updated_at` — use that.

```rust
// Draft Studio routes (new canonical paths)
.route("/api/drafts", get(content::list_studio_drafts).post(content::create_studio_draft))
.route("/api/drafts/{id}", get(content::get_studio_draft))
.route("/api/drafts/{id}/autosave", patch(content::autosave_draft))
.route("/api/drafts/{id}/meta", patch(content::patch_draft_meta))
.route("/api/drafts/{id}/schedule", post(content::schedule_studio_draft))
.route("/api/drafts/{id}/unschedule", post(content::unschedule_studio_draft))
.route("/api/drafts/{id}/archive", post(content::archive_studio_draft))
.route("/api/drafts/{id}/restore", post(content::restore_studio_draft))
.route("/api/drafts/{id}/duplicate", post(content::duplicate_studio_draft))
.route("/api/drafts/{id}/revisions", get(content::list_draft_revisions).post(content::create_draft_revision))
.route("/api/drafts/{id}/activity", get(content::list_draft_activity))
```

**Backward compatibility:** The existing `/api/content/drafts` routes remain as-is. They are separate handlers in `drafts.rs` and will continue to work. No aliases needed — the old routes stay, the new routes are additive.

**Route ordering:** All literal sub-paths (`/api/drafts/{id}/autosave`, `/api/drafts/{id}/meta`, etc.) naturally don't conflict because `{id}` is in the middle, not at the end competing with a literal segment. Axum handles `/{id}/meta` vs `/{id}/schedule` fine.

**Design decision — PATCH path for autosave:**
The tech architecture spec says `PATCH /api/drafts/:id` for autosave, but this conflicts with the legacy `PATCH /api/content/drafts/:id` which already handles content edits. Two options:
1. Use `PATCH /api/drafts/{id}` and accept that semantics differ from legacy edit (legacy requires content/blocks, autosave uses `updated_at` conflict detection).
2. Use a dedicated `PATCH /api/drafts/{id}/autosave` to avoid confusion.

**Decision:** Use `PATCH /api/drafts/{id}` as spec'd. The `/api/drafts/` path is a completely separate namespace from `/api/content/drafts/`. No ambiguity. The client will migrate to the new namespace over time.

### Step 7: Create blank-draft support in storage

**Decision:** The spec says `POST /api/drafts` should create a "blank" draft. Currently `insert_draft_for` requires non-empty content. Two options:
1. Allow empty content in the storage layer.
2. Insert with a single space `" "` as placeholder content.

**Decision:** Use `" "` (single space) as the placeholder. This avoids changing the storage-layer contract that other callers depend on. The autosave PATCH will overwrite it immediately when the user starts typing. The `content_preview` will show as empty string after trimming.

## Key Design Decisions

### 1. Separate autosave from edit semantics
The existing `edit_draft` handler (`PATCH /api/content/drafts/:id`) requires either `content` or `blocks` and validates content length. The new autosave handler (`PATCH /api/drafts/:id`) takes `content` + `content_type` + `updated_at` and skips validation (drafts can be work-in-progress). This separation ensures autosave never triggers workflow side effects.

### 2. No content validation on autosave
Autosave should accept any content state — even empty or over-limit. The user is in the middle of editing. Validation happens at schedule/publish time, not during autosave.

### 3. `content_preview` server-side truncation
The tech architecture spec calls for truncating content to ~60 chars for list summaries. This is done server-side in the list handler rather than storing a separate column. For threads (JSON blocks), extract the first block's text before truncating.

### 4. Activity logging on create
When a draft is created via the new `/api/drafts` endpoint, log a `created` activity entry. The legacy `/api/content/drafts` endpoint does NOT log activity (preserving backward compat).

### 5. Revision on schedule/unschedule only via new endpoints
The new `schedule_studio_draft` and `unschedule_studio_draft` handlers create revisions. The legacy `schedule_draft` handler does not (no behavior change to existing code).

### 6. No search/tag filtering in SQL initially
For the list endpoint, `search` and `tag` filtering will be done in-application code (filter Vec after loading). With typical draft counts (<200), this is fine. Can optimize to SQL later if needed.

### 7. Backward compatibility via coexistence
Old `/api/content/drafts/*` routes stay exactly as-is. New `/api/drafts/*` routes are a parallel, richer API. The frontend `api.drafts.*` namespace continues to work. The new `api.draftStudio.*` namespace targets the new endpoints. Migration happens organically as UI components switch.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `updated_at` granularity (second-level) causes false negatives in conflict detection | Low | Medium | In practice, two autosave writes from different tabs in the same second is unlikely. Could add a `revision_counter` column later if needed. |
| Thread content preview extraction is fragile | Low | Low | Use `serde_json::from_str` with fallback to raw content truncation if parsing fails. |
| Test DB doesn't run migrations for new tables | Low | High | `init_test_db()` already runs all migrations including the Session 02 migration. Verify in first test run. |
| `unschedule_draft_for` storage function doesn't exist yet | Certain | Medium | Must add it in Step 2 before the handler can work. Small change (~10 lines). |
