# Session 5 Implementation Plan — Composer Binding and Server Autosave

## Mission

Refactor the Draft Studio so that a selected server draft record — not local component-only state — is the source of truth for the composer. Implement debounced server autosave with visible sync status, demote localStorage autosave to crash recovery, and prevent cross-draft state leakage.

## Key Design Decisions

### 1. Server draft is canonical; localStorage is crash recovery only

**Rationale:** The session instructions state "selected server draft record is the source of truth" and "local autosave only as crash recovery fallback." This means:
- On draft selection: fetch full content via `api.draftStudio.get(id)`, hydrate into ComposeWorkspace.
- On edit: debounced PATCH to server (1500ms), immediate localStorage save (500ms) as crash safety net.
- On reload/reconnect: compare localStorage timestamp vs server `updated_at`; only offer recovery if local is newer.

### 2. `DraftSaveManager` class in composerAutosave.ts

Rather than scattering autosave logic across ComposeWorkspace and the store, introduce a `DraftSaveManager` class that encapsulates:
- Draft-ID-scoped localStorage keys (`tuitbot:compose:draft:{id}`)
- Debounced server PATCH (1500ms)
- Debounced localStorage save (500ms)
- `updated_at` tracking for conflict detection
- Sync status reporting (callback to store's `setSyncStatus`)
- Cleanup on draft switch / destroy

**Rationale:** ComposeWorkspace is already 893 lines. Adding server save, conflict handling, and draft-scoped recovery inline would push it over the 500-line Rust limit equivalent. A class also makes the lifecycle explicit: `new DraftSaveManager(id, updatedAt)` → `.save(content)` → `.destroy()`.

### 3. ComposeWorkspace gets optional `draftId` + `initialContent` props

Rather than having ComposeWorkspace fetch its own data (which couples it to the draft studio), the shell fetches the full draft via `api.draftStudio.get(id)` and passes hydration data down:
- `draftId?: number` — when set, enables server autosave mode
- `initialContent?: { mode, tweetText, threadBlocks, attachedMedia, updatedAt }` — hydration payload

**Rationale:** Keeps ComposeWorkspace reusable for non-draft-studio contexts (ComposeModal, home embed). The shell handles the fetch-and-hydrate orchestration.

### 4. Draft switch fully replaces editor state

When `draftId` changes (via `{#key}` or reactive effect):
1. Flush any pending save for the old draft
2. Destroy the old `DraftSaveManager`
3. Clear undo snapshots, recovery banners, media previews, timers
4. Create new `DraftSaveManager` with new draft's `updated_at`
5. Hydrate from the new draft's content

**Implementation:** Use `{#key draftId}` on ComposeWorkspace in DraftStudioShell so Svelte destroys and recreates the component. This is the safest approach — no partial state leaks between drafts.

### 5. Conflict resolution: user-facing choice

On 409 stale_write:
- Set `syncStatus = 'conflict'`
- Show conflict UI in DraftSyncBadge: "Another change was saved. [Use mine] [Reload server]"
- "Use mine": re-fetch to get new `updated_at`, then PATCH again with local content
- "Reload server": re-fetch and hydrate, discarding local changes

**Rationale:** The api-sync-contract.md specifies 409 with `stale_write` response. Silent overwrites risk data loss.

### 6. Content serialization for PATCH

The autosave PATCH sends `{ content, content_type, updated_at }`. The `content` field format:
- **tweet mode:** plain text string
- **thread mode:** `JSON.stringify(blocks.map(b => b.text))` (matching `buildComposeRequest` in composeHandlers.ts)

The full `ThreadBlock[]` with IDs, media_paths, and order is saved to localStorage for crash recovery but the server PATCH only sends the text array (matching the existing content format in `scheduled_content`).

## Detailed Implementation Steps

### Step 1: Extend `composerAutosave.ts` with `DraftSaveManager`

**What:** Add a new `DraftSaveManager` class alongside existing functions (which remain for non-draft-studio use).

```typescript
export class DraftSaveManager {
  private draftId: number;
  private lastServerUpdatedAt: string;
  private localTimer: ReturnType<typeof setTimeout> | null = null;
  private serverTimer: ReturnType<typeof setTimeout> | null = null;
  private onSyncStatus: (status: SyncStatus) => void;
  private destroyed = false;

  constructor(draftId: number, serverUpdatedAt: string, onSyncStatus: (s: SyncStatus) => void);

  // Called on every content change
  save(mode: 'tweet' | 'thread', tweetText: string, blocks: ThreadBlock[], media: AttachedMedia[]): void;

  // Flush pending saves immediately (for draft switch / component destroy)
  flush(): Promise<void>;

  // Tear down timers, flush pending
  destroy(): Promise<void>;

  // Force-save with conflict override (after user chooses "use mine")
  forceServerSave(content: string, contentType: string): Promise<void>;

  // Get last known server updated_at
  getLastServerUpdatedAt(): string;
}
```

Key behaviors:
- `save()` debounces localStorage at 500ms, server PATCH at 1500ms
- Server PATCH: sets syncStatus to `'saving'`, PATCHes, on success sets `'saved'` and updates `lastServerUpdatedAt`, on 409 sets `'conflict'`, on network error sets `'offline'`
- localStorage key: `tuitbot:compose:draft:${draftId}` (scoped per draft)
- `destroy()` flushes pending server save, clears timers, removes draft-scoped localStorage entry

Also add:
- `readDraftAutoSave(draftId: number): AutosavePayload | null` — read draft-scoped recovery data
- `clearDraftAutoSave(draftId: number): void` — clear draft-scoped recovery data

The existing `saveAutoSave` / `readAutoSave` / `clearAutoSave` functions remain unchanged for ComposeModal and home embed use.


6. Parse server content into ComposeWorkspace format:
   - `content_type === 'tweet'`: `{ mode: 'tweet', tweetText: content, threadBlocks: [] }`

**Content parsing helper** (inline in shell or extracted to utility):
```typescript
function parseServerDraft(draft: ScheduledContentItem): HydrationPayload {
  if (draft.content_type === 'thread') {
    const texts: string[] = JSON.parse(draft.content || '[]');
    return {
      mode: 'thread',
      tweetText: '',
      threadBlocks: texts.map((text, i) => ({
        id: crypto.randomUUID(), text, media_paths: [], order: i
      })),
      attachedMedia: [],
      updatedAt: draft.updated_at
    };
  }
  return {
    mode: 'tweet',
    tweetText: draft.content || '',
    threadBlocks: [],
    attachedMedia: [],
    updatedAt: draft.updated_at
  };
}
```

### Step 4: Add `draftId` and `initialContent` props to `ComposeWorkspace.svelte`

**What:** Allow the composer to be hydrated from server data and emit changes to a `DraftSaveManager`.

Props to add:
```typescript
draftId?: number;
initialContent?: {
  mode: 'tweet' | 'thread';
  tweetText: string;
  threadBlocks: ThreadBlock[];
  attachedMedia: AttachedMedia[];
  updatedAt: string;
};
```

Lifecycle changes in `onMount`:
1. If `draftId` is set and `initialContent` provided:
   - Initialize state from `initialContent` (skip recovery check for this draft)
   - Create `DraftSaveManager(draftId, initialContent.updatedAt, setSyncStatus)`
   - Check for crash recovery: `readDraftAutoSave(draftId)`. If found and its timestamp > server `updatedAt`, show recovery banner. Otherwise discard it.
2. If `draftId` is not set: existing behavior (global localStorage autosave)

Change the `autoSave()` function:
```typescript
function autoSave() {
  if (autoSaveTimer) clearTimeout(autoSaveTimer);
  if (draftSaveManager) {
    // Draft Studio mode: delegate to DraftSaveManager
    draftSaveManager.save(mode, tweetText, threadBlocks, attachedMedia);
  } else {
    // Legacy mode: global localStorage
    autoSaveTimer = setTimeout(() => {
      saveAutoSave(mode, tweetText, threadBlocks, attachedMedia);
    }, AUTOSAVE_DEBOUNCE_MS);
  }
}
```

Change `onDestroy`:
```typescript
onDestroy(() => {
  if (draftSaveManager) {
    draftSaveManager.destroy(); // flushes pending server save
  } else {
    flushAutoSave();
    markSessionActive();
  }
  // ... rest unchanged
});
```

The `handleSubmit` function when `draftId` is set should:
- Flush the `DraftSaveManager` before submitting
- Not clear global autosave (since it uses draft-scoped keys)

### Step 6: Handle conflict resolution in the shell

When the user clicks "Use mine" in the conflict badge:
1. Re-fetch the draft to get current `updated_at`
2. Call `draftSaveManager.forceServerSave()` with local content and new `updated_at`
3. On success: `setSyncStatus('saved')`

When the user clicks "Reload server":
1. Re-fetch the draft via `api.draftStudio.get(id)`
2. Parse and re-hydrate ComposeWorkspace (simplest: change the `{#key}` to force remount)
3. `setSyncStatus('saved')`

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `{#key draftId}` causes visible flash/remount on draft switch | Medium | Low | The remount is fast since no network fetch happens during mount (data pre-fetched). Can add a CSS transition on the composer-zone if needed. |
| Thread content format mismatch between server and composer | Medium | High | Validate parsing by checking `JSON.parse` result is an array of strings. Fall back to raw text if parsing fails. Add a test case. |
| Race condition: slow PATCH response arrives after user switched drafts | Medium | Medium | `DraftSaveManager.destroy()` sets `destroyed = true`. The PATCH callback checks this flag and discards results if destroyed. |
| `syncStatus` reactivity across component boundaries | Low | Medium | `getSyncStatus()` getter function already exists. The badge reads via this getter. If issues arise, pass as prop from shell. |
| File size: ComposeWorkspace is already 893 lines | High | Medium | The added code is ~30 lines (prop declarations, manager creation in onMount, modified autoSave function, modified onDestroy). Stays under budget. Most new logic lives in `DraftSaveManager`. |
| Undo snapshots from previous draft leak into new draft | Low | High | `{#key draftId}` destroys and recreates the component, clearing all local state including undo snapshots, timers, and recovery data. |

## Content Format Reference

For the PATCH autosave, the `content` field must match what `buildComposeRequest` produces:
- **tweet:** `tweetText.trim()`
- **thread:** `JSON.stringify(validBlocks.map(b => b.text))` — an array of strings

For localStorage crash recovery, the full `AutosavePayload` is stored (with `ThreadBlock[]` including IDs, media_paths, order) so the editor can fully restore thread state.
