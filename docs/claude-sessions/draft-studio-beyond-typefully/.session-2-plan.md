## Design Decisions

### 1. Migration naming: `20260308000100` (not `20260308000022`)

The session instructions specify the filename `20260308000100_draft_studio_foundation.sql`. The handoff from Session 01 suggested `20260308000022`, but we follow the session instructions since they override. The `000100` sequence number provides ample room for any future insertions.

### 2. File size: extract `scheduled_content.rs` into a module directory

The existing file is 637 lines. Adding ~12 storage functions + tests will push it well past the 500-line Rust file limit. Split into:

```
crates/tuitbot-core/src/storage/scheduled_content/
├── mod.rs          — struct definition, serialize helper, existing functions, re-exports
├── drafts.rs       — new draft studio functions (archive, restore, duplicate, meta update)
├── revisions.rs    — revision insert/list
├── activity.rs     — activity insert/list
├── tags.rs         — tag CRUD, assignment
└── tests.rs        — all tests (existing + new)
```

All public items re-exported from `mod.rs` so `use crate::storage::scheduled_content::*` keeps working. No breaking change to any consumer.

### 3. Struct additions are `Option<String>` with `#[serde(skip_serializing_if = "Option::is_none")]`

New fields on `ScheduledContent`:
- `title: Option<String>` — draft title for the rail display
- `notes: Option<String>` — free-form scratchpad
- `archived_at: Option<String>` — NULL means not archived

Using `skip_serializing_if` avoids sending null fields to existing frontend code that doesn't expect them yet.

### 4. `list_drafts_for` filter change is safe

Adding `AND archived_at IS NULL` excludes archived drafts. Since `archived_at` defaults to NULL on all existing rows, no existing draft is filtered out. The filter is a no-op until the first archive action.

### 5. New structs for supporting tables

```rust
pub struct ContentRevision {
    pub id: i64,
    pub content_id: i64,
    pub account_id: String,
    pub content: String,
    pub content_type: String,
    pub trigger: String,        // ai_rewrite | schedule | unschedule | manual
    pub created_at: String,
}

pub struct ContentTag {
    pub id: i64,
    pub account_id: String,
    pub name: String,
    pub color: Option<String>,
}

pub struct ContentActivity {
    pub id: i64,
    pub content_id: i64,
    pub account_id: String,
    pub action: String,
    pub detail: Option<String>,
    pub created_at: String,
}
```

No struct needed for `content_tag_assignments` — it's a join table operated by `assign_tag_for` / `unassign_tag_for`.

### 6. `duplicate_draft_for` copies content, not metadata

Duplicating a draft creates a new row with the same `content`, `content_type`, and `title` (with " (copy)" suffix). It does NOT copy: tags, revisions, activity, `scheduled_for`, `archived_at`, QA fields. Status is always `draft`.

### 7. `archive_draft_for` sets `archived_at` only, does NOT change `status`

This preserves the ability to archive both drafts and scheduled items. The archive tab in the UI queries `WHERE archived_at IS NOT NULL`. The active/scheduled tabs query `WHERE archived_at IS NULL AND status = '...'`.

This is a deliberate design choice: archiving is orthogonal to status. A scheduled item can be archived (hidden from view but still scheduled), though the UI may prevent this initially.

## Migration SQL

```sql
-- Draft Studio foundation: additive columns + supporting tables.

-- 1. Additive columns on scheduled_content
ALTER TABLE scheduled_content ADD COLUMN title TEXT DEFAULT NULL;
ALTER TABLE scheduled_content ADD COLUMN notes TEXT DEFAULT NULL;
ALTER TABLE scheduled_content ADD COLUMN archived_at TEXT DEFAULT NULL;

-- 2. Revision snapshots
CREATE TABLE IF NOT EXISTS content_revisions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id) ON DELETE CASCADE,
    account_id TEXT NOT NULL,
    content TEXT NOT NULL,
    content_type TEXT NOT NULL,
    trigger_kind TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_content_revisions_content_id ON content_revisions(content_id);

-- 3. Tags
CREATE TABLE IF NOT EXISTS content_tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL,
    name TEXT NOT NULL,
    color TEXT DEFAULT NULL,
    UNIQUE(account_id, name)
);

-- 4. Tag assignments (many-to-many)
CREATE TABLE IF NOT EXISTS content_tag_assignments (
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id) ON DELETE CASCADE,
    tag_id INTEGER NOT NULL REFERENCES content_tags(id) ON DELETE CASCADE,
    PRIMARY KEY (content_id, tag_id)
);

-- 5. Activity log
CREATE TABLE IF NOT EXISTS content_activity (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content_id INTEGER NOT NULL REFERENCES scheduled_content(id) ON DELETE CASCADE,
    account_id TEXT NOT NULL,
    action TEXT NOT NULL,
    detail TEXT DEFAULT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_content_activity_content_id ON content_activity(content_id);
```

**Note:** The technical architecture doc uses `trigger` as the column name in `content_revisions`, but `trigger` is a SQL reserved word in SQLite. Using `trigger_kind` instead. The Rust struct field can still be named `trigger_kind` for clarity, and the API layer (Session 03) can map it to `trigger` in JSON if needed.

### Step 3: Add new structs

In `scheduled_content/mod.rs`, add `ContentRevision`, `ContentTag`, `ContentActivity` structs with `sqlx::FromRow` and `serde::Serialize`.

### Step 4: Add draft studio storage functions in submodules

**`scheduled_content/drafts.rs`:**
- `archive_draft_for(pool, account_id, id)` — `UPDATE scheduled_content SET archived_at = datetime('now'), updated_at = datetime('now') WHERE id = ? AND account_id = ? AND archived_at IS NULL`
- `restore_draft_for(pool, account_id, id)` — `UPDATE scheduled_content SET archived_at = NULL, updated_at = datetime('now') WHERE id = ? AND account_id = ? AND archived_at IS NOT NULL`
- `duplicate_draft_for(pool, account_id, id)` — SELECT original, INSERT copy with `status = 'draft'`, `title = original.title + " (copy)"`, return new id
- `update_draft_meta_for(pool, account_id, id, title, notes)` — `UPDATE scheduled_content SET title = ?, notes = ?, updated_at = datetime('now') WHERE id = ? AND account_id = ?`

**`scheduled_content/revisions.rs`:**
- `insert_revision_for(pool, account_id, content_id, content, content_type, trigger_kind)` — INSERT into content_revisions
- `list_revisions_for(pool, account_id, content_id)` — SELECT * WHERE content_id = ? AND account_id = ? ORDER BY created_at DESC

**`scheduled_content/activity.rs`:**
- `insert_activity_for(pool, account_id, content_id, action, detail)` — INSERT into content_activity
- `list_activity_for(pool, account_id, content_id)` — SELECT * WHERE content_id = ? AND account_id = ? ORDER BY created_at DESC

**`scheduled_content/tags.rs`:**
- `create_tag_for(pool, account_id, name, color)` — INSERT into content_tags, return id
- `list_tags_for(pool, account_id)` — SELECT * WHERE account_id = ? ORDER BY name
- `assign_tag_for(pool, content_id, tag_id)` — INSERT OR IGNORE into content_tag_assignments
- `unassign_tag_for(pool, content_id, tag_id)` — DELETE from content_tag_assignments

All functions return `Result<_, StorageError>` using `StorageError::Query` for DB errors.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| SQLx `FromRow` fails with new nullable columns | Low | High | `Option<String>` maps to `TEXT DEFAULT NULL` cleanly. Test immediately after struct change. |
| `trigger` is SQL reserved word | High | Medium | Use `trigger_kind` as column name instead. Document the divergence from technical-architecture.md. |
| Module directory refactor breaks imports | Low | Medium | Re-export everything from `mod.rs`. Run `cargo build` immediately after refactor, before adding new code. |
| `scheduled_content.rs` at 637 lines + new functions exceeds 500-line limit | Certain | Low | Module directory split is the first step, not deferred. |
| `duplicate_draft_for` race condition (read-then-write) | Very Low | Low | SQLite serializes writes with WAL mode. Single-user desktop app. Acceptable. |
| Root `migrations/` is missing migration 21 | Pre-existing | Low | Note in handoff. Don't fix — out of scope. New migration is correctly mirrored. |
