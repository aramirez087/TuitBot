# Session 7 Implementation Plan — Metadata, Filters, and Organization

## Summary

Ship per-draft organizational tools: inline metadata editing (title, notes, tags, source badge, ready-state), a filter/sort bar in the rail, and a responsive details panel. All metadata editing stays live and inline — no modal forms.

## Design Decisions

### 1. Details panel placement: right third column vs. inside composer zone
**Decision**: Right column in the shell grid, collapsible. On mobile, bottom drawer.

**Rationale**: The composer's existing inspector (voice, AI, revisions) is already right-side inside ComposeWorkspace. Adding a *separate* details panel to the right of the composer creates a natural information hierarchy: rail (navigation) → composer (writing) → details (organization). The details panel only shows when a draft is selected and the user toggles it or it auto-shows on wider viewports.

**Implementation**: Shell grid changes from `260px 1fr` to `260px 1fr auto`. Details panel is 280px when open, 0 when closed. Toggle via `Cmd+Shift+I` or a button in the composer header.

### 2. Tags: server-joined on list vs. separate fetch
**Decision**: Fetch tags for the selected draft separately via `GET /api/drafts/:id/tags`. Don't add tags to the list summary endpoint.

**Rationale**: Adding tags to every DraftSummary requires a JOIN on every list call and complicates the server query. Tags are an organizational detail — needed when viewing a specific draft, not when scanning the rail. The filter-by-tag query param already exists on the list endpoint (via tag ID), so the rail can filter without knowing every draft's tags.

### 3. Filter/sort: client-side vs. server-side
**Decision**: Hybrid. Search uses the existing server `?search=` param. Sort and content-type filter are client-side on the already-loaded collection. Tag filter uses server `?tag=` param.

**Rationale**: The collection is small enough (typically <200 drafts) for client-side sort. Search benefits from server-side because the server has full content, not just preview. Tag filter needs server because tag assignments aren't on the summary.

### 4. Sort options
- Updated (default) — `updated_at DESC`
- Created — `created_at DESC`
- Title A-Z — `title ?? content_preview` ascending
- Scheduled — `scheduled_for ASC` (null-last, only meaningful on scheduled tab)

### 5. Ready-state affordance
A simple visual indicator on the rail item showing whether a draft is "ready" (has content, title, and scheduled time) vs. "in progress". This is a derived client-side check, not a server field.

### 6. Source badge
Display `draft.source` (manual/assist/discovery) as a small badge on the rail item and in the details panel. Already in the DB (`source` column on `scheduled_content`), but not yet in `DraftSummary` or `ScheduledContentItem` frontend types.

```typescript
function sortDrafts(drafts: DraftSummary[], by: string): DraftSummary[] {
  switch (by) {
    case 'created': return [...drafts].sort((a, b) => b.created_at.localeCompare(a.created_at));
    case 'title': return [...drafts].sort((a, b) =>
      (a.title ?? a.content_preview).localeCompare(b.title ?? b.content_preview));
    case 'scheduled': return [...drafts].sort((a, b) =>
      (a.scheduled_for ?? 'z').localeCompare(b.scheduled_for ?? 'z'));
    default: return [...drafts].sort(byUpdatedDesc);
  }
}

function filterBySearch(drafts: DraftSummary[], q: string): DraftSummary[] {
  if (!q.trim()) return drafts;
  const needle = q.toLowerCase();
  return drafts.filter(d =>
    (d.title ?? '').toLowerCase().includes(needle) ||
    d.content_preview.toLowerCase().includes(needle)
  );
}
```

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Shell grid change breaks existing composer layout | Medium | High | Test with details panel both open and closed. Use CSS `auto` for details column width so it collapses cleanly. |
| Tag dropdown N+1 — listing all tags for the tag picker | Low | Low | Account tags are loaded once on mount. Drafts rarely have >20 tags total. |

## Non-Goals for Session 7

- **Bulk tag operations**: No multi-select tagging. One draft at a time.
- **Tag colors in the picker**: Tags have an optional `color` field but the picker shows simple text. Color picker UI deferred.
- **Tags on rail items**: Tags only visible in details panel and as filter chips. No tag pills on `DraftRailItem` to avoid clutter and N+1.
- **Mobile details drawer**: Responsive CSS hides the panel below 1024px. A proper bottom drawer is Session 10.
- **Posted tab server filter**: The posted tab shows client-filtered `status === 'posted'` items. Server-side `?status=posted` already works but the collection fetch doesn't include posted items by default. This is noted but not fixed here (Session 8 concern).
