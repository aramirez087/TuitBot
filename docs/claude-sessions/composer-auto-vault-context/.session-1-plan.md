## Objective

## Codebase Audit Summary

### The Gap

The autopilot draft workflow (`workflow/draft.rs:51-68`) builds full RAG context:
```
config.business.{product_keywords, competitor_keywords, effective_industry_topics()}
  -> winning_dna::build_draft_context(db, keywords, 5, 14.0)
    -> Tier 1: analytics::get_scored_ancestors() [engagement-weighted]
    -> Tier 2: watchtower::get_seeds_for_context() [cold-start fallback]
  -> DraftContext.prompt_block  (~2000 chars max)
  -> ContentGenerator::generate_reply_with_context(..., rag_prompt)
```

The composer assist endpoints (`routes/assist.rs:52-168`) bypass this entirely:
| Endpoint | Method Called | RAG? |
|----------|-------------|------|
| `POST /api/assist/tweet` (line 59) | `gen.generate_tweet(&topic)` | No |
| `POST /api/assist/reply` (line 95) | `gen.generate_reply(...)` | No |
| `POST /api/assist/thread` (line 127) | `gen.generate_thread(&topic)` | No |
| `POST /api/assist/improve` (line 161) | `gen.improve_draft(draft, context)` | No (only user-supplied tone cue) |

### Structural Constraints

1. **ContentGenerator is DB-unaware**: Holds `Box<dyn LlmProvider>` + `BusinessProfile` only. No pool.
2. **BusinessProfile.business is private in ContentGenerator**: The merged keyword set used by `draft.rs` is constructed outside the generator.
3. **No `improve_draft_with_context()`**: Unlike tweet/reply/thread which all have `_with_context` variants, `improve_draft()` has no such overload.
4. **AppState has both db and config_path**: Server handlers can access `state.db` for `build_draft_context()` and `state.config_path` to load business profile keywords.

## Design Decisions

### 1. Helper Placement

**Decision**: Add a shared `resolve_composer_rag_context()` async helper in `routes/assist.rs` that all assist handlers call.

**Rationale**: Keeps RAG resolution at the server layer where `state.db` and `state.config_path` are available. The `ContentGenerator` stays DB-unaware per the three-layer architecture (Toolkit is stateless, Workflow has DB access, but `ContentGenerator` lives at the Toolkit boundary).

**Signature sketch**:
```rust
async fn resolve_composer_rag_context(
    state: &AppState,
    _account_id: &str,
) -> Option<String>
```
Returns `Option<String>` — the formatted prompt block from `DraftContext`, or `None` on any error.

### 2. Keyword Sourcing

**Decision**: Load `Config` from `state.config_path` (same pattern as `get_mode` at `assist.rs:252`) and extract merged keywords from `config.business`.

**Why not expose keywords through ContentGenerator?**: That would add a DB-aware method to a Toolkit-layer struct, violating the three-layer model. The draft workflow already assembles keywords from `Config` at the Workflow layer. The server resolver mirrors this at the server layer.

**Keyword assembly** (identical to `draft.rs:52-54`):
```rust
let mut keywords: Vec<String> = config.business.product_keywords.clone();
keywords.extend(config.business.competitor_keywords.clone());
keywords.extend(config.business.effective_industry_topics().to_vec());
```

**Alternative considered**: Add `BusinessProfile::draft_context_keywords() -> Vec<String>` as a shared helper. This is a Session 02 refinement — the charter should recommend it to avoid keyword-assembly duplication between `draft.rs` and the new resolver.

### 3. `improve_draft` Dual-Context Design

**Decision**: Add `improve_draft_with_context(draft, tone_cue, rag_context)` to `ContentGenerator`. Automatic vault context goes into the system prompt as a `rag_section` (same as tweet/thread). User-supplied `context` remains the tone/style directive.

**Prompt layout for improve with both contexts**:
```
System: You are {product_name}'s social media voice. {description}.
{voice_section}
{persona_section}
{rag_section}           <-- NEW: automatic vault context

Task: Rewrite and improve the draft tweet below...

Tone/style directive: {user_context}    <-- existing: user-supplied quick cue
```

**Coexistence rule**: RAG context is injected into the system prompt. User context is a directive in the task section. They occupy separate prompt slots and cannot conflict.

### 4. Fail-Open Behavior

**Decision**: The resolver catches all errors from `build_draft_context()`, logs at `warn` level, and returns `None`. All assist handlers check for `Some(prompt_block)` and pass it to `_with_context` variants or fall through to the plain variants.

This matches `draft.rs:56-63` which uses `.ok()` to silently discard errors.

### 5. Request/Response Shape Preservation

**Decision**: Zero changes to request or response JSON schemas. All four assist endpoints keep their current `Deserialize`/`Serialize` types unchanged.

### 6. Non-Goals

- No new UI elements, toggles, or note pickers
- No manual vault-note selection in composer
- No new API endpoints
- No changes to request/response shapes for `/api/assist/tweet`, `/api/assist/thread`, `/api/assist/improve`
- No changes to the `/api/assist/reply` endpoint (reply assist is already separate from the draft workflow; it could be enhanced later but is out of scope)
- No multi-account redesign
- No logging of note content beyond what existing prompt assembly permits

### Step 2: Write `charter.md`

Contents to cover:
1. **Problem statement**: Composer assist endpoints bypass the draft-context RAG path that makes autopilot replies higher quality.
2. **Goal**: Automatic, transparent injection of vault context (winning ancestors + content seeds) into all composer generation flows.
3. **Architecture**:
   - Shared `resolve_composer_rag_context()` helper in `routes/assist.rs`
   - Keywords sourced from `Config.business` via `state.config_path`
   - `build_draft_context()` called with standard constants

5. **Non-goals**: No UI, no manual picker, no request changes, no reply-assist changes, no multi-account
6. **Tradeoffs**:
   - Config re-read per assist call (acceptable: file-based config read is fast, and alternatives like caching add complexity)
   - Keyword assembly duplicated between `draft.rs` and the resolver (Session 02 should extract a shared helper on `BusinessProfile`)
   - RAG context adds latency (one DB query per assist call — acceptable since `build_draft_context` typically returns in <10ms)

Risks:
- **Config read failure**: If `config_path` is inaccessible, the resolver returns `None` (fail-open). Low probability since server startup already reads config.
- **Empty vault**: New accounts with no ingested notes and no posting history get `None` context. Generation quality is unchanged from current behavior.
- **Prompt length**: RAG block is capped at `RAG_MAX_CHARS = 2000`. Combined with existing system prompt sections, total prompt stays well within LLM context limits.
- **Keyword overlap**: Merging product_keywords + competitor_keywords + industry_topics may produce duplicates. `get_scored_ancestors()` handles this gracefully (SQL LIKE matching is duplicate-tolerant).

## Key Design Rationale Summary

| Decision | Why |
|----------|-----|
| Resolver in `routes/assist.rs`, not in `ContentGenerator` | Three-layer architecture: ContentGenerator is Toolkit (stateless). DB access belongs at Workflow or Server layer. |
| Config re-read per call | Simple, correct, no cache invalidation bugs. File reads are <1ms. Same pattern used by `get_mode`. |
| `BusinessProfile::draft_context_keywords()` in Session 02 | Eliminates keyword-assembly duplication between `draft.rs` and the resolver. Single source of truth. |
| Separate `improve_draft_with_context()` | Preserves backward compatibility of existing `improve_draft()`. Internal delegation avoids duplication. |
| Fail-open with `warn` log | Matches `draft.rs` pattern (`.ok()`). Generation must never fail due to RAG unavailability. |
| No reply-assist changes | `/api/assist/reply` is used for discovery feed replies, which have a different context model. Out of scope. |
