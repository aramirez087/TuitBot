# Session 5 — Wire Improve and From-Notes: Implementation Plan

## Objective

Wire automatic vault context (RAG) into the `/api/assist/improve` endpoint so that all three composer assist handlers (tweet, thread, improve) benefit from vault-derived context. Preserve the existing meaning of the user-supplied `context` field (tone cue / from-notes instructions) — RAG context is additive only.

## Current State (Post-Session 04)

### What's done
- **Session 02**: `improve_draft_with_context(draft, tone_cue, rag_context)` added to `ContentGenerator` in `mod.rs:290-297`. Full test coverage: prompt injection (`improve_draft_with_context_injects_rag_in_prompt`), absence (`improve_draft_with_context_no_rag_when_none`), dual-context success (`improve_draft_with_context_success`), and base equivalence (`improve_draft_with_context_none_matches_base`).
- **Session 03**: `resolve_composer_rag_context()` helper added to `assist.rs:44-79`. Three fail-open tests (config missing, DB empty, no keywords).
- **Session 04**: `assist_tweet` and `assist_thread` wired to use resolver + `_with_context` variants. Dead-code suppressions removed.

### What remains
- `assist_improve` handler (lines 199-214) still calls `gen.improve_draft(&body.draft, body.context.as_deref())` — no RAG context.

## Key Design Decisions

### 1. User `context` vs automatic `rag_context` are independent parameters

The `AssistImproveRequest.context` field serves three distinct purposes in the frontend:

| Call site | `context` value | Purpose |
|-----------|----------------|---------|
| Inline improve (`⌘J`, ComposeWorkspace:433) | `voiceCue \|\| undefined` | Tone cue (e.g., "more casual") |
| From-notes (ComposeWorkspace:476) | `"${voiceCue}. Expand these rough notes into a polished tweet"` or `"Expand these rough notes into a polished tweet"` | Task instruction + optional tone |
| AI Improve button (ComposeWorkspace:529) | `voiceCue \|\| undefined` | Tone cue |
| Thread inline improve (ThreadFlowLane:304) | `voiceCue \|\| undefined` | Tone cue |
| Drafts page (drafts/+page.svelte:88) | `undefined` | No cue |

In all cases, `context` is a user-supplied directive that the generator inserts as a `Tone/style directive (MUST follow)` instruction (see `improve_draft_inner` at mod.rs:317-322). This is semantically different from the RAG context, which provides background knowledge about winning patterns.

**Decision**: Pass both through as separate parameters. The generator's `improve_draft_inner` already handles them independently — `tone_cue` goes into a "MUST follow" directive block, while `rag_context` goes into the `format_rag_section` helper that renders it as a knowledge context block.

### 2. No frontend change required

The `api.assist.improve(draft, context?)` call signature (`api.ts:1088-1092`) sends `{ draft, context }`. The server's `AssistImproveRequest` struct already deserializes both fields. The backend change is purely server-side: resolving RAG context and passing it as the third parameter to `improve_draft_with_context`. The request and response shapes are unchanged.

### 3. Same call ordering as tweet/thread handlers

Follow the Session 04 pattern: `get_generator()` first (fast, in-memory), then `resolve_composer_rag_context()` (disk I/O + DB), then the `_with_context` call. This ensures missing-LLM errors short-circuit without wasting I/O.

### 4. No new tests needed in this session

Existing coverage is comprehensive:
- **Generator-level**: 6 tests for `improve_draft` / `improve_draft_with_context` including prompt injection and absence assertions.
- **Resolver-level**: 3 fail-open tests in `assist.rs::tests`.
- **Integration-level**: Deferred to Session 06 (server regression tests).

## Files to Modify

### 1. `crates/tuitbot-server/src/routes/assist.rs` — `assist_improve` handler

**Lines 199-214.** This is the only code change in the session.

**Before:**
```rust
pub async fn assist_improve(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
    Json(body): Json<AssistImproveRequest>,
) -> Result<Json<AssistImproveResponse>, ApiError> {
    let gen = get_generator(&state, &ctx.account_id).await?;

    let output = gen
        .improve_draft(&body.draft, body.context.as_deref())
        .await
        .map_err(|e| ApiError::Internal(e.to_string()))?;

    Ok(Json(AssistImproveResponse {
        content: output.text,
    }))
}
```

**After:**
```rust
pub async fn assist_improve(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
    Json(body): Json<AssistImproveRequest>,
) -> Result<Json<AssistImproveResponse>, ApiError> {
    let gen = get_generator(&state, &ctx.account_id).await?;
    let rag_context = resolve_composer_rag_context(&state, &ctx.account_id).await;

    let output = gen
        .improve_draft_with_context(
            &body.draft,
            body.context.as_deref(),
            rag_context.as_deref(),
        )
        .await
        .map_err(|e| ApiError::Internal(e.to_string()))?;

    Ok(Json(AssistImproveResponse {
        content: output.text,
    }))
}
```

**Rationale:**
- Two lines added: `let rag_context = ...` and the call changes from `improve_draft` to `improve_draft_with_context` with three parameters.
- `body.context.as_deref()` (the user tone cue / from-notes instruction) is preserved as the second parameter.
- `rag_context.as_deref()` (vault-derived context) is the third parameter.
- Both `Option<&str>` — either can be `None` independently. The generator handles all four combinations.

## Behavior After the Change

### All five `api.assist.improve()` call sites

| Call site | `body.draft` | `body.context` | `rag_context` (new, server-side) | Net effect |
|-----------|-------------|----------------|----------------------------------|------------|
| Inline improve (`⌘J`) — tweet mode | Selected text or full tweet | Voice cue or `None` | Vault data or `None` | Draft improved with persona + vault knowledge |
| Inline improve (`⌘J`) — thread mode | Selected text or card text | Voice cue or `None` | Vault data or `None` | Same as tweet mode |
| From-notes | Raw notes input | `"${voiceCue}. Expand..."` or `"Expand these rough notes..."` | Vault data or `None` | Notes expanded with instruction + vault knowledge |
| AI Improve button | Full tweet text | Voice cue or `None` | Vault data or `None` | Same as inline improve on full text |
| Drafts page improve | Draft content | `None` | Vault data or `None` | Draft improved with vault knowledge (no voice cue) |

### Fail-open behavior

When vault data is unavailable (no config, no keywords, empty DB), `rag_context` is `None`. The generator's `format_rag_section(None)` returns an empty string. The system prompt is identical to the pre-change version. No degradation.

### Dual-context prompt assembly

In `improve_draft_inner`, the system prompt structure is:

```
You are {product_name}'s social media voice. {product_description}.
{voice_section}         ← brand_voice from config
{persona_section}       ← opinions, experiences, pillars
{rag_section}           ← NEW: vault-derived winning patterns (or empty)

Task: Rewrite and improve the draft tweet below...
{tone_instruction}      ← user-supplied context ("Tone/style directive (MUST follow): {cue}")

Rules:
- Maximum 280 characters.
...
```

The `rag_section` and `tone_instruction` are in separate parts of the prompt. The rag section provides background knowledge (examples of winning tweets), while the tone instruction is a binding directive. They don't conflict.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Resolver latency in interactive path | Low | Low | Same ~5-15ms overhead as tweet/thread handlers. Immaterial for interactive improve calls (<1/s). |
| RAG context conflicts with from-notes task instructions | Very Low | Medium | The task instruction is a "MUST follow" directive placed after the rag section. The LLM will prioritize the explicit instruction. |
| No server integration test for full improve path | Medium | Medium | Deferred to Session 06. Generator-level and resolver-level tests independently cover both layers. |
| Stale vault data leading to irrelevant context in prompts | Low | Low | Inherited from Session 03 design: fail-open, recency weighting, and keyword relevance filtering already handle this. |
