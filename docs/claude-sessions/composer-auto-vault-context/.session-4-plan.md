```rust
pub async fn assist_tweet(
    State(state): State<Arc<AppState>>,
    ctx: AccountContext,
    Json(body): Json<AssistTweetRequest>,
) -> Result<Json<AssistTweetResponse>, ApiError> {
    let gen = get_generator(&state, &ctx.account_id).await?;
    let rag_context = resolve_composer_rag_context(&state, &ctx.account_id).await;

    let output = gen
        .generate_tweet_with_context(&body.topic, None, rag_context.as_deref())
        .await
        .map_err(|e| ApiError::Internal(e.to_string()))?;

    Ok(Json(AssistTweetResponse {
        content: output.text,
        topic: body.topic,
    }))
}
```

**Design decisions:**
- `format: None` — the composer UI doesn't expose format selection; `None` preserves the current behavior of no format constraint (matching the previous `generate_tweet` call which delegates to `generate_tweet_inner(topic, None, None)`).
- `rag_context.as_deref()` — converts `Option<String>` to `Option<&str>` as required by the generator signature.
- Resolver is called **after** `get_generator()` — generator validation failure should short-circuit before the DB/config read.

## Key Design Decisions

### D1: Call Ordering — Generator First, Then Resolver

The resolver is called **after** `get_generator()` succeeds. If the LLM isn't configured, the handler returns `ApiError::BadRequest` immediately without wasting a config read + DB query on RAG resolution.

**Alternative considered:** Calling resolver first (or in parallel). Rejected because:
- The generator check is fast (in-memory map lookup).
- The resolver involves disk I/O (config read) + DB I/O (vault query).
- On the happy path, both are needed; on the error path, failing fast on missing LLM is better UX.

### D2: Passing `None` for Format/Structure

The `_with_context` variants accept optional `TweetFormat` / `ThreadStructure` parameters. The composer UI has no mechanism to select these, so we pass `None` — matching the behavior of the previous `generate_tweet` / `generate_thread` calls which also pass `None` internally.

If the composer UI later wants to expose format/structure selection, the handler signatures don't need to change — just pass `body.format` / `body.structure` instead of `None` and add the fields to the request struct.

### D3: No New Tests in This Session

The existing test coverage is sufficient:
- **Generator-level:** `generate_tweet_with_context_injects_rag` and `generate_thread_with_context_injects_rag` (Session 02) verify RAG injection.
- **Resolver-level:** Three fail-open tests (Session 03) verify the resolver returns `None` on all error paths.
- **Integration gap:** No server-level integration test exercises the full handler → resolver → generator path. This is intentional — Session 06 is specifically scoped for regression tests.

### D4: `assist_improve` Deferred to Session 05

The session instructions scope tweet and thread only. The improve handler has a more complex signature (dual context: user-supplied `tone_cue` + automatic `rag_context`), warranting its own focused session. The handoff will document the exact remaining work.

## Risks and Mitigations

### R1: Clippy Warning After Dead-Code Removal

**Risk:** Removing `#[allow(dead_code)]` might surface other warnings if the function signature doesn't match callers.
**Likelihood:** Very low — the function signature is `async fn resolve_composer_rag_context(state: &AppState, _account_id: &str) -> Option<String>` and callers pass `&state` and `&ctx.account_id`.
**Mitigation:** Quality gate will catch this.

### R2: Resolver Latency in Interactive Path

**Risk:** Adding config read + DB query to every tweet/thread assist call adds ~5-15ms latency.
**Likelihood:** Certain, but immaterial for interactive use (<1 req/s, user won't perceive <20ms).
**Mitigation:** Already accepted in implementation plan's Risk R5. Can be optimized later with caching if needed.

### R3: Missing Import Removal May Cause Compile Error

**Risk:** The `#[allow(unused_imports)]` on the `winning_dna` import (line 14-16) was added because the import was only used in `#[cfg(test)]` blocks. With the resolver now called from live handlers, the import is needed in non-test builds too.
**Likelihood:** Zero — the resolver uses `winning_dna::build_draft_context`, `winning_dna::MAX_ANCESTORS`, and `winning_dna::RECENCY_HALF_LIFE_DAYS` directly. These are referenced in non-test code.
**Mitigation:** Quality gate compilation will verify.
