## Detailed Steps (Execution Order)

### Step 1: Add `BusinessProfile::draft_context_keywords()`

**File:** `crates/tuitbot-core/src/config/types.rs`
**Location:** Inside `impl BusinessProfile` block (after `is_enriched()`, line ~167)

**What to add:**
```rust
/// Returns the merged keyword set used for draft-context retrieval.
///
/// Combines `product_keywords`, `competitor_keywords`, and the
/// effective industry topics into a single owned `Vec<String>`.
/// This is the single source of truth for keyword assembly across
/// draft workflows, composer RAG resolution, and engagement scoring.
pub fn draft_context_keywords(&self) -> Vec<String> {
    let mut keywords: Vec<String> = self.product_keywords.clone();
    keywords.extend(self.competitor_keywords.clone());
    keywords.extend(self.effective_industry_topics().to_vec());
    keywords
}
```

**Design decisions:**
- Returns `Vec<String>` (owned) because all existing callers need an owned Vec for downstream functions like `build_draft_context()` and `ScoringEngine::new()`.
- Preserves exact assembly order from `draft.rs:52-54`: product → competitor → industry.
- Uses `effective_industry_topics()` so it automatically falls back to product_keywords when industry_topics is empty, matching current behavior.
- Does NOT dedup keywords (existing callers don't dedup; adding it would change semantics).

**Rationale:**
- Eliminates the duplicated 3-line keyword assembly in `draft.rs:52-54`, `engagement.rs:64-66`, and the upcoming composer resolver (Session 03).
- The `discover.rs:94-101` version uses iterator chaining for a slightly different purpose (scoring engine), but could also be refactored in a follow-up. Not blocking for this session since it uses `.cloned().collect()` which is functionally identical.

### Step 3: Refactor `improve_draft()` into inner delegation pattern

**File:** `crates/tuitbot-core/src/content/generator/mod.rs`
**Location:** Lines 274-318 (the current `improve_draft` method)

**What to do:**

1. Create a new private `improve_draft_inner()` method that accepts `rag_context: Option<&str>`:

```rust
/// Internal draft improvement with optional RAG context.
async fn improve_draft_inner(
    &self,
    draft: &str,
    tone_cue: Option<&str>,
    rag_context: Option<&str>,
) -> Result<GenerationOutput, LlmError> {
    tracing::debug!(
        draft_len = draft.len(),
        tone_cue = ?tone_cue,
        has_rag_context = rag_context.is_some(),
        "Improving draft",
    );

    let voice_section = self.format_voice_section();
    let persona_section = self.format_persona_context();
    let rag_section = Self::format_rag_section(rag_context);

    let tone_instruction = match tone_cue {
        Some(cue) if !cue.is_empty() => {
            format!("\n\nTone/style directive (MUST follow): {cue}")
        }
        _ => String::new(),
    };

    let system = format!(
        "You are {}'s social media voice. {}.\
         {voice_section}\
         {persona_section}\
         {rag_section}\n\n\
         Task: Rewrite and improve the draft tweet below. \
         Keep the core message but make it sharper, more engaging, \
         and better-written.{tone_instruction}\n\n\
         Rules:\n\
         - Maximum 280 characters.\n\
         - Do not use hashtags.\n\
         - Output only the improved tweet text, nothing else.",
        self.business.product_name, self.business.product_description,
    );

    let user_message = format!("Draft to improve:\n{draft}");
    let params = GenerationParams {
        max_tokens: 150,
        temperature: 0.7,
        ..Default::default()
    };

    self.generate_single(&system, &user_message, &params).await
}
```

2. Refactor existing `improve_draft()` to delegate:

```rust
pub async fn improve_draft(
    &self,
    draft: &str,
    tone_cue: Option<&str>,
) -> Result<GenerationOutput, LlmError> {
    self.improve_draft_inner(draft, tone_cue, None).await
}
```

3. Add new public `improve_draft_with_context()`:

```rust
/// Rewrite/improve an existing draft tweet with optional RAG context
/// injected into the system prompt.
pub async fn improve_draft_with_context(
    &self,
    draft: &str,
    tone_cue: Option<&str>,
    rag_context: Option<&str>,
) -> Result<GenerationOutput, LlmError> {
    self.improve_draft_inner(draft, tone_cue, rag_context).await
}
```

**Design decisions:**
- Follows the exact same `_inner` delegation pattern used by `generate_reply_inner`, `generate_tweet_inner`, and `generate_thread_inner`.
- RAG section is placed between `{persona_section}` and the task block in the system prompt, matching charter specification (charter.md line ~95-99). This is the same position as in tweet/thread generators.
- The existing `format_rag_section()` static method is reused — no new formatting logic.
- `improve_draft()` remains source-compatible: same signature, same behavior (passes `None` for rag_context).

**RAG placement rationale:**
The `rag_section` goes after `{persona_section}` and before the task block. This is identical to how tweet/reply/thread generators place it. The RAG content provides background reference material in the system context, while the task instruction ("Rewrite and improve...") and tone_cue remain as direct instructions. They don't conflict because they serve different roles (context vs. directive).

## Key Design Decisions

### 1. `draft_context_keywords()` does NOT dedup

The existing `draft.rs:52-54` code produces duplicates when `industry_topics` is empty (because `effective_industry_topics()` falls back to `product_keywords`, which are already in the Vec). The new helper preserves this exact behavior. Dedup would change the proven autopilot keyword matching semantics. Downstream consumers (RAG retrieval SQL, scoring engine) are tolerant of duplicate keywords.

### 2. RAG section placement in improve prompt

The `rag_section` is placed after `{persona_section}` and before the task block ("Rewrite and improve..."). This matches the pattern in `generate_tweet_inner`, `generate_reply_inner`, and `generate_thread_inner`. The user's tone_cue remains in the task instruction section as a directive, separate from the RAG context.

### 3. `business()` getter over `pub` field

A read-only accessor preserves encapsulation of the `ContentGenerator` struct. Server routes need read-only access to call `draft_context_keywords()` — they should not be able to mutate the business profile on an existing generator instance.

### 4. `PromptCapturingProvider` for prompt content assertions

The existing `MockProvider` discards system prompts. For testing that RAG sections actually appear in the assembled prompt, we introduce a simple prompt-capturing variant. This is essential for verifying the core contract — "RAG context is injected via format_rag_section() into the system prompt" — rather than just testing that generation succeeds.

### 5. `engagement.rs` refactoring is in scope

Although the session instructions focus on `draft.rs`, `engagement.rs:64-66` has the exact same 3-line pattern and benefits equally from the new helper. Including it reduces total duplication from 3 call sites to 1 (the helper itself). This is a low-risk pure refactor with no semantic change.

## Risks and Mitigations

### R1: Duplicate keywords in `draft_context_keywords()` when `industry_topics` is empty

**Risk:** The fallback in `effective_industry_topics()` causes product_keywords to appear twice.
**Impact:** Minimal — RAG retrieval and keyword matching tolerate duplicates.
**Mitigation:** Document in `core-contract.md`. If it becomes a problem later, add a dedup pass as a separate change with its own test coverage.

### R2: `PromptCapturingProvider` test could be brittle

**Risk:** Tests asserting prompt content are coupled to exact prompt format strings.
**Impact:** Low — prompt format changes are intentional and rare.
**Mitigation:** Assert on substring presence (e.g., `contains("Winning patterns")`) not exact match. This catches injection presence without being fragile to surrounding text changes.

### R3: `improve_draft()` callers must not break

**Risk:** Refactoring `improve_draft()` to delegate to `improve_draft_inner()` could subtly change behavior.
**Impact:** High if it happens — the improve endpoint is user-facing.
**Mitigation:** The delegation passes `None` for `rag_context`, which produces an empty string from `format_rag_section(None)`, which is identical to the current code that doesn't call `format_rag_section` at all (both produce no RAG section in the prompt). The `voice_section`, `persona_section`, `tone_instruction` assembly is copied verbatim.

### R4: `discover.rs` not refactored — inconsistency

**Risk:** `discover.rs:94-101` still uses the old pattern, creating divergence.
**Impact:** Low — it produces the same keyword set, just via iterator chaining instead of Vec extension.
**Mitigation:** Note in handoff. Can be cleaned up in a follow-up session if desired. Not blocking for the core contract.
