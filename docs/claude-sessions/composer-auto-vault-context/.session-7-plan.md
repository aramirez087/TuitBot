```markdown
### Vault Context (Automatic)

When you use AI Assist — whether generating a tweet, generating a thread, or improving a draft — the backend automatically enriches the LLM prompt with context from your vault:

- **Winning patterns:** Your historically best-performing tweets (scored by engagement) provide examples of content that resonated with your audience.
- **Relevant ideas:** Notes ingested into the vault via content sources are surfaced as topic seeds when they match your configured product, competitor, or industry keywords.

This context is injected into the LLM system prompt alongside your existing voice settings and persona — it augments generation without replacing any user-supplied input.

#### How it works

1. When an assist endpoint is called, the server loads your business profile keywords (`product_keywords`, `competitor_keywords`, `industry_topics` from `config.toml`).
2. It queries the local SQLite database for winning ancestors (high-engagement tweets) and content seeds (vault notes matching your keywords).
3. The resulting context block (capped at ~2000 characters) is injected into the system prompt before the generation task.
4. The response shape is unchanged — no new fields, no new request parameters.

#### Affected endpoints

| Endpoint | Context injected |
|----------|-----------------|
| `POST /api/assist/tweet` | Vault context (automatic) |
| `POST /api/assist/thread` | Vault context (automatic) |
| `POST /api/assist/improve` | Vault context (automatic) + user-supplied tone cue (if provided) |
| `POST /api/assist/reply` | No vault context (uses different generation path) |

#### Fallback behavior

If vault data is unavailable — no keywords configured, empty database, config error, or a fresh account with no posting history — generation proceeds exactly as before. The feature is additive: its absence has zero impact on existing functionality.

#### No UI changes required

The vault context is resolved server-side and injected transparently. The existing quick-cue input, "From Notes" panel, and AI Generate/Improve actions in the composer all work through the same endpoints with unchanged request payloads. No frontend code was modified for this feature.
```

**Rationale:**
- Placed under AI Assist because vault context is a property of how AI Assist works, not a separate feature.
- Explicitly mentions `assist_reply` exclusion to prevent confusion.
- Documents fallback behavior for new-account cold-start.
- States "No UI changes" to align with the charter's non-goals and prevent future scope creep.

## Key Design Decisions

### D1: Subsection placement in `composer-mode.md`

**Decision:** Add vault context as a subsection under "AI Assist" rather than a top-level section.

**Rationale:** Vault context is a backend behavior of AI Assist, not a user-facing feature with its own UI or interaction model. Placing it under AI Assist keeps the doc's information architecture aligned with the user's mental model: "when I use AI Assist, the system automatically includes my vault data."

### D2: No mention of `build_draft_context()` internals in user-facing docs

**Decision:** The `composer-mode.md` update describes the feature in terms of "winning patterns" and "relevant ideas" (the prompt headers) rather than module names or function signatures.

**Rationale:** `composer-mode.md` is user-facing documentation. Implementation details belong in the roadmap artifacts (which already have full technical coverage in `charter.md`, `core-contract.md`, `server-rag-contract.md`, `improve-flow.md`, `tweet-thread-wiring.md`, and `test-matrix.md`).

### D3: Release notes scope includes test architecture

**Decision:** The release notes briefly mention the test infrastructure (12 HTTP-level tests, mock LlmProvider) in the session history table.

**Rationale:** This is a roadmap-facing artifact for maintainers, not user-facing. Knowing the test architecture exists and where to find it is important for future contributors.

### D4: No changes to CLAUDE.md or README

**Decision:** This session does not modify CLAUDE.md or any README files.

**Rationale:** CLAUDE.md describes project structure and commands, not feature behavior. The vault context feature doesn't change any commands, crate responsibilities, or architectural layers. The feature is entirely within the existing Workflow/Server pattern described in CLAUDE.md.
