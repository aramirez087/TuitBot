## Key Design Decisions

### 1. Dedicated test file, not appended to api_tests.rs

**Decision:** Create `crates/tuitbot-server/tests/assist_rag_tests.rs` as a standalone integration test file.

**Rationale:** The existing `api_tests.rs` is already large (67KB) and covers many endpoints. A dedicated file keeps the RAG-specific tests cohesive, easy to locate, and avoids inflating the existing file. This follows the same pattern as `compose_contract_tests.rs` (dedicated to compose/draft contracts).

### 2. PromptCapturingProvider at the server test level

**Decision:** Define a `PromptCapturingProvider` (similar to the one in `generator/tests.rs`) that captures the system prompt sent to the LLM and returns deterministic canned responses. This lives inside the test file.

**Rationale:** To verify that vault context actually reaches the LLM prompt through the full HTTP stack (handler → resolver → generator), we need to inspect what the generator sends to the provider. A canned-response mock isn't sufficient alone — we need to capture and assert on the system prompt content. The pattern already exists in `crates/tuitbot-core/src/content/generator/tests.rs:368-410` and can be adapted.

**Implementation:** The provider stores an `Arc<tokio::sync::Mutex<Vec<String>>>` to capture all system prompts across calls (thread generation may call the LLM multiple times). Test assertions check that `"Winning patterns"` or `"Relevant ideas"` appears in the captured prompt when vault data is seeded, and doesn't appear when vault data is absent.

### 3. Test state builder with tempdir-based config

**Decision:** Build a `test_state_with_generator` helper that:
1. Creates a `tempfile::TempDir`
2. Writes a `config.toml` with `product_keywords` that enable the RAG resolver
3. Uses `init_test_db()` for an in-memory SQLite DB
4. Registers a `ContentGenerator` backed by the prompt-capturing mock in the `content_generators` map (keyed by the default account ID)
5. Returns `(Router, Arc<Mutex<Vec<String>>>)` — the router and captured-prompts handle

**Rationale:** The `resolve_composer_rag_context()` function loads config from `state.config_path` and queries `state.db`. To exercise the full path, both must be real. Using `tempfile::TempDir` ensures test isolation. The default account ID (`00000000-0000-0000-0000-000000000000`) is used so no `X-Account-Id` header is needed (matching existing test patterns in `api_tests.rs`).

### 4. Vault data seeding strategy

**Decision:** Seed vault data using the same low-level storage functions used in `source/tests/integration.rs`:
- `watchtower::insert_source_context()` — register a source
- `watchtower::upsert_content_node()` — insert a content node
- `watchtower::insert_draft_seed_with_weight()` — insert a seed directly

For the "with ancestors" path, seed data via:
- `INSERT INTO original_tweets` + `analytics::upsert_tweet_performance()` + `analytics::update_tweet_engagement_score()` — matching the pattern in `winning_dna/tests.rs:617-638`

**Rationale:** This exercises the real `build_draft_context()` code path. We need data in the DB that produces a non-empty `prompt_block`. Two paths exist:
1. **Ancestors path:** Tweets with performance data → `format_ancestors_prompt()` → `"Winning patterns"` header
2. **Seeds (cold-start) path:** Content seeds without ancestors → `format_seeds_prompt()` → `"Relevant ideas"` header

We'll use the seeds path for the "happy path" tests (simpler setup — no need for performance data) and verify the `"Relevant ideas"` string appears in the captured prompt. One test can optionally use the ancestors path to verify that path too.

### 5. Thread response parsing

**Decision:** The mock provider for thread generation must return a valid thread format (6 `---`-separated segments) to avoid the retry/failure logic in `generate_thread_with_context`. The captured prompts list may contain multiple entries due to retries.

**Rationale:** `ContentGenerator::generate_thread_with_context` has retry logic — if the LLM returns fewer than 5 tweets, it retries up to 2 times. By returning a valid 6-segment thread, we avoid spurious retries and keep the test deterministic. The RAG context assertion checks that *any* captured prompt contains the expected string.

### 6. Dual-context test (improve only)

**Decision:** Include a dedicated test for the improve handler where both `body.context` (tone cue) and vault-seeded RAG context are present. Assert the captured prompt contains both the tone directive text and the vault context text.

**Rationale:** This validates the dual-context independence documented in `improve-flow.md`. It's the only handler that accepts both user-supplied context and automatic vault context.

## Risks and Mitigations

### Risk 1: Test isolation — config file collisions
**Impact:** Medium. Tests share the process but each needs its own config file path.
**Mitigation:** Each test helper creates its own `tempfile::TempDir`. The tempdir is held alive by the test function scope (returned alongside the router or stored in a local variable). No cross-test contamination possible.

### Risk 2: Thread generation retry behavior
**Impact:** Low. If the mock response doesn't produce 5+ tweets, the test will see retries and the mock might produce 3 calls instead of 1.
**Mitigation:** Return a 6-segment thread in the mock. The `PromptCapturingProvider` captures all calls, and assertions check `any()` rather than index [0].

### Risk 3: `resolve_composer_rag_context` ignores `_account_id`
**Impact:** Low. The resolver currently has an unused `_account_id` parameter. The seeded DB data isn't account-scoped for vault content (watchtower tables don't filter by account). This is correct behavior for single-account desktop use.
**Mitigation:** Tests use the default account ID consistently. No special account creation needed.

### Risk 4: async_trait dependency for mock provider in server tests
**Impact:** Low. The `LlmProvider` trait uses `#[async_trait::async_trait]`, so the mock must also use it.
**Mitigation:** `async-trait` is a transitive dependency through `tuitbot-core`. The test file imports it directly. If it's not re-exported, add `async-trait = "0.1"` to `[dev-dependencies]` in the server's `Cargo.toml`.

### Risk 5: `analytics::upsert_tweet_performance` and `analytics::update_tweet_engagement_score` visibility
**Impact:** Medium. These functions are used in `winning_dna` tests (inside tuitbot-core) but may not be public.
**Mitigation:** Check if they're `pub`. If not, use direct SQL inserts (like the `winning_dna` tests do with `sqlx::query`). The `sqlx` crate is available through `tuitbot-core` re-exports or as a dev-dependency.

#### a) PromptCapturingProvider
A mock `LlmProvider` that:
- Stores `Arc<Mutex<Vec<String>>>` for captured system prompts
- Accepts a `Vec<String>` of canned responses (cycles through them)
- For thread tests: returns a valid 6-segment `---`-delimited string
- For tweet/improve tests: returns a short string under 280 chars

#### b) Helper: `seed_vault_data_seeds(pool)`
Inserts enough watchtower data to produce a non-empty cold-start prompt_block:
1. `watchtower::insert_source_context(pool, "local_fs", "{}") → src_id`
2. `watchtower::upsert_content_node(pool, src_id, "notes.md", "hash1", Some("Test Notes"), "body", None, None)`
3. `watchtower::insert_draft_seed_with_weight(pool, node_id, "Test hook about growth strategies", Some("tip"), 0.5)`

After this, `build_draft_context(pool, &["test"], 5, 14.0)` should return a `DraftContext` with non-empty `prompt_block` containing `"Relevant ideas"`.

#### c) Helper: `seed_vault_data_ancestors(pool)`
Inserts performance data to exercise the ancestors (non-cold-start) path:
1. `INSERT INTO original_tweets (...) VALUES (DEFAULT_ACCOUNT_ID, 'tw1', 'Testing makes everything better', 'test', 'sent', '2026-02-28T10:00:00Z')`
2. `analytics::upsert_tweet_performance(pool, 'tw1', 10, 5, 3, 500, 82.0)`
3. `analytics::update_tweet_engagement_score(pool, 'tw1', 0.85)`

After this, `build_draft_context(pool, &["test"], 5, 14.0)` should return ancestors with `"Winning patterns"` in the prompt_block.

#### d) Helper: `test_router_with_generator(responses, seed_fn)` → `(Router, Arc<Mutex<Vec<String>>>)`
1. Creates `tempfile::TempDir`
2. Writes `config.toml` with `product_keywords = ["test"]`
3. Calls `init_test_db()`
4. Optionally calls `seed_fn(pool)` to populate vault data
5. Creates `PromptCapturingProvider` with given responses
6. Creates `ContentGenerator` with mock provider + test business profile
7. Inserts generator into `content_generators` map keyed by `DEFAULT_ACCOUNT_ID`
8. Builds `AppState` and returns `(build_router(state), captured_prompts)`

#### e) Helper: `test_router_no_generator()` → `Router`
Same as above but without inserting a generator. For testing the `400 Bad Request` path.

#### f) Helper: `post_json(router, path, body)` → `(StatusCode, serde_json::Value)`
Standard POST with Bearer auth header (reused pattern from `api_tests.rs` / `compose_contract_tests.rs`).
