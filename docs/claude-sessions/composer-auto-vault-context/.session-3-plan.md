# Session 3 Implementation Plan — Add Composer RAG Context Resolver

## Objective

## Key Design Decisions

### 1. Helper Placement: Module-Private Function in `assist.rs`

The resolver is a private `async fn` in `routes/assist.rs`, co-located with the handlers that will call it. Rationale:
- It mirrors the existing `get_generator()` helper pattern at `assist.rs:24-35`.
- It needs `&AppState` (server-layer struct) — not appropriate for `tuitbot-core`.
- All consumers are in the same file.
- No `pub` visibility needed until another module requires it.

### 2. Signature

```rust
async fn resolve_composer_rag_context(
    state: &AppState,
    _account_id: &str,
) -> Option<String>
```

- Returns `Option<String>` — `Some(prompt_block)` when RAG data is available and non-empty, `None` otherwise.
- `_account_id` is kept as a parameter for forward-compatibility (multi-account config loading in future sessions) but unused initially — config is loaded from `state.config_path` (single-account pattern matching `get_mode` at line 252).
- No `Result` wrapper: fail-open is the only mode. All error paths return `None` with a `tracing::warn!`.

### 3. Config Loading: Re-Read Per Call (Same as `get_mode`)

Load `Config` from `state.config_path` on every call using `Config::load(Some(&state.config_path.to_string_lossy()))`. This is the identical pattern used by `get_mode` at `assist.rs:252`.

Tradeoff accepted: Sub-millisecond filesystem read. Caching would require invalidation machinery that isn't justified yet.

### 4. Constants from `winning_dna` Module

Use `winning_dna::MAX_ANCESTORS` and `winning_dna::RECENCY_HALF_LIFE_DAYS` directly — no new magic numbers. These are the same constants used by the autopilot draft workflow in `workflow/draft.rs:57-58`.

### 5. Fail-Open Error Handling

Two failure points, both handled with `warn!` + `None`:

1. **Config load failure** — `Config::load()` returns `Err`. Log: `warn!("composer RAG: failed to load config: {err}")`. Return `None`.
2. **`build_draft_context()` failure** — Storage/DB error. Log: `warn!("composer RAG: failed to build draft context: {err}")`. Return `None`.

Additionally, empty `prompt_block` (no ancestors and no seeds) returns `None` — this is the cold-start case with an empty DB, matching `draft.rs:63-66` which filters on `!s.is_empty()`.

### 6. No Tests in This Session (Deferred to Session 06)

The session-02 handoff suggests "Add a unit/integration test verifying the resolver returns `None` when config is missing or DB has no data." However, after inspection:

- The server crate has minimal test infrastructure (only `mcp.rs` has a trivial non-async `#[test]`).
- Testing `resolve_composer_rag_context()` requires constructing an `AppState` with a real `DbPool` and a temp config file — integration test territory.
- The session instructions list Session 06 as "Add server regression tests."
- The helper's logic is straightforward (3 calls: load config, build context, extract prompt_block) and each component is individually well-tested in core.

**Decision:** Add a minimal unit test that exercises the function with a test DB and a temp config file using `init_test_db()` (available via `test-helpers` feature, already in server's dev-dependencies). This verifies the happy path (empty DB → `None`) and validates the wiring without needing an LLM. If the test setup proves too complex for the scope, document it as deferred to Session 06.

### Step 2: Implement `resolve_composer_rag_context()`

Add the helper in the `// Helpers` section of `assist.rs`, after the existing `get_generator()` function (after line 35):

```rust
/// Resolve optional RAG context from the vault for composer assist handlers.
///
/// Loads the business profile's keyword set, queries winning ancestors and
/// content seeds via `build_draft_context()`, and returns the formatted
/// prompt block. Returns `None` (fail-open) on any error or when no
/// relevant context exists.
async fn resolve_composer_rag_context(
    state: &AppState,
    _account_id: &str,
) -> Option<String> {
    // Load config from disk (same pattern as get_mode).
    let config = match tuitbot_core::config::Config::load(
        Some(&state.config_path.to_string_lossy()),
    ) {
        Ok(c) => c,
        Err(e) => {
            tracing::warn!("composer RAG: failed to load config: {e}");
            return None;
        }
    };

    let keywords = config.business.draft_context_keywords();
    if keywords.is_empty() {
        return None;
    }

    let draft_context = match winning_dna::build_draft_context(
        &state.db,
        &keywords,
        winning_dna::MAX_ANCESTORS,
        winning_dna::RECENCY_HALF_LIFE_DAYS,
    )
    .await
    {
        Ok(ctx) => ctx,
        Err(e) => {
            tracing::warn!("composer RAG: failed to build draft context: {e}");
            return None;
        }
    };

    if draft_context.prompt_block.is_empty() {
        None
    } else {
        Some(draft_context.prompt_block)
    }
}
```

Key implementation notes:
- Early return on empty keywords avoids a pointless DB query.
- Uses `match` instead of `.ok()` to log the specific error (more useful than silent swallowing).
- Final empty check mirrors `draft.rs:66` (`filter(|s| !s.is_empty())`).

## Risks and Mitigations

### Risk 1: `AppState` Field Count Drift

The `test_state()` helper must construct a full `AppState`. If upstream changes add fields to `AppState`, the test won't compile — but this is a compile-time error, not a runtime risk. **Mitigation:** This is actually desirable as a canary. If the test becomes too fragile, extract `test_state()` into a shared test helper in a future session.

### Risk 2: Config Re-Read Latency

Each assist call reads `config.toml` from disk. For typical usage (interactive composer, <1 req/s), this is negligible. **Mitigation:** Already accepted in charter (Design Decision #2). If profiling shows this is hot, caching can be added later without changing the resolver's signature.

### Risk 3: Empty Keywords Short-Circuit

The early return on `keywords.is_empty()` means a brand-new account with no product keywords gets no RAG context. This is correct behavior — no keywords means no topics for retrieval — but worth documenting. **Mitigation:** Document in `server-rag-contract.md`.

### Risk 4: No Handler Wiring Yet

This session adds the resolver but does NOT wire it into `assist_tweet`, `assist_thread`, or `assist_improve`. The resolver sits unused until Session 04. This is intentional — the exit criteria require "one shared composer-RAG resolver and no endpoint yet duplicates raw `build_draft_context()` plumbing." **Mitigation:** Session 04's handoff will specify exact integration points.
