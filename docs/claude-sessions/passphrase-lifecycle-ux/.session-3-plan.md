### Step 1: Add `passphrase_hash_mtime()` to tuitbot-core

**File:** `crates/tuitbot-core/src/auth/passphrase.rs`

**What:** Add a public function that returns the modification time of the `passphrase_hash` file.

**Code to add** (after `reset_passphrase()`, before `#[cfg(test)]`):

```rust
/// Return the modification time of the `passphrase_hash` file.
///
/// Returns `None` if the file does not exist or its metadata cannot be read.
pub fn passphrase_hash_mtime(data_dir: &Path) -> Option<std::time::SystemTime> {
    let hash_path = data_dir.join("passphrase_hash");
    std::fs::metadata(&hash_path)
        .ok()
        .and_then(|m| m.modified().ok())
}
```

**Rationale:** Stateless utility function in the Toolkit layer. No DB or LLM dependency. Returns `Option` because the file may not exist (unclaimed instance).

**Tests:** Add a unit test in the existing `mod tests` block:

```rust
#[test]
fn passphrase_hash_mtime_returns_some_after_create() {
    let dir = tempfile::tempdir().unwrap();
    assert!(passphrase_hash_mtime(dir.path()).is_none());
    ensure_passphrase(dir.path()).unwrap();
    assert!(passphrase_hash_mtime(dir.path()).is_some());
}

#[test]
fn passphrase_hash_mtime_changes_after_reset() {
    let dir = tempfile::tempdir().unwrap();
    ensure_passphrase(dir.path()).unwrap();
    let mtime1 = passphrase_hash_mtime(dir.path()).unwrap();
    // Small sleep to ensure filesystem mtime granularity advances
    std::thread::sleep(std::time::Duration::from_millis(50));
    reset_passphrase(dir.path()).unwrap();
    let mtime2 = passphrase_hash_mtime(dir.path()).unwrap();
    assert!(mtime2 >= mtime1);
}
```

**Risk:** Filesystem mtime granularity on some systems (e.g., ext3 has 1-second granularity). Mitigated by using `>=` comparison everywhere and the 50ms sleep in tests.

### Step 4: Mtime check in login handler

**File:** `crates/tuitbot-server/src/auth/routes.rs`

**What:** Before verifying the passphrase, check if the `passphrase_hash` file has a newer mtime than the last-observed value. If so, re-read the file and update both the in-memory hash and mtime.

**Current code (lines 116-124):**
```rust
    // Check if passphrase auth is configured
    let hash = state.passphrase_hash.read().await;
    let Some(ref hash) = *hash else {
        return (
            StatusCode::SERVICE_UNAVAILABLE,
            axum::Json(json!({"error": "passphrase authentication not configured"})),
        )
            .into_response();
    };
```

**Replace with:**

```rust
    // Check for out-of-band passphrase resets (e.g., CLI --reset-passphrase).
    // A single stat() syscall per login attempt is negligible overhead.
    {
        let disk_mtime = passphrase::passphrase_hash_mtime(&state.data_dir);
        let cached_mtime = *state.passphrase_hash_mtime.read().await;
        let needs_reload = match (disk_mtime, cached_mtime) {
            (Some(disk), Some(cached)) => disk >= cached && disk != cached,
            (Some(_), None) => true,   // File appeared (first claim or reset after deletion)
            (None, Some(_)) => true,   // File disappeared (factory reset)
            (None, None) => false,     // No file, nothing cached — unchanged
        };
        if needs_reload {
            if let Ok(new_hash) = passphrase::load_passphrase_hash(&state.data_dir) {
                let mut hash_guard = state.passphrase_hash.write().await;
                *hash_guard = new_hash;
                let mut mtime_guard = state.passphrase_hash_mtime.write().await;
                *mtime_guard = disk_mtime;
                tracing::info!("passphrase hash reloaded from disk (out-of-band change detected)");
            }
        }
    }

    // Check if passphrase auth is configured
    let hash = state.passphrase_hash.read().await;
    let Some(ref hash) = *hash else {
        return (
            StatusCode::SERVICE_UNAVAILABLE,
            axum::Json(json!({"error": "passphrase authentication not configured"})),
        )
            .into_response();
    };
```

**Design decisions:**

1. **`>=` with `!=` guard:** The charter says "use `>=` comparison" to handle filesystem granularity. However, on the very first login after startup, `disk_mtime == cached_mtime` (we just read it). We don't want to reload every time. So the condition is `disk >= cached && disk != cached`, which is equivalent to `disk > cached` but written to be clear about the intent. Wait — actually, re-reading the charter: "Use `>=` comparison to handle filesystem granularity." The concern is that on some filesystems, writing a file twice in quick succession might produce the same mtime (second-level granularity). With `>=`, any reload triggered by a stale mtime will catch up. But `>=` alone would mean we reload every login because `disk == cached`. We need `>` as the baseline and handle the edge case where mtimes collide by comparing the hash content. However, this adds complexity.

   **Revised approach:** Use `disk_mtime != cached_mtime` as the trigger. This handles:
   - File modified (mtime advances) → reload
   - File replaced with same mtime (extremely unlikely) → miss, but next modification catches it
   - First login after startup (mtime matches) → no reload (correct)

   If we want to handle the filesystem-granularity edge case (two writes in the same second produce same mtime), we accept that this is a nearly impossible scenario for passphrase reset (user runs CLI reset twice in one second). The `!=` comparison is correct for this use case.

   **Final condition:**
   ```rust
   let needs_reload = match (disk_mtime, cached_mtime) {
       (Some(disk), Some(cached)) => disk != cached,
       (Some(_), None) => true,
       (None, Some(_)) => true,
       (None, None) => false,
   };
   ```

2. **Block scope:** The reload logic is in a `{}` block so the write guards are dropped before the subsequent `read()` — otherwise we'd deadlock on `RwLock`.

3. **Error handling:** If `load_passphrase_hash` fails, we silently skip the reload and proceed with the cached value. This is safe — the worst case is verifying against a stale hash, which will fail, and the user retries.

4. **Import needed:** Add `use tuitbot_core::auth::passphrase;` — already imported on line 16 (`use tuitbot_core::auth::{passphrase, session};`).

## Key Design Decisions

### D1: No tracing output during --reset-passphrase

The tracing subscriber is initialized before the early exit (needed if `reset_passphrase()` fails and anyhow formats the error). But the `tracing::info!("starting tuitbot server")` log is moved below the early exit. The `println!` outputs only the passphrase — no "Web login passphrase (reset):" prefix, no newlines, no formatting. This makes the output scriptable.

### D2: Mtime comparison uses `!=` not `>=`

The charter recommended `>=` for filesystem granularity. However, `>=` would cause a reload on every single login (since `disk == cached` after startup). The `!=` comparison correctly detects changes while avoiding unnecessary reloads. The edge case of two writes producing the same mtime (sub-second on most modern filesystems) is not a realistic concern for passphrase reset operations.

### D3: Reload scope is login-only

The mtime check runs only in the `login` handler, not in the auth middleware or status endpoint. Login is the only path that verifies the passphrase against the hash. The auth middleware checks session cookies and bearer tokens — it never touches the passphrase hash. This keeps the overhead contained.

### D4: LAN reset endpoint also updates mtime

Even though the mtime reload in login would eventually catch up, explicitly updating the mtime in the LAN reset endpoint prevents a redundant file read on the next login and avoids a confusing "passphrase hash reloaded" log line.

### D5: Settings init (claim) and factory reset also update mtime

Same rationale as D4. Any in-process mutation of the passphrase hash should keep the mtime cache in sync.
