## Root-Cause Analysis: Three Reported Failures

### Failure 1: No passphrase after onboarding

**Symptom:** A user completes the onboarding wizard but is never shown a usable passphrase or a clear recovery path.

**Root cause in code:**

The `alreadyClaimed` path in onboarding skips the ClaimStep entirely. This occurs when `config.toml` is deleted (e.g., user wants to reconfigure) but `passphrase_hash` still exists on disk.

- `dashboard/src/routes/+layout.svelte:50-58` — When `configStatus()` returns `configured: false, claimed: true`, the layout redirects to `/onboarding?claimed=1`.
- `dashboard/src/routes/onboarding/+page.svelte:31-32` — `alreadyClaimed = $page.url.searchParams.get('claimed') === '1'` and `showClaimStep = !isTauri && !alreadyClaimed`. This results in 8 steps (no "Secure" step).
- `dashboard/src/routes/onboarding/+page.svelte:152-154` — The submit function skips the claim when `showClaimStep` is false.

**Result:** The user completes onboarding, gets redirected to `/content?compose=true`, and has no idea what their passphrase is. If their session expires, they're locked out with no self-service recovery path visible during the onboarding flow.

**Secondary scenario (Tauri):** In Tauri mode, `showClaimStep` is false because `isTauri` is true. The user never sees a passphrase. This is intentional per Design Decision 5, but if they later switch to LAN mode, they have no passphrase. The LAN settings page allows generating one, but this isn't communicated during onboarding.

### Failure 2: Noisy `--reset-passphrase` output

**Symptom:** Running `tuitbot-server --reset-passphrase` produces server startup noise (DB init, API token creation, config loading, LLM initialization, watchtower startup) and then fails with "Address already in use" if the server is already running.

**Root cause in code:**

`crates/tuitbot-server/src/main.rs:78-83` — The `--reset-passphrase` flag is handled WITHIN the normal server startup flow, not as an early exit:

```rust
let passphrase_hash = if cli.reset_passphrase {
    let new_passphrase = passphrase::reset_passphrase(db_dir)?;
    println!("\n  Web login passphrase (reset): {new_passphrase}\n");
    tracing::info!("Passphrase has been reset");
    passphrase::load_passphrase_hash(db_dir)?
} else if cli.host == "0.0.0.0" { ... }
```

After this block, main.rs continues to:
- Line 71: `storage::init_db()` — SQLite pool creation + migration
- Line 74: `auth::ensure_api_token()` — API token file I/O
- Lines 112-233: Broadcast channels, config loading, LLM provider init, watchtower, AppState
- Line 245: `TcpListener::bind()` — **fails** if port 3001 is already in use

The user sees: tracing startup log → passphrase printed → more startup logs → "Address already in use" error. The new passphrase IS printed, but buried in noise, and the process exits with an error code despite the reset succeeding on disk.

**Note:** Lines 64-69 show tracing init and DB path derivation happen BEFORE the passphrase reset block, and line 71 (`storage::init_db`) happens BEFORE it too. So even the "successful" part includes DB initialization overhead.

### Failure 3: Stale in-memory auth after out-of-band reset

**Symptom:** After running `tuitbot-server --reset-passphrase` as a separate process (while the server is running), the new passphrase doesn't work for login until the server is restarted.

**Root cause in code:**

- `crates/tuitbot-server/src/state.rs:40` — `passphrase_hash: RwLock<Option<String>>` holds the bcrypt hash in memory.
- `crates/tuitbot-server/src/auth/routes.rs:117` — The login handler reads `state.passphrase_hash.read().await` and verifies against the IN-MEMORY value.
- `crates/tuitbot-core/src/auth/passphrase.rs:163-179` — `reset_passphrase()` writes the new hash directly to disk at `data_dir/passphrase_hash`.

The running server loaded the old hash into `state.passphrase_hash` at startup (main.rs:99-101) and never re-reads the disk file. There is no filesystem watcher, no periodic refresh, and no mechanism for the CLI reset process to notify the running server.

**Contrast with LAN reset:** `crates/tuitbot-server/src/routes/lan.rs:43-55` — The LAN reset endpoint correctly calls `passphrase::reset_passphrase()` AND then updates `state.passphrase_hash.write().await` with the new hash from disk. This works because it runs in-process.

**The gap:** Out-of-band resets (CLI) modify the file but cannot update the in-memory state of a separate process.

### 3. Design Decisions

**Decision A: Show passphrase recovery guidance in claimed-but-unconfigured onboarding**
- When `alreadyClaimed` is true, the onboarding flow should display explicit guidance about the passphrase (not skip it silently)
- Options: (a) show existing passphrase info with "you already have a passphrase, use it to log in after onboarding" message, (b) show a summary step that reminds the user they can reset via CLI or LAN settings
- Recommended: Option (b) — a lightweight "Your Instance is Secured" summary step that shows recovery paths without regenerating the passphrase
- Files: `dashboard/src/routes/onboarding/+page.svelte`, `dashboard/src/lib/components/onboarding/ClaimStep.svelte`

**Decision B: Make `--reset-passphrase` a fast-path maintenance command**
- The flag should trigger passphrase reset and exit BEFORE DB init, API token, config loading, LLM setup, watchtower, and port binding
- Move the reset logic to an early guard in main.rs, after CLI parsing and data_dir derivation but before anything else
- The output should be clean: just the new passphrase and a success message
- Files: `crates/tuitbot-server/src/main.rs`

**Decision C: Live-reload passphrase_hash from disk on login**
- The login handler should re-read the passphrase_hash from disk when the file's mtime is newer than the last load time
- This avoids a filesystem watcher (unnecessary complexity for a single file) while ensuring out-of-band resets take effect immediately
- Alternative considered: filesystem watcher (notify crate) — rejected as overengineered for a single file read on login attempts
- Alternative considered: signal-based reload (SIGHUP) — rejected as not portable to Windows/Tauri
- Implementation: Add a `passphrase_hash_mtime` to AppState, check mtime on each login attempt, reload from disk if changed
- Files: `crates/tuitbot-server/src/auth/routes.rs`, `crates/tuitbot-server/src/state.rs`, `crates/tuitbot-core/src/auth/passphrase.rs`

## Key Design Decisions and Rationale

### Decision A: Recovery guidance in onboarding (not passphrase regeneration)

**Rationale:** When the instance is already claimed, we cannot show the existing passphrase (we only have the bcrypt hash). Regenerating the passphrase here would bypass the one-shot claim security model. Instead, we show clear guidance: "Your instance is already secured. After setup, log in with your passphrase, or reset it via CLI/settings."

**Alternative rejected:** Allow re-claiming when `alreadyClaimed` is true. This would weaken the security model by allowing any browser to overwrite an existing passphrase.

### Decision B: Early fast path for --reset-passphrase

**Rationale:** The reset operation only needs `data_dir` (to find `passphrase_hash`). It doesn't need a database, API token, config, LLM provider, watchtower, or network binding. Moving it before all of these eliminates noise and prevents "Address already in use" errors.

**Implementation detail:** After `Cli::parse()`, derive `db_dir` from `cli.config` (lines 58-62), then:
```rust
if cli.reset_passphrase {
    let new_passphrase = passphrase::reset_passphrase(db_dir)?;
    println!("{new_passphrase}");
    return Ok(());
}
```

This is 4 lines. The rest of main.rs is unchanged.

### Decision C: Mtime-based refresh on login

**Rationale:** Login attempts are infrequent (rate-limited to 5/minute per IP). A single `stat()` syscall on each login attempt adds negligible overhead and catches out-of-band resets immediately. A filesystem watcher would add a dependency (notify crate), a background task, and complexity for a problem that only manifests on login.

**Implementation detail:** In the login handler, before verifying the passphrase:
1. `stat()` the `data_dir/passphrase_hash` file to get mtime
2. Compare with `state.passphrase_hash_mtime`
3. If newer (or first check), re-read the file and update both the hash and mtime in AppState
4. Proceed with verification against the (possibly refreshed) hash

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| ClaimStep changes break the working claim flow | High | Regression test: fresh web install must still work end-to-end |
| Early exit for --reset-passphrase changes error codes | Low | Test that reset returns exit code 0 |
| Mtime granularity issues on some filesystems | Low | Use `>=` comparison, not `>`, and always reload on first login |
| Parallel login + file read race condition | Very Low | RwLock ensures atomic read/write; worst case is one stale verification attempt |
| Session 02 frontend changes introduce Svelte 5 rune issues | Medium | Run `npm run check` and `npm run build` as quality gates |
