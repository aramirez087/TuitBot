# Session 2 Implementation Plan — Onboarding and Recovery UX

**Date:** 2026-03-01
**Scope:** Frontend only — no backend changes
**Branch:** fix/uninstall (continuing from Session 01)

## Summary

This session fixes two problems:
1. The `alreadyClaimed` onboarding path skips the Secure step entirely, leaving users without passphrase knowledge or recovery guidance.
2. After completing onboarding in the `alreadyClaimed` path, the user is redirected to `/content?compose=true`, which bounces to `/login` via the layout gate — an unnecessary double redirect.

The fix adds a second mode to `ClaimStep.svelte` that shows recovery information instead of passphrase generation, and adjusts the onboarding page to always include the Secure step in web mode.

## Design Decisions

### D1: Reuse `ClaimStep.svelte` with a mode prop (not a new component)

**Rationale:** The two modes share the same position in the wizard (step 9), the same styling conventions, and conceptually represent the same "security" concern. A prop-based branch keeps the component count low and avoids duplicating the step's structural markup.

### D2: Reuse the `saved` bindable for the acknowledgment checkbox

**Rationale:** In fresh-claim mode, `saved` means "I've saved my passphrase." In `alreadyClaimed` mode, it means "I understand my recovery options." The parent only cares about a boolean gate for advance — reusing the same binding avoids adding a new prop or state channel between parent and child.

### D3: Redirect `alreadyClaimed` users to `/login` after submit (not `/content`)

**Rationale:** The `alreadyClaimed` path does not create a session (no claim, no csrf_token in response). Redirecting to `/content?compose=true` would cause the layout gate to detect no session and bounce to `/login` — a confusing double redirect. Going directly to `/login` is explicit and matches acceptance scenario A2: "completes config, redirected to login."

### D4: No changes to login page

**Rationale:** The login page at `dashboard/src/routes/login/+page.svelte` already has:
- "Forgot your passphrase?" section (line 92)
- CLI reset command (line 94, line 96)
- Clear, actionable copy

The charter marks this as "likely no-op." The existing copy is accurate and consistent with the recovery guidance added to `ClaimStep`.

### Step 2: Modify `ClaimStep.svelte`

**File:** `dashboard/src/lib/components/onboarding/ClaimStep.svelte`

**Changes:**

1. **Add `alreadyClaimed` prop** to the `$props()` destructure:
   ```typescript
   let {
       passphrase = $bindable(''),
       saved = $bindable(false),
       alreadyClaimed = false,
   }: {
       passphrase: string;
       saved: boolean;
       alreadyClaimed?: boolean;
   } = $props();
   ```

2. **Guard passphrase generation** — skip `onMount` generation when `alreadyClaimed`:
   ```typescript
   onMount(() => {
       if (!alreadyClaimed) {
           generatedPhrase = generatePassphrase();
           passphrase = generatedPhrase;
       }
       return () => {
           if (copyTimeout) clearTimeout(copyTimeout);
       };
   });
   ```

3. **Branch the template** — wrap existing claim UI in `{#if !alreadyClaimed}` and add a new `{:else}` block for recovery guidance:

   **Recovery guidance block content:**
   - **Title:** "Your Instance is Secured"
   - **Description:** "This instance already has a passphrase set. Since passphrases are stored as one-way hashes, we can't display the original. After setup, you'll need your passphrase to log in."
   - **Recovery paths list:**
     - "If you remember your passphrase, use it to log in after setup completes."
     - "Reset from the terminal:" + `<code>tuitbot-server --reset-passphrase</code>`
     - "Or if using cargo:" + `<code>cargo run -p tuitbot-server -- --reset-passphrase</code>`
     - "Reset from Settings > LAN Access once logged in."
   - **Info box** (styled like the existing warning-box but with info tone): "After setup, you'll be redirected to the login page. Use your passphrase or reset it using the methods above."
   - **Acknowledgment checkbox** reusing the `saved` binding: "I understand how to access my instance"

   **Structural approach:** The top-level `<div class="step">` stays. Inside it:
   ```svelte
   {#if alreadyClaimed}
       <!-- Recovery guidance variant -->
       <h2 class="step-title">Your Instance is Secured</h2>
       <p class="step-description">...</p>
       <div class="recovery-paths">...</div>
       <div class="info-box">...</div>
       <label class="save-checkbox">
           <input type="checkbox" bind:checked={saved} />
           <span>I understand how to access my instance</span>
       </label>
   {:else}
       <!-- Existing claim UI (unchanged) -->
       <h2 class="step-title">Secure Your Instance</h2>
       ... (all existing content from line 76-145 moves here)
   {/if}
   ```

4. **Add styles** for the new elements:
   - `.recovery-paths` — vertical list with gap
   - `.recovery-path` — styled similarly to existing `.warning-box` items
   - `.info-box` — styled like `.warning-box` but with accent/info color instead of warning
   - `.recovery-code` — styled like the login page's `.help-code` (mono font, subtle background)
   - Reuse existing `.step`, `.step-title`, `.step-description`, `.save-checkbox` styles

**What does NOT change:**
- The entire `{:else}` (non-alreadyClaimed) branch is identical to the current file
- No changes to passphrase generation logic, copy, regenerate, toggleCustom, handleCustomInput, or copyToClipboard
- All existing styles remain unchanged

### Step 3: Modify `onboarding/+page.svelte`

**File:** `dashboard/src/routes/onboarding/+page.svelte`

**Changes:**

1. **Fix `showClaimStep` derivation** (line 32):
   ```typescript
   // Before:
   let showClaimStep = $derived(!isTauri && !alreadyClaimed);
   // After:
   let showClaimStep = $derived(!isTauri);
   ```
   This ensures the 'Secure' step is always included in web mode, whether the instance is already claimed or not.

2. **Fix `canAdvance()` case 8** (line 72-73):
   ```typescript
   // Before:
   case 8: // Secure (claim) — only when not Tauri
       return claimPassphrase.trim().length >= 8 && passphraseSaved;
   // After:
   case 8: // Secure — recovery acknowledgment or passphrase save
       if (alreadyClaimed) return passphraseSaved;
       return claimPassphrase.trim().length >= 8 && passphraseSaved;
   ```
   When `alreadyClaimed`, we don't require a passphrase — just the acknowledgment checkbox. `passphraseSaved` is reused as the acknowledgment flag (see Design Decision D2).

3. **Fix redirect after submit** (around line 170):
   ```typescript
   // Before:
   onboardingData.reset();
   goto('/content?compose=true');
   // After:
   onboardingData.reset();
   if (alreadyClaimed) {
       goto('/login');
   } else {
       goto('/content?compose=true');
   }
   ```

4. **Pass `alreadyClaimed` prop to ClaimStep** (line 238):
   ```svelte
   <!-- Before: -->
   <ClaimStep bind:passphrase={claimPassphrase} bind:saved={passphraseSaved} />
   <!-- After: -->
   <ClaimStep bind:passphrase={claimPassphrase} bind:saved={passphraseSaved} {alreadyClaimed} />
   ```

5. **Update the comment** on line 28-29 to reflect the new logic:
   ```typescript
   // Tauri/bearer mode skips the claim step. Web mode always shows it —
   // either as passphrase generation (fresh) or recovery guidance (already claimed).
   ```

**What does NOT change:**
- Steps array logic: `showClaimStep ? [...BASE_STEPS, 'Secure'] : BASE_STEPS` — still correct, now 'Secure' shows for both fresh and claimed web mode
- `isClaimStep` derivation: `showClaimStep && currentStep === steps.length - 1` — still correct

- Race condition retry logic — still correct (no `config.claim` in `alreadyClaimed` mode)
