# Session 3 Implementation Plan ŌĆö Thread Splitting & Flow Lane

## Mission

Turn thread writing into a fast continuous lane that beats Typefully on split speed, clarity, and keyboard flow while preserving current data contracts (`ThreadBlock[]`, `ComposeRequest`, autosave, approval queue).

## Current State Audit

### Component Hierarchy (thread path)

```
ComposeWorkspace.svelte (741 lines)
  ŌööŌöĆ ThreadComposer.svelte (434 lines)     ŌåÉ orchestrator: state, CRUD, split/merge, keyboard, validation
       ŌööŌöĆ ThreadFlowLane.svelte (137 lines) ŌåÉ presentational: iterates blocks, "Add tweet" button
            ŌööŌöĆ ThreadFlowCard.svelte (295 lines) ├Ś N ŌåÉ textarea, MediaSlot, separator tools
```

### Problems with this structure

1. **Dual-state ownership**: ComposeWorkspace holds `threadBlocks`, ThreadComposer holds its own `blocks` synced via `$effect` on `initialBlocks`. This means every AI generation path pushes blocks from workspace ŌåÆ ThreadComposer (by ID comparison), introducing sync fragility.
2. **Intermediary indirection**: ComposeWorkspace calls `threadComposerRef.handlePaletteAction()` and `threadComposerRef.handleInlineAssist()` ŌĆö the workspace can't directly manipulate thread state.
3. **"Form builder" feel**: ThreadFlowCard has `border-left: 2px solid` boxing, a `border-bottom` separator line, and ThreadFlowLane has a dashed "Add tweet" button at the bottom ŌĆö all contributing to a form-like card stack.
4. **Split shortcut is Cmd+Shift+Enter** (three keys), not the benchmark's `Cmd+Enter` (two keys). No paragraph-aware paste splitting.
5. **"Add tweet card" language** appears in ThreadFlowLane button and CommandPalette actions.

### What works well (preserve)

- `ThreadBlock[]` contract and `ComposeRequest` serialization in `handleSubmit`
- `data-block-id` DOM attribute pattern for cursor management
- Split word-boundary heuristic (┬▒10 chars)
- Merge with previous (Backspace at pos 0) and merge with next
- Alt+Arrow reorder
- Tab/Shift+Tab navigation between blocks
- Drag-and-drop reorder with live `dropTarget` visual
- MediaSlot per block
- Autosave debounce with 7-day TTL
- Validation: Ōēź2 non-empty blocks, per-block char limit, Ōēż4 media

## Architecture Decision: Absorb ThreadComposer into ThreadFlowLane

### Decision

Eliminate the `ThreadComposer` intermediary. Lift its logic into two places:

1. **`threadOps.ts`** (NEW) ŌĆö pure, testable functions for block CRUD, split, merge, reorder, validate.
2. **`ThreadFlowLane.svelte`** ŌĆö becomes the smart thread orchestrator (absorbs ThreadComposer's state, keyboard handling, drag logic, focus management, and exported methods).

ComposeWorkspace renders ThreadFlowLane directly instead of ThreadComposer.

### Rationale

- Eliminates dual-state sync (`$effect` on `initialBlocks` goes away).
- ThreadFlowLane already receives all the callback props ŌĆö upgrading it to own state is a natural evolution.
- Pure utility extraction to `threadOps.ts` makes thread logic testable without Svelte.
- ComposeWorkspace changes are minimal: swap one import, same callback API.
- ThreadComposer can remain in the codebase (untouched) for reference or later deletion.

## Design Decisions

### D1: Cmd+Enter splits, Cmd+Shift+Enter submits (in thread mode)

**Current**: `Cmd+Enter` = submit globally; `Cmd+Shift+Enter` = split/add in ThreadComposer.

**New**: In thread mode, `Cmd+Enter` splits at cursor (the fast path). Submit in thread mode becomes `Cmd+Shift+Enter` or the button. In tweet mode, `Cmd+Enter` remains submit.

**Implementation**: ComposeWorkspace's `handleKeydown` checks mode for `cmd+enter`:
- Tweet mode: `handleSubmit()` (unchanged)
- Thread mode: no-op (let the event propagate to ThreadFlowLane's card keydown handler, which catches it and splits)

For `cmd+shift+enter` in thread mode: ComposeWorkspace catches this and calls `handleSubmit()`.

**CommandPalette update**: The `submit` action gets shortcut `cmd+shift+enter` when in thread mode. Add a new `split` entry showing `cmd+enter` for thread mode.

### D2: Paragraph-aware paste auto-split

When the user pastes text containing `\n\n` (double newline / paragraph break) into an **empty** block:
1. Split the pasted text on `\n\n`
2. Set the current block's text to the first paragraph
3. Create new blocks for subsequent paragraphs
4. Focus the last created block

**Constraints**:
- Only triggers when the target block is empty (prevents disrupting in-progress editing)
- Minimum 2 paragraphs needed to trigger (single paragraph = normal paste)
- Each resulting block is trimmed
- Empty paragraphs are discarded
- Media stays on the original block

**Implementation**: Add a `paste` event handler on the textarea in ThreadFlowCard that calls a new `onpaste(id, text)` callback. ThreadFlowLane handles the split logic using `threadOps.splitFromPaste()`.

### D3: Double-empty-line auto-split (deferred to Session 4)

This feature has too many edge cases for Session 3:
- Detecting intent (intentional blank lines vs. thread break signal)
- Interaction with undo
- IME composition interruption
- User confusion when text disappears

**Scope cut**: Session 3 adds only `Cmd+Enter` split and paste auto-split. Double-empty-line auto-split is a candidate for Session 4 as a progressive enhancement.

### D4: Spine visual ŌĆö persistent left line with neutral dot markers

**Visual spec**:
- A 1px vertical line in `--color-border-subtle` runs the full height of the lane
- Positioned at `left: 20px` (center of a 40px gutter)
- Each block gets a 12px neutral dot (circle) at the top of its writing area, centered on the spine
- The dot uses `--color-border-subtle` by default, `--color-accent` when focused
- No user avatar images (too noisy, requires API fetch) ŌĆö just dots

**Mobile (Ōēż640px)**: Spine and dots hide; the existing left accent bar returns but thinner (1px solid instead of 2px).

**CSS approach**: The spine is a `::before` pseudo-element on `.flow-lane`. Each block's dot is a `::before` on `.card-writing-area`. The writing area gets `padding-left: 40px` (desktop) to make room for the spine gutter.

### D5: Between-block affordance replaces "Add tweet" button

**Current**: Dashed "Add tweet" button at the bottom of the lane.

**New**: A subtle "+" circle appears on hover/focus between any two adjacent blocks, centered on the spine line. Clicking it inserts a new block at that position.

**Implementation**: ThreadFlowCard renders a `.between-block-zone` below itself (except for the last card, which renders it without the "above" line). The zone is 24px tall, with a 16px `+` circle that fades in on hover. Clicking calls a new `onaddafter` callback.

**The old "Add tweet" button at the bottom of ThreadFlowLane is removed.**

### D6: Keyboard shortcut changes summary

| Shortcut | Old behavior | New behavior |
|----------|-------------|-------------|
| `Cmd+Enter` (thread) | Submit | Split at cursor |
| `Cmd+Shift+Enter` (thread) | Split / add after | Submit |
| `Cmd+Enter` (tweet) | Submit | Submit (unchanged) |
| `Cmd+Shift+Enter` (tweet) | ŌĆö | Submit (unchanged) |

All other shortcuts remain the same: `Backspace@0` merges, `Alt+Arrow` reorders, `Tab`/`Shift+Tab` navigates, `Cmd+D` duplicates, `Cmd+Shift+S` splits (secondary), `Cmd+Shift+M` merges next.

## Cursor Management Edge Cases (to document in handoff)

### Documented behaviors

1. **Split at cursor**: `splitBlockAt` uses the textarea's `selectionStart`. If the cursor is within 10 chars of a word boundary, it snaps to the boundary. Both halves must be non-empty after trim, or split is a no-op.

2. **Merge cursor placement**: After merge, `requestAnimationFrame` ŌåÆ `document.querySelector('[data-block-id="..."] textarea')` ŌåÆ `setSelectionRange(joinPoint, joinPoint)`. The join point is the length of the first block's text plus the separator (0 or 1 char for `\n`).

3. **Focus after add/split/duplicate**: New block gets focus via `focusBlock(newId)` which uses `requestAnimationFrame` + DOM query. Cursor goes to position 0.

4. **Paste split**: `e.preventDefault()` stops the browser's default paste. Text is programmatically distributed across blocks. Cursor lands at end of the last new block.

5. **Tab navigation wraps**: Tab at last block does nothing (doesn't leave the thread). Shift+Tab at first block does nothing.

### Known edge cases requiring follow-up

1. **IME composition**: Japanese/Chinese input uses composition events. `Cmd+Enter` during composition might interfere. **Mitigation**: Check `e.isComposing` before processing split shortcuts.

2. **Undo/Redo**: Browser's native Ctrl+Z doesn't understand multi-block splits. After a split, Ctrl+Z restores the new block's text but doesn't re-merge. **Follow-up**: Session 4+ could add a custom undo stack for thread operations.

3. **Selection spanning blocks**: If a user could select across multiple textareas (they can't with standard textareas), split/merge would need different handling. **Non-issue**: Separate `<textarea>` elements have independent selections.

4. **Textarea auto-resize**: Currently `rows={3}` is fixed. Text longer than 3 rows overflows or scrolls. **Follow-up**: Session 4+ should add auto-resize (listen to `input` events, adjust `rows` based on `scrollHeight`).

5. **Paste with media**: Pasting images from clipboard isn't handled (only text paste is intercepted). **Scope**: Image paste is a separate feature, not part of this session.

## Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|------------|
| Cmd+Enter conflict with global handler | Split doesn't fire because window handler captures first | Medium | Window handler checks `mode === 'tweet'` before intercepting; textarea handler uses `e.stopPropagation()` after handling |
| ThreadFlowLane grows too large | >400 line guideline for components | Low | The 400-line limit applies to `+page.svelte`, not components; 380 lines is acceptable for an orchestrator. Can extract more to `threadOps.ts` if needed |
| Paste auto-split fires unexpectedly | User loses pasted content | Medium | Only intercepts when block is empty AND paste has `\n\n`. Falls through to normal paste otherwise. E2E testing. |
| Spine visual conflicts with RTL/accessibility | Visual-only left alignment breaks for RTL users | Low | Spine is decorative (`aria-hidden`); functional interaction is via textareas which handle bidi natively |
