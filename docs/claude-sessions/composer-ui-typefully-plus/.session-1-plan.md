## Key Design Decisions

### 1. Dual-Surface Home via Route-Level Rendering
**Decision**: The `+page.svelte` home route renders two possible surfaces (`composer` or `analytics`) based on a persisted `home_surface` preference, NOT via separate SvelteKit routes.

**Rationale**:
- A single `/` route avoids redirect flicker on load and keeps the sidebar "Dashboard" nav item consistent.
- The preference controls which surface is the default view; the other surface is always accessible (e.g., via a tab or sidebar link).
- No backend API needed — `persistGet('home_surface', 'composer')` is sufficient.

### 2. Shared Compose Workspace Extracted from Modal
**Decision**: Extract the compose orchestration logic into a shared `ComposeWorkspace.svelte` component usable in both the modal and the full-page home surface.

**Rationale**:
- `ComposeModal.svelte` at 678 lines contains all the compose state management, AI assist, autosave, and submission logic. This cannot be duplicated.
- The workspace component encapsulates: mode state, text/blocks state, autosave, AI handlers, submit flow, and inspector content.
- The modal becomes a thin shell that wraps `ComposeWorkspace` in a `ComposerShell` dialog.
- The home surface embeds `ComposeWorkspace` directly in the page layout (no modal/backdrop).

### 3. `home_surface` Preference
**Decision**: Store `home_surface: 'composer' | 'analytics'` via `persistSet/persistGet` (Tauri plugin-store). Default: `'composer'`.

**Rationale**:
- Operator rules require "fresh-install default" of `composer`.
- Operator rules require a Settings-level override back to `analytics`.
- No backend API needed per hard constraints ("Do not add backend APIs for UI-only preferences unless the current frontend persistence path is demonstrably insufficient").
- `persistence.ts` already handles this pattern (see `sidebar_collapsed`).

### 4. Full-Page Canvas Layout
**Decision**: The home composer surface uses the full `main-content` area (not a modal) with:
- Dark canvas background (inherits `--color-base` from layout).
- Centered writing lane: `max-width: 860px; margin: 0 auto;` with the same thread flow components.
- Top-right action cluster: Schedule pill (warm), Publish pill (cool), icon tools.
- Inspector as a right rail (not a drawer), since we have full-page width.

**Rationale**:
- The existing `ComposerCanvas` already implements the main+inspector flex layout.
- A full-page context gives us ~860px writing lane plus ~260px inspector, totaling ~1120px — comfortable at standard viewport widths.
- The centered-lane design matches the Typefully benchmark while using Tuitbot's existing component set.

### 5. Avatar Spine Instead of Standalone Cards
**Decision**: Add a left "spine" with small avatar pucks (16-20px circles) aligned to each tweet segment, connected by a subtle vertical line.

**Rationale**:
- Operator rules specify "centered thread lane with a left spine, avatar pucks, and low-noise separators."
- ThreadFlowCard already uses a left border accent bar — the spine replaces this with a continuous vertical line + avatar dots.
- This is a CSS/template change to `ThreadFlowLane` and `ThreadFlowCard`, not a new component.

### 6. Action Cluster Replaces Floating Submit
**Decision**: Replace the single floating submit pill with a top-right action cluster containing `Schedule` (warm color) and `Publish` (cool color) pills, plus icon buttons for preview, focus, AI, and settings.

**Rationale**:
- Operator rules: "Top-right actions use a premium cluster: warm `Schedule`, cool `Publish`, then quiet icon tools."
- The current `ComposerHeaderBar` already has icon tools; the submit pill is in `ComposerCanvas`.
- For the full-page surface, merge these into a unified header/action bar.

**Shared ComposeWorkspace extraction plan:**
- State lifted from ComposeModal: `mode`, `tweetText`, `threadBlocks`, `selectedTime`, `attachedMedia`, `focusMode`, `voiceCue`, etc.
- Handlers lifted: `handleSubmit`, `handleInlineAssist`, `handleAiAssist`, `handleGenerateFromNotes`, autosave logic.

**`home_surface` preference data flow:**
```
App loads → +page.svelte onMount → persistGet('home_surface', 'composer')
  → sets local state → renders composer or analytics surface

Settings page → "Home Surface" dropdown → persistSet('home_surface', value)
  → next navigation to / picks up new value
```

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| **ComposeWorkspace extraction breaks modal** | High | Session 02 must maintain the existing `ComposeModal` external API (`open`, `onclose`, `onsubmit` props). Extract state/handlers into `ComposeWorkspace`, then have `ComposeModal` delegate to it. Run full CI after extraction. |
| **Autosave conflict between modal and home** | Medium | Use the same `tuitbot:compose:draft` localStorage key. The home composer and modal should not coexist (home replaces the need for the modal on `/`). When navigating away from `/`, autosave persists; when opening modal from another page, it uses the same recovery flow. |
| **`home_surface` preference not loading fast enough** | Medium | `persistGet` is async (Tauri store load). The `+page.svelte` must show a loading skeleton or default surface while the preference resolves. Use `composer` as the synchronous default to avoid layout shift. |
| **Thread flow components not designed for full-page width** | Low | ThreadFlowLane/Card are already width-flexible (no fixed width). The 760-860px centered lane is achieved via `max-width` on the parent, not changes to the components themselves. |
| **Settings page already at 345 lines** | Low | Adding a home-surface toggle is ~20 lines. Could go in an existing section (Business Profile has a "general" feel) or a new tiny "Appearance" section. Either keeps the page well under 400 lines. |
| **Sidebar nav label confusion** | Low | The "Dashboard" nav item should change label to "Home" or show different labels based on `home_surface`. This is a ~5-line change to `Sidebar.svelte`. |
