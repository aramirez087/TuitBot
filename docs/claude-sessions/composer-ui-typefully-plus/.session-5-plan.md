## Detailed Implementation Plan

### Step 1: Expand `persistence.ts` with `localStorage` fallback

**File**: `dashboard/src/lib/stores/persistence.ts`

**Current problem**: When not in Tauri (browser-only dev), `getStore()` returns `null`, so `persistGet` returns the hardcoded `fallback` and `persistSet` is a no-op. This means no persistence across page refreshes in development.

**Change**: Add `localStorage` fallback when Tauri store is unavailable. This mirrors how `theme.ts` and `accounts.ts` already use `localStorage` directly.

```typescript
// persistence.ts â€” enhanced

let storeInstance: any = null;
let useFallback = false;

async function getStore() {
    if (storeInstance) return storeInstance;
    if (useFallback) return null;

    try {
        const { load } = await import('@tauri-apps/plugin-store');
        storeInstance = await load('ui-state.json', { autoSave: true, defaults: {} } as any);
        return storeInstance;
    } catch {
        useFallback = true;
        return null;
    }
}

export async function persistGet<T>(key: string, fallback: T): Promise<T> {
    const store = await getStore();
    if (store) {
        try {
            const value = await store.get(key);
            return value !== null && value !== undefined ? (value as T) : fallback;
        } catch {
            return fallback;
        }
    }

    // localStorage fallback (browser-only dev)
    try {
        const raw = localStorage.getItem(`tuitbot:ui:${key}`);
        if (raw === null) return fallback;
        return JSON.parse(raw) as T;
    } catch {
        return fallback;
    }
}

export async function persistSet<T>(key: string, value: T): Promise<void> {
    const store = await getStore();
    if (store) {
        try {
            await store.set(key, value);
            await store.save();
        } catch {
            // Silently ignore persistence failures.
        }
        return;
    }

    // localStorage fallback (browser-only dev)
    try {
        localStorage.setItem(`tuitbot:ui:${key}`, JSON.stringify(value));
    } catch {
        // Silently ignore quota errors.
    }
}
```

**Key decisions**:
- Prefix localStorage keys with `tuitbot:ui:` to avoid collisions with other localStorage usage (e.g. autosave uses `tuitbot:compose:draft`, theme uses `tuitbot-theme`).
- Cache the `useFallback` flag so we don't re-attempt the Tauri import on every call.
- `JSON.parse`/`JSON.stringify` for localStorage values since they may be non-string types (booleans, objects).
- This enhancement benefits ALL existing `persistGet`/`persistSet` callers: sidebar collapsed state, tips dismissed state, and the new home surface preference.

**Risk**: Existing callers (`sidebar_collapsed`, `home_tips_dismissed`) previously returned in-memory defaults in browser mode. Now they'll persist to localStorage. This is strictly better â€” no behavioral regression possible since the fallback value is the same.

### Step 2: Create `homeSurface.ts` store

**File**: `dashboard/src/lib/stores/homeSurface.ts`

**Purpose**: Centralized reactive store for the home surface preference. Exposes:
- `homeSurface` â€” writable Svelte store with current value
- `homeSurfaceReady` â€” derived boolean, true after async load completes
- `loadHomeSurface()` â€” async init, reads from persistence
- `setHomeSurface(value)` â€” writes to persistence and updates store

```typescript
import { writable, derived } from 'svelte/store';
import { persistGet, persistSet } from './persistence';

export type HomeSurface = 'composer' | 'analytics';

const PREF_KEY = 'home_surface';

// Synchronous default is 'composer' â€” prevents layout shift while loading
const _surface = writable<HomeSurface>('composer');
const _loaded = writable(false);

export const homeSurface = { subscribe: _surface.subscribe };
export const homeSurfaceReady = derived(_loaded, ($l) => $l);

export async function loadHomeSurface(): Promise<void> {
    const value = await persistGet<HomeSurface>(PREF_KEY, 'composer');
    _surface.set(value);
    _loaded.set(true);
}

export async function setHomeSurface(value: HomeSurface): Promise<void> {
    _surface.set(value);
    await persistSet(PREF_KEY, value);
}
```

**Design decisions**:

- **Read-only `homeSurface` export**: The store's `subscribe` is exposed directly but not `set`/`update`. All mutations go through `setHomeSurface()` to ensure persistence is always written. This prevents accidental unsaved changes.
- **`homeSurfaceReady` as derived boolean**: The home route can subscribe to this to know when the async load is complete, preventing flicker.
- **Synchronous default `'composer'`**: The writable starts with `'composer'` so if rendering happens before `loadHomeSurface()` resolves, the user sees the composer (the intended default). This eliminates flash-of-wrong-surface.
- **No backend API involvement**: This is a pure frontend preference, matching `sidebar_collapsed`, `home_tips_dismissed`, and the spec in `home-surface-plan.md`.

### Step 3: Create `WorkspaceSection.svelte`

**File**: `dashboard/src/routes/(app)/settings/WorkspaceSection.svelte`

**Purpose**: Settings section for the "Default landing page" preference. Uses the same `SettingsSection` wrapper pattern as all other settings sections (e.g., `LanAccessSection.svelte`, `BusinessProfileSection.svelte`).

**Critical constraint**: This section must NOT use the `settings` store (`draft`/`updateDraft`/`saveSettings`) because the home surface preference is a UI-only preference stored in the Tauri plugin-store / localStorage, not part of the backend `TuitbotConfig`. It operates independently of the Save/Discard flow.

**Template structure**:

```svelte
<script lang="ts">
    import { LayoutDashboard } from 'lucide-svelte';
    import SettingsSection from '$lib/components/settings/SettingsSection.svelte';
    import { homeSurface, setHomeSurface, loadHomeSurface, type HomeSurface } from '$lib/stores/homeSurface';
    import { onMount } from 'svelte';

    let current = $state<HomeSurface>('composer');
    let loaded = $state(false);

    onMount(async () => {
        await loadHomeSurface();
        // Read current value from store subscription
        const unsub = homeSurface.subscribe((v) => { current = v; });
        loaded = true;
        return unsub;
    });

    async function handleChange(value: HomeSurface) {
        current = value;
        await setHomeSurface(value);
    }
</script>

<SettingsSection
    id="workspace"
    title="Workspace"
    description="Configure your default workspace experience"
    icon={LayoutDashboard}
>
    <div class="field-group">
        <h3 class="group-title">Default Landing Page</h3>
        <p class="group-desc">Choose what you see when you open Tuitbot</p>

        <div class="option-cards">
            <button class="option-card" class:selected={current === 'composer'}
                onclick={() => handleChange('composer')}>
                <div class="option-radio" class:checked={current === 'composer'}></div>
                <div class="option-content">
                    <span class="option-label">Composer home</span>
                    <span class="option-desc">Jump straight into writing â€” recommended</span>
                </div>
            </button>

            <button class="option-card" class:selected={current === 'analytics'}
                onclick={() => handleChange('analytics')}>
                <div class="option-radio" class:checked={current === 'analytics'}></div>
                <div class="option-content">
                    <span class="option-label">Analytics overview</span>
                    <span class="option-desc">See your growth dashboard first</span>
                </div>
            </button>
        </div>

        <p class="field-hint">Takes effect on next visit to the home page</p>
    </div>
</SettingsSection>
```

**Design decisions**:

- **Radio-card pattern instead of `<select>`**: A clickable card with a radio indicator is more visually scannable and touch-friendly than a dropdown for a two-option preference. Each option gets a label + description. The `home-surface-plan.md` spec mentioned a `<select>`, but cards are more consistent with the app's UI quality bar.
- **Instant save**: Changes are saved immediately via `setHomeSurface()` â€” no dependency on the "Save Changes" button flow. This is consistent with how `sidebar_collapsed` and theme preferences work (immediate, no save bar). A subtle "Takes effect on next visit" hint communicates this.
- **`recommended` badge on composer option**: The description says "recommended" to make composer the opinionated default, matching the session spec ("copy that makes composer the recommended default").
- **Separate from backend config**: The section uses `homeSurface` store (persistence.ts), NOT the `settings` store (api.settings.*). This ensures no backend schema changes are needed.
- **Icon**: Uses `LayoutDashboard` from lucide-svelte (already imported in Sidebar). Could also use `Home` â€” choosing `LayoutDashboard` for visual consistency with the sidebar nav item.

**Estimated lines**: ~130 (script ~25, template ~45, styles ~60).

### Step 5: Update home route to use `homeSurface` store

**File**: `dashboard/src/routes/(app)/+page.svelte`

**Current state**: Local `homeSurface` state variable with inline `persistGet` in `onMount`. Works but duplicates the preference logic that should live in the store.

**Change**: Replace inline state with the centralized `homeSurface` store. Use `$homeSurface` and `$homeSurfaceReady` for reactive rendering.

```svelte
<script lang="ts">
    import { onMount } from 'svelte';
    import { api, type ComposeRequest, type ScheduleConfig } from '$lib/api';
    import ComposeWorkspace from '$lib/components/composer/ComposeWorkspace.svelte';
    import AnalyticsHome from '$lib/components/home/AnalyticsHome.svelte';
    import { homeSurface, homeSurfaceReady, loadHomeSurface } from '$lib/stores/homeSurface';

    let schedule = $state<ScheduleConfig | null>(null);

    onMount(async () => {
        await loadHomeSurface();
        try {
            const cfg = await api.content.schedule();
            schedule = cfg;
        } catch {
            // Schedule loading is non-critical
        }
    });

    async function handleSubmit(data: ComposeRequest) {
        await api.content.compose(data);
    }
</script>

<svelte:head>
    <title>{$homeSurface === 'composer' ? 'Compose' : 'Dashboard'} â€” Tuitbot</title>
</svelte:head>

{#if !$homeSurfaceReady}
    <div class="loading-skeleton"></div>
{:else if $homeSurface === 'composer'}
    <div class="home-composer">
        <ComposeWorkspace
            {schedule}
            onsubmit={handleSubmit}
            embedded={true}
        />
    </div>
{:else}
    <AnalyticsHome />
{/if}
```

**Design decisions**:

- **`$homeSurfaceReady` guards the skeleton**: The loading skeleton shows only until the store has loaded the preference. Since the store defaults to `'composer'` synchronously, the skeleton is only visible if the async `loadHomeSurface()` takes measurable time (rare).
- **Reactive via store subscription**: Using `$homeSurface` means if another component calls `setHomeSurface()` (e.g., from Settings), the home route will reactively update when navigated back to. No stale state.
- **Schedule loading runs concurrently**: `loadHomeSurface()` is awaited first (fast â€” localStorage/plugin-store read), then schedule loads. Schedule is non-blocking for the UI since it only affects the composer inspector.
- **No redirect loops**: The route always renders one of two surfaces based on the store value. No `goto()` calls, no redirects.

## Key Design Decisions

### D1: localStorage fallback in persistence.ts (not a separate system)

**Why**: The session spec says "persist in both Tauri and browser dev". The current `persistence.ts` already handles Tauri, but silently drops writes in browser mode. Adding localStorage fallback in the same module means ALL persist callers (sidebar, tips, home surface) gain browser persistence with zero caller changes.

**Alternative considered**: A separate `browserStorage.ts` module. Rejected because it would create parallel persistence paths and force callers to choose. Better to enhance the existing module.

### D2: Instant-save in WorkspaceSection (no Save bar dependency)

**Why**: The workspace preference is a UI-only setting stored in plugin-store/localStorage. It should not participate in the server settings Save/Discard flow which validates and writes to `config.toml`. Mixing them would require:
- Adding `home_surface` to the `TuitbotConfig` type (backend schema change)
- Validating it server-side
- Persisting it to TOML (irrelevant for a UI preference)

Instant-save matches how `sidebar_collapsed` and theme already work.

### D3: Store module vs inline state

**Why**: A centralized `homeSurface.ts` store allows the home route, settings page, and any future component to read/write the preference reactively. With inline state (current approach), the settings page would need to use `persistSet` directly and the home route would need a page refresh to pick up the change. The store enables reactive propagation.

### D4: Radio cards vs select dropdown

**Why**: Two options is the ideal use case for radio-style cards â€” each option gets a label, description, and visual treatment. A `<select>` would hide the options behind a click and provide no room for the "recommended" hint. Cards are more consistent with the app's design quality.

### D5: Sidebar rename from "Dashboard" to "Home"

**Why**: The default surface is now the composer, not a dashboard. "Dashboard" implies analytics. "Home" is surface-agnostic and accurate regardless of the user's preference.

## Risks and Mitigations

### R1: localStorage key collisions

**Risk**: The `tuitbot:ui:` prefix for localStorage keys could theoretically collide with other apps on the same origin.

**Mitigation**: The prefix is specific enough (`tuitbot:ui:`) and the app runs on `localhost:5173` (dev) or inside Tauri (production). No real collision risk.

### R2: Race between store load and render

**Risk**: If `loadHomeSurface()` is slow, the user briefly sees the loading skeleton.

**Mitigation**: The store defaults to `'composer'` synchronously. `persistGet` reads localStorage (< 1ms) or Tauri plugin-store (< 5ms). The skeleton will flash imperceptibly or not at all. For extra safety, we could remove the skeleton and render `composer` immediately (since it's the sync default), but keeping the `homeSurfaceReady` gate is more correct and handles edge cases.

### R3: WorkspaceSection renders inside `$draft` gate

**Risk**: If the backend settings API fails, the entire settings page shows an error â€” including the workspace preference which doesn't depend on the backend at all.

**Mitigation**: This is acceptable because: (a) the error state already has a "Retry" button, (b) if the backend is unreachable, the user has bigger problems than the home surface preference, (c) rendering it outside the gate would require duplicating loading/error state handling. Low risk, low impact.

### R4: Preference not persisting in SSR/prerender

**Risk**: SvelteKit SSR could attempt to read localStorage before it exists.

**Mitigation**: The app is SPA mode (no SSR per CLAUDE.md: "SPA mode (no SSR)"). All code runs in the browser. `persistGet`/`persistSet` are async and called from `onMount` or event handlers, never at module scope. No SSR risk.

### R5: Stale preference after changing settings

**Risk**: User changes preference in Settings, then navigates home. The old surface flashes briefly.

**Mitigation**: The `homeSurface` store is reactive. When `setHomeSurface()` is called in Settings, the store value updates immediately. When the user navigates to `/`, `loadHomeSurface()` is called in `onMount`, which re-reads from persistence. But since the store was already updated by `setHomeSurface()`, the correct surface renders immediately. No flash.
