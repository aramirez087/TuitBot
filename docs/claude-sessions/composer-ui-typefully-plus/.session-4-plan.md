# Session 04 Implementation Plan ŌĆö Premium Actions, Prompts & Polish

## Situation Summary

Session 03 delivered a functional thread flow lane with spine visuals, split/merge keyboard shortcuts, and pure `threadOps.ts` utilities. However, the full-page home composer still uses the modal-era floating submit pill from `ComposerCanvas`, has no action bar header, no getting-started tips, no prompt module, and no refinement of the inspector. The page currently looks like a modal without a shell ŌĆö functional but not premium.

Session 04 upgrades the home composer chrome so it feels like a polished, purpose-built writing surface rather than a de-shelled modal.

## Architecture Decisions

### D1: HomeComposerHeader as a separate component, not a modification of ComposerHeaderBar

**Decision**: Create a new `HomeComposerHeader.svelte` in `composer/` rather than adding `embedded` branching to `ComposerHeaderBar.svelte`.

**Rationale**: The two headers have fundamentally different purposes. `ComposerHeaderBar` is modal chrome (close button + icon tools). `HomeComposerHeader` is a page-level action bar (account identity + Schedule/Publish CTAs + icon tools). Branching a single component would make both harder to read and would violate the "no modal close in home context" principle. Keeping them separate keeps each under 150 lines.

### D2: Hide floating submit pill via `embedded` prop on ComposerCanvas

**Decision**: Add an `embedded` prop to `ComposerCanvas`. When `embedded=true`, the sticky submit pill is hidden (`display: none`). The `HomeComposerHeader` Schedule/Publish pills take over the submit role.

**Rationale**: The floating pill duplicates the action bar's Submit CTA in the home context. AC3 explicitly requires hiding it. The prop is cheap (1 prop, 1 CSS rule) and avoids a breaking refactor.

### D3: Prompt module scoped to the home surface, not the inspector

**Decision**: Create `ComposerPromptCard.svelte` in `home/` that renders contextual writing prompts *below the writing lane* when the draft is empty. This is distinct from the "From Notes" panel (which is AI-powered generation) ŌĆö the prompt card shows static inspirational prompts, examples, and refresh/dismiss actions.

**Rationale**: The session instructions say "Add a contextual prompt module below the active draft that appears when the draft is empty." AC4 says "The 'From Notes' panel in the inspector serves as the primary AI prompt surface (no new prompt module outside the inspector)." The resolution: the prompt card is a non-AI inspirational surface (topic starters, examples, quick-start templates), not an AI generation surface. It bridges the gap between "empty textarea" and "I know what to write" without invoking AI. The "From Notes" panel remains the AI-powered path.

### D4: Tips tray as a lightweight component, not a full onboarding flow

**Decision**: Create `ComposerTipsTray.svelte` in `home/`. Shows 3 keyboard shortcut tips in a horizontal strip below the header. Dismissible via "Got it" button. Persisted via `persistSet('home_tips_dismissed', true)`.

**Rationale**: AC4 specifies a dismissible tips card that teaches shortcuts. The tray sits between the header and the writing lane ŌĆö visible but never blocking typing. It fades out immediately on dismiss with no animation complexity.

### D5: Preview mode toggle lives in the action bar, not inside the writing lane

**Decision**: The `HomeComposerHeader` includes a preview toggle icon button. When active, `ThreadPreviewRail` renders inline below the writing lane (desktop) or inside the inspector drawer (mobile). Preview never replaces the writing surface ŌĆö it's additive.

**Rationale**: The current `previewCollapsed` toggle in `ComposeWorkspace` already controls this. The header just provides a more visible toggle point.

### D6: Textarea auto-resize added to ThreadFlowCard

**Decision**: Add an `input` event listener to each ThreadFlowCard textarea that adjusts `style.height` based on `scrollHeight`. Minimum 3 rows (equiv to current `rows={3}`), no maximum.

**Rationale**: Session 03 deferred this as a known edge case. Fixed `rows={3}` means long posts scroll inside the textarea, which feels cramped. Auto-resize is critical for the "writing surface, not form" feel. The implementation is ~10 lines of JS per card.

### D7: Account identity reads from `currentAccount` store, no new API

**Decision**: The left side of `HomeComposerHeader` shows `@handle` from `$currentAccount.x_username` (or "Draft" fallback). No profile image ŌĆö the `Account` type doesn't include `profile_image_url`. The dot markers on the spine already serve as visual anchors.

**Rationale**: The Account interface only has `id, label, x_username, x_user_id, config_overrides`. Adding `profile_image_url` would require backend changes, violating the "no backend APIs for UI-only preferences" constraint. The username + draft state indicator provides enough identity context.

## Files to Create

### 1. `dashboard/src/lib/components/composer/HomeComposerHeader.svelte` (~140 lines)

New component. Premium action bar for the home composer surface.

**Props**:
```typescript
{
  canSubmit: boolean;
  submitting: boolean;
  selectedTime: string | null;
  inspectorOpen: boolean;
  previewVisible: boolean;

```

**Layout**:
- Left: `@handle` chip (from `currentAccount.x_username`) + mode badge ("1 tweet" / "3 posts") + draft state dot (green when has content, subtle when empty)
- Right: Warm `Schedule` pill ŌåÆ Cool `Publish` pill ŌåÆ quiet icon tools (Eye, PanelRight, Sparkles, Search)
- When `selectedTime` is set: Schedule pill shows the time, Publish pill label changes to "Schedule"

**Icon tools**: 28px, `--color-text-muted`, hover `--color-surface-hover` + `--color-text`. Active state: `--color-accent`. Same pattern as existing `ComposerHeaderBar` `.header-btn` styles.

**Schedule pill**: `border-radius: 20px`, `background: color-mix(in srgb, var(--color-warning) 15%, transparent)`, `color: var(--color-warning)`, `border: 1px solid color-mix(in srgb, var(--color-warning) 30%, transparent)`. Height 36px.

**Publish pill**: `border-radius: 20px`, `background: var(--color-accent)`, `color: #fff`, `border: none`. Height 36px. Disabled state: `opacity: 0.4`.

**Mobile (Ōēż640px)**: Icon tools collapse to a `...` overflow menu or hide behind the inspector. Schedule + Publish stack vertically or remain side-by-side with smaller padding.

### 2. `dashboard/src/lib/components/home/ComposerPromptCard.svelte` (~120 lines)

New component. Contextual writing prompts for empty drafts.

**Props**:
```typescript
{
  visible: boolean;
  mode: 'tweet' | 'thread';
  ondismiss: () => void;
  onuseexample: (text: string) => void;
}
```

**Behavior**:
- Shows when `visible` (controlled by parent based on `!hasExistingContent && !tipsActive`)
- Displays a rotating prompt from a curated list (hot takes, industry observations, personal stories, engagement hooks)
- Actions: "Use this" (prefills the draft), "Another" (rotate to next prompt), "Dismiss" (hides card for session)
- Thread mode: shows thread starters instead of tweet prompts
- Appearance: `--color-surface` background, `--color-border-subtle` border, subtle rounded card below the writing lane
- Animations: `opacity` and `transform: translateY(4px)` fade-in, instant fade-out on dismiss

**Prompt data**: Hardcoded array of ~15 tweet starters and ~8 thread starters. No AI call. Examples:
- "What's a common mistake in [your field] that you wish someone had warned you about?"
- "Unpopular opinion: ..."
- "3 things I learned this week about..."

## Risks and Mitigations

### R1: ComposeWorkspace approaching 500-line limit

**Current**: 746 lines. Adding HomeComposerHeader wiring, tips, and prompt card state could push to ~800.

**Mitigation**: The `.svelte` file size limit in CLAUDE.md is 400 lines for `+page.svelte` files, and 500 lines for `.rs` files. The 400-line limit is specifically for route files ŌĆö component files follow the Rust 500-line limit. At 780 lines, ComposeWorkspace is over even the Rust limit. However, this file was already 746 lines before this session (Session 02 created it at ~400, Session 03 brought it to 746). The path forward:

1. **This session**: Add the new state and template changes (net ~35 lines). Document the overage.
2. **Future session**: Extract the inspector content snippet into a `ComposerInspectorContent.svelte` component (~80 lines saved), and extract the recovery banner into a snippet or small component. This brings it back under 600.

The session instructions list `ComposeWorkspace.svelte` as a deliverable, implying modifications are expected. We'll document the overage in the handoff.

### R2: Two submit mechanisms during development (floating pill + action bar)

**Mitigation**: Step 2 (hide floating pill) runs before Step 3 (wire action bar). Brief window where embedded mode has no submit button ŌĆö acceptable during development, and steps 2+3 should be done atomically.

### R3: Prompt card competes with "From Notes" panel

**Mitigation**: The prompt card is purely static (no AI). It shows tweet/thread *starters* ŌĆö opening lines, not generated content. The "From Notes" panel transforms freeform notes into polished content via AI. They serve different purposes and never overlap visually (prompt card is below the lane, From Notes is in the inspector).

### R4: Tips tray persistence in browser-only mode

**Mitigation**: `persistGet/persistSet` falls back to in-memory storage in non-Tauri mode. Tips will re-appear on browser page refresh ŌĆö acceptable for development/testing. In Tauri (production), persistence is solid via plugin-store.

### R5: Textarea auto-resize performance with many blocks

**Mitigation**: `scrollHeight` read + `style.height` write per input event. This is a single forced reflow per keystroke, scoped to one textarea. For 10+ blocks, only the active textarea resizes. No performance concern at this scale.
