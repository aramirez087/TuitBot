## Key Design Decisions

### D1: ComposeWorkspace owns ComposerShell conditionally

Rather than having ComposeModal own the shell and workspace as siblings, ComposeWorkspace conditionally renders ComposerShell based on the `embedded` prop. This keeps all compose state and its dependent UI in one component, eliminating cross-component state sharing.

**Tradeoff**: ComposeWorkspace grows to ~420 lines (near the 400-line guideline). This is acceptable because it's fundamentally a state orchestrator â€” the visual components it delegates to (ComposerCanvas, ComposerHeaderBar, etc.) remain small. If it exceeds 400 lines significantly, the inspector snippet content could be extracted to a dedicated `InspectorPanels.svelte` component.

### D2: Recovery banner moves from ComposerShell to ComposeWorkspace

ComposeWorkspace manages `showRecovery` and `recoveryData` state, so it should render the banner. This simplifies ComposerShell (which becomes purely structural) and allows the banner to work identically in both modal and embedded contexts.

### D3: `embedded` prop rather than mode-specific workspace variants

A single `ComposeWorkspace` with an `embedded` boolean is simpler than separate `ModalComposeWorkspace` and `PageComposeWorkspace` components. The behavioral differences are small (focus mode, escape handling, shell rendering) and well-encapsulated by conditionals on a single prop.

### D4: Synchronous `'composer'` default, async preference override

The `+page.svelte` uses a `loaded` flag to avoid rendering anything until `persistGet` resolves. This prevents the brief flash described in Risk 3 from the handoff. The trade-off is a few milliseconds of empty content area on route load, but in practice the Tauri store resolves in <10ms, making this imperceptible.

### D5: Autosave key is shared

Both the modal and full-page workspace use the same `tuitbot:compose:draft` localStorage key. This is intentional â€” a draft started on the home surface can be recovered if the user navigates away and opens the modal from the calendar. There's no scenario where both surfaces are simultaneously open (the modal doesn't open on `/` when the home composer is active).

### D6: `handleSubmit` callback in home route resets workspace

When the user submits from the home composer, the workspace should reset to a clean state. Since ComposeWorkspace mounts once and doesn't remount (unlike the modal which remounts each open), the workspace needs an explicit `reset()` method or the `onsubmit` handler needs to trigger a reset.

**Approach**: After `onsubmit` completes successfully in ComposeWorkspace, the workspace calls `clearAutoSave()` and resets its own state (tweetText='', threadBlocks=[], mode='tweet', etc.). This self-reset happens in the workspace's `handleSubmit` function, after calling the external `onsubmit` prop. This is identical to how the modal works â€” the external `onsubmit` (in the calendar) handles the API call, then closes the modal which unmounts the workspace, clearing state. For the embedded workspace, we explicitly reset state instead of relying on unmount.

## Risks and Mitigations

### Risk 1: ComposeWorkspace extraction breaks autosave (Severity: High)

**Problem**: The current `$effect` on `open` (line 147-165) resets state and checks recovery when the modal opens. This lifecycle doesn't exist in the embedded context.

**Mitigation**:
- Modal context: `ComposeModal` uses `{#if open}` to mount/unmount ComposeWorkspace. Each mount triggers `onMount` which does recovery check + state reset. Unmount triggers `onDestroy` which cleans up timers and object URLs.
- Embedded context: `ComposeWorkspace` mounts once with the route. `onMount` checks recovery. State persists until explicit reset (on submit) or route navigation (unmount).
- Autosave `$effect` fires identically in both contexts â€” it watches mode/text/blocks.

**Verification**: After extraction, test:
1. Open modal â†’ type â†’ close â†’ reopen â†’ recovery banner appears
2. Open modal â†’ type â†’ submit â†’ close â†’ reopen â†’ no recovery (autosave cleared)
3. Navigate to home â†’ type â†’ navigate away â†’ navigate back â†’ recovery banner appears
4. Navigate to home â†’ type â†’ submit â†’ content resets, no recovery on next visit

### Risk 2: ComposerShell lost recovery banner (Severity: Medium)

**Problem**: Moving the recovery banner out of ComposerShell changes the visual hierarchy.

**Mitigation**: In modal context (`embedded=false`), ComposeWorkspace renders the recovery banner as its first child inside the ComposerShell's children slot, before ComposerHeaderBar. This produces identical visual output: shell border-radius â†’ recovery banner â†’ header bar â†’ canvas.

**Verification**: Compare screenshots of the modal with recovery banner before and after the extraction.

### Risk 3: Inspector snippet ref sharing (Severity: Medium)

**Problem**: The `voicePanelRef` (used in `saveCueToHistory()`) is referenced from handler functions AND rendered in the inspector snippet. If the snippet is defined inside ComposeWorkspace but rendered inside ComposerCanvas or ComposerInspector, the ref binding must still work.

**Mitigation**: The snippet is defined in ComposeWorkspace's template (where the `bind:this={voicePanelRef}` lives) and passed to ComposerCanvas and ComposerInspector via their snippet props. Svelte 5 snippets capture the defining component's scope, so `bind:this` works correctly even when the snippet is rendered inside a child component.

**Verification**: Test AI improve with voice cue â†’ verify `saveCueToHistory()` is called.

### Risk 4: ComposeWorkspace exceeds 400-line Svelte guideline (Severity: Low)

**Problem**: The workspace will contain ~420 lines of orchestration logic.

**Mitigation**: This is within acceptable range (the guideline says 400 for page files; this is a component that is primarily a state orchestrator). If it grows further in later sessions, extract `InspectorPanels.svelte` to separate the inspector content.

### Risk 5: Home route padding conflict (Severity: Low)

**Problem**: The `+layout.svelte` `.main-content` has `padding: 24px 32px`. The full-page composer needs edge-to-edge feel.

**Mitigation**: The `.home-composer` wrapper uses `margin: -24px -32px` to negate the layout padding, then applies its own padding for the writing lane centering. Or, more simply, the existing padding is acceptable for the centered writing lane (860px max-width within a padded container is fine). The writing lane centering handles this naturally.

**Decision**: Accept the layout padding. The 32px horizontal padding creates a comfortable margin around the 860px centered lane. No negative margin hack needed.
