## Design Decisions

### D1: ComposeWorkspace extraction strategy (852 → ~480 lines)

The file is 852 lines, violating the 500-line component limit. Three extractions bring it under:

1. **Inspector content snippet (lines 448–499) → `InspectorContent.svelte` (~80 lines)**
   - The `{#snippet inspectorContent()}` block renders Schedule, Voice, and AI sections
   - Extract to a standalone component that receives callbacks and state as props
   - Both desktop rail (via ComposerCanvas `inspector` snippet) and mobile drawer (via ComposerInspector) render the same component
   - Props: `schedule`, `selectedTime`, `voiceCue`, `assisting`, `hasExistingContent`, `showFromNotes`, `showUndo`, `mode`, + event callbacks
   - Savings: ~52 lines of template + ~50 lines of inspector styles = ~100 lines

2. **Recovery banner (lines 502–510) → `RecoveryBanner.svelte` (~50 lines)**
   - Thin component: shows "Unsaved draft found" with Recover/Discard buttons
   - Props: `onrecover`, `ondismiss`
   - Savings: ~10 lines template + ~40 lines styles = ~50 lines

3. **AI/submit handler logic (lines 227–439) → `composeHandlers.ts` (~140 lines)**
   - Extract pure logic functions: `buildComposeRequest()`, `performInlineAssist()`, `performAiAssist()`, `performGenerateFromNotes()`
   - These take state parameters and API references, return results
   - ComposeWorkspace calls them, applying results to its reactive state
   - Savings: ~150 lines of handler logic moved out, replaced by ~30 lines of thin wrappers
   - Net savings: ~120 lines

**Total reduction: ~270 lines → 852 - 270 = ~582 lines.** With the style lines that move to InspectorContent (~80 lines of `.inspector-*` and `.ai-*` styles), final size: **~500 lines**.

**Rationale**: This extraction preserves the single-orchestrator pattern while respecting the file size limit. The three extracted pieces have clear boundaries and minimal coupling.

The `SHORTCUT_CATALOG` in `shortcuts.ts` gets matching label updates.

### D5: `Cmd+N` global shortcut

Add a `<svelte:window onkeydown>` handler in the app layout (`(app)/+layout.svelte`) or Sidebar that:
- On `Cmd+N`: if on home route with composer surface, focuses the first textarea; otherwise opens ComposeModal or navigates to `/`
- This is a lightweight addition — no new component, just a handler

**Decision**: Implement as a `<svelte:window>` handler in `(app)/+layout.svelte` that dispatches a custom event or uses SvelteKit's `goto('/')`. The home page's ComposeWorkspace already auto-focuses the textarea on mount.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| ComposeWorkspace extraction breaks reactive bindings | High — autosave, submit, AI assist could break | Extract incrementally: inspector content first (pure render), then recovery banner (simple), then handlers (most complex). Run `npm run check` after each. |
| `bind:this` refs (`voicePanelRef`, `threadFlowRef`, `tweetEditorRef`) can't be passed through extracted components easily | Medium — breaks AI assist pipeline | Keep refs in ComposeWorkspace; pass callbacks to InspectorContent rather than refs |
| Command palette action ID changes break palette dispatch | Low — palette IDs are strings, all in one switch statement | Keep IDs unchanged; only change labels. No breaking changes. |
| `Cmd+N` conflicts with browser's "new window" shortcut | Medium — poor UX on web dev mode | Use `e.preventDefault()` only when in Tauri context (check `window.__TAURI__`). In browser mode, let native behavior pass through. |
| Inspector content extraction loses `$derived` reactivity | Medium — inspector might not update when state changes | Pass all reactive values as props; Svelte 5's `$props()` is inherently reactive |
| File size estimate off — ComposeWorkspace still >500 after extraction | Low — cosmetic guideline violation | Measure actual line count after each extraction. If still over, move `.recovery-*`, `.undo-*`, and `.preview-*` styles to extracted components or a shared CSS module |
