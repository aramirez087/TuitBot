# Session 05 Implementation Plan: Progressive Activation & Capability Gating

## Mission

Allow users to complete a short X-first onboarding without being forced to fill every advanced field. Introduce explicit capability tiers (profile-ready → exploration → generation-ready → posting-ready) so the app knows what the user can do and surfaces missing prerequisites as resumable prompts rather than blocking validation errors.

---

## Key Design Decisions

### D1: Capability Tier Model (4 tiers, computed server-side)

Define four deterministic capability tiers in `tuitbot-core`:

| Tier | Name | Required Fields | Unlocks |
|------|------|----------------|---------|
| 0 | `unconfigured` | Nothing | Redirect to onboarding |
| 1 | `profile-ready` | `business.product_name`, `product_description`, `product_keywords`, `industry_topics` | Dashboard access, view settings |
| 2 | `exploration-ready` | Tier 1 + X credentials (client_id for x_api, or scraper backend) + OAuth tokens on disk | Discovery, search, scoring |
| 3 | `generation-ready` | Tier 2 + valid LLM config (`llm.provider` + `llm.api_key` for cloud providers, or ollama) | Draft generation, reply composition |
| 4 | `posting-ready` | Tier 3 + `can_post` (OAuth tokens valid for x_api, or scraper session) | Scheduled posting, autopilot |

**Rationale:** These tiers mirror the natural progression. A user who connects X and fills their profile can explore content without LLM. LLM enables generation. Posting requires valid tokens. Each tier is a strict superset of the previous.

### D2: Shortened onboarding — only tiers 1-2 required at submit time

The onboarding wizard will require only:
- **X Access** (mode selection + optional OAuth) — required for tier 1-2
- **Profile** (name, description, keywords, topics) — required for tier 1

LLM, Language/Brand, Vault, and Validation steps become **optional** — the user can skip forward from Profile directly to Review.

**Rationale:** The fastest path to value is seeing discovery results for their profile. LLM and vault are enrichment that improves quality but shouldn't block first entry.

### D3: Config validation split — `validate_minimum()` vs `validate()`

Add a new `validate_minimum()` method to `Config` that only checks tier-1 fields (business profile). The existing `validate()` remains for full validation. `init_settings` will use `validate_minimum()` so partially-configured configs are accepted.

**Rationale:** The existing `validate()` rejects configs missing LLM API keys or X client_id. We need a softer check for the shortened onboarding path. Full `validate()` is still used by `PATCH /api/settings` and manual validation endpoints.

### D4: Capability tier computed from config + filesystem state

The tier is not stored — it's derived every time from `Config` + token file existence. This is a pure function with no extra persistence.

**Rationale:** Storing the tier would create stale-state bugs. Computing it is cheap (config parse + 1-2 file existence checks) and always correct.

### D5: Frontend capability store — derived, not manually set

A new `capabilityTier` store in the dashboard derives the tier from the existing `runtime/status` endpoint response (which already returns `can_post`, `provider_backend`, and `capabilities`). We extend the endpoint to also return `capability_tier`.

### D6: Onboarding steps become skippable via "Skip & finish" flow

After the Profile step, a "Skip to finish" link appears that jumps directly to Review. The user still sees Language/Brand, Vault, LLM, and Validation steps in the progress bar but can bypass them. Skipped steps show as "skipped" (not completed) in progress dots.

### D7: ValidationStep becomes optional and LLM-aware

ValidationStep currently auto-runs an LLM connection test. When LLM isn't configured, it should show "LLM not configured — you can set this up later in Settings" instead of failing. The step is skippable.

## Implementation Order

### Phase 1: Core capability tier model (Rust)

**Step 1.1: Create `crates/tuitbot-core/src/config/capability.rs`**

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum CapabilityTier {
    Unconfigured = 0,
    ProfileReady = 1,
    ExplorationReady = 2,
    GenerationReady = 3,
    PostingReady = 4,
}
```

Add `compute_tier(config: &Config, can_post: bool) -> CapabilityTier`:
- Check business profile fields for tier 1
- Check X API config (client_id present or scraper mode) for tier 2
- Check LLM config (provider + api_key for cloud, or ollama) for tier 3
- Check `can_post` flag for tier 4
- Return highest achieved tier

Add `CapabilityTier::label()`, `description()`, `missing_for_next()` methods.

**Step 1.2: Add `validate_minimum()` to `Config` in `validation.rs`**

Only validates:
- `business.product_name` non-empty
- `business.product_description` non-empty (trimmed)
- `business.product_keywords` or `business.competitor_keywords` non-empty
- `business.industry_topics` non-empty
- `storage.db_path` not empty/dir (structural requirement)
- Schedule constraints (these have defaults, so they pass)
- Content sources deployment mode compatibility (if any sources present)

Skips: LLM api_key check, X API client_id check.

**Step 1.3: Register module in `crates/tuitbot-core/src/config/mod.rs`**

Add `pub mod capability;` and re-export `CapabilityTier`.

### Phase 2: Server endpoint changes

**Step 2.1: Modify `init_settings` in `settings.rs`**

Change line ~206 from `config.validate()` to `config.validate_minimum()`. This allows configs without LLM keys or X client_id to be written.

Keep the existing `validate()` path for `PATCH /api/settings` and `POST /api/settings/validate` — those are for complete configurations.

**Step 2.2: Add `capability_tier` to `config_status` response**

In `config_status()`, after checking `configured`:
- If configured, load config and compute tier
- Add `"capability_tier": tier` to response JSON
- This lets the frontend know the tier immediately on load

**Step 2.3: Add `capability_tier` to runtime status response**

In the runtime status handler (routes/runtime.rs), compute and include the tier alongside the existing `can_post` field.

### Phase 3: Frontend capability store

**Step 3.1: Add `CapabilityTier` to `api/types.ts`**

```typescript
export type CapabilityTier =
  'unconfigured' | 'profile_ready' | 'exploration_ready' |
  'generation_ready' | 'posting_ready';
```

Extend `RuntimeStatus` with `capability_tier: CapabilityTier`.

**Step 3.2: Create `dashboard/src/lib/stores/capability.ts`**

Derived store from runtime status:
```typescript
export const capabilityTier = derived(runtimeStatus, ($status) =>
  $status?.capability_tier ?? 'unconfigured'
);

// Convenience checks
export const canExplore = derived(capabilityTier, (t) => tierRank(t) >= 2);
export const canGenerate = derived(capabilityTier, (t) => tierRank(t) >= 3);
export const canPublish = derived(capabilityTier, (t) => tierRank(t) >= 4);
```

**Step 3.3: Expose tier in `runtime.ts`**

Ensure the runtime status polling picks up and stores `capability_tier`.

### Phase 4: Onboarding flow refactor

**Step 4.1: Refactor `+page.svelte` step flow**

The new step flow for both modes:

**API mode (shortened):**
```
Required: Welcome → X Access → [LLM] → [Analyze] → Profile → Review [→ Secure]
Optional: Language, Vault, Validate (accessible but skippable)
```

**Scraper mode (shortened):**
```
Required: Welcome → X Access → Profile → Review [→ Secure]
Optional: LLM, Language, Vault, Validate (accessible but skippable)
```

Implementation approach:
- Keep all steps in the `steps` array
- Add a `requiredSteps` set: `{'Welcome', 'X Access', 'Profile', 'Review', 'Secure'}`
  - API mode also requires: `{'LLM', 'Analyze'}` (LLM is needed for profile analysis)
- After Profile step, show a "Skip optional steps" button alongside "Next"
- Clicking "Skip" jumps `currentStep` to the Review step index
- `canAdvance()` for optional steps always returns `true` (they're skippable)
- Progress dots show optional steps as dimmed/hollow when skipped

**Step 4.2: Update `canAdvance()` logic**

For the shortened flow, `canAdvance()` needs changes:
- `LLM` step: In scraper mode, always returnable `true` (optional). In API mode, still required for analysis.
- `Language` step: Always `true` (has defaults)
- `Vault` step: Always `true` (already skippable)
- `Validate` step: Always `true` (optional when LLM not configured)

**Step 4.3: Modify `submit()` to handle partial config**

The `submit()` function builds the config payload. Changes:
- Don't include `llm` section if provider/key are empty (let defaults apply)
- Don't include `content_sources` if no vault configured
- Don't include `x_api.client_id` if scraper mode and no client_id entered
- Let `validate_minimum()` on the server accept the partial payload

**Step 4.4: Update ReviewStep to show tier status**

ReviewStep should show:
- Which capabilities are active (green check)
- Which are deferred (amber "Set up later" tag)
- Current tier label ("You'll start in exploration mode")
- Clear messaging about what's available vs what needs setup

### Phase 5: ValidationStep update

**Step 5.1: Handle missing LLM in ValidationStep**

Current behavior: auto-runs LLM test on mount.

New behavior:
- If `$onboardingData.llm_provider` is empty or no API key: show info card "LLM not configured — content generation will be available after you set up an LLM provider in Settings"
- If LLM is configured: run test as before
- Always show summary section with current config state
- Remove the step as a blocker — `canAdvance()` always `true`

### Phase 6: Tests

**Step 6.1: Rust tests for capability tier computation**

In `capability.rs` tests:
- `test_unconfigured_tier` — empty config → Unconfigured
- `test_profile_ready_tier` — business profile filled, no X/LLM → ProfileReady
- `test_exploration_ready_tier` — profile + X client_id → ExplorationReady
- `test_generation_ready_tier` — profile + X + LLM configured → GenerationReady
- `test_posting_ready_tier` — all above + can_post=true → PostingReady
- `test_tier_ordering` — verify PartialOrd works correctly
- `test_scraper_mode_exploration_ready` — scraper backend counts for tier 2

**Step 6.2: Rust tests for `validate_minimum()`**

In `validation.rs` tests:
- `test_minimum_valid_no_llm` — config without LLM passes minimum validation
- `test_minimum_valid_no_x_api` — config without client_id (scraper mode) passes
- `test_minimum_fails_no_product_name` — still requires business profile
- `test_minimum_fails_no_description` — still requires description
- `test_full_validate_still_rejects` — full validate rejects what minimum allows

**Step 6.3: Frontend checks (via `npm run check`)**

TypeScript type safety covers the frontend. No runtime tests needed beyond `svelte-check`.

### Phase 7: Documentation

**Step 7.1: Write `progressive-activation.md`**

Design doc covering:
- Capability tier definitions and transitions
- What's required vs optional during onboarding
- How tiers map to UI states
- Server-side computation logic
- Measurement plan for tier transitions

**Step 7.2: Write `session-05-handoff.md`**

Standard handoff covering changes, decisions, open issues, and Session 06 inputs.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `validate_minimum()` allows broken configs that crash at runtime | High | Only relaxes LLM and X API checks; all structural fields (db_path, schedule, limits) still validated. Runtime code already handles missing LLM gracefully (returns error on generation attempt). |
| Existing users' full validate() path breaks | Medium | `validate()` is unchanged. `validate_minimum()` is additive. PATCH and manual validate still use full. |
| Tier computation disagrees between config_status and runtime_status | Low | Both call the same `compute_tier()` function. config_status passes `can_post=false` (no runtime), runtime_status passes actual `can_post`. |
| Skip flow in onboarding creates confusion | Medium | "Skip optional steps" is a secondary action. Primary flow is still "Next". ReviewStep clearly shows what's configured vs deferred. |
| Content sources with empty vault_path pass minimum validation | Low | Empty content_sources section is valid. Onboarding already treats vault as optional. |
