# Session 06 Implementation Plan: First-Run Checklist & Unlocks

## Mission

Build a post-onboarding activation checklist that shows users which capabilities are unlocked, which are deferred, and what the next best action is — using existing settings surfaces instead of rebuilding credential flows.

---

## Key Design Decisions

### D1: ActivationChecklist is a standalone component, not a full page
The checklist lives at `dashboard/src/lib/components/onboarding/ActivationChecklist.svelte` and renders as a card/banner inside the home page (`/`) and optionally the content calendar. This avoids creating a new route and keeps users in the main app shell where they can immediately explore.

**Rationale:** Users who completed onboarding land at `/content?compose=true` which redirects to draft studio. The checklist should be visible on the main surfaces they'll navigate to — not buried in a separate onboarding page they might never revisit.

### D2: Checklist items link directly to Settings sections via hash anchors
Each deferred capability (LLM, Browser Session, Content Sources) links to `/settings#<section-id>` (e.g., `/settings#llm`, `/settings#xapi`, `/settings#sources`). The Settings page already uses IntersectionObserver-based section tracking and has `id` attributes on each `<section>`.

**Rationale:** Reuses existing credential flows entirely. No mini-wizards, no duplication. The settings page already handles save/discard, validation, and scope management.

### D3: Checklist state is derived from capability stores, never persisted
The checklist computes its items from `capabilityTier`, `canExplore`, `canGenerate`, `canPublish`, and `deploymentMode` stores. No localStorage flags or API state for "dismissed" — the checklist automatically disappears when all items are complete (tier 4).

**Rationale:** Follows Session 05's D1 principle (tiers are computed, never stored). Prevents stale-state bugs. Users can always see what's missing.

### D4: Deployment-mode filtering is done in the checklist component
Desktop users see local folder and browser session items. Self-host users see server folder and browser session. Both skip cloud-only items. The `deploymentMode` store already provides this.

### D5: First-value actions are always visible regardless of tier
Even at tier 1 (profile_ready), users can browse settings, view their profile, and explore the dashboard. The checklist shows "what you can do now" alongside "what to set up next." No dead ends.

### D6: Checklist is dismissible per-session but re-shows on tier change
A `$state` boolean tracks whether the user dismissed the checklist banner. It resets when the tier changes (detected via `$effect`). This prevents annoyance while still surfacing new unlocks.

## Files to Create

### 1. `dashboard/src/lib/components/onboarding/ActivationChecklist.svelte`

New component (~200 lines). Structure:

```
Props: none (reads from stores directly)
Stores used:
  - capabilityTier, canExplore, canGenerate, canPublish (from capability.ts)
  - deploymentMode (from runtime.ts)
  - canPost (from runtime.ts)

State:
  - dismissed: boolean ($state, resets on tier change)
  - previousTier: string (for detecting transitions)

Computed checklist items (each has):
  - id: string
  - label: string
  - description: string
  - completed: boolean (derived from tier)
  - href: string (settings section link)
  - deploymentModes: string[] (which modes show this item)
  - tier: number (which tier this unlocks)

Rendering:
  - Tier indicator badge (current tier label + color)
  - "What you can do now" section with current capabilities
  - "Next steps" section with incomplete items as actionable cards
  - Each card has: icon, label, description, "Set up" link button
  - Auto-hides when tier === 'posting_ready'
```

Checklist items definition:

| Item | Completes at | Links to | Desktop | Self-host |
|------|-------------|----------|---------|-----------|
| Business profile | tier >= 1 | `/settings#business` | yes | yes |
| X credentials | tier >= 2 | `/settings#xapi` | yes | yes |
| LLM provider | tier >= 3 | `/settings#llm` | yes | yes |
| Browser session (scraper) | tier >= 4 | `/settings#xapi` | yes | yes |
| OAuth tokens (x_api) | tier >= 4 | `/settings#xapi` | yes | yes |
| Knowledge vault | optional | `/settings#sources` | yes | yes |

"What you can do now" cards by tier:

| Tier | Available actions |
|------|------------------|
| 1 (profile_ready) | View dashboard, edit settings, review profile |
| 2 (exploration_ready) | + Browse discovery, view targets, score content |
| 3 (generation_ready) | + Create AI drafts, compose replies |
| 4 (posting_ready) | + Schedule posts, enable autopilot |

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Checklist doesn't update after settings change | User sees stale state | Tier is derived from runtime status which re-fetches on settings save. `reloadCapabilities()` already exists and is called after settings PATCH. |
| Settings hash links don't scroll | User lands at top of settings | Settings page already uses `IntersectionObserver` with section `id` attributes. Browser native hash scrolling works. Verify during testing. |
| Checklist obscures draft studio on home page | Poor UX for returning users | Checklist auto-hides at tier 4. Dismissible per-session. Compact rendering (not a full-page takeover). |
| `deploymentMode` not loaded when checklist renders | Shows wrong items | `loadCapabilities()` is called in the app layout's `onMount`. Add a loading guard in the checklist. |
| Content calendar `?compose=true` redirect conflicts with new home-page redirect | Double redirect or lost draft creation | Remove the `?compose=true` handling from content page since we now redirect to `/` after onboarding. The draft studio shell on `/` handles first-run experience. |
