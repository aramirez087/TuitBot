# Session 3 Implementation Plan — Profile Analysis Pipeline

## Mission

Build the profile-analysis endpoint that reads the connected X account's profile + recent tweets, runs a two-pass inference pipeline (deterministic heuristics → LLM enrichment), and returns normalized `InferredProfile` suggestions with confidence and provenance.

---

## Key Design Decisions

### D1: Two-pass architecture (heuristics-first, LLM-enrichment-second)

**Decision:** Implement heuristic extraction first, then optionally enrich with LLM.

**Rationale:**
- Heuristics alone can fill `product_url` (profile URL), `product_name` (display name), `product_description` (bio), and `account_type` (bio keyword scan) with zero LLM cost.
- If LLM is not yet configured during onboarding (LLM step comes later in the flow), the endpoint still returns useful defaults.
- LLM enrichment upgrades `target_audience`, `product_keywords`, `industry_topics`, and `brand_voice` — the fields that benefit most from semantic analysis.
- The endpoint accepts an optional `llm_config` body parameter. If provided and valid, LLM enrichment runs. If absent or LLM call fails, heuristic-only results are returned with `status: "partial"` and a warning.

### D2: Fetch up to 50 recent authored tweets

**Decision:** Call `get_user_tweets(user_id, 50, None)` to get recent tweet content.

**Rationale:**
- 50 tweets is enough for tone/topic analysis without excessive API cost (1 API call).
- The inference contract specifies confidence thresholds at 5 and 10 tweets — 50 gives ample headroom.
- Only `text` and `public_metrics` from tweets are used; no storage of raw tweet content beyond the request lifecycle.

### D3: Profile inference lives in `tuitbot-core::toolkit` (not `workflow`)

**Decision:** Place `profile_inference.rs` in `crates/tuitbot-core/src/toolkit/` as a toolkit-level module.

**Rationale:**
- Workflow layer rules say "Workflow MUST call X API through `toolkit::*`". Profile inference directly uses XApiClient, making it toolkit-level.
- It has no DB dependency (no persistence) — it's a stateless function: `(User, Vec<Tweet>, Option<LlmProvider>) → InferredProfile`.
- Keeps workflow layer reserved for DB+LLM composite operations as per the charter.

### D4: No persistence of inference results

**Decision:** The `InferredProfile` is returned to the client and held in the `onboarding-session` store. It is NOT written to disk or DB.

**Rationale:**
- Onboarding data store already holds the final values after user review (Session 04 concern).
- Persisting intermediate inference adds complexity and stale-data risks.
- The endpoint is idempotent — re-calling it re-analyzes from fresh X data.

### D5: Sparse-account fallback strategy

**Decision:** Degrade gracefully at each level:
- **No bio + no tweets:** Return all-defaults with `confidence: "low"`, `provenance: "default"`. Status: `"partial"` with warning.
- **Bio only, no tweets:** Heuristic extraction from bio. `brand_voice` and `industry_topics` are `null`/empty with low confidence.
- **Tweets only, no bio:** LLM enrichment (if available) uses tweet text. `product_url` defaults to null.
- **Auth failure:** Return `status: "x_api_error"` immediately.

### D6: LLM config comes from the request body (not server config)

**Decision:** The analyze endpoint accepts optional LLM config in the request body since the server may not be fully configured yet during onboarding.

**Rationale:**
- During onboarding, the user has connected X but may not have completed the LLM configuration step yet.
- If the frontend has already collected LLM settings (from an earlier step), it can pass them.
- If not, heuristic-only analysis runs — still useful.
- This avoids coupling the analysis endpoint to server-side config state that may not exist.

## Files to Create

### 1. `crates/tuitbot-core/src/toolkit/profile_inference.rs` (NEW)

**Purpose:** Stateless profile inference engine. Contains types and functions.

**Contents:**
- `Confidence` enum (`High`, `Medium`, `Low`) — from inference contract
- `Provenance` enum (`Bio`, `Tweets`, `BioAndTweets`, `ProfileUrl`, `DisplayName`, `Default`) — from inference contract
- `InferredField<T>` struct — generic wrapper with `value`, `confidence`, `provenance`
- `InferredProfile` struct — all 8 fields per the contract
- `ProfileInput` struct — `user: User`, `tweets: Vec<Tweet>`, wraps the raw data
- `fn extract_heuristics(input: &ProfileInput) -> InferredProfile` — deterministic pass
  - `account_type`: Scan bio for business indicators (CEO, founder, "we build", .com, ™, ®)
  - `product_name`: Business → extract company name from bio; Individual → display name
  - `product_description`: Bio text if >10 chars, else empty with low confidence
  - `product_url`: Profile `url` field, else scan bio for URLs
  - `target_audience`: Placeholder "Twitter users" with low confidence
  - `product_keywords`: Extract hashtags + capitalized multi-word terms from bio
  - `industry_topics`: Empty vec with low confidence (needs LLM)
  - `brand_voice`: None with low confidence (needs tweets + LLM)
- `async fn enrich_with_llm(base: InferredProfile, input: &ProfileInput, llm: &dyn LlmProvider) -> Result<InferredProfile, LlmError>` — sends structured prompt per contract, parses JSON response, upgrades fields where LLM provides better data
- `fn build_llm_prompt(input: &ProfileInput) -> (String, String)` — returns (system, user) prompts
- `fn parse_llm_response(text: &str, input: &ProfileInput) -> Result<LlmInferenceResult, String>` — JSON parsing with markdown fence stripping
- `fn merge_llm_into_heuristics(base: InferredProfile, llm: LlmInferenceResult, input: &ProfileInput) -> InferredProfile` — confidence assignment per contract rules
- `fn compute_base_confidence(bio_len: usize, tweet_count: usize) -> Confidence` — per contract algorithm

### 6. `crates/tuitbot-server/src/routes/onboarding.rs`

**Change:** Add `POST /api/onboarding/analyze-profile` handler.

**Handler signature:**
```rust
pub async fn analyze_profile(
    State(state): State<Arc<AppState>>,
    Json(body): Json<AnalyzeProfileRequest>,
) -> Result<Json<Value>, ApiError>
```

**Request body:**
```rust
#[derive(Deserialize)]
pub struct AnalyzeProfileRequest {
    /// Optional LLM config for enrichment. If absent, heuristic-only.
    pub llm: Option<LlmConfigInput>,
}

#[derive(Deserialize)]
pub struct LlmConfigInput {
    pub provider: String,
    pub api_key: Option<String>,
    pub model: String,
    pub base_url: Option<String>,
}
```

**Handler logic:**
1. Load onboarding tokens from `onboarding_token_path(data_dir)`
2. If no tokens → return `{ status: "x_api_error", error: "Not connected" }`
3. Create `XApiHttpClient` from access token
4. Call `client.get_me()` to get user profile
5. Call `client.get_user_tweets(&user.id, 50, None)` to get recent tweets
6. If X API fails → return `{ status: "x_api_error", error: "..." }`
7. Build `ProfileInput` from user + tweets
8. Run `extract_heuristics(&input)` → base profile
9. If `body.llm` is provided:
   a. Create LLM provider via `create_provider(&llm_config)`
   b. Run `enrich_with_llm(base, &input, &provider).await`
   c. If LLM fails → use heuristic base, add warning
10. Return `{ status: "ok"|"partial", profile: InferredProfile, warnings: [...] }`

## Risks and Mitigations

### R1: X API rate limits during tweet fetch
**Risk:** `get_user_tweets` may be rate-limited if the user has been actively using the app.
**Mitigation:** The endpoint catches X API errors and returns `x_api_error` status. The frontend falls back to manual entry. Onboarding tokens are fresh (just authenticated), so rate limit collision is unlikely.

### R2: LLM JSON parsing failures
**Risk:** LLM returns malformed JSON, incomplete fields, or unexpected values.
**Mitigation:** `parse_llm_response()` strips markdown fences, uses `serde_json::from_str` with a lenient intermediate struct (all fields `Option`), and falls back to heuristic values for any missing field. Tests cover malformed responses.

### R3: Onboarding tokens expired between auth and analyze
**Risk:** User connected X, left onboarding idle, tokens expired (2h TTL).
**Mitigation:** Load tokens and check `expires_at`. If expired, return `x_api_error` with message suggesting re-authentication. The frontend can re-trigger the auth flow.

### R4: Empty/private accounts returning no tweets
**Risk:** `get_user_tweets` returns empty `data` for accounts with no tweets or protected accounts.
**Mitigation:** The sparse-account fallback strategy (D5) handles this. Bio-only heuristics still provide useful defaults. Low confidence is communicated to the UI.

### R5: Profile inference module exceeding 500-line limit
**Risk:** The module with types + heuristics + LLM prompt + parsing + tests may exceed 500 lines.
**Mitigation:** If approaching 500 lines, split into a module directory: `profile_inference/mod.rs` (types + public API), `profile_inference/heuristics.rs`, `profile_inference/llm.rs`, `profile_inference/tests.rs`.

### R6: Onboarding analyze endpoint is unauthenticated
**Risk:** Anyone can call `/api/onboarding/analyze-profile` without auth, potentially abusing X API tokens.
**Mitigation:** The endpoint only works if `onboarding_tokens.json` exists on disk — this requires the user to have completed the OAuth flow on this machine. There's no way to supply arbitrary tokens. Additionally, the endpoint is idempotent and read-only (no mutations). For future hardening, rate-limit by IP.

## What Session 04 Needs from This Session

Session 04 builds the single-screen onboarding UI that consumes the analysis API:

1. **API call:** `api.onboarding.analyzeProfile(llmConfig?)` → returns `InferredProfile` with confidence/provenance per field
2. **Store integration:** `onboardingSession.inferred_profile` holds the result; `onboardingSession.analyzing` tracks loading state
3. **Field mapping:** Each `InferredField` has `.value`, `.confidence`, `.provenance` — the UI should:
   - Pre-fill form fields with `.value`
   - Show confidence badges (high=green, medium=yellow, low=gray)
   - Show provenance tooltips ("Inferred from bio", "Inferred from tweets", etc.)
   - All fields are editable — inference is a suggestion, not a commitment
4. **Fallback:** If `status === "partial"` or `status === "x_api_error"`, show manual entry (existing BusinessStep behavior) with a notice explaining why prefill didn't work
5. **Timing:** Call `analyzeProfile()` immediately after X OAuth completes (auto-trigger), or on explicit "Analyze my profile" button click
