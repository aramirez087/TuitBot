## Part 1: Analytics Event Infrastructure

### Decision D1: Client-side event log, not a third-party SDK

Tuitbot is a self-hosted/desktop app with no cloud telemetry service. The analytics system should:
- Log structured events to `console.info` with a consistent prefix (e.g., `[tuitbot:funnel]`)
- Optionally emit events to a new lightweight `funnelEvents` writable store so the UI or a future backend endpoint can consume them
- NOT add any external analytics dependency (no Posthog, Amplitude, etc.)

This matches the existing `console.info('[draft-studio]', ...)` pattern already in `content/+page.svelte:60`.

### Decision D2: Event taxonomy uses kebab-case string names

All events follow `{domain}:{action}` naming: `onboarding:started`, `onboarding:x-auth-success`, etc.

### Task 1.1: Create `dashboard/src/lib/analytics/funnel.ts`

A minimal tracking utility:

```typescript
export interface FunnelEvent {
  event: string;
  properties?: Record<string, unknown>;
  timestamp: string;
}

export function trackFunnel(event: string, properties?: Record<string, unknown>): void {
  const entry: FunnelEvent = {
    event,
    properties,
    timestamp: new Date().toISOString(),
  };
  console.info('[tuitbot:funnel]', entry);
  // Future: push to a store or backend endpoint
}
```

**File**: `dashboard/src/lib/analytics/funnel.ts` (NEW, ~25 lines)

## Part 2: Harden Failure States

### Task 2.1: Auth loss during onboarding

**File**: `dashboard/src/lib/components/onboarding/XApiStep.svelte` (MODIFY)

**Problem**: If the OAuth window is closed or the server restarts mid-flow, the polling loop may hang or error silently.

**Fix**:
- Add a 5-minute timeout on the polling loop; show a "Connection timed out" message with retry button
- If `authStatus` returns an error response (not just `connected: false`), surface the error message instead of continuing to poll
- Add an explicit "Cancel" button during the polling phase that stops the interval and shows the initial form

### Task 2.2: Sparse/empty X profiles

**File**: `dashboard/src/lib/components/onboarding/ProfileAnalysisState.svelte` (MODIFY)
**File**: `dashboard/src/lib/components/onboarding/PrefillProfileForm.svelte` (MODIFY)

**Problem**: Users with empty bios and no tweets get an `InferredProfile` where everything is Low confidence. The UI doesn't call this out clearly.

**Fix**:
- In `ProfileAnalysisState`, if result.status is `'partial'` or all fields are Low confidence, show a notice: "We couldn't find much on your profile. You'll fill in the details next."
- In `PrefillProfileForm`, if no fields were inferred (all empty/default), hide confidence badges and show the form in "manual entry" mode with a brief helper: "Tell us about your product or personal brand."

### Task 2.3: Low-confidence inference display

**File**: `dashboard/src/lib/components/onboarding/PrefillProfileForm.svelte` (MODIFY)

**Problem**: Low-confidence fields should be visually distinct from high-confidence ones so users know which need attention.

**Fix**:
- Low-confidence fields get a subtle highlight (e.g., dashed border or amber left-border) with tooltip "This was guessed — please review"
- High-confidence fields show a green check badge (existing pattern)
- Medium-confidence fields show no special styling (neutral)

### Task 2.4: Server unavailable during init

**File**: `dashboard/src/routes/onboarding/+page.svelte` (MODIFY)

**Problem**: Network errors during `submit()` show raw error messages.

**Fix**:
- Catch network/fetch errors specifically (e.g., `TypeError: Failed to fetch`) and show: "Can't reach the Tuitbot server. Check that it's running and try again."
- Add a retry button alongside the error banner (currently only shows text)
- Preserve form state across retries (already works since `onboardingData` store persists)

### Task 2.5: Unsupported deployment mode guard

**File**: `dashboard/src/routes/onboarding/+page.svelte` (MODIFY)

**Problem**: If `configStatus` returns an unrecognized deployment mode, the onboarding flow may behave unexpectedly.

**Fix**:
- On mount, check `api.settings.configStatus()` — if `configured: true`, redirect to `/` immediately (already initialized)
- If deployment mode is not one of `desktop`, `self_host`, `cloud`, show a fallback message: "This deployment mode is not supported for browser onboarding. Use `tuitbot init` from the command line."

### Task 2.6: Analysis endpoint error differentiation

**File**: `dashboard/src/lib/components/onboarding/ProfileAnalysisState.svelte` (MODIFY)

**Problem**: The error card shows the same UI for X API errors (tokens expired) vs LLM errors (wrong key) vs network errors.

**Fix**:
- For `x_api_error`: "Couldn't access your X account. Your tokens may have expired." + "Retry" and "Continue manually"
- For `llm_error`: "AI analysis wasn't available, but we extracted what we could from your profile." + auto-continue with partial results
- For network error: "Can't reach the server. Check your connection." + "Retry"
- For `partial` status: auto-advance (don't block — partial is still useful)

### Guiding principles (from product context)
- Direct, practical language. No marketing fluff.
- "Tuitbot" not "TuitBot" in user-facing copy.
- Use "X" not "Twitter" except parenthetical "(Twitter)" on first mention.
- Action-oriented CTAs. Say what happens, not what the thing is.

### Task 4.2: Create `docs/roadmap/x-profile-prefill-onboarding/funnel-metrics.md`

**File**: `docs/roadmap/x-profile-prefill-onboarding/funnel-metrics.md` (NEW)

Document the funnel stages, key metrics, and measurement approach:

**Funnel Stages:**
1. Entry (`onboarding:started`) — User begins onboarding
2. X Auth (`onboarding:x-auth-success` / `onboarding:scraper-selected`) — X connection established
3. Analysis (`onboarding:analysis-success` / `onboarding:analysis-skipped`) — Profile analyzed or skipped
4. Profile Complete (`onboarding:step-entered` where step='Review') — Required fields filled
5. Submission (`onboarding:submitted`) — Init payload sent
6. Completion (`onboarding:completed`) — Config created, redirected to app
7. First Checklist View (`activation:checklist-viewed`) — User sees activation checklist
8. Tier 2 Unlock (`activation:tier-changed` to `exploration_ready`) — X credentials configured
9. Tier 3 Unlock (`activation:tier-changed` to `generation_ready`) — LLM configured
10. Tier 4 Unlock (`activation:tier-changed` to `posting_ready`) — Full posting access

**Key Activation Metrics:**
- **Onboarding completion rate**: `onboarding:completed` / `onboarding:started`
- **X auth rate**: `onboarding:x-auth-success` / `onboarding:started`
- **Analysis utilization**: `onboarding:analysis-success` / `onboarding:x-auth-success`
- **Profile edit rate**: `onboarding:profile-edited` / `onboarding:analysis-success`
- **Skip rate**: `onboarding:step-skipped` / `onboarding:started`
- **Time to posting_ready**: timestamp delta from `onboarding:completed` to `activation:tier-changed{to:posting_ready}`
- **Checklist engagement**: `activation:checklist-item-clicked` / `activation:checklist-viewed`

**Measurement Method:**
- All events logged to `console.info('[tuitbot:funnel]', ...)` with structured JSON
- Grep-able from Tauri logs or browser DevTools
- Future: optional server-side event sink at `POST /api/telemetry/funnel` (not in scope for Session 8)

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| `$effect` for step tracking fires on initial render | Spurious `step-entered` events | Guard with `if (currentStep > 0)` or track previous step |
| XApiStep polling timeout conflicts with slow OAuth | User sees timeout when still completing OAuth in browser | Set timeout to 5 min (generous), show "Still waiting..." at 2 min |
| Copy changes break long strings in narrow viewports | Layout overflow | Test at 320px width during verification |
| PrefillProfileForm confidence styling adds visual noise | Cognitive overload | Only highlight Low confidence, leave Medium/High neutral |
| funnel.ts import adds bundle size | Negligible | <1KB, no external deps |
| Error retry in submit() could cause double-init | 409 already handled | Existing 409 recovery redirects to `/` |
