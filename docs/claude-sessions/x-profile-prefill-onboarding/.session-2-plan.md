# Session 02 Implementation Plan: Unified Entry & X Auth Bootstrap

## Mission

Build the unified onboarding entry path so users in every deployment mode can start with X sign-in instead of the long wizard. This session introduces onboarding-specific OAuth endpoints, a short-lived onboarding session store, and the shared "Continue with X" entry point.

---

## Key Design Decisions

### D1: New `/api/onboarding/x-auth/*` endpoints (not reusing account-scoped)

**Decision:** Create new onboarding-specific OAuth endpoints at `/api/onboarding/x-auth/start` and `/api/onboarding/x-auth/callback`.

**Rationale:**
- The existing account-scoped endpoints (`/api/accounts/{id}/x-auth/*`) call `accounts::get_account()` which requires a DB account to exist. During onboarding, no account exists yet.
- Using `DEFAULT_ACCOUNT_ID` would save tokens to the default account path immediately, which is correct for the happy path but conflates "temporary onboarding tokens" with "permanent account tokens" before setup is finalized.
- New endpoints share the same core PKCE infrastructure (`generate_pkce`, `build_auth_url`, `exchange_auth_code`) but store tokens in a temporary onboarding-specific location.
- After `POST /api/settings/init` succeeds, the tokens are migrated from the temp path to the account token path.

### D2: Temporary token storage at `data_dir/onboarding_tokens.json`

**Decision:** Store onboarding OAuth tokens at `{data_dir}/onboarding_tokens.json`.

**Rationale:**
- Single well-known path, easy to find and clean up.
- Migrated to `account_token_path(data_dir, DEFAULT_ACCOUNT_ID)` during `POST /api/settings/init`.
- Cleaned up on: successful migration, server restart, or if file is older than 1 hour.
- No need for a DB table since onboarding is ephemeral and single-user.

### D3: Skip `entities` field on User struct (just add `description`, `location`, `url`)

**Decision:** Add only `description: Option<String>`, `location: Option<String>`, `url: Option<String>` to the `User` struct. Skip `entities` for now.

**Rationale:**
- `entities` is a deeply nested type (`entities.url.urls[].expanded_url`, `entities.description.urls[]`, etc.) that adds significant complexity.
- `description`, `location`, and `url` cover 95%+ of inference value for the profile prefill use case.
- `entities` can be added in a follow-up if URL expansion from bio links proves important.

### D4: Auth middleware exemption for onboarding endpoints

**Decision:** Add `/onboarding/x-auth/start`, `/onboarding/x-auth/callback`, and `/onboarding/x-auth/status` to `AUTH_EXEMPT_PATHS`.

**Rationale:**
- Onboarding happens before any auth is configured (no bearer token, no session cookie).
- The existing exemptions for `/settings/init` and `/settings/status` establish this pattern.
- The endpoints are safe to expose unauthenticated because they only interact with temporary onboarding state.

### D5: Onboarding session store is client-side only (no server session)

**Decision:** The onboarding session state lives entirely in the Svelte store (`onboarding-session.ts`), not in a server-side session.

**Rationale:**
- Onboarding is a single-page flow; no need to survive full page reloads (the store already uses Svelte writable).
- Server-side onboarding state is limited to: pending PKCE challenges (already in `AppState.pending_oauth`) and the temporary token file.
- A server-side session would add complexity (new DB table, session cookies before claim) for no clear benefit at this stage.

### Phase 2: Onboarding Server Endpoints (Rust)

#### Step 2.1: Create `onboarding.rs` route module
**File:** `crates/tuitbot-server/src/routes/onboarding.rs`

Endpoints:

1. **`POST /api/onboarding/x-auth/start`**
   - No account validation needed (no account exists yet).
   - Requires `x_client_id` to be non-empty in AppState (returns 400 if missing).
   - Generates PKCE challenge, stores in `pending_oauth` with `account_id = "__onboarding__"`.
   - Returns `{ authorization_url, state }`.

2. **`POST /api/onboarding/x-auth/callback`**
   - Validates state parameter matches a pending PKCE entry with `account_id == "__onboarding__"`.
   - Exchanges code for tokens.
   - Saves tokens to `{data_dir}/onboarding_tokens.json`.
   - Calls `get_me()` using the new access token to fetch user identity.
   - Returns `{ status: "connected", user: { id, username, name, profile_image_url, description, location, url } }`.

3. **`GET /api/onboarding/x-auth/status`**
   - Checks if `onboarding_tokens.json` exists and is valid.
   - If valid, loads tokens and calls `get_me()` to return user identity.
   - Returns `{ connected: bool, user?: { ... } }`.

Key implementation details:
- Reuse `generate_pkce()`, `build_auth_url()`, `build_redirect_uri()`, `exchange_auth_code()` from `tuitbot_core::startup`.
- Reuse `auth::Tokens`, `auth::save_tokens()`, `auth::load_tokens()` for token persistence.
- Create a temporary `XApiHttpClient` with the onboarding tokens to call `get_me()`.
- The `__onboarding__` sentinel in `PendingOAuth.account_id` distinguishes onboarding flows from account flows, preventing cross-flow state theft.

Helper function:
```rust
fn onboarding_token_path(data_dir: &Path) -> PathBuf {
    data_dir.join("onboarding_tokens.json")
}
```

### Phase 3: Frontend Store & API (TypeScript)

#### Step 3.1: Create `onboarding-session.ts`
**File:** `dashboard/src/lib/stores/onboarding-session.ts`

```typescript
import { writable, derived, get } from 'svelte/store';

export interface OnboardingXUser {
    id: string;
    username: string;
    name: string;
    profile_image_url: string | null;
    description: string | null;
    location: string | null;
    url: string | null;
}

export interface OnboardingSession {
    // X Auth state
    x_connected: boolean;
    x_user: OnboardingXUser | null;
    oauth_state: string;
    auth_url: string;
    auth_error: string;
    auth_loading: boolean;
}

function createOnboardingSession() {
    const initial: OnboardingSession = {
        x_connected: false,
        x_user: null,
        oauth_state: '',
        auth_url: '',
        auth_error: '',
        auth_loading: false,
    };

    const { subscribe, update, set } = writable<OnboardingSession>(initial);

    return {
        subscribe,
        set,
        update,
        setConnected: (user: OnboardingXUser) => {
            update(s => ({ ...s, x_connected: true, x_user: user, auth_loading: false, auth_error: '' }));
        },
        setAuthUrl: (url: string, state: string) => {
            update(s => ({ ...s, auth_url: url, oauth_state: state, auth_loading: false }));
        },
        setError: (error: string) => {
            update(s => ({ ...s, auth_error: error, auth_loading: false }));
        },
        setLoading: (loading: boolean) => {
            update(s => ({ ...s, auth_loading: loading }));
        },
        reset: () => set(initial),
    };
}

export const onboardingSession = createOnboardingSession();
```

OAuth flow mechanics:
1. User clicks "Connect with X" button.
2. Frontend calls `POST /api/onboarding/x-auth/start` (auth-exempt).
3. Frontend opens `authorization_url` in a popup/new tab.
4. User authorizes on X, gets redirected to callback URL.
5. The callback page (existing at `/callback`) needs to handle the onboarding case.
6. Frontend polls `GET /api/onboarding/x-auth/status` until connected or timeout.

**Important:** The existing OAuth callback flow uses a local HTTP server at `127.0.0.1:8080/callback` that captures the code and state. For onboarding, we need the frontend to handle the callback code. Two approaches:

**Approach A (recommended for desktop):** The existing Tauri callback handler at port 8080 can be reused. After receiving the code, the frontend calls `POST /api/onboarding/x-auth/callback` with the code and state.

**Approach B (for web/LAN):** The callback redirects to the dashboard with query params (`?code=...&state=...`), and the onboarding page detects and processes them.

For this session, implement polling-based detection: after opening the auth URL, poll `/api/onboarding/x-auth/status` every 2s. The callback handler (existing Tauri mechanism or manual paste) calls the callback endpoint, and the poll detects the result.

## Risks and Mitigations

### R1: Callback URL routing during onboarding
**Risk:** The existing OAuth callback mechanism (Tauri local server at port 8080) may not work during onboarding if the config isn't fully loaded yet.

**Mitigation:** The callback server is started by the Tauri app, not the API server. In desktop mode, it should be available. In web/LAN mode, the user needs to manually paste the callback URL or the server-side callback handler needs to work. For this session, implement polling-based detection as the primary mechanism, which works regardless of callback delivery method.

### R2: Temporary token file left behind after abandoned onboarding
**Risk:** User starts onboarding, connects X, but never finishes setup. `onboarding_tokens.json` sits on disk with valid tokens.

**Mitigation:**
1. On server startup, delete `onboarding_tokens.json` if it exists and is older than 1 hour.
2. The `init_settings` endpoint migrates or removes it.
3. Tokens have a 2-hour expiry anyway, so stale files are harmless after that.

### R3: Concurrent onboarding sessions
**Risk:** Two browser tabs both start onboarding OAuth flows.

**Mitigation:** Only one `onboarding_tokens.json` can exist at a time. The last callback to complete wins. This is acceptable for single-user desktop/self-host deployments.

### R4: `get_me()` call in callback adding latency
**Risk:** The callback endpoint calls `get_me()` to fetch user identity, adding ~200-500ms latency to the callback response.

**Mitigation:** Acceptable latency for a one-time onboarding step. The alternative (returning just `status: linked` and fetching identity on a separate poll) adds complexity without meaningful UX improvement since the user is already waiting.

## API Contracts for Session 03

### `POST /api/onboarding/x-auth/start`
**Auth:** Exempt
**Request:** Empty body
**Response:**
```json
{
  "authorization_url": "https://twitter.com/i/oauth2/authorize?...",
  "state": "random_state_string"
}
```
**Errors:** 400 if `x_client_id` not configured.

### `POST /api/onboarding/x-auth/callback`
**Auth:** Exempt
**Request:**
```json
{ "code": "auth_code_from_x", "state": "matching_state" }
```
**Response:**
```json
{
  "status": "connected",
  "user": {
    "id": "123456",
    "username": "handle",
    "name": "Display Name",
    "profile_image_url": "https://...",
    "description": "Bio text here",
    "location": "City, Country",
    "url": "https://t.co/..."
  }
}
```
**Errors:** 400 invalid/expired state, 500 token exchange failure.

### `GET /api/onboarding/x-auth/status`
**Auth:** Exempt
**Response:**
```json
{
  "connected": true,
  "user": { ... }
}
```
or
```json
{
  "connected": false
}
```
