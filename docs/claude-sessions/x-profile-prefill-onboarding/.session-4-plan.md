# Session 4 Implementation Plan — Single-Screen Onboarding UI

## Mission

Replace the multi-step onboarding wizard's Profile step with a single-screen editable prefill experience. After X OAuth + LLM setup, users see all inferred fields pre-populated with confidence/provenance metadata, edit anything they want, and advance — no more stepping through empty manual forms.

---

## Architecture Overview

### Current Step Flow
```
0: Welcome → 1: X Access → 2: Profile (BusinessStep) → 3: LLM → 4: Language → 5: Vault → 6: Validate → 7: Review → [8: Secure]
```

### Target Step Flow (per target-flow.md)
```
0: Welcome → 1: X Access → 2: LLM → 3: Analyze → 4: Profile (PrefillProfileForm) → 5: Language → 6: Vault → 7: Validate → 8: Review → [9: Secure]
```

### Key Changes
- **LLM moves before Profile** (step 2→3 becomes step 3→2) so analysis can use LLM config
- **New analysis step** (step 3) auto-triggers `analyzeProfile()` and auto-advances
- **BusinessStep replaced** by `PrefillProfileForm` showing inferred fields with confidence badges
- **New `ProfileAnalysisState` component** handles the loading/progress UI for the analysis step
- **Scraper mode** skips analysis step and shows empty PrefillProfileForm (manual entry fallback)

## Files to Create

### 1. `dashboard/src/lib/components/onboarding/ProfileAnalysisState.svelte`

**Purpose:** Loading screen for step 3 (Analyze). Shows animated progress: fetching profile → reading tweets → analyzing strategy. Auto-triggers the API call and auto-advances when done.

**Props:**
```typescript
interface Props {
  oncomplete: () => void;  // called when analysis finishes (success or fallback)
  onerror: (msg: string) => void;  // called on unrecoverable error
}
```

**Behavior:**
1. On mount, call `api.onboarding.analyzeProfile(llmConfig)` where `llmConfig` is read from `$onboardingData` (provider, api_key, model, base_url)
2. Show 3 progress steps with checkmarks/spinner animation:
   - "Fetching your X profile..." → check after ~1s (simulated, actual is instant)
   - "Reading your recent tweets..." → check after ~1.5s
   - "Analyzing your content strategy..." → check when API returns
3. On success (`status === 'ok'` or `'partial'`): store results via `onboardingSession.setInferredProfile()`, call `oncomplete()`
4. On error (`status === 'x_api_error'`): show error message + "Continue manually" button that calls `oncomplete()`
5. If `brand_voice` is inferred, apply it to `onboardingData.brand_voice` (mapping: "professional"→"balanced", "casual"→"bold", "formal"→"conservative")

**Design:** Centered layout with stacked progress items. Each item: icon (spinner or checkmark) + label. Uses existing `Loader2`, `CheckCircle2` icons from lucide-svelte. Matches the ValidationStep animation pattern.

**Size estimate:** ~120 lines (well under 400-line Svelte limit)

**Confidence badges:**
- `high` → small green dot (6px circle, `var(--color-success)`)
- `medium` → small yellow dot (6px circle, `var(--color-warning)`)
- `low` → small gray dot (6px circle, `var(--color-text-subtle)`)
- Only shown when inference data exists for that field

**Provenance tags:**
- Small text next to label: "from bio", "from tweets", "from bio + tweets", "from profile URL", "from display name"
- Color: `var(--color-text-subtle)`, font-size: 11px
- Only shown when inference data exists

**Connected account card:**
- Only shown when `$onboardingSession.x_connected` is true

- If `$onboardingSession.analysis_warnings.length > 0`, show a yellow warning banner at the bottom

**Field sync to store:**
- Each field uses `$state()` initialized from `$onboardingData` values
- If `$onboardingSession.inferred_profile` exists, use an `$effect()` on mount to prefill fields from inferred values (only if current store value is empty/default — don't overwrite user edits if they navigate back)
- Each field syncs back to `onboardingData` via `$effect()` (same pattern as current BusinessStep)

**Reuses:** `TagInput` component for keywords and topics

**Size estimate:** ~300 lines (under 400-line limit). If it grows, the connected-account card can be extracted.

### 3. `dashboard/src/routes/onboarding/+page.svelte`

**Changes:**

a) **Update step labels:**
```typescript
// Before:
const BASE_STEPS = ['Welcome', 'X Access', 'Profile', 'LLM', 'Language', 'Vault', 'Validate', 'Review'];

// After — dynamic based on mode:
const API_STEPS = ['Welcome', 'X Access', 'LLM', 'Analyze', 'Profile', 'Language', 'Vault', 'Validate', 'Review'];
const SCRAPER_STEPS = ['Welcome', 'X Access', 'Profile', 'LLM', 'Language', 'Vault', 'Validate', 'Review'];
// Use $derived to pick based on $onboardingData.provider_backend
```

Wait — re-evaluating. The target-flow says scraper users skip the Analyze step and see the old BusinessStep layout. The simplest approach: always use the same step array but conditionally show/skip the Analyze step for scraper mode.

**Revised approach:** Use a single step list but make the Analyze step conditional:
```typescript
let isScraperMode = $derived($onboardingData.provider_backend === 'scraper');
let BASE_STEPS = $derived(
  isScraperMode
    ? ['Welcome', 'X Access', 'Profile', 'LLM', 'Language', 'Vault', 'Validate', 'Review']
    : ['Welcome', 'X Access', 'LLM', 'Analyze', 'Profile', 'Language', 'Vault', 'Validate', 'Review']
);
```

This means step indices shift based on mode:
- **API mode:** 0=Welcome, 1=XAccess, 2=LLM, 3=Analyze, 4=Profile, 5=Language, 6=Vault, 7=Validate, 8=Review, [9=Secure]
- **Scraper mode:** 0=Welcome, 1=XAccess, 2=Profile, 3=LLM, 4=Language, 5=Vault, 6=Validate, 7=Review, [8=Secure]

To keep the step-switch logic manageable, use step names rather than raw indices:
```typescript
let currentStepName = $derived(steps[currentStep]);
```

Then in the template:
```svelte
{#if currentStepName === 'Welcome'}
  <WelcomeStep />
{:else if currentStepName === 'X Access'}
  <XApiStep />
{:else if currentStepName === 'LLM'}
  <LlmStep />
{:else if currentStepName === 'Analyze'}
  <ProfileAnalysisState oncomplete={next} />
{:else if currentStepName === 'Profile'}
  <PrefillProfileForm />
{:else if currentStepName === 'Language'}
  ...
```

b) **Update `canAdvance()` to use step names:**
```typescript
function canAdvance(): boolean {
  const data = $onboardingData;
  switch (currentStepName) {
    case 'Welcome': return true;
    case 'X Access': ...same logic...
    case 'Profile': ...same validation (name, desc, audience, keywords, topics required)...
    case 'LLM': ...same logic...
    case 'Analyze': return false; // auto-advances via oncomplete
    case 'Language': return true;
    case 'Vault': return true;
    case 'Validate': return true;
    case 'Review': return true;
    case 'Secure': ...same logic...
    default: return false;
  }
}
```

c) **Update imports:**
- Add `ProfileAnalysisState` and `PrefillProfileForm` imports
- Remove `BusinessStep` import (or keep for scraper fallback — but PrefillProfileForm handles both cases)

d) **Hide Next button on Analyze step** (it auto-advances):
- In the actions section, hide Next when `currentStepName === 'Analyze'`

e) **Auto-advance from Analyze step:** The `oncomplete` callback calls `next()` directly.

### 4. `dashboard/src/lib/stores/onboarding.ts`

**Changes:** Minimal. The store shape is already correct — all profile fields exist. One small addition:

- Add a `prefillFromInference(profile: InferredProfile)` method that maps inferred values into the store fields, only setting values that are currently empty/default:
```typescript
prefillFromInference: (profile: InferredProfile) => {
  update((data) => ({
    ...data,
    account_type: profile.account_type.value,
    product_name: data.product_name || profile.product_name.value,
    product_description: data.product_description || profile.product_description.value,
    product_url: data.product_url || profile.product_url.value || '',
    target_audience: data.target_audience || profile.target_audience.value,
    product_keywords: data.product_keywords.length > 0 ? data.product_keywords : profile.product_keywords.value,
    industry_topics: data.industry_topics.length > 0 ? data.industry_topics : profile.industry_topics.value,
  }));
}
```

This centralizes the prefill logic so both `ProfileAnalysisState` (after API success) and `PrefillProfileForm` (on mount) don't duplicate logic.

## Key Design Decisions

### D1: Step names instead of indices
Using step name strings for the switch/case logic eliminates brittle index arithmetic when steps shift between API and scraper modes. This is the biggest structural change to `+page.svelte`.

### D2: PrefillProfileForm handles both inferred and manual modes
Rather than maintaining two separate components (BusinessStep for manual, ProfileReviewStep for inferred), a single `PrefillProfileForm` works for both. When no inferred data exists, confidence badges and provenance tags simply don't render, and the form behaves identically to the old BusinessStep.

### D3: Prefill happens in the Analyze step's oncomplete, not in the form
`ProfileAnalysisState` calls `onboardingData.prefillFromInference(profile)` immediately after receiving the API response, before auto-advancing. This means `PrefillProfileForm` just reads from `$onboardingData` (already populated) and `$onboardingSession.inferred_profile` (for metadata display only).

### D4: brand_voice mapping from inference
The `brand_voice` field from inference returns values like "professional", "casual", "formal", "witty". These map to the existing brand voice options: "conservative", "balanced", "bold". Mapping: professional→balanced, casual→bold, formal→conservative, witty→bold. Applied during prefill in `ProfileAnalysisState`.

### D5: No separate Analyze step for scraper mode
Scraper users skip directly from X Access to Profile (same step order as current). The `BASE_STEPS` derivation handles this conditionally.

### D6: Connected account identity card in PrefillProfileForm
Showing the connected X account (avatar + @username) at the top of the profile form reinforces the link between "we analyzed your account" and "here are the results." This is a lightweight copy of the connected-card from XApiStep.

## Risks and Mitigations

### R1: Step index confusion during refactor
**Risk:** Changing from index-based to name-based step logic could introduce bugs in `canAdvance()`, `isLastStep`, or claim step detection.
**Mitigation:** The `currentStepName` derivation makes logic readable. The `isClaimStep` derivation stays the same (`showClaimStep && currentStep === steps.length - 1`). All conditions are straightforward.

### R2: PrefillProfileForm exceeds 400-line Svelte limit
**Risk:** With confidence badges, provenance tags, connected-account card, and all fields, the component could grow large.
**Mitigation:** Connected-account card is ~20 lines of template. Confidence/provenance helpers are ~5 lines each. Total estimate is ~300 lines. If it exceeds 400, extract the connected-account card into a small inline snippet or a separate component.

### R3: Race condition — user navigates back to X Access during analysis
**Risk:** User clicks Back while analysis is running.
**Mitigation:** The analysis call is fire-and-forget from the component lifecycle. If the user navigates away, the component unmounts and the `oncomplete` callback won't fire (component destroyed). If they come back to Analyze, it re-runs. The store's `analyzing` flag prevents double-trigger.

### R4: Brand voice mapping is lossy
**Risk:** The inference returns arbitrary brand_voice strings that may not map cleanly to the 3 options.
**Mitigation:** Use a mapping table with a fallback to "balanced". The mapping only runs if inference provides a value. Users can always change it on the Language step.

### R5: TagInput needs to show inference metadata
**Risk:** The existing `TagInput` component doesn't support confidence dots or provenance labels.
**Mitigation:** Don't modify TagInput. Instead, show the confidence dot and provenance label on the field label line (above the TagInput), not inside it. This is consistent with how text fields show their metadata.
