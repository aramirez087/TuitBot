## Mission

Produce four planning documents under `docs/roadmap/x-profile-prefill-onboarding/` that define the deployment-unified onboarding flow starting with X sign-in, profile inference, and editable prefill.

## Current State Analysis

### Current Onboarding Flow (8–9 steps)

The existing flow in `dashboard/src/routes/onboarding/+page.svelte` is a linear wizard:

1. **Welcome** — static splash (no friction)
2. **X Access** — user must manually enter X API Client ID (or choose scraper mode). No OAuth sign-in happens here — just credential input.
3. **Profile** (BusinessStep) — 6 required fields all typed manually: account_type, product_name, product_description, target_audience, product_keywords[], industry_topics[]
4. **LLM** — provider selection + API key
5. **Language & Brand** — language + brand_voice (defaults available, low friction)
6. **Vault** (Sources) — optional content sources
7. **Validate** — LLM connection test
8. **Review** — summary + approval_mode toggle
9. **Secure** (ClaimStep, web-only) — passphrase for self-host

### Key Friction Points

- **Step 2 (X Access)**: Requires user to create an X Developer App, configure OAuth callback, and copy Client ID — before they see any value.
- **Step 3 (Profile)**: All 6 fields manual with no hints or prefill. This is where most drop-off likely occurs.
- **No OAuth flow during onboarding**: X auth happens post-setup via `tuitbot auth` CLI or the accounts panel.
- **No profile data is fetched from X**: The `User` struct only has `id`, `username`, `name`, `profile_image_url`, `public_metrics` — no bio/description, location, or URL fields.

### Available Infrastructure

- **OAuth PKCE flow**: Already implemented in `x_auth.rs` with `start_link`, `complete_link`, `link_status`
- **`get_me`** and **`get_user_tweets`** in toolkit: Can fetch authenticated user profile and recent tweets
- **`sync_profile`** in accounts.rs: Already syncs X profile data to account record
- **BusinessProfile struct**: Has `product_name`, `product_description`, `product_url`, `target_audience`, `product_keywords[]`, `industry_topics[]`, plus enrichment fields (`brand_voice`, `reply_style`, `content_style`, `persona_opinions`, `persona_experiences`, `content_pillars`)
- **DeploymentMode**: `Desktop` / `SelfHost` / `Cloud` with capability gating already in place
- **Config validation**: Requires `product_name`, `product_keywords OR competitor_keywords`, `product_description`, `industry_topics` as non-empty

### Critical Gaps

1. **X API `User` type missing bio fields**: The `User` struct in `x_api/types.rs` doesn't include `description`, `location`, `url`, or `entities` — all available from X API v2 `/users/me` with `user.fields=description,location,url,entities`.
2. **No profile analysis endpoint**: No server endpoint accepts X profile data and returns inferred business profile fields.
3. **No LLM-based inference**: No existing code analyzes recent tweets to extract keywords, topics, audience, or tone.
4. **Onboarding store has no inference state**: No concept of inferred values, confidence, or provenance.

## Key Design Decisions

### 1. X Sign-In First (Not Client ID First)

**Decision**: Move OAuth PKCE sign-in to be the first substantive step. Client ID entry still needed, but immediately followed by "Connect your X account" which performs OAuth and fetches profile.

**Rationale**: Sign-in gives us identity + profile data for prefill. Current flow asks for Client ID but doesn't use it until after onboarding.

### 2. Extend `User` Type with Bio Fields

**Decision**: Add `description`, `location`, `url`, `entities` to the `x_api::types::User` struct and request `user.fields` in the API call.

**Rationale**: X API v2 returns these when requested. The bio is the primary input for inferring product_description, target_audience, and keywords.

### 3. LLM-Based Profile Inference in `tuitbot-core`

**Decision**: Create a new module `core::workflow::profile_inference` that:
- Takes `User` (with bio) + recent tweets (10–20)
- Calls LLM with a structured prompt
- Returns `InferredProfile` with per-field confidence + provenance

**Rationale**: Business logic belongs in core (per architecture.md), not in server handlers or frontend.

### 4. New Server Endpoint: `POST /api/onboarding/analyze-profile`

**Decision**: New endpoint that:
1. Uses OAuth tokens from the just-completed auth flow
2. Calls `get_me` to fetch profile with expanded fields
3. Calls `get_user_tweets` for recent 10-20 tweets
4. Runs `profile_inference` with the configured LLM
5. Returns `InferredProfile` JSON

**Rationale**: Keeps server thin (routing + delegation). Could also be done client-side via multiple API calls, but a single endpoint is simpler and avoids exposing raw tweet data to frontend.

### 5. Single-Screen Editable Prefill

**Decision**: Replace the current multi-step Profile wizard (BusinessStep) with a single review screen showing all inferred fields with edit capability. Each field shows its inferred value, confidence indicator, and source provenance.

**Rationale**: Users should see all inferences at once and correct anything wrong, not step through fields one by one.

### 6. Shared Flow, Minimal Branching

**Decision**: Desktop and SelfHost share identical onboarding steps. Only divergences:
- **ClaimStep** (web-only, already existing)
- **Content Sources** default (local_fs for Desktop, google_drive for SelfHost — already handled by capabilities)

Cloud compatibility note: Future cloud mode would skip Client ID entry (managed by platform) and ClaimStep (managed auth), but the profile inference flow would be identical.

### 7. Scraper Mode Compatibility

**Decision**: For scraper/no-key mode users, profile inference is unavailable (no OAuth = no `get_me`). They get the current manual entry flow as fallback. The UI gracefully degrades.

### inference-contract.md

Define `InferredProfile` response contract:

```typescript
interface InferredField<T> {
  value: T;
  confidence: "high" | "medium" | "low";
  provenance: string; // e.g., "X bio", "recent tweets", "bio + tweets", "default"
}

interface InferredProfile {
  account_type: InferredField<"individual" | "business">;
  product_name: InferredField<string>;
  product_description: InferredField<string>;
  product_url: InferredField<string | null>;
  target_audience: InferredField<string>;
  product_keywords: InferredField<string[]>;
  industry_topics: InferredField<string[]>;
  brand_voice: InferredField<string | null>;
  // Future fields (not in MVP)
  // content_pillars: InferredField<string[]>;
  // persona_opinions: InferredField<string[]>;
}
```

Inference rules per field:
- **account_type**: "business" if bio contains product/company indicators; "individual" otherwise. Source: bio.
- **product_name**: X display name or product name extracted from bio. Source: bio.
- **product_description**: First sentence/clause of bio. Source: bio.
- **product_url**: `url` field from X profile entities. Source: profile URL.
- **target_audience**: Inferred from tweet engagement patterns and bio keywords. Source: bio + tweets.
- **product_keywords**: Top 3–7 terms from bio + frequent tweet topics. Source: bio + tweets.
- **industry_topics**: Distinct from keywords — broader themes. Source: tweets.
- **brand_voice**: Tone classification from tweet samples. Source: tweets.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| X API rate limits during onboarding | Analysis fails, user stuck | Cache profile data; single retry with backoff; graceful fallback to manual entry |
| LLM unavailable when inference runs | No prefill | Fall back to manual entry with helpful placeholders; LLM step comes after profile, so it may not be configured yet — use a lightweight default model or skip inference |
| Bio too short or empty | Poor inference quality | Show low-confidence badges; fall back to manual entry for missing fields |
| User has no tweets | No tweet-based inference | Inference from bio only; confidence downgraded |
| OAuth tokens expire during analysis | API calls fail | Tokens are fresh (just completed OAuth); add token refresh before analysis |
| Scraper mode users get degraded experience | Two-tier onboarding | Clear messaging about benefits of API mode; manual entry is the current experience so no regression |

### LLM Availability Ordering Problem

**Critical risk**: The current onboarding has LLM setup AFTER profile entry. For inference, we need LLM BEFORE profile. Options:
1. **Reorder steps**: Move LLM setup before profile inference. Con: user hasn't connected LLM yet when they connect X.
2. **Use a bundled/free model**: Use Ollama or a lightweight local model for inference only. Con: not all users have Ollama.
3. **Move LLM step to before analysis**: Ask for LLM right after X auth, before analysis.
4. **Two-pass**: Do basic bio extraction without LLM (regex/heuristics), then enrich with LLM after it's configured.

**Recommended**: Option 3 — reorder so LLM setup comes right after X auth. The flow becomes: Welcome → X Access+Auth → LLM → Analyze Profile → Review Prefill → Language/Brand → Sources → Validate → Review → Secure.

Alternatively, Option 4 (two-pass) is more resilient: do heuristic extraction first, then offer "re-analyze" after LLM is configured.

**Decision for charter**: Document both options; recommend Option 3 (reorder) for simplicity, with Option 4 as fallback if user testing shows LLM-first is confusing.
