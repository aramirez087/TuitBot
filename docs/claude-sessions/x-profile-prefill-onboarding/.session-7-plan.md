# Session 07 Implementation Plan: Provisioning and First-Value Launch

## Mission

Turn the completed onboarding form into a fully provisioned Tuitbot starter state. After submit, the user lands on a configured home page with their X identity active, safe defaults enforced, and no dead ends — regardless of deployment mode.

---

## Current State Analysis

### What already works
- `POST /api/settings/init` writes config.toml, migrates onboarding tokens to default account path, optionally claims + creates session
- Frontend `submit()` in `+page.svelte` builds config JSON from onboarding store, calls `api.settings.init()`, redirects to `/`
- `(app)/+layout.svelte` calls `initAccounts()` → `syncCurrentProfile()` on mount
- `syncCurrentProfile()` calls `POST /api/accounts/{id}/sync-profile` which fetches X profile and writes `x_username`, `x_display_name`, `x_avatar_url` to DB
- Capability tier is derived from config, gates are computed client-side
- ActivationChecklist shows on home + content pages when tier < posting_ready

### What's missing (this session's scope)
1. **No X profile data on default account after onboarding** — `init_settings` migrates tokens but doesn't populate `x_user_id`, `x_username`, `x_display_name`, `x_avatar_url` on the default account row. `syncCurrentProfile()` runs later but may fail silently.
2. **No server-side profile sync during init** — The onboarding callback already has the user data (from `complete_onboarding_auth`), but it's not persisted to the accounts table. The frontend would need to re-fetch.
3. **Frontend doesn't pass X user data to init** — The onboarding form has X user info available (from the auth callback) but doesn't include it in the init payload.
4. **No idempotency for repeated init calls** — Double-submit returns 409, but there's no recovery path (user is stuck on onboarding page if redirect fails).
5. **No provisioning tests** — `fresh_install_auth.rs` tests claim flow but not the full provisioning pipeline (token migration + account profile sync).
6. **`initAccounts()` doesn't trigger profile sync reliably** — It calls `syncCurrentProfile()` but doesn't await it, and errors are silently swallowed.

---

## Design Decisions

### D1: Pass X user data in onboarding init payload
The frontend already has X user data from the auth callback response. Pass it in the init body as an `x_profile` field. `init_settings` extracts it (like `claim`) before TOML conversion and writes it to the default account row.

**Rationale:** Avoids a separate sync-profile API call that could fail. The data is already available; passing it is cheaper and more reliable than re-fetching from X API.

### D2: Server-side account profile population in init_settings
After token migration succeeds, if `x_profile` is present in the request, update the default account with `x_user_id`, `x_username`, `x_display_name`, `x_avatar_url`.

**Rationale:** Single atomic provisioning step. No race between config creation and profile sync.

### D3: Frontend stores X user data in onboarding store
Add `x_user_id`, `x_username`, `x_display_name`, `x_avatar_url` fields to `OnboardingData`. Populated by the X auth callback response. Included in the init payload.

### D4: Idempotent completion with 409 recovery
When the frontend gets a 409 from `init_settings`, it means config already exists (e.g., double-submit, browser back/forward). Instead of showing an error, redirect to `/` since provisioning already succeeded.

### D5: Frontend profile sync as fallback
Keep `syncCurrentProfile()` in `(app)/+layout.svelte` as a fallback. If the server-side profile population in D2 was skipped (e.g., no X auth during onboarding), the existing sync will fill it in.

### D6: No changes to deployment mode handling
The current `init_settings` already handles deployment-mode-specific behavior (token migration, capabilities). Approval mode defaults to `true` in the onboarding store. No additional deployment-mode branching needed.

## Risks and Mitigations

### R1: X profile data not available during onboarding
**Scenario:** User skips X auth or uses scraper mode (no OAuth callback).
**Mitigation:** `x_profile` is optional in the init payload. Existing `syncCurrentProfile()` in `(app)/+layout.svelte` serves as fallback. Scraper mode users get profile synced when they configure browser session later.

### R2: Database not ready when init_settings writes account profile
**Scenario:** `ensure_default_account` hasn't run, so `update_account` fails.
**Mitigation:** The default account is seeded during DB init (`ensure_default_account` in migrations). The `init_settings` handler runs after server startup, so this should always be ready. Add a warn-and-continue fallback.

### R3: Token migration race with profile sync
**Scenario:** Frontend redirects to `/`, `syncCurrentProfile()` fires before token migration completes.
**Mitigation:** Token migration happens synchronously in `init_settings` before the response is sent. By the time the frontend redirects, tokens are already in place.

### R4: Double-submit 409 loses session
**Scenario:** User double-clicks "Start Tuitbot", first request creates config + session, second returns 409. Frontend error handler might not have the session.
**Mitigation:** Detect 409 in the catch block and redirect to `/` (config already exists, user already has session from first request via Set-Cookie header).

### R5: Test isolation
**Scenario:** Provisioning tests share state with existing tests.
**Mitigation:** Each test uses `tempfile::tempdir()` and `init_test_db()` for full isolation. This pattern is already proven in `fresh_install_auth.rs`.
