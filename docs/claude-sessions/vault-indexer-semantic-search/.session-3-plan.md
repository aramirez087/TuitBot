# Session 3 Plan: Hybrid Retrieval & API Contract

## Mission

Implement the hybrid retrieval and API layer that turns indexed vault content into fast, explainable semantic evidence results. Add `GET /api/vault/evidence` and `GET /api/vault/index-status` endpoints, blend semantic + keyword + graph signals with per-result `match_reason`, and ensure clean fallback when the semantic index is empty or unavailable.

---

## Key Design Decisions

### Decision 1: MatchReason enum in tuitbot-core

**Choice:** Add a `MatchReason` enum (`Semantic`, `Keyword`, `Graph`, `Hybrid`) to `retrieval.rs` and extend `VaultCitation` with optional `match_reason` and `score` fields.

**Rationale:** The implementation map specifies this location. Adding to the existing `VaultCitation` struct (with `Option` fields and `skip_serializing_if`) preserves backward compatibility — existing endpoints that don't set these fields will serialize identically to before.

### Decision 2: Semantic search as a pure function in a new module

**Choice:** Create `semantic_search.rs` in `tuitbot-core/src/context/` with a `semantic_search()` function that takes `&SemanticIndex`, a query embedding, and limit, returning scored chunk IDs. The function does NOT call the embedding provider — the caller embeds the query first.

**Rationale:** Separating embedding from search keeps the function testable without mocking HTTP. The server layer handles "embed query → search index" orchestration.

### Decision 3: Hybrid blending with RRF (Reciprocal Rank Fusion)

**Choice:** Use Reciprocal Rank Fusion to merge semantic and keyword result lists. Formula: `score = Σ 1/(k + rank_i)` where `k=60` (standard RRF constant). Results that appear in both lists get `match_reason: Hybrid`.

**Rationale:** RRF is the standard approach for combining heterogeneous ranking signals without score normalization. It's simple, well-understood, and performs well empirically. The `k=60` constant is from the original RRF paper and prevents high-ranked items from dominating.

### Decision 4: Evidence endpoint is a single unified surface

**Choice:** `GET /api/vault/evidence?q=&limit=&mode=` serves all consumers (Ghostwriter, hook picker, thread editor, selection review). No per-surface bespoke endpoints.

**Rationale:** Operator rules say "extend request and response types so selection review, hook generation, tweet editing, and slot-targeted thread actions can issue evidence queries without bespoke APIs for each surface." One endpoint with optional scope filters meets this.

### Decision 5: Fail-open when embedding provider is None

**Choice:** When `AppState.embedding_provider` is `None` or `AppState.semantic_index` is `None`, `hybrid_search()` silently falls back to keyword-only results. The response includes `index_status.freshness_pct: 0.0` so the frontend can show the appropriate degraded state.

**Rationale:** Operator constraint: "If semantic indexing is stale or unavailable, fall back cleanly to the current vault behavior."

### Decision 6: Cloud mode omits relative_path from evidence results

**Choice:** Evidence results in Cloud mode omit `relative_path` (set to `None`), matching the existing pattern in `note_neighbors` and `vault_sources`.

**Rationale:** Operator constraint: "Do not expose raw note bodies from read APIs beyond existing privacy rules."

### 1. `crates/tuitbot-core/src/context/semantic_search.rs`

**Purpose:** Pure semantic search function that queries the in-memory `SemanticIndex`.

**Contents:**
```rust
// Functions:
pub async fn semantic_search(
    index: &SemanticIndex,
    query_embedding: &[f32],
    limit: usize,
) -> Vec<SemanticHit>

pub struct SemanticHit {
    pub chunk_id: i64,
    pub distance: f32,       // cosine distance (0 = identical)
    pub similarity: f32,     // 1.0 - distance
}
```

**Key behaviors:**
- Delegates to `SemanticIndex::search()` which returns `(chunk_id, distance)` pairs
- Converts distances to similarity scores (`1.0 - distance`)
- Returns empty vec if index is empty (fail-open)
- No DB access, no HTTP calls — pure computation

**Estimated size:** ~60 lines + ~80 lines tests

### 2. `crates/tuitbot-core/src/context/hybrid_retrieval.rs`

**Purpose:** Blend semantic, keyword, and graph signals into a unified ranked result list with per-result `match_reason`.

**Contents:**
```rust
pub async fn hybrid_search(
    pool: &DbPool,
    account_id: &str,
    query: &str,
    semantic_hits: Option<&[SemanticHit]>,  // None = no semantic available
    selected_node_ids: Option<&[i64]>,
    limit: u32,
) -> Result<Vec<EvidenceResult>, StorageError>

pub struct EvidenceResult {
    pub chunk_id: i64,
    pub node_id: i64,
    pub heading_path: String,
    pub source_path: String,
    pub source_title: Option<String>,
    pub snippet: String,
    pub score: f64,
    pub match_reason: MatchReason,
    pub node_title: Option<String>,
}
```

**Algorithm:**
1. Parse query into keywords (whitespace split)
2. Run keyword search via existing `watchtower::search_chunks_with_context()` (limit * 2 for headroom)
3. If `semantic_hits` is `Some`, build a semantic chunk_id→score map
4. If `selected_node_ids` is `Some`, fetch graph neighbor chunks via existing `graph_expansion::expand_graph_neighbors()`
5. Merge all results using RRF:
   - Rank each result list independently (1-indexed)
   - For each chunk_id: `rrf_score = Σ 1/(60 + rank_i)` across all lists it appears in
   - Classify `match_reason`:
     - Appears in semantic only → `Semantic`
     - Appears in keyword only → `Keyword`
     - Appears in graph only → `Graph`
     - Appears in 2+ lists → `Hybrid`
6. Sort by rrf_score DESC, truncate to `limit`
7. Enrich each result with chunk metadata (heading_path, snippet, source_path, node_title) from the keyword or graph results, or by fetching from DB if only semantic

**Estimated size:** ~200 lines + ~150 lines tests

### 3. `crates/tuitbot-server/src/routes/vault/evidence.rs`

**Purpose:** `GET /api/vault/evidence` endpoint handler.

**Contents:**
```rust
#[derive(Deserialize)]
pub struct EvidenceQuery {
    pub q: String,                          // required
    pub limit: Option<u32>,                 // default 8, max 20
    pub mode: Option<String>,               // "hybrid" (default), "semantic", "keyword"
    pub scope: Option<String>,              // optional: "selection:{session_id}" for selection-scoped
}

#[derive(Serialize)]
pub struct EvidenceResponse {
    pub results: Vec<EvidenceResultItem>,
    pub query: String,
    pub mode: String,
    pub index_status: IndexStatusSummary,
}

#[derive(Serialize)]
pub struct EvidenceResultItem {
    pub chunk_id: i64,
    pub node_id: i64,
    pub heading_path: String,
    pub snippet: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub relative_path: Option<String>,      // omitted in Cloud mode
    pub match_reason: String,               // "semantic", "keyword", "graph", "hybrid"
    pub score: f64,
    pub node_title: Option<String>,
}

#[derive(Serialize)]
pub struct IndexStatusSummary {
    pub total_chunks: i64,
    pub embedded_chunks: i64,
    pub freshness_pct: f64,
}
```

**Key behaviors:**
- Validates query is non-empty (returns 400 otherwise)
- Clamps limit to `min(limit.unwrap_or(8), 20)`
- When `mode=semantic` and no embedding provider → returns empty results + degraded index_status
- When `mode=keyword` → skips semantic search entirely
- When `mode=hybrid` (default) → runs both semantic and keyword, blends with RRF
- When `scope=selection:{session_id}` → resolves selection's node_id for graph context
- Cloud mode: omits `relative_path` from all results
- Fetches `IndexStats` from `embeddings::get_index_stats_for()` for the response

**Handler flow:**
1. Parse and validate query params
2. If mode != "keyword" and embedding_provider is Some:
   - Embed the query text: `provider.embed(vec![q.clone()]).await`
   - Search the semantic index: `semantic_search(&index, &embedding, limit)`
   - On embedding error → log warn, fall back to keyword-only
3. If scope starts with "selection:" → parse session_id, resolve node_id for graph hints
4. Call `hybrid_search()` with semantic hits (or None) and optional selected_node_ids
5. Build response with privacy filtering

**Estimated size:** ~180 lines + ~120 lines tests

## Risks and Mitigations

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **Embedding the query adds latency** | Medium | Embedding a single short query is ~50ms for OpenAI, ~20ms for Ollama. Log latency. If slow, the frontend can show a loading skeleton (Session 4 concern). |
| 2 | **RRF scores are not intuitive to users** | Low | Scores are relative rank-based, not absolute similarity. Document this in ranking spec. Frontend can show "High/Medium/Low" relevance badges instead of raw scores. |
| 3 | **Large keyword result sets from LIKE queries** | Medium | `search_chunks_with_context` already has a limit parameter. Pass `limit * 2` for RRF headroom, never unbounded. |
| 4 | **SemanticIndex lock contention** | Low | Read lock for search, write lock only during indexing. Search is fast (<10ms for 50K vectors). Embedding worker holds write lock briefly per batch. |
| 5 | **hybrid_retrieval.rs exceeding 500-line limit** | Medium | Keep RRF logic tight. If tests push over 500 lines, split tests into `hybrid_retrieval/tests.rs` submodule per CLAUDE.md convention. |
| 6 | **Embedding provider returns dimension mismatch at query time** | Low | Catch `DimensionMismatch` error, log warning, fall back to keyword-only. Don't propagate as 500 to the client. |
| 7 | **Frontend type drift** | Low | Types are additive (new optional fields, new interfaces). No breaking changes to existing types. `npm run check` validates. |
