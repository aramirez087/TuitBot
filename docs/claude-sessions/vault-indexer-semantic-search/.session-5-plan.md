# Session 05 Plan: Draft & Slot Actions for Semantic Evidence

## Mission

Connect semantic evidence to tweet and thread editing so users can strengthen whole drafts or specific slots while keeping provenance and undo honest.

---

## Current State Assessment

### What exists

1. **Evidence Rail** (Session 4): `EvidenceRail.svelte` renders in the composer sidebar with search, pin/dismiss, auto-query, and a single "Apply to draft" button per card.
2. **handleApplyEvidence()** in `ComposerInspector.svelte:92-142`: Already applies pinned evidence to the focused slot via `api.assist.improve()`, pushes a `DraftInsert`, and records a `ProvenanceRef` with `source_role: 'semantic_evidence'`.
3. **draftInsertStore.ts**: Pure function state management with push/pop/undoById, block-level lookup, slot labels.
4. **CitationChips.svelte**: Renders two strips — "Based on:" (vault citations) and "Related notes:" (graph inserts with undo).
5. **ComposeWorkspace.svelte:201-231**: Reads `inspectorRef.getVaultProvenance()` and `inspectorRef.getDraftInsertState()` at submit time, passes provenance into `buildComposeRequest()`.
6. **Backend compose routes** (`compose/mod.rs`): Already accept `provenance: Vec<ProvenanceRef>` and persist via `build_provenance_input()` → `approval_queue::enqueue_with_provenance_for()`.

### What's missing (Session 5 scope)

1. **Slot picker for evidence**: The "Apply to draft" button currently targets only the focused block (or block 0). Users can't choose which slot to target from the evidence card itself — no dropdown or slot picker UI.
2. **Whole-draft improvement**: No "Strengthen entire draft" action that applies evidence to all blocks at once (or the single tweet) as a batch operation.
3. **Evidence citation strip in CitationChips**: Evidence-sourced inserts have `source_role: 'semantic_evidence'` but CitationChips only shows "Based on:" (vault citations) and "Related notes:" (graph inserts). No third strip for evidence contributions.
4. **focusedBlockIndex propagation**: `ComposerInspector` doesn't pass `focusedBlockIndex` to `InspectorContent` → `EvidenceRail`. The prop exists on EvidenceRail but is never wired. Auto-query and slot targeting default to block 0.
5. **Evidence provenance in DraftInsert**: `buildInsert()` only supports `edgeType`/`edgeLabel` params; evidence inserts use `handleApplyEvidence()` directly but don't flow `match_reason` or `similarity_score` into the `DraftInsert.provenance`.
6. **Tests for evidence → slot → undo flow**: No tests exist for `handleApplyEvidence()`, evidence-specific undo behavior, or evidence citation rendering.
7. **Documentation**: No `provenance-and-slot-actions.md` design doc.

## Design Decisions

### D1: Slot Picker Dropdown on Evidence Cards

Add a slot picker dropdown to `EvidenceCard` when in thread mode. When the user clicks "Apply to slot", show a small dropdown listing all thread blocks by slot label ("Opening hook", "Tweet 2", ..., "Closing takeaway"). In tweet mode, keep the current single-click "Apply to tweet" behavior.

**Rationale:** Follows the existing `SlotTargetPanel` pattern from `VaultSelectionReview` but lighter — a dropdown instead of a full panel. Matches the UX spec Section 5 which says "Apply to slot: Opens a dropdown showing available thread slots."

### D2: "Strengthen Draft" Whole-Draft Action

Add a "Strengthen draft" button to the pinned evidence section header. This applies all pinned evidence snippets as context to improve the entire draft (all blocks or single tweet) in one operation. Creates one `DraftInsert` per block modified, so each can be individually undone.

**Rationale:** Users who pin 3-4 evidence items want a one-click way to enrich the whole draft. Per-block inserts preserve undo granularity. The "strengthen" verb is intentional — it implies additive support, not replacement.

### D3: Evidence Inserts in CitationChips

Add a third strip to `CitationChips`: "Evidence used:" with a purple tint (matching the semantic badge color). Evidence inserts display `sourceTitle → slotLabel` with an undo button, same pattern as graph inserts.

**Rationale:** Evidence-sourced changes must be visually traceable. Purple color distinguishes from blue (vault) and graph (purple-ish but distinct hue). Same interaction pattern (undo button) keeps the UI consistent.

### D4: Wire focusedBlockIndex Through the Component Chain

Add `focusedBlockIndex` tracking from `ThreadFlowLane` → `ComposeWorkspace` → `ComposerInspector` → `InspectorContent` → `EvidenceRail`. This enables precise auto-query targeting and correct default slot selection.

**Rationale:** Required for auto-query to fire on the correct block and for "Apply to draft" to target the user's active block by default.

### D5: Enrich DraftInsert Provenance for Evidence

Extend `buildInsert()` to accept optional `matchReason`, `similarityScore`, and `chunkId` params. Evidence inserts will carry richer provenance than graph neighbor inserts.

**Rationale:** Analytics and citation rendering need to distinguish evidence-sourced inserts from graph-sourced ones. The `source_role` field alone is insufficient for the citation UI.

### D6: No Backend Changes

All Session 5 work is frontend-only. The existing `ProvenanceRef` type already supports `match_reason`, `similarity_score`, `source_role`, and `chunk_id`. The compose routes already persist arbitrary `ProvenanceRef` arrays. No new endpoints or schema changes needed.

#### Step 3.3: Implement handleStrengthenDraft in ComposerInspector

**File:** `dashboard/src/lib/components/composer/ComposerInspector.svelte`
**Changes:**
- Add `handleStrengthenDraft()` that iterates over blocks (or tweet text), calling `api.assist.improve()` for each with combined pinned evidence as context
- Creates one `DraftInsert` per block, each independently undoable
- Adds `ProvenanceRef` entries for all pinned evidence used
- Shows batch undo message: "Strengthened N blocks with evidence."

```typescript
async function handleStrengthenDraft() {
    const pinned = evidenceState.pinned;
    if (pinned.length === 0) return;

    const evidenceContext = pinned
        .map(p => `"${p.node_title ?? 'vault'}": ${p.snippet}`)
        .join('\n');

    assisting = true;
    try {
        if (mode === 'tweet') {
            if (!tweetText.trim()) return;
            const context = `Strengthen this tweet using these evidence points:\n${evidenceContext}`;
            const result = await api.assist.improve(tweetText, context);
            const insert = buildInsert({
                blockId: 'tweet',
                slotLabel: 'Tweet',
                previousText: tweetText,
                insertedText: result.content,
                sourceNodeId: pinned[0].node_id,
                sourceTitle: `${pinned.length} evidence items`,
                matchReason: 'semantic',
            });
            tweetText = result.content;
            draftInsertState = pushInsert(draftInsertState, insert);
        } else {
            // Thread: strengthen each non-empty block sequentially
            for (let i = 0; i < threadBlocks.length; i++) {
                const block = threadBlocks[i];
                if (!block.text.trim()) continue;
                const slotLabel = getSlotLabel(i, threadBlocks.length);
                const context = `This is the "${slotLabel}" of a thread. Strengthen it using these evidence points:\n${evidenceContext}`;
                const result = await api.assist.improve(block.text, context);
                const insert = buildInsert({
                    blockId: block.id,
                    slotLabel,
                    previousText: block.text,
                    insertedText: result.content,
                    sourceNodeId: pinned[0].node_id,
                    sourceTitle: `${pinned.length} evidence items`,
                    matchReason: 'semantic',
                });
                threadBlocks = threadBlocks.map((b, j) =>
                    j === i ? { ...b, text: result.content } : b
                );
                draftInsertState = pushInsert(draftInsertState, insert);
            }
        }

        // Add provenance for all pinned evidence used
        for (const pin of pinned) {
            vaultProvenance = [
                ...vaultProvenance,
                {
                    node_id: pin.node_id,
                    chunk_id: pin.chunk_id,
                    heading_path: pin.heading_path,
                    snippet: pin.snippet,
                    match_reason: pin.match_reason,
                    similarity_score: pin.score,
                    source_role: 'semantic_evidence',
                },
            ];
        }

        undoMessage = mode === 'tweet'
            ? 'Strengthened tweet with evidence.'
            : `Strengthened ${threadBlocks.filter(b => b.text.trim()).length} blocks with evidence.`;
        startUndoTimer();
    } catch (e) {
        onsubmiterror?.(e instanceof Error ? e.message : 'Strengthen failed');
    } finally {
        assisting = false;
    }
}
```

#### Step 4.2: Filter evidence inserts from DraftInsertState

**Where evidence inserts are passed to CitationChips:**

The parent component (likely `ComposerCanvas` or the compose body rendering) needs to split `DraftInsertState.history` into:
- `graphInserts`: inserts where `provenance.source_role !== 'semantic_evidence'`
- `evidenceInserts`: inserts where `provenance.source_role === 'semantic_evidence'`

Add a helper to `draftInsertStore.ts`:

```typescript
/** Partition inserts by source role. */
export function partitionInserts(state: DraftInsertState): {
    graphInserts: DraftInsert[];
    evidenceInserts: DraftInsert[];
} {
    const graphInserts: DraftInsert[] = [];
    const evidenceInserts: DraftInsert[] = [];
    for (const insert of state.history) {
        if (insert.provenance.source_role === 'semantic_evidence') {
            evidenceInserts.push(insert);
        } else {
            graphInserts.push(insert);
        }
    }
    return { graphInserts, evidenceInserts };
}
```

### Phase 5: Enrich buildInsert for Evidence Provenance

**Goal:** Evidence inserts carry full provenance metadata.

#### Step 5.1: Extend buildInsert params

**File:** `dashboard/src/lib/stores/draftInsertStore.ts`
**Changes:**
- Add optional params: `matchReason`, `similarityScore`, `chunkId`, `sourceRole`, `headingPath`, `snippet`
- Populate `ProvenanceRef` with all available fields

```typescript
export function buildInsert(params: {
    blockId: string;
    slotLabel: string;
    previousText: string;
    insertedText: string;
    sourceNodeId: number;
    sourceTitle: string;
    edgeType?: string;
    edgeLabel?: string;
    matchReason?: string;
    similarityScore?: number;
    chunkId?: number;
    sourceRole?: string;
    headingPath?: string;
    snippet?: string;
}): DraftInsert {
    const provenance: ProvenanceRef = {
        node_id: params.sourceNodeId,
        chunk_id: params.chunkId,
        edge_type: params.edgeType,
        edge_label: params.edgeLabel,
        match_reason: params.matchReason as MatchReason | undefined,
        similarity_score: params.similarityScore,
        source_role: params.sourceRole,
        heading_path: params.headingPath,
        snippet: params.snippet,
    };
    return {
        id: crypto.randomUUID(),
        blockId: params.blockId,
        slotLabel: params.slotLabel,
        previousText: params.previousText,
        insertedText: params.insertedText,
        sourceNodeId: params.sourceNodeId,
        sourceTitle: params.sourceTitle,
        provenance,
        timestamp: Date.now(),
    };
}
```

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Strengthen draft makes N sequential API calls for N blocks — slow for long threads | Medium | Show per-block progress indicator. If any call fails, stop and show partial result with undo available for completed blocks. Cap at 10 blocks. |
| Slot picker dropdown z-index conflicts with evidence rail scroll | Low | Use `position: absolute` within the card, `overflow: visible` on the card list. Test with 5+ results scrolled. |
| focusedBlockIndex goes stale when blocks are reordered | Low | Derive from `focusedBlockId` (stable UUID) rather than numeric index. Recalculate on each use. |
| `buildInsert` signature change breaks existing callers | Low | All new params are optional. Existing callers (handleSlotInsert, handleApplyEvidence) continue to work unchanged. |
| File size limits (500 lines Rust, 400 lines Svelte) | Medium | ComposerInspector is at 528 lines (slightly over). If adding handleStrengthenDraft pushes it further, extract strengthen logic into a utility function in a new file `dashboard/src/lib/utils/strengthenDraft.ts`. |
