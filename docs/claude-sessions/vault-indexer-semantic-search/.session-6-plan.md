- `hybrid_search()` call — log total search latency

```rust

let search_start = std::time::Instant::now();
let semantic_hits = if mode != "keyword" { ... };
let semantic_elapsed = search_start.elapsed();
let did_fallback = mode != "keyword" && semantic_hits.is_none();

    latency_ms = semantic_elapsed.as_millis() as u64,
    fallback = did_fallback,
    mode = mode,
    result_count = semantic_hits.as_ref().map(|h| h.len()).unwrap_or(0),

```

## Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | No feature flag — ship default-on | Semantic search is additive and fail-open. If no provider is configured, Evidence Rail is hidden. If provider is down, keyword fallback is silent. Feature flags add complexity without safety benefit. |
| D2 | Instrumentation via tracing spans (backend) + funnel events (frontend) | Matches existing patterns. Backend uses `tracing::info!` for structured logging. Frontend uses `trackFunnel()` + backend relay via `POST /api/telemetry/events`. No new infrastructure. |
| D3 | Privacy labels computed from deployment_mode in frontend | Privacy envelope is already available via runtime store. Adding `deployment_mode` to `IndexStatusResponse` lets the badge/settings compute labels without extra API calls. |
| D4 | No `semantic_search_enabled` config flag (deviation from implementation-map.md) | The map specified a config flag, but the implementation already has a natural feature gate: `embedding_provider.is_some()` on the backend and `provider_configured` on the frontend. Adding a redundant flag would require config migration and UI without adding safety. |
| D5 | Extend `IndexStatusResponse` rather than creating a new endpoint | Keeps the API surface minimal. The three new fields (`deployment_mode`, `search_available`, `provider_name`) are lightweight additions to an existing response. |
| D6 | Backend relay for evidence telemetry events | Enables server-side aggregation without building a dedicated metrics pipeline. Events flow through the existing `POST /api/telemetry/events` endpoint. |

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `evidenceFunnel` backend relay increases telemetry volume | Low | Low | Buffer flushes at 10 events (matching backlink pattern). Telemetry endpoint is structured-log only, no DB writes. |
| `ContentSourcesSection.svelte` grows beyond 400-line limit | Medium | Low | The semantic index section is ~40 lines of template. Current file is ~300 lines. Should stay under limit. Monitor during implementation. |
| `evidence.rs` tests module grows large | Low | Low | Evidence tests are concise (each ~25 lines). Adding 3 tests adds ~75 lines. File stays under 500 total. |
| Privacy labels become stale if deployment mode changes mid-session | Low | Low | `IndexStatusBadge` re-fetches status on mount. Settings page re-fetches on mount. Users would need to close and reopen to see updated mode, which is the expected flow. |
| Cloud mode lacks runtime E2E validation | Medium | Medium | Same risk as obsidian-ghostwriter-edge release. Mitigated by unit tests asserting `is_cloud` path (already tested in `cloud_mode_omits_relative_path`). Non-blocking for release. |
