# Session 4 Implementation Plan: Ghostwriter Evidence Panel

## Overview

Add the Evidence Rail — a semantic evidence search panel inside the existing Ghostwriter composer sidebar. Users can search their vault by concept, pin relevant evidence, and feed it into draft generation — all without leaving the compose flow.

**Guiding principle:** The Evidence Rail is additive. It sits between Voice and From Vault in InspectorContent, uses the same visual language (cards, badges, pin/dismiss), and degrades gracefully when the semantic index is unavailable.

## Design Decisions

### D1: Evidence state lives in ComposerInspector, not a global store

**Decision:** Pinned evidence, dismissed chunk IDs, and auto-query toggle are managed as `$state()` runes inside `ComposerInspector`, same pattern as `vaultProvenance` and `draftInsertState`.

**Rationale:** Evidence is session-scoped (cleared when composer closes). A global store would require manual lifecycle cleanup. The pure function helpers in `evidenceStore.ts` handle the logic; the component owns the state.

### D2: EvidenceRail is a collapsible section, not a panel mode

**Decision:** The Evidence Rail renders as an always-present (when configured) collapsible section in InspectorContent, independent of `notesPanelMode`. It does NOT participate in the `'notes' | 'vault' | null` toggle.

**Rationale:** The UX spec says "always available when the composer is open." Making it a panel mode would conflict with From Vault / From Notes, forcing users to choose. As a collapsible section, evidence search coexists with active vault selection review.

### D3: Auto-query seeds from focused block text, not full draft

**Decision:** When auto-query is enabled, extract text from the currently focused block (tweet text in tweet mode, or the focused thread block). Use that as the query, not the entire draft concatenated.

**Rationale:** Block-level text produces more focused semantic matches. Full-draft queries would return broad, unfocused results. The 800ms debounce fires on the block that the user is actively editing.

### D4: Deduplication by chunk_id against graph neighbors

**Decision:** When rendering evidence results, filter out any `chunk_id` that already appears in the current `VaultSelectionReview`'s accepted graph neighbors. This is a client-side filter using `acceptedNeighbors` from the selection session.

**Rationale:** UX spec Section 9 requires this. Graph neighbors take priority since they include richer relationship context (edge_type, intent).

### D5: Pinned evidence flows into generation via ComposerInspector export

**Decision:** `ComposerInspector` exports a new `getPinnedEvidence(): PinnedEvidence[]` function. The parent (`ComposeWorkspace`) reads pinned evidence when building the compose request and includes chunk_ids + snippets in the LLM context.

**Rationale:** Follows the same pattern as `getVaultProvenance()` and `getVaultHookStyle()`. The parent orchestrates what goes into the generation request; ComposerInspector just tracks the state.

### D6: No backend changes needed for this session

**Decision:** All Session 4 work is frontend-only. The `/api/vault/evidence` and `/api/vault/index-status` endpoints were built in Session 3 and are ready to consume.

**Rationale:** The API contract is complete. This session focuses on the UX surface that calls those endpoints.

### D7: Auto-query uses AbortController for cancellation

**Decision:** Each auto-query keystroke debounce creates an AbortController. When a new debounce fires, the previous controller is aborted. This prevents stale results from appearing after the user has moved on.

**Rationale:** Standard pattern for typeahead search. Without cancellation, a slow semantic query could resolve after a fast one, showing outdated results.

### Phase 1: Evidence Store (pure functions)

**File:** `dashboard/src/lib/stores/evidenceStore.ts`

```typescript
// Pure functions, no Svelte stores (matches draftInsertStore pattern)

export interface PinnedEvidence {
  chunk_id: number;
  node_id: number;
  heading_path: string;
  snippet: string;
  match_reason: MatchReason;
  score: number;
  node_title: string | null;
  relative_path?: string;
}

export interface EvidenceState {
  pinned: PinnedEvidence[];           // ordered, max 5
  dismissedChunkIds: Set<number>;     // per-session
  autoQueryEnabled: boolean;
  lastManualQuery: string;
}

export function createEvidenceState(): EvidenceState
export function pinEvidence(state: EvidenceState, result: EvidenceResult): EvidenceState
export function unpinEvidence(state: EvidenceState, chunkId: number): EvidenceState
export function dismissEvidence(state: EvidenceState, chunkId: number): EvidenceState
export function toggleAutoQuery(state: EvidenceState): EvidenceState
export function filterResults(
  results: EvidenceResult[],
  state: EvidenceState,
  graphChunkIds: Set<number>
): EvidenceResult[]
export function isPinned(state: EvidenceState, chunkId: number): boolean
export function canPin(state: EvidenceState): boolean  // pinned.length < 5
```

### Phase 3: IndexStatusBadge Component

**File:** `dashboard/src/lib/components/composer/IndexStatusBadge.svelte`

Props:
- `status: IndexStatusResponse | null` — fetched from `/api/vault/index-status`
- `compact?: boolean` — badge-only (no popover)

Logic:
- Derive badge color from `freshness_pct` and `provider_configured`:
  - `!provider_configured` → gray, "No embedding provider"
  - `!index_loaded && total_chunks === 0` → gray, "No index"
  - `freshness_pct >= 95` → green
  - `freshness_pct >= 50` → amber (pulsing)
  - `freshness_pct < 50` → red
- Tooltip shows chunk counts and freshness percentage
- Click opens popover with model info, last indexed timestamp, and "Reindex Now" button (deferred — button shown but not wired; reindex API is Session 6)

### Phase 5: EvidenceRail Component

**File:** `dashboard/src/lib/components/composer/EvidenceRail.svelte`

Props:
```typescript
{
  tweetText: string;
  threadBlocks: ThreadBlock[];
  mode: 'tweet' | 'thread';
  focusedBlockIndex?: number;         // which thread block is focused (for auto-query)
  hasExistingContent: boolean;
  selectionSessionId?: string | null;
  graphNeighborChunkIds?: Set<number>; // for dedup
  evidenceState: EvidenceState;        // from parent (ComposerInspector)
  onevidence: (newState: EvidenceState) => void;  // state update callback
  onapplytoSlot?: (evidence: PinnedEvidence, slotIndex: number, slotLabel: string) => void;
}
```

Internal state:
- `collapsed: boolean` — section collapse state, starts based on visibility rules
- `searchQuery: string` — search input value
- `results: EvidenceResult[]` — current search results
- `loading: boolean` — search in progress
- `error: string | null` — search error
- `indexStatus: IndexStatusResponse | null` — fetched on mount
- `autoQueryController: AbortController | null` — for cancellation
- `isSuggested: boolean` — whether current results are from auto-query

Key behaviors:
1. **Mount:** Fetch `/api/vault/index-status`. If `!provider_configured`, hide entirely.
2. **Manual search:** 300ms debounce, call `api.vault.searchEvidence({ q, limit: 8, mode: 'hybrid' })`, filter results through `evidenceStore.filterResults()`.
3. **Auto-query:** When `evidenceState.autoQueryEnabled` and text changes, 800ms debounce, extract focused block text, call with `mode: 'semantic'` and `limit: 5`. Use AbortController. Mark results as `suggested`.
4. **Pin:** Call `onevidence(pinEvidence(evidenceState, result))`, track analytics.
5. **Dismiss:** Call `onevidence(dismissEvidence(evidenceState, result.chunk_id))`, track analytics.
6. **Apply to slot:** Call `onapplytoSlot()` (parent handles the slot insert via existing `handleSlotInsert` pattern).
7. **Keyboard:** `Cmd+Shift+E` toggles collapse. `Escape` in search clears + collapses.

Empty/degraded states (from UX spec Section 7):
- No config: hidden (handled by visibility check)
- Index empty: building message with progress bar (`embedded_chunks / total_chunks`)
- Provider unavailable: warning message, keyword fallback
- Stale (<50%): amber warning "Index may show outdated results"
- No results: "No matching evidence found" message (hidden for auto-query)

Estimated lines: ~380 (within 400-line Svelte limit)

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| EvidenceRail.svelte exceeds 400-line limit | Medium | Track line count during implementation. If approaching limit, extract EvidenceSearchBar into its own component (~60 lines saved). |
| Auto-query fires too aggressively, causing lag | Medium | 800ms debounce + AbortController + `mode: 'semantic'` (server-side fast path). Disable auto-query on mobile. |
| `focusedBlockIndex` not available from ComposeWorkspace | Low | If not wired, auto-query falls back to tweetText in tweet mode. Thread block focus tracking may need a small prop addition to ComposeWorkspace — check during implementation. |
| Pinned evidence not consumed by generation endpoint | Low | Session 5 handles the full generation integration. For now, `getPinnedEvidence()` is exported and ready. The parent can include pinned snippets in the compose request's context field. |
| Graph neighbor chunk_id set unavailable outside VaultSelectionReview | Low | Build the set in ComposerInspector from accepted neighbors. If no selection session is active, the set is empty and no dedup is needed. |
