# Session 2 Implementation Plan: Indexer & Storage Foundation

## Mission

Build the background vault indexing foundation: embedding provider abstraction, `chunk_embeddings` schema, HNSW in-memory index, dirty-state tracking, and background `EmbeddingWorker` — all wired into the existing Watchtower lifecycle. No frontend. No search API.

---

## Key Design Decisions

### D1: `usearch` for HNSW — evaluate at integration time

The architecture spec names `usearch ^2.0` as a "pure Rust crate." This needs verification — `usearch` has historically relied on C++ FFI. If it fails to compile cross-platform in CI (macOS/Linux/Windows), fall back to:
1. `hnsw_rs` — pure Rust HNSW implementation
2. Brute-force cosine scan behind the same `SemanticIndex` trait (acceptable for <10K vectors)

**Mitigation:** Wrap all HNSW interaction behind `SemanticIndex` methods so the index backend can be swapped without changing callers.

### D2: `EmbeddingWorker` as independent task, not inside `WatchtowerLoop`

The `WatchtowerLoop` already manages filesystem watching, remote polling, and chunking. Adding embedding inside it would violate separation of concerns and make the loop harder to test. Instead:
- `EmbeddingWorker` is a separate struct with its own `run(cancel)` loop (same pattern as `SeedWorker`)
- Spawned alongside the Watchtower in `main.rs`, sharing the same `CancellationToken`
- Polls `get_dirty_chunks_for()` on a 5-second interval (vs SeedWorker's 5-minute interval)

### D3: SQLite is source of truth — HNSW rebuilt on startup

Per architecture spec: no periodic HNSW persist. SQLite stores all embeddings; HNSW is rebuilt from `get_all_embeddings_for()` at startup. This simplifies crash recovery and eliminates file format versioning.

### D4: Embedding config is `Option<EmbeddingConfig>` on `Config`

When `None`, all embedding infrastructure is skipped: no worker spawned, no index built, no `SemanticIndex` in AppState. The system degrades cleanly to existing keyword+graph retrieval.

### D5: Account-scoped everything

Every storage function uses `account_id` parameter. Every query includes `WHERE account_id = ?`. No cross-account leakage possible. Follows existing watchtower storage patterns.

### D6: Provider implementations use `reqwest` (already in workspace)

Both OpenAI and Ollama embedding providers use the existing `reqwest` dependency. No new HTTP client crate needed.

### Step 1: Migration — `chunk_embeddings` table

**Create:** `crates/tuitbot-core/migrations/20260322000200_chunk_embeddings.sql`

```sql
CREATE TABLE IF NOT EXISTS chunk_embeddings (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    chunk_id        INTEGER NOT NULL REFERENCES content_chunks(id) ON DELETE CASCADE,
    account_id      TEXT    NOT NULL DEFAULT 'default',
    embedding       BLOB    NOT NULL,
    model_id        TEXT    NOT NULL,
    dimension       INTEGER NOT NULL,
    embedding_hash  TEXT    NOT NULL,
    generation      INTEGER NOT NULL DEFAULT 1,
    created_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT    NOT NULL DEFAULT (datetime('now')),
    UNIQUE(chunk_id, account_id)
);

CREATE INDEX IF NOT EXISTS idx_chunk_embeddings_account
    ON chunk_embeddings(account_id);
CREATE INDEX IF NOT EXISTS idx_chunk_embeddings_generation
    ON chunk_embeddings(account_id, generation);
CREATE INDEX IF NOT EXISTS idx_chunk_embeddings_model
    ON chunk_embeddings(model_id);
```

**Rationale:** Pure additive migration. No existing tables modified. CASCADE delete ensures chunk deletion auto-removes embeddings. Naming follows existing pattern: `YYYYMMDDNNNNNN_description.sql`. Latest migration is `20260322000100`, so `20260322000200` is next.

**Tests:**
- Migration applies cleanly on fresh DB
- Migration is idempotent (IF NOT EXISTS)
- CASCADE: deleting a content_chunk cascades to chunk_embeddings
- UNIQUE constraint prevents duplicate (chunk_id, account_id) pairs

### Step 7: Embeddings storage CRUD

**Create:** `crates/tuitbot-core/src/storage/watchtower/embeddings.rs`

Row types and structs:

```rust
pub struct ChunkEmbeddingRow {
    pub id: i64,
    pub chunk_id: i64,
    pub account_id: String,
    pub embedding: Vec<u8>,  // Raw BLOB bytes
    pub model_id: String,
    pub dimension: i64,
    pub embedding_hash: String,
    pub generation: i64,
    pub created_at: String,
    pub updated_at: String,
}

pub struct DirtyChunk {
    pub chunk_id: i64,
    pub chunk_text: String,
    pub chunk_hash: String,
    pub node_id: i64,
}

pub struct IndexStats {
    pub total_chunks: i64,
    pub embedded_chunks: i64,
    pub dirty_chunks: i64,
    pub freshness_pct: f64,
    pub last_indexed_at: Option<String>,
    pub model_id: Option<String>,
}
```

Functions (all account-scoped):

1. **`upsert_chunk_embedding(pool, chunk_id, account_id, embedding_bytes, model_id, dimension, embedding_hash, generation)`**
   - INSERT OR REPLACE on UNIQUE(chunk_id, account_id)
   - Updates `updated_at` on conflict

2. **`get_all_embeddings_for(pool, account_id) → Vec<ChunkEmbeddingRow>`**
   - Used at startup to hydrate HNSW index
   - Ordered by id for deterministic rebuild

3. **`get_dirty_chunks_for(pool, account_id, limit) → Vec<DirtyChunk>`**
   - LEFT JOIN content_chunks ON chunk_embeddings
   - WHERE: chunk has no embedding row OR chunk_hash != embedding_hash
   - AND content_chunks.status = 'active'
   - LIMIT for batch processing

4. **`get_index_stats_for(pool, account_id) → IndexStats`**
   - Three COUNT queries (could be optimized to one with subqueries)
   - Computes freshness_pct from counts

5. **`delete_embeddings_by_model(pool, account_id, model_id) → u64`**
   - Used during model switch: purge old-model embeddings

6. **`delete_embedding_for_chunk(pool, chunk_id, account_id)`**
   - Used when a chunk is explicitly invalidated

- Add `pub mod embeddings;` and `pub use embeddings::*;`

### Step 8: `SemanticIndex` — in-memory HNSW wrapper

**Create:** `crates/tuitbot-core/src/context/semantic_index.rs`

```rust
pub struct SemanticIndex {
    index: usearch::Index,         // or alternative HNSW impl
    key_to_chunk_id: HashMap<u64, i64>,
    chunk_id_to_key: HashMap<i64, u64>,
    next_key: AtomicU64,
    dimension: usize,
    model_id: String,
    capacity: usize,               // default: 50_000
}

pub enum SemanticSearchError {
    DimensionMismatch { expected: usize, actual: usize },
    IndexFull,
    Internal(String),
}
```

Methods:

1. **`new(dimension, model_id, capacity) → Self`**
   - Creates empty HNSW index with given dimension and capacity
   - Uses cosine metric (normalized vectors)

2. **`insert(chunk_id, embedding) → Result<(), SemanticSearchError>`**
   - Assigns next_key, inserts into HNSW and both HashMaps
   - Returns `IndexFull` if at capacity

3. **`remove(chunk_id) → bool`**
   - Removes from HNSW and both mappings
   - Returns false if chunk_id not in index

4. **`search(query, k) → Vec<(i64, f32)>`**
   - Returns (chunk_id, distance) pairs, ascending distance
   - If index is empty, returns empty Vec (no error)

5. **`len()`, `is_empty()`, `model_id()`, `dimension()`** — accessors

6. **`rebuild_from(embeddings: Vec<(i64, Vec<f32>)>)`**
   - Clears index and rebuilds from a batch of (chunk_id, embedding) pairs
   - Used at startup and after model switch

**Modify:** `crates/tuitbot-core/src/context/mod.rs` — add `pub mod semantic_index;`

**Modify:** `crates/tuitbot-core/Cargo.toml` — add `usearch` dependency (or alternative)

**Fallback plan if `usearch` fails:**
Create a `BruteForceIndex` behind the same API that does linear cosine scan. Performance is acceptable for <10K vectors. Replace later when a working HNSW crate is confirmed.

**Tests:**
- Insert and search returns the inserted vector as nearest neighbor
- Insert multiple vectors, search finds correct top-K
- Remove makes vector unfindable
- Dimension mismatch on insert is caught
- Empty index search returns empty vec
- Rebuild replaces all contents
- Capacity limit respected

### Step 9: `EmbeddingWorker` — background indexer

**Create:** `crates/tuitbot-core/src/automation/watchtower/embedding_worker.rs`

```rust
pub struct EmbeddingWorker {
    pool: DbPool,
    provider: Arc<dyn EmbeddingProvider>,
    index: Arc<RwLock<SemanticIndex>>,
    batch_size: usize,
    account_id: String,
}

pub const EMBEDDING_WORKER_INTERVAL_SECS: u64 = 5;
```

**`run(&self, cancel: CancellationToken, scheduler: LoopScheduler)`:**
- Same loop pattern as `SeedWorker::run()`:
  ```rust
  loop {
      tokio::select! {
          () = cancel.cancelled() => break,
          () = scheduler.tick() => {}
      }
      match self.process_dirty_batch().await { ... }
  }
  ```

**`process_dirty_batch(&self) → Result<u32, EmbeddingError>`:**
1. `get_dirty_chunks_for(pool, account_id, batch_size)` → dirty chunks
2. If empty, return Ok(0)
3. Extract `chunk_text` from each dirty chunk
4. Call `provider.embed(texts)` → embedding response
5. For each (chunk, embedding):
   a. Compute `embedding_hash = SHA256(chunk_text)` (reuse `sha2` crate)
   b. Serialize embedding as little-endian f32 bytes
   c. `upsert_chunk_embedding(...)` with `generation = current_max + 1`
   d. `index.write().insert(chunk_id, embedding_vec)`
6. Log batch stats via `tracing::info!`
7. Return Ok(count)

**Error handling:**
- Provider unavailable: log warning, skip batch, return Ok(0)
- Partial failure: not applicable (batch is all-or-nothing for simplicity in v1)
- Rate limited: log, sleep for retry_after, then return
- Dimension mismatch: log error, halt (return Err) — requires config fix

**Modify:** `crates/tuitbot-core/src/automation/watchtower/mod.rs`
- Add `pub mod embedding_worker;` declaration

**Tests (unit, with mock provider):**
- Worker processes dirty chunks and calls provider.embed
- Worker inserts embeddings into storage and index
- Worker handles provider error gracefully (logs, skips)
- Worker handles empty dirty set (no-op)
- Worker respects cancellation token
- Idempotent: re-processing same chunk updates rather than duplicates

8. Spawn with same cancel token as Watchtower
9. Store index and provider in AppState

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `usearch` crate won't compile on Windows CI | Medium | High | Wrap behind `SemanticIndex` abstraction. Fallback: brute-force cosine scan (linear, O(n)). Test `cargo build --target x86_64-pc-windows-msvc` early. |
| `usearch` API differs from docs | Medium | Medium | Read actual crate source/examples before coding. The `usearch` crate has gone through breaking API changes. Pin exact version. |
| SQLite BLOB read perf at 50K rows on startup | Low | Medium | Measure startup time. If >5s, add lazy loading (build index in background, serve keyword-only until ready). |
| Embedding provider HTTP errors in tests | Low | Low | Use `wiremock` for all provider tests (already in dev-deps). No real API calls in tests. |
| Coverage drop below 75% from new untested code | Medium | Medium | Write tests for every function. Storage tests use temp SQLite DB. Provider tests use wiremock. Index tests use deterministic vectors. |
| `Config` struct changes break existing config.toml files | Low | Medium | `embedding` is `Option<EmbeddingConfig>` with `#[serde(default)]` — existing configs without `[embedding]` section parse fine as `None`. |
