# Session 1 Implementation Plan: Semantic Evidence Charter & UX

## Objective

Produce six documentation deliverables that define the product charter, UX, architecture, and implementation roadmap for a background semantic vault index that upgrades Ghostwriter into an evidence-first drafting workspace. **No source code changes.**

---

## Pre-Work: Codebase Audit Summary

### Current Compose Flow (verified by reading live code)

1. **Selection ingress** — Obsidian plugin → `POST /api/vault/send-selection` → 30-min session with `resolved_node_id` → WebSocket `SelectionReceived`
2. **Composer hydration** — `ComposerInspector` loads session → `VaultSelectionReview` auto-expands 1-hop graph neighbors via `resolve_graph_suggestions()`
3. **Hook/Angle generation** — User accepts neighbors → calls `api.assist.angles()` (Hook Miner) or `api.assist.hooks()` → hook picker UI
4. **Draft creation** — Selected hook → `handleGenerateFromVault()` → vault fragments injected into LLM prompt via `format_fragments_prompt()` (capped at 1000 chars)
5. **Slot refinement** — `SlotTargetPanel` maps unapplied neighbors → specific thread slots → `handleSlotInsert()` → `api.assist.improve()` with neighbor context → undo stack
6. **Provenance** — `vaultProvenance[]` tracks `ProvenanceRef` entries with edge_type, edge_label, source_role

### Current Retrieval Stack

| Layer | Mechanism | File |
|-------|-----------|------|
| Keyword search | SQLite FTS5 on content_nodes | `storage/watchtower/nodes.rs` |
| Chunk search | `search_chunks_with_context()` | `storage/watchtower/chunks.rs` |
| Graph expansion | 1-hop wikilinks + backlinks + shared tags | `context/graph_expansion.rs` |
| Fragment retrieval | `retrieve_vault_fragments()` — selected nodes first, then keyword fill | `context/retrieval.rs` |
| LLM context | `format_fragments_prompt()` — 5 fragments, 1000 chars | `context/retrieval.rs` |

### What's Missing

- No vector embeddings, no embedding storage, no dense similarity search
- No semantic re-ranking (BM25 or learned-to-rank)
- No chunk-level embedding columns in schema
- No embedding provider config in `LlmConfig` or `config/types.rs`
- Graph expansion is deterministic (score = 3×direct + 2×backlink + 1×shared_tag + 0.5×chunk_boost) — no semantic signal
- `retrieval_boost` is a static DB column, never dynamically tuned

## Key Design Decisions

### D1: Embedding Provider Abstraction
**Decision:** Add an `EmbeddingProvider` trait alongside the existing `LlmProvider` trait in `crates/tuitbot-core/src/llm/`. Concrete implementations: OpenAI `text-embedding-3-small` (1536d), Ollama `nomic-embed-text` (768d), and a future Anthropic implementation.

**Rationale:** The existing `LlmProvider` trait + factory pattern (`llm/factory.rs`) proves this project already embraces provider abstraction. Embeddings are a different API shape (batch input → batch output, no streaming) so a separate trait is cleaner than overloading `LlmProvider`. The factory pattern extends naturally: `create_embedding_provider(config)`.

### D2: Index Backend — SQLite + In-Process HNSW (usearch)
**Decision:** Store raw embedding vectors as BLOBs in SQLite (`chunk_embeddings` table) and load them into an in-process HNSW index (`usearch` crate) at startup. Rebuild the index on startup from persisted vectors; incremental inserts/deletes handled via usearch's mutable index API.

**Rationale:**
- SQLite is the existing storage backend — no new infrastructure dependency
- `usearch` is a single-crate HNSW implementation (Rust-native, no C++ dependency hell) with mutable index support
- Desktop/self-host users don't want to run a separate vector DB
- Cloud mode can later swap to pgvector via the trait abstraction
- In-process search at <10ms for 100K vectors (typical vault size) — well within UX budget
- Alternative considered: `qdrant` — rejected because it adds a server process, which violates local-first constraint

### D3: Semantic Unit Granularity — Chunk-Level Embeddings
**Decision:** Embed at the chunk level (heading-delimited sections from `chunker.rs`), not note-level or paragraph-level.

**Rationale:** Chunks are the existing retrieval unit (`content_chunks` table, `FragmentContext`, `VaultCitation`). Embedding at chunk level preserves the 1:1 mapping between semantic search results and provenance citations. Note-level would be too coarse; paragraph-level would explode the index without proportional recall gain.

### D4: Dirty-State Tracking — Hash-Based with Generation Counter
**Decision:** Add `embedding_hash` (SHA256 of chunk_text) and `embedding_generation` (monotonic counter) columns to track which chunks need re-embedding. The Watchtower ingest pipeline already computes content hashes — extend this to chunk level.

**Rationale:** When `ingest_content()` re-chunks a note, any chunk whose text hash differs from `embedding_hash` is marked dirty. The background indexer processes dirty chunks in batches. The generation counter enables "index is N% fresh" status reporting without scanning all rows.

### D5: Hybrid Retrieval — Semantic + Graph + Keyword with Explainable Blending
**Decision:** Semantic search results carry a `match_reason` enum: `Semantic`, `Graph`, `Keyword`, `Hybrid(reasons)`. The retrieval pipeline runs all three in parallel, deduplicates by chunk_id, and produces a blended score with per-signal weights exposed in the API response.

**Rationale:** The operator rules require "semantic retrieval as complementary to current graph and keyword retrieval, not a silent replacement." Explainable reasons align with the existing `SuggestionReason` / `SuggestionIntent` pattern in `graph_expansion.rs`. Users can see *why* a result surfaced.

### D6: UX Integration Point — Evidence Rail in Composer Inspector
**Decision:** Add a collapsible "Evidence" rail inside the existing `ComposerInspector` component (alongside the current vault panel), not a separate page or modal. Evidence appears contextually: auto-queried from draft text during editing, manually searchable, and surfaced alongside graph suggestions.

**Rationale:** The operator rules say "semantic search should feel like a power tool, not a parallel workflow." The existing `FromVaultPanel` → `VaultSelectionReview` → `SlotTargetPanel` flow is the proven compose pattern. Evidence slots into this flow as a new source of suggestions that feeds into the same hook/angle/slot-refinement pipeline.

### D7: Auto-Query Behavior — Opt-In with Debounced Triggers
**Decision:** Auto-query is off by default. When enabled via a toggle, it fires a semantic search 800ms after the user stops typing in a thread block, using the block's text as the query. Results appear in the evidence rail with a "suggested" badge. Users can pin results to lock them across edits.

**Rationale:** Auto-query without opt-in would violate "never injected without a visible action." The debounce prevents query spam. Pin/dismiss actions mirror the existing accept/dismiss pattern on `GraphSuggestionCards`.

### D8: Deployment Behavior
**Decision:**
- **Desktop:** Embedding computed locally via Ollama (if configured) or queued for cloud API. Index built and searched in-process.
- **Self-host:** Same as desktop but embedding provider configurable in `config.toml`.
- **Cloud:** Embedding via server-side provider. Privacy envelope enforced — raw chunk text never returned in search API responses (only snippet + metadata, same as current `CITATION_SNIPPET_LEN = 120`).

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| usearch crate stability | Index corruption on crash | Persist vectors in SQLite as source of truth; rebuild HNSW on startup from persisted data |
| Embedding API cost for large vaults | Unexpected spend for Desktop users | Default to Ollama (free, local); batch size limits; skip unchanged chunks via hash |
| HNSW memory for large vaults | OOM on constrained machines | Cap index at 50K vectors (~100MB); paginate beyond that with SQLite-only cosine scan |
| Auto-query latency | Typing feels laggy | 800ms debounce; cancel in-flight queries on new keystroke; show loading skeleton |
| Cross-platform SQLite BLOB handling | Windows/macOS differences | Use sqlx's native BLOB binding; test in CI on all 3 platforms |
| Privacy leakage in cloud mode | Raw note text exposed | Same `CITATION_SNIPPET_LEN = 120` truncation; no raw embedding vectors in API responses |
